開発スキル（全体・習慣編）

HTML/CSS のコーディングから業務システムのプログラミングまでを含む、開発全般に共通する習慣・姿勢・プロセスのスキル定義。

2026-05-24 ドラフト（言語別の細かいスキルは別シートで作成予定）

対象スコープ

本シートは「**開発者として共通で持つべき習慣**」を扱います。HTML、CSS、JavaScript、PHP、SQL 等の 言語固有のスキルは別シート（今後作成）。

評価ルール: PC操作スキルと同じ（5段階評価／重要度別 段階的進行）

重要度の凡例: A 基礎（必須） B 業務効率（推奨） C 専門（応用）

1. 要件理解・コミュニケーション（5項目）

重要度 項目
A機能の 目的・背景 を理解しようと努めてから着手する
A仕様の 不明点・曖昧点を質問 してから実装する（推測で進めない）
Bクライアント・社内の要望を 技術要件に翻訳 できる
B開発前に スコープ・優先度 を明確に確認する習慣がある
C要件の 妥当性・実現可能性を判断 し、代替案を提案できる

2. 設計（6項目）

重要度 項目
A実装前に 「何を作るか」を整理 してから着手する
A既存のコード構造を 理解してから 変更を加える
B機能を 小さな単位に分割 して設計できる
B命名（変数・関数・クラス） に時間をかけ、意味の通る名前を付ける
Bデータ構造・処理フローを意識して設計する
C拡張性・保守性 を考慮した設計（疎結合、責任分離等）ができる

3. コーディング習慣（7項目）

重要度 項目
A既存の コードスタイル・規約 に合わせて書く
A動作確認をしてから次のタスクに進む（「動くはず」で渡さない）
Aハードコードを避け、定数・変数化 する
B関数を 小さく保つ（1関数=1責任）
B重複コードを発見したら リファクタ する
Bエラーハンドリング を意識した実装ができる（例外・null チェック等）
Cパフォーマンス・セキュリティを意識した実装ができる（N+1、SQL インジェクション等）

4. テスト・品質管理（5項目）

重要度 項目
A機能完成後、複数ケース で動作確認する（正常系だけで終わらせない）
Aエッジケース（空・最大・最小・異常値・特殊文字）を想定する
B自動テストの価値を理解しており、書いたコードに対してテストを書く意識がある
Cテスト駆動開発（TDD）を実践できる
CCI/CD やコードカバレッジ・自動化されたチェックを意識・活用できる

5. バージョン管理（Git）（7項目）

重要度 項目
Agit status / log / diff を 頻繁に使い、変更内容を常に把握する
Aコミットメッセージを 意味のあるもの にする（「修正」だけで終わらせない）
A機能単位・修正単位で 適切な粒度でコミット する
Bブランチを 目的別 に分けて運用できる（feature ブランチ等）
Bコンフリクトを 落ち着いて解決 できる
Brebase / merge を場面で使い分けられる
C高度な操作（cherry-pick、bisect、reflog 等）で問題解決できる

6. デバッグ・問題解決（6項目）

重要度 項目
Aエラーメッセージを 必ず読む 習慣がある（読まずに勘で直さない）
Aログ・コンソール出力・dump を活用して状態を可視化できる
B問題を 小さく切り分け て原因を特定できる（再現条件の絞り込み）
Bブラウザの DevTools、IDE のデバッガを使える
B同じエラーを 二度起こさない仕組み を作る（メモ、テスト、定数化等）
Cスタックトレースから原因を推測できる／再現性のないバグを系統的に追える

7. ドキュメンテーション（5項目）

重要度 項目
Aコードに必要なコメントを書ける（「なぜ」を残す、「何を」はコードで分かる）
A仕様変更時に ドキュメントも更新 する（コードと仕様の乖離を防ぐ）
BREADME・API 仕様書・運用手順書を書ける
B図（フロー図・構成図・ER 図等）を使って説明できる
C他の開発者がすぐ理解できる構造で書ける（読み手を意識）

8. AI活用（開発における）（8項目）

PC操作の「AI活用（業務における）」とは別軸で、開発業務での AI 活用に特化したスキル。

重要度 項目
AAIに 「エラーの原因調査・解決策」 を相談できる
AAIに 「コードのレビュー・改善提案」 を依頼できる
AAIの提案を 鵜呑みにせず、自分で検証・テストする
BAIに 叩き台 を作らせ、自分で仕上げる流れができる
BAIに 技術調査（ライブラリ比較・最新手法等）を依頼できる
BClaude Code 等の AI エージェント を開発業務に活用できる
CAIと ペアプログラミング ができる（対話的にコードを育てる）
CAIを使った 開発ワークフローを設計 し、他社員に展開できる

9. 学習・継続的改善（5項目）

重要度 項目
A公式ドキュメント を読む習慣がある（解説サイトだけに頼らない）
A知らない技術・用語を 即時に調べる 習慣がある
B業界トレンド・新技術に 定期的に 目を通す（週次など）
B新しい技術・ツールを 試す時間 を業務内で確保している
C自分の経験を 言語化・共有 できる（社内勉強会・ブログ・ドキュメント化）

項目数 集計

重要度 項目数 備考
A 基礎（必須）21初回評価対象
B 業務効率（推奨）22Aクリア後
C 専門（応用）11Bクリア後
合計54

レビュー観点（社長判断用）

  • 初回評価対象（A 21項目）は社員1人あたり 5〜10分。気軽に開始できる規模
  • 9カテゴリで開発全般の習慣を網羅、A:B:C = 21:22:11 のバランス
  • 「AI活用（開発における）」を独立セクションに — はなさかの方針（AI活用重視）を強く反映。Claude Code への言及あり
  • 「テスト・品質管理」は A が2項目のみ（複数ケース確認・エッジケース）。社員の現状に合わせて調整余地あり
  • 「ドキュメンテーション」の「なぜを残す」は、はなさかの「CLAUDE.md 中心の運用」とも整合
