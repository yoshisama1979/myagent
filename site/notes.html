開発オートパイロット構想 — 人間判断を最小化しAIに一括自走させる仕組み（社長の大タスク）

項目 内容
カテゴリ開発・運用基盤／AI活用ワークフロー
起案者社長
ステータス構想段階。社長の「大タスク」の一つとして仕組み構築に取り組む意向
適用範囲まず社内の開発プロセス改善。確立後は将来クライアントへの提供（プロダクト／サービス化）も視野（確認済み）
既存 ai-loop との関係別の新しい仕組みとして考える（確認済み）。発想は近いが用途が異なる（ai-loop はプロジェクト自体のレビュー、本構想は開発作業そのものの自走化）

課題・背景

  • 現在 3台のPCで別案件を並行してシステム構築中（確認済み）
  • 一つ一つの確認に人間が都度対応するため、そこがボトルネックになっている
  • 確認をAIに一気に任せられれば、並行作業がもっと進む
  • 狙い：AIをよりよく活用するため、人間の判断を極力小さくする

仕組みの基本ループ（社長案）

1. ゴールとなる仕様書を作る ← ここだけ人間と一緒に行う
   │
   ▼
2. その仕様書を「全て解決するまで」AIを動かす（自走）
   │
   ▼
3. 問題が出たり、人間の判断が必要なことが出たら
   → どこかに残しておく（溜める。途中で人間を止めない）
   │
   ▼
4. 一通りの作業が終わったら、溜めた確認事項を
   → 人間がまとめて確認・判断
   │
   ▼
5. 人間の判断をすべて反映し、それをもとに次のゴールへ向かう
   │
   ▼（2〜5を繰り返す）
   ▼
6. すべての問題が解決したとき → 完成

ポイント：人間の関与を 「最初の仕様書づくり」と「区切りごとの一括確認」 の2点に集約し、その間はAIを止めずに走らせ続ける。確認待ちで作業を分断しないことで、3台並行の効率を上げる狙い。

棚卸し時の検討事項・次のアクション（Main モード向け）

  • 「人間判断が必要なこと」を溜めておく場所・形式の設計（例：案件ごとの保留キュー／チェックリスト。ai-loop の conversation.md とは別建て）
  • AIが「人間判断が必要」と自己判断する基準の定義（どこで止めて溜める／どこは自走で進めてよいか）
  • 「全て解決した＝完成」の判定条件の定義（テスト全パス・保留キューが空 等）。前項 2026-05-29「品質保証方針（テストピラミッド＋AI探索テスト）」と連動させると、完成判定の自動化に使える
  • 3台並行を 1人の人間がまとめて確認する運用の負荷・タイミング設計（案件横断で一括確認するか、案件ごとか）
  • 将来のクライアント提供を見据えた場合の プロダクト化の切り出し方（社内ツールとして磨きつつ汎用化）

具体化①：Codex によるデザイン/UX 自動修正環境

本構想の具体ピースの一つ。開発の中で デザイン的なところも自動的に修正される環境 を作りたい。

システム開発の品質保証方針 — 手動テスト最小化 × AIエージェント活用（調査メモ）

項目 内容
カテゴリ開発・運用基盤／品質保証（テスト自動化方針）
起案者社長
ステータス調査・アイデア段階（今後システムに導入していきたい意向）
適用対象クライアント納品システム全般の標準テスト方針として組み込む想定（確認済み）
出典社長が外部AIに「手動テストを減らしつつ品質を高めたい」と相談して得た回答（調査結果）

ゴール

  • 人間が行う手動テストを極力減らし、かつ品質を高く保つ
  • E2Eテストを軸にしつつ、すべてをE2Eに頼らず「適材適所の自動化」と「AIエージェントの賢い配置」で実現する

1. テスト自動化の黄金比（テストピラミッド）

層 役割・特徴
ユニットテスト（底辺・大量）ロジック・関数の検証。ミリ秒単位で高速、原因特定が容易。バグの大部分をここで堰き止める
結合テスト（中間）API連携・DBとのやり取りが正しいか検証
E2Eテスト（頂点・厳選）ユーザーの主要導線（ログイン〜決済等）に絞ってデプロイ時に自動実行。遅く Flaky（不安定）になりやすいので全カバーしない

→ 「ユニット/結合で大半のバグを潰し、E2Eで最後の導線を確認」が、打鍵コスト（手動テスト）を減らす近道。

2. AIエージェント（画面操作AI）の使いどころ

E2Eを整備しても残る「人間がやらざるを得なかったテスト」こそ、画面操作AIエージェントの主戦場：

  • デザイン・レイアウトの崩れチェック：スマホでのボタン文字被り、デザインカンプとの違和感など視覚確認
  • 文言・文脈のニュアンス確認：エラーメッセージの日本語の不自然さ、文脈的におかしい挙動
  • 探索的テスト（モンキーテスト）：無効値の入力・ボタン連打など、テストケース化しにくい意地悪なテスト

→ 仕様書・指示書を渡して裏で実行させ、これまでの目視「受け入れテスト」の時間を大幅削減。

3. 理想の開発ワークフロー（CI/CD組み込み）

[コード変更]
   │
   ▼
1. ユニット・結合テスト（GitHub Actions等で自動 / 1〜2分）
   仕様通りのロジックかを瞬時に検証
   │
   ▼
2. 従来のE2Eテスト（Playwright等で自動 / 3〜5分）
   主要導線・ブラウザ基本動作のデグレ防止を厳密に検証
   │
   ▼
3. AIによる探索・視覚テスト（画面操作エージェント / 5〜10分）
   UIの崩れ・エッジケース不具合を自律テスト。バグ発見時は修正パッチまで提案
   │
   ▼
【人間は最終確認のみ】
  • 従来E2E ＝「デグレード（先祖返り）の完全防止」を担保
  • AIエージェント ＝「人間による目視・レビューの自動化」を担う
  • この2段構えで、テストコードのメンテからもリリース前の泥臭い手動テストからも解放され、本質的な機能開発に集中できる

補足・前提（確認済み）

  • 適用先は クライアント納品システム全般の標準テスト方針として組み込む想定（確認済み）
  • 文中の「Codex」は 特定ツールの指定ではなく、ツール選定は要調査（確認済み）。Codex / Playwright MCP / その他の画面操作AIを比較検討する課題として残す

棚卸し時の検討事項・次のアクション（Main モード向け）

  • 画面操作AIエージェントのツール選定（Codex / Playwright MCP 等の比較・検証）
  • 最初の自動化・AI置き換えスポットの特定 — 「現在の手動テストで最も時間がかかる部分／毎回同じことを確認していて退屈な部分」を洗い出す（外部AIからの問いかけ。社長未回答）
  • 「はなさか 業務改善システム構築ポリシー」（2026-05-08 メモ【C-1】）への テスト・品質保証セクションとしての組み込み可否
  • 運用管理サービス（月額50,000円）の品質保証施策と連動させるか

A-1. 月額50,000円に含むサービス範囲の確定

「何を含めて、何を別料金にするか」の境界線を明文化する。曖昧だと工数が膨張するリスクあり。

含めるかどうかを 1つずつ判断したい候補:

候補 含める / 別料金 / 採用しない
軽微改修対応(月◯時間まで)未決
死活監視・異常通知(営業時間内 / 24h)未決
SSL証明書・ドメイン更新(実費別 / 込み)未決
月次稼働レポート未決
セキュリティパッチ適用未決(含めるべき)
ユーザー操作サポート(軽微)未決
月次定例MTG(30分)未決

→ 確定後、「サービス定義書」テンプレート を作成し、提案・契約時に使う

A-2. 初期構築費の想定レンジ

  • 小規模 / 中規模 / 大規模 ごとの目安価格
  • 工数換算 + 利益率の基準
  • 提案テンプレに「初期費用 + 月額50,000円〜」をセットする際の数字

ステージ1(汎用スキル版)の進行ステップ

  1. 社長から過去のスキルシート資料を受領(着手の前提)
  2. 資料を下敷きにスキル項目を整理(追加・削除・更新)
  3. レベル定義(例: 5段階)とフォーマットを策定
  4. 社員に自己評価を実施
  5. 社長レビュー・フィードバック → 改善
  6. 本運用開始(定期更新サイクルを設計)

汎用スキル項目の候補(資料受領前のたたき台)

※ 単なるPC操作ではなく、AI活用を前提とした現代的な汎用スキルとして設計する。
AIにできる作業は積極的に任せ、人は判断と仕上げに集中することを基本姿勢とする。

A. AI活用力(最重要・現代の前提スキル)

  • AI移譲の判断: 面倒な作業・繰り返し作業をAIにやらせる発想
  • 限界の見極め: AIが苦手な作業(事実確認、最新情報、機微な判断等)を識別
  • 役割分担の判断: 「人がやるべきか / 工夫してAIにやらせるか」を場面ごとに選べる
  • プロンプト設計: 目的を達成するための指示の出し方・分解の仕方
  • 出力の検証: AI出力を鵜呑みにせず、ファクト・品質を評価できる
  • ツール使い分け: ChatGPT / Claude / Gemini / 画像生成系 / Claude Code 等を用途で選べる

B. PC操作(基礎)

  • ショートカット、タイピング速度、ファイル管理
  • 複数アプリ・ウィンドウの切替効率

C. 作業スピード・段取り

  • 業務処理量、見積/段取り力
  • AIに任せる前提でのタスク分解・並行処理

D. ビジネス基礎

  • 報連相、メール作法
  • Slack / Chatwork 等のチャット運用

E. ドキュメント作成

  • Word、Excel、Google スプレッドシート
  • Markdown(AIとの相性も良い)

F. 情報収集力

  • 検索スキル(Web、社内資料、AI検索)
  • 一次情報を辿る習慣

G. セキュリティ意識

  • パスワード管理、機密情報取り扱い
  • AIに渡してはいけない情報の判別
