評価・スキル機能（2026-05-09 決定）

方針

  • 人事評価機能を HANAツール に組み込む（システム化）
  • 検討時、スプレッドシート運用も候補に挙がったが、システム化のメリット（履歴管理・自動集計・運用負荷軽減・全社員横断の一元管理）を優先

評価方式

項目 内容
形式スキルチェックシート（質問に対する評価）
評価段階5段階（2026-05-24 変更、当初3段階）
実施頻度半期に1度
評価主体社員の自己評価 ＋ 社長によるレビュー
結果の用途教育・意識向上 ＋ 人事査定 の両方

設計思想

  • 質問項目に 「会社として身に着けてほしい項目」 を含めることで、社員の自然なベース向上を狙う
  • スキル可視化 ＋ 育成・査定の基盤として機能させる
  • 2026-05-08 「スキルシート構想」（→ notes.html）と一体の取り組み

5段階評価の定義（2026-05-24 確定）

レベル 定義
1知らない
2知っているができていない
3使うように努力している
4積極的に使っている
5良さを人に伝えられる（マスター・教える側）

学習プロセス型評価。レベル5の「人に伝えられる」を最上位に置くことで、組織全体の底上げループを作る設計。

重要度別 段階的運用ルール（2026-05-24 確定）

各項目に 重要度（A基礎／B業務効率／C専門） を付け、A から段階的に評価対象を広げる。

重要度の定義

重要度 内容
A. 基礎（必須）業務遂行に必須、できないと業務効率が大きく落ちる
B. 業務効率（推奨）業務スピードを大きく上げる
C. 専門（応用）専門性・差別化につながる高度なスキル

段階的進行ルール

[Step 1] 重要度A の項目のみ評価
              ↓
[判定] 全社員の80%以上が「4以上」を達成？
   No  → 改善期間 → 次回半期評価で再判定
   Yes → 重要度A は基礎クリア（常時 4以上 維持）
              ↓
[Step 2] 重要度B の項目を評価範囲に追加
              ↓
[判定] 80%が4以上？
   No  → 改善期間
   Yes → 重要度B クリア → 重要度C へ

この設計の効果

  • ゲーミフィケーション：次の重要度が「解放」される達成感
  • チーム目標：「全員80%」が共通ゴール、協力動機が生まれる
  • 評価工数の最小化：常に「次の重要度」だけ評価
  • 段階的ステップアップ：基礎→応用→専門と自然に深化

運用フロー（2026-05-17 追加）

評価タイミング

  • 定期：年2回（3月末・9月末） — 決算サイクル準拠
  • 初回（既存社員）：スキルシート完成後 すぐ実施（定期スケジュール待たない）
  • 初回（入社者）：入社時にベースライン評価 → 6ヶ月後に成長確認

査定（賃金改定）との連動

  • 評価：年2回（成長フィードバック中心）
  • 査定（賃金改定）：年1回（決算後＝4月想定）
  • 半期評価は「軌道修正・育成」、年次評価が「給与反映」というレイヤー分け

評価運用フロー

[評価期間終了の1ヶ月前]
  社長: スキルシート配布 → 社員: 自己評価(1週間)
              ↓
[終了2週間前]
  社長: 社長評価実施 → ギャップ把握
              ↓
[評価期間終了]
  1on1ミーティング(30分〜1時間):
   ・自己評価 vs 社長評価のすり合わせ

              ↓

目標管理機能（2026-05-24 設計方針）

背景

  • 2026-05-24 にHTMLベースの目標ツリー（business/goals.html）を新設したが、運用負荷を下げ、達成度の自動集計・履歴管理・複数人運用に対応するため、HANAツール本体への組み込みを企図
  • 達成記録の蓄積を「組織の学習資産」として残し、停滞感の防止と次の打ち手の判断材料にする（→ 全社運用ルールとして反映済み）

設計の本質3原則

原則 外すと…
目標は「正解」ではなく「仮説」目標変更が「悪」になり、現実とズレた目標を持ち続ける
達成も未達成も等しく資産「未達成だった」を記録しないと組織が学習しない
レビューしない目標は存在しないのと同じ立派な目標ツリーが「飾り」になる

採用フレームワーク：OKR-Lite + 達成ログ併走型

  • 世界的に検証された OKR（Google/Intel発祥）をベースに、はなさかの規模に合わせて軽量化
  • O（Objective：定性目標） と KR（Key Results：定量指標） を分離管理
  • KRの60〜70%達成で成功とするストレッチ文化（100%必達にしない）
  • 達成ログを別エンティティとして管理（完了 ≠ 削除）

階層モデル（5層、運用は2-3層）

1. Vision        — 「どこに向かう会社か」          (10年)
2. Strategic     — 「3年でここに到達」              (1-3年)
3. Objective     — 「今期これを達成」              (3-12ヶ月)
4. Key Result    — 「これが動けば達成と言える」    (1-3ヶ月)
5. Action        — 「そのために今これをやる」      (1-4週)
  • 階層は 3〜5層が運用限界。深くしすぎない
  • 5層目（Action）と「日々のタスク」は別物。タスクは目標管理システムでは扱わない

各目標ノードの属性

種類 属性
アイデンティティID / 名前 / 親ID / 階層レベル
時間軸開始日 / 期限 / 想定期間
責任オーナー（必ず1人）
数値目標値 / 現状値 / 単位 / 方向（↑↓→）
質的説明なぜ（why） / 達成判定基準
ステータス構想中 / 進行中 / 保留 / 達成 / 部分達成 / 未達成 / 中止 / 持ち越し
履歴ステータス変遷ログ / 数値の時系列 / 達成記録
関連関連プロジェクトID / 関連ドキュメントURL

⚠ ステータスは「未達成」「中止」「持ち越し」も明示的に持つ。削除・非表示にしない。

指標（KR）の設計原則

原則 内容
Leading と Lagging を混ぜる結果指標だけだと打ち手が遅れる
1目標に2〜3指標まで多すぎると焦点がブレる
入力可能なものに絞る計測できない指標は無意味
方向と期間を明示「↑ Q3末までに ¥X」のように記述

レビューサイクル（運用の心臓部）

頻度 対象階層 内容 所要時間
毎日（朝）KR / Action今日やることの確認、昨日の達成記録5分
毎週（月曜）KR今週のフォーカス決定、KR進捗更新30分
毎月（月初）Objective月次振り返り、目標の追加・削除・調整60-90分
四半期Objective 全面評価達成度判定、次四半期のObjective設定半日
半期Strategic戦略の妥当性検証、必要なら方針転換1日

達成記録（Achievement Log）の運用

  1. 完了 ≠ 削除。完了 = 達成ログに転記
    • 達成時にログエントリ生成（達成日 / 結果値 / 目標値 / 達成率 / 主な貢献 / 学び）
    • Goalツリー上では「達成」状態で表示続ける
  2. 達成ログは検索・絞り込み可能に
    • 期間・カテゴリ・オーナーで絞れる
    • 月次／年次レビューで自動参照
  3. 部分達成・中止・持ち越しも記録
    • 「やめた理由」「届かなかった理由」を明示的に残す → 組織の学習資産化

よくある失敗パターン（先回りで避ける）

失敗 仕組みでの予防
目標が増え続けて回らないActive状態の上限を設ける（例：Objective 10個まで）
KRに「やった/やらない」しかない必ず数値化。数値化できないものはKRにしない
ストレッチ過剰 or 安全すぎるストレッチKRと通常KRの混在を推奨
親なし目標が増える親IDを必須にする（最上位以外）
レビューしても何も変わらないレビューで「追加/削除/期限変更」を1つ以上必須化
オーナー不在オーナーは必ず1人（共同オーナー禁止）

実装フェーズ（MVP優先順位）

Phase 1（MVP・これだけで実用化）

  1. 目標のCRUD（階層構造、親子関係）
  2. 指標の登録（目標値・現状値・単位）
  3. ステータス管理（8状態）
  4. 達成ログ機能（記録できる場所）
  5. 月次レビュー画面（今月のObjective一覧 → 達成度更新）

Phase 2（運用が回り始めたら）

  1. 達成度の自動計算（指標から）
  2. 時系列グラフ（数値の推移）
  3. 通知・リマインダー（月次レビュー時期）
  4. レビュー記録（議事録形式）

Phase 3（複数人運用時）

  1. 個人目標と組織目標の連携
  2. メンバー別の目標管理
  3. CRM・タスク管理ツールとの連携

API設計の方針（Claude協働を見据えて）

  • 目標一覧・詳細取得APIを bin/hana-api.sh から呼び出せるように整備
  • Claudeが 登録済み目標を参照しながら作業 できるようにする（議論時の前提共有・整合性チェック）
  • 達成ログの登録APIも用意し、Claude経由でも記録可能に
  • READのみのAPIキー / 全権APIキーを分離（用途に応じて使い分け）
