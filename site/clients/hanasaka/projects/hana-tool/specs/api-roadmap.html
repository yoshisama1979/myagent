P1Priority 1：即効性が高い・今すぐ困る

P1-1. ToDo更新 API

エンドポイント案PATCH /api/external/todos/{id}
更新可能フィールド案content / description / next_work_date / due_date / assignee_user_id
なぜ秘書として必要「あのタスクの期日を木曜に変えて」「内容ちょっと書き直して」が今できない。会話で発生するメンテを即時反映できない
使用シーン会議中の調整、クライアント連絡後のタスク微調整、ToDo整理

P1-2. ToDo完了化 API

エンドポイント案PATCH /api/external/todos/{id}/complete（or P1-1 の status 変更で兼用）
挙動completed_at に現在日時を設定。再開時は /api/external/todos/{id}/reopen で null 戻し
なぜ秘書として必要「これ終わったから完了にしておいて」が今できない。完了登録のハードルが下がれば 達成記録が貯まりやすくなる（達成記録併走の原則と直結）
使用シーン会話中の完了報告の即時反映、週次レビュー時の一括完了化

P1-3. users 一覧 API

エンドポイント案GET /api/external/users
取得項目id, name, email, role, 在籍状況 等
なぜ秘書として必要担当者を振る際、社員リストが分からない。「松木さんに振っておいて」→ 松木さんのID が取れない（現状 .env のデフォルト1人のみ把握）。ToDo登録の正確性が大きく上がる
セキュリティemail等の機密性に応じて返却項目を絞る検討要

P2Priority 2：重要だが今すぐは凌げる

P2-1. work（案件）一覧 API

エンドポイント案GET /api/external/works（フィルタ：status, project_id, client_id, due_date_from/to）
なぜ必要現状 clients/search 経由でしか案件IDが分からない。「進行中の全案件」「今月期日の案件」を俯瞰したい。月曜定例の事前準備に有用

P2-2. ToDo の日付範囲フィルタ

追加パラメータ案completed_at_from / completed_at_to / due_date_from / due_date_to / created_at_from / created_at_to
なぜ必要今は全件取得して app 側で絞っている（先週分の取得もそう）。件数が増えるとパフォーマンス・データ転送量に影響
使用シーン月次レビュー、週次振り返り、期限切れアラート

P2-3. ToDoキーワード検索

追加パラメータ案q パラメータで content / description を部分一致検索
なぜ必要「あのバナー削除の件どこだっけ」と聞かれた時に直接引ける。記憶補助としての役割

P2-4. ToDoの複数work_id指定

パラメータ案?work_ids[]=49&work_ids[]=140
なぜ必要「HANAツール関連全部」「特定クライアントの全案件のToDo」など、案件横断で見たい

P3Priority 3：あると便利・将来的に

機能エンドポイント案用途
ToDoへのコメント追加POST /api/external/todos/{id}/comments「進捗どう？」のような会話履歴をToDoに残せる
work詳細取得GET /api/external/works/{id}予算・期日・関係者情報。秘書として濃い議論ができる
稼働時間取得（Toggl連携）GET /api/external/work-times「先週HANAツールに何時間使った」→ 利益率の議論に使える
日報取得・投稿GET/POST /api/external/daily-reports「今日の日報書いておいて」を会話から実行
ダッシュボード集計GET /api/external/stats/...案件別完了率、月次推移など。毎回計算せずに取れる
クライアント別の累積売上GET /api/external/stats/clients/{id}/revenueクライアント単価の判断材料

秘書として欲しい情報（API機能より一段先の論点）

API機能ではなく「情報そのものがどこにあるか」「Claudeがそれにアクセスできるか」の論点。HANAツールに集約するか、個別連携するかの方針判断が必要。

欲しい情報現状の所在私の活用方法
クライアント別の累積売上マネーフォワード or スプレッドシート「このクライアント単価高い／低い」の判断材料
進行中案件の予算・実績工数Toggl + 案件管理「赤字案件」の早期発見
メール／Chatworkの未対応一覧各ツール直接「対応漏れない？」のチェック
Googleカレンダーの予定Google直接「今週何時間が会議で潰れる」を見て、現実的なフォーカス3つを提案
外注先の稼働状況・残作業個別やりとり外注リソースの最適配分

議論ポイント（後日決定）

  1. Priority 1 から着手するか？
    • パートナー推奨：YES。ToDo更新 + users一覧で「会話↔︎HANAツールの双方向更新」が成立する
    • 判断：☐ YES（実装する） ☐ NO（後回し） ☐ 一部のみ
  2. 目標管理機能 MVP との実装順序
    • 選択肢A：目標管理MVP優先 → 落ち着いてからAPI拡張
    • 選択肢B：API拡張Priority 1だけ先に → 目標管理MVPは後（並行作業可？）
    • 選択肢C：両方並行（要工数）
  3. 「情報の所在」論点の方針
    • 選択肢A：HANAツールに段階的に集約（自社で抱える、自由度高い）
    • 選択肢B：個別ツールAPIを Claude が直接叩く（既存ツール活用、複雑）
    • 選択肢C：ハイブリッド（読み取り頻度の高いものはHANAツールにキャッシュ）
  4. users一覧のセキュリティポリシー
    • email・電話など機密情報を含めて返す？
    • 外部APIで返す範囲をどこまでにするか
  5. API設計の統一規約
    • 更新系：PATCH か PUT か
    • 状態変更：専用エンドポイント（/complete）か汎用PATCH（status: "completed"）か
    • ページネーション：必要か、必要ならどの方式（offset / cursor）
