目標管理機能 MVP 実装指示書

HANAツール内に目標ツリーと達成記録を扱う機能を新設するための、Phase 1 MVP 範囲の技術仕様。

作成日：2026-05-25 ／ 設計方針：decisions.html「目標管理機能」 ／ 上位の概念整理：business/goals.html

1. 目的とスコープ

解決したい課題

  • 現状の目標管理（business/goals.html 手書き）では、達成度の更新・履歴管理・複数人運用ができない
  • 達成記録が蓄積されないと「組織の学習資産化」が進まない
  • レビューが回らないと、目標管理が「飾り」になる

Phase 1 MVP の範囲

機能含む除外
目標CRUD階層構造、親子関係、ステータス8種並び替え・コピー機能
指標管理目標値・現状値・単位、複数指標可自動計算（数値ロールアップ）
達成ログ達成日・結果値・主な貢献・学び の記録承認フロー
レビュー画面月次：今月のObjective一覧 → 達成度更新四半期/半期/年次レビュー画面（Phase 2）
APIREAD + WRITE の REST エンドポイントGraphQL、Webhook
UI基本CRUDとツリー表示、月次レビューガントチャート、グラフ、ダッシュボード（Phase 2）
権限ログインユーザー単位（既存認証流用）RBAC、目標別ACL（Phase 3）

想定する技術スタック

  • PHP 8.x / Laravel（既存プロジェクト準拠）
  • MySQL（既存DB準拠）
  • API認証：既存の external API トークン方式（/api/external/... パターン）
  • UI：既存のフロントエンド方式（Blade / Vue 等、現行に合わせる）

2. データモデル

下記6テーブルで構成。すべて id（bigint, AI, PK）、created_at、updated_at、deleted_at（soft delete）を共通カラムとして持つ。

2.1 goals（目標本体）

カラム型説明
parent_idbigint nullable, FK→goals.id親目標のID。最上位はNULL
leveltinyint階層レベル（1=Strategic, 2=Objective, 3=KR, 4=Action）
titlevarchar(200)目標名（例：「HANAチャット販売拡大」）
descriptiontext nullable詳細説明（なぜ・判定基準）
owner_user_idbigint, FK→users.idオーナー（必須、1人のみ）
statusvarchar(20)draft / active / paused / done / partial / missed / cancelled / carried_over
period_startdate開始日
period_enddate期限
achievement_ratedecimal(5,2) nullable達成率%（0.00〜100.00。Phase 1は手動入力、Phase 2で指標から自動算出）
sort_orderint同一親内での表示順

📌 階層はself-referenceで実装。深さ制限は app層 で5階層までに制限（DBは無制限）。

2.2 goal_metrics（指標）

カラム型説明
goal_idbigint, FK→goals.id紐づく目標
namevarchar(100)指標名（例：「月間契約数」）
unitvarchar(20)単位（例：「件」「円」「%」）
target_valuedecimal(15,2)目標値
current_valuedecimal(15,2) nullable現状値（最新の goal_measurements を反映）
directionvarchar(10)up / down / maintain
metric_typevarchar(20)leading / lagging
sort_orderint表示順

📌 1目標に2〜3指標までを推奨（4以上は警告表示。バリデーションはしない）。

2.3 goal_measurements（指標の時系列値）

カラム型説明
goal_metric_idbigint, FK→goal_metrics.id紐づく指標
measured_atdate計測日
valuedecimal(15,2)実測値
notetext nullable備考
recorded_by_user_idbigint, FK→users.id記録者

2.4 goal_achievement_logs（達成記録）

カラム型説明
goal_idbigint, FK→goals.id達成された目標
achieved_atdate達成日
result_statusvarchar(20)done / partial / missed / cancelled
achievement_ratedecimal(5,2)達成率%
contributionstext nullable主な貢献（何が効いたか）
lessonstext nullable学び（次に活かすこと）
recorded_by_user_idbigint, FK→users.id記録者

📌 「未達成」「中止」も 明示的に記録。達成ログは消さない（組織の学習資産）。

2.5 goal_status_history（ステータス変遷ログ）

カラム型説明
goal_idbigint, FK→goals.id対象目標
from_statusvarchar(20) nullable変更前ステータス
to_statusvarchar(20)変更後ステータス
changed_atdatetime変更日時
changed_by_user_idbigint, FK→users.id変更者
reasontext nullable変更理由

2.6 goal_reviews（レビュー記録、Phase 1 では月次のみ）

カラム型説明
review_typevarchar(20)monthly（Phase 1）/ quarterly / semi-annual / annual（Phase 2以降）
review_periodvarchar(20)例：「2026-06」「2026-Q3」
summarytext振り返り総括
decisionstext nullableレビューで決まった目標調整内容
reviewer_user_idbigint, FK→users.idレビュー実施者
reviewed_atdate実施日

2.7 ER図（テキスト表現）

users
  └─ owns ─ goals
                ├─ parent_id ─ goals (self-ref)
                ├─ has many ─ goal_metrics
                │              └─ has many ─ goal_measurements
                ├─ has many ─ goal_achievement_logs
                └─ has many ─ goal_status_history

goal_reviews（特定の goal とは紐づかない、期間単位のスナップショット）

3. API 仕様

既存 /api/external/... パターンに合わせる。認証は既存トークン方式を流用。

エンドポイント一覧

メソッドパス用途
GET/api/external/goals一覧取得（フィルタ：status, parent_id, owner_user_id, period）
GET/api/external/goals/treeツリー構造で全件取得（ネストJSON）
GET/api/external/goals/{id}詳細取得（指標・達成ログ・履歴を含む）
POST/api/external/goals新規作成
PATCH/api/external/goals/{id}更新（部分更新可）
DELETE/api/external/goals/{id}削除（soft delete）
PATCH/api/external/goals/{id}/statusステータス変更（status_history に自動記録）
POST/api/external/goals/{id}/metrics指標追加
PATCH/api/external/goal-metrics/{id}指標更新
POST/api/external/goal-metrics/{id}/measurements計測値の記録
POST/api/external/goals/{id}/achievement-logs達成ログ登録
GET/api/external/goals/achievements達成ログ横断検索（period, owner, level でフィルタ）
POST/api/external/goal-reviewsレビュー記録の登録

4. UI 仕様

画面一覧（Phase 1）

#画面パス目的
1目標ツリー一覧/goals全目標を階層構造で俯瞰
2目標詳細・編集/goals/{id}個別目標の詳細表示・編集、達成ログ確認
3目標新規作成/goals/create新規目標の作成（親選択 → 詳細入力）
4達成ログ登録/goals/{id}/achievement-logs/create達成（または部分達成・中止）の記録
5月次レビュー/goals/reviews/monthly今月のObjective一覧 → 達成度更新 → レビュー記録

画面1：目標ツリー一覧

  • 左ペイン：ツリー表示（折りたたみ可能、ステータスバッジ付き）
  • 右ペイン：選択中の目標の概要（指標・直近の達成ログ）
  • フィルタ：ステータス（active/done/all）、オーナー、期間
  • 並び順：sort_order → level → title
  • 「新規作成」ボタン（親選択モーダル）

画面2：目標詳細・編集

  • ヘッダー：タイトル、ステータスバッジ、オーナー、期間
  • セクション1：説明（なぜ・判定基準）
  • セクション2：指標一覧（目標値・現状値・進捗バー）、追加ボタン
  • セクション3：子目標一覧（リンク）
  • セクション4：達成ログ履歴（最新10件、もっと見る）
  • セクション5：ステータス変遷履歴
  • 操作：編集 / ステータス変更 / 達成記録 / 削除（オーナーのみ可）

画面5：月次レビュー

  • 表示対象：当月期間中の Objective レベル目標（level=2）
  • 各目標について：現在の指標値を入力 → 達成率を計算（手動 or 自動）→ ステータス更新
  • レビュー総括欄（テキスト）→ goal_reviews に保存
  • 過去レビュー一覧へのリンク

5. 実装順序（推奨イテレーション）

Iter.内容完了の判定
1DBマイグレーション（6テーブル）、シーダー（テストデータ）artisan migrate / seed が通る、テーブル確認
2REST API：goals CRUD（基本）curl で create/read/update/delete が通る
3REST API：metrics、measurements、status変更curl テストパス、status_history が記録される
4REST API：achievement_logs、reviewscurl テストパス、ログ検索が動く
5UI：画面1（一覧ツリー）と画面2（詳細）ブラウザで目標を作成・編集できる
6UI：画面3（新規作成）、画面4（達成ログ登録）UI から CRUD・達成記録の全フローが動く
7UI：画面5（月次レビュー）月次レビューの一連の操作が完結する
8data移行：business/goals.html の内容を初期データとして投入HTML版の目標がツール内に再現される
9bin/hana-api.sh に goals 用コマンドを追加Claude が CLI で目標一覧を取得できる

📌 各イテレーションごとに動作確認 → コミット。Big bang を避ける。

6. 受入条件（Phase 1 完了の判定）

  • ☐5階層までの目標ツリーをUIで作成・編集・削除できる
  • ☐各目標に指標を複数登録でき、目標値・現状値を更新できる
  • ☐達成ログを登録でき、過去の達成ログを期間・オーナーで絞り込み表示できる
  • ☐ステータスを変更すると status_history に自動記録される
  • ☐月次レビュー画面から当月Objectiveを一覧 → 一括更新 → レビュー記録ができる
  • ☐全APIエンドポイントが認証付きで動作する
  • ☐business/goals.html の現状目標案がツール内に投入され、参照可能
  • ☐Claude Code から bash bin/hana-api.sh goals で目標一覧が取得できる

7. 既存システムとの統合点

  • users テーブル：オーナーは既存ユーザーから選択
  • 外部APIトークン：既存の /api/external/... 認証を流用
  • Chatwork通知：月次レビュー期限が近付いた時に通知（Phase 2）
  • ToDoとの連携：将来、目標の Action 層を ToDo と連携検討（Phase 3）
  • クライアント・プロジェクト：目標に関連プロジェクトを紐付ける外部キー追加検討（Phase 2）

8. Phase 2・3 への布石

Phase 2（運用が回り始めたら）

  • 達成度の自動計算（指標値から達成率を算出）
  • 時系列グラフ（measurements の推移を可視化）
  • 四半期・半期・年次レビュー画面
  • Chatworkでのレビュー時期リマインダー
  • 関連プロジェクトとの紐付け

Phase 3（複数人運用が定着したら）

  • 個人目標と組織目標の連携（個人OKR）
  • 権限管理（オーナー以外の閲覧・編集権限）
  • CRM・タスク管理ツールとの連携
  • 達成記録のサマリー自動生成（AIによる年次振り返り）

9. 実装時の注意事項

  • 「完了≠削除」を死守：達成・中止・持ち越しのいずれも履歴に残し、検索・参照可能に
  • 階層深度の制限：UI上で5階層を超える作成をブロック（バリデーション）
  • オーナーは必ず1人：共同オーナー禁止（UIで複数選択不可）
  • 削除はsoft delete：物理削除しない（誤操作時のリカバリ・履歴保全のため）
  • API ドキュメント：Swagger等で公開し、Claude協働で参照可能に
