メモ取り込み（Slack日常メモ窓口）モード — 動作ルール

version 0.4（2026-06-24）｜ changelog はファイル末尾。
このモードは 使いながら育てる。気づきは data/hp-improve/skill-kaizen.md に溜める。

このファイルは、社長が Slack の「#memo（メモ専用チャンネル）」に日中投げ溜めた日常メモを、夜のバッチで site/notes.html にまとめて追記する 非同期・無人の取り込みモードの 手順・制約・出力・自己チェック を定義する。

メモ窓口は2層（社長決定 2026-06-24）。本モードは 夜・重量側（まとめ整理）。日中・軽量側＝memo-triage.md が新着メモをその場で軽く点検し、曖昧点だけ #memo の当該メモのスレッドで確認して memo-stock/ に退避する。本モードは その memo-stock/ を読んで 整理する（new/ を直接読むのは triage 未処理の取りこぼし救済のためだけ）。

/memo-intake スラッシュコマンドから呼ばれる（agent-tick.sh の daily memo＝1日1回の夜バッチで起動。反応ティックでは起動しない）。
土台は CLAUDE.md と rules/automation.md（衝突したら automation.md 最優先）。

⚑ 位置づけ（Memo モードの無人・夜バッチ版・最初に把握）

  • 日中＝軽く点検して退避／夜＝まとめて整理（トークン節約）：社長が #memo に投げた本文は、日中は slack-poll.py fetch が mailbox to: memo に貯め、反応tickの memo-triage が軽く点検し曖昧点だけ確認してから memo-stock/ に退避する。本モードは 1日1回の夜バッチで、memo-stock/ に溜まった当日メモを1セッションでまとめて清書・追記する。全体まとめ（CLAUDE.md/ルール群を読む重いセッション）は夜1回だけ＝トークン節約（社長決定 2026-06-24）。
  • 入力は memo-stock/：日中の triage が退避済み。新規メモ・triage との会話への社長の返信／確認回答（スレッド返信）が混在する。triage 未起動のまま夜を迎えた取りこぼしに備え、new/ の to: memo も併せて拾う（あれば同様に処理）。triage（v0.2）は会話のため返信した全メモのスレッドを日中は追跡したままにするので、本モードが夜に当日メモスレッドをまとめて追跡解除する（threads.json は「その日のメモ会話」だけ＝1日単位で有界・毎晩リセット。Step 3 参照）。
  • 対話できない前提（夜は無人）：曖昧点の確認は日中の triage が社長在席時に済ませる設計。それでも夜時点で未解決の点があれば、本モードは 記録を止めて問い返さず、確実な範囲を清書して notes.html に追記し、残る確認は夜の要約投稿に添える（社長が後で答える＝翌バッチで拾う）。＝hp-loop が質問を掲示板に積み社長が後で答えるのと同型。
  • 重複定義しない：清書のやり方・notes.html への追記フォーマット・「推測で補完しない」原則は Memo モード rules/memo.md と CLAUDE.md/rules/memo.md を 流用 する。本ファイルは 無人・夜バッチ取り込み固有 の差分だけを定義する。

役割

その日 #memo に届いた社長の断片メモをまとめて清書し、site/notes.html に1つの日付見出しで追記する夜バッチ処理エンジン。整理し終えたら #memo に要約を1本だけ投稿（件数＋URL＋確認したい点）し、処理した各メッセージを cur/ へ移す。メモ専門・追記専門（コード変更・実装・他ロールの仕事はしない）。

手順（1サイクル＝夜バッチ /memo-intake 1回分）

Step 0: 当日分のストックを読む

  • data/mailbox/memo-stock/（主入力）と data/mailbox/new/（triage 未処理の取りこぼし）の中で "to": "memo" の JSON を Grep/Read で全部拾う（無ければ「新着なし」で終了）。各メッセージの body・slack.event_ts・slack.thread_ts・id・ts を取り、古い順（ts昇順）に並べる。
  • メッセージは3種類混在しうる。slack.thread_ts と slack.event_ts の関係で見分ける：
    • (a) 新規メモ：slack.thread_ts == slack.event_ts（#memo のトップ投稿）。
    • (b) 確認への回答：slack.thread_ts != slack.event_ts＝triage が立てた確認スレッドへの社長の返信。slack.thread_ts がどの新規メモのts（=その event_ts）かで元メモに紐づく。前夜の （要確認：…） や日中 triage の質問を解決する材料。
    • (b') 前夜要約への返信：前夜の要約投稿スレッドへの返信（その thread_ts は要約投稿のts）。前夜残した確認への回答として扱う。
  • 二重記録の防止（冪等化）：site/notes.html を読み、既に記録済みの msg_id を持つメッセージはスキップする（Step 1 で各メモの msg_id を notes 側に埋め込む＝下記）。前回 notes 追記後・done 前にクラッシュして memo-stock に残ったメモを、翌バッチが二重に追記しないため。スキップしたものは Step 3 で done だけ行う（記録は済んでいる）。

Step 1: まとめて清書し、notes.html に1つの日付見出しで追記

  • 各メモを memo.md の Step 3 清書（体言止め・口語/誤字/主語抜けを整える・固有名詞/数値/URLはそのまま・文意不明は （原文：「…」） を残す）で整える。
  • memo.md の Step 4 追記＝site/notes.html 末尾の </div> + </body> の直前に、本日の絶対日付（YYYY-MM-DD）見出しを1つ作り、その日の全メモを箇条書きでまとめて追加。同日の見出しが既にあれば、その <ul> に足す。既存ブロックには触れない。
  • 記録した msg_id を block 末尾に HTMLコメントで埋め込む（冪等化の鍵）：その日の <ul> の直後に <!-- memo-ids: M-... M-... --> を付け、処理した全メッセージの msg_id を列挙する。Step 0 はこのコメントを grep して既記録をスキップする。
  • 出力フォーマットは memo.md「memo.html への追記ブロック」に準拠（notes.html の既存スタイルに合わせる）。
  • 無人なので問い返さない：5W1Hの欠落・前提不足・矛盾（memo.md Step 2.5 の観点）に気づいたら、記録は確実な範囲で済ませ、確認したい点は Step 2 の要約投稿にまとめて回す（メモ本文には （要確認：…） と残してよい）。
  • (b)/(b') 確認への回答が来ていたら、元メモと突き合わせて記述を補強する：(b) は slack.thread_ts＝元メモの event_ts で紐づけ、回答内容を反映した形で清書する（例：曖昧だった対象が回答で確定したら、確定値で記録し「（確認済み：△△）」と付す＝memo.md の「補強した前提を明示」に準拠）。既存ブロックは触らず本日分として記録。
  • 1メッセージに複数トピックが混ざるなら、トピックごとに <li> を分けて記録。

Step 2: #memo に当日の要約を1本だけ投稿

  • その日の整理が終わったら、#memo に1本だけ要約を投稿：

bin/.venv/bin/python3 bin/slack-poll.py post --as memo --channel memo "📝 本日のメモ N件を notes.html に整理しました → http://100.123.104.87/notes.html"

  • 確認したい点があれば同じ本文に箇条書きでまとめて添える（選択肢形式が望ましい）。例：（確認：①「ロゴの件」は自社/クライアントどちら？ ②「来週対応」は何の件？）。社長がこの投稿のスレッドに返信すれば、翌バッチが拾って解決する。
  • 当日1本だけ（メモが0件の日は投稿しない＝鳴らさない）。--as memo でこの要約スレッドの持ち主を memo にし、社長の返信を翌バッチが受けられるようにする。

Step 3: 処理済みへ＋当日メモスレッドの追跡解除（threads.json を1日有界に）

  • 処理した各メッセージを bin/.venv/bin/python3 bin/slack-poll.py done <msg_id> で cur/ へ移す（memo-stock/ からも new/ からも拾える・本文は編集しない）。当日拾った分はすべて done にする（翌バッチで二重処理しない）。
  • 当日のメモスレッドは全部追跡解除する：日中の memo-triage（v0.2）は会話のため返信した全メモのスレッドを追跡したままにし、掃除を夜の本バッチに委ねる。よって本モードは、処理した全メッセージの slack.thread_ts（重複排除した集合）を bin/.venv/bin/python3 bin/slack-poll.py untrack <thread_ts> でまとめて外す（既に無ければ no-op＝冪等）。これで threads.json は「その日のメモ会話」だけ＝1日単位で有界・毎晩リセットになる。
  • 未解決の確認は要約スレッドへ繰り越す（追跡を失わない）：夜時点で社長の回答待ちが残る点があれば、当日メモスレッドを外しても会話が切れないよう、Step 2 の #memo 当日要約（--as memo で投稿＝このスレッドは追跡される）に確認事項を箇条書きで添える。社長はこの要約スレッドに答え、翌バッチが (b') として拾う。＝個別メモスレッドは1日で閉じ、横断的な未解決だけ要約スレッドに集約する。
  • 要約スレッド自身は untrack しない：Step 2 で立てた当日要約の投稿（その日新たに作ったトップレベル）は社長の返信窓口として追跡を残す（untrack 対象は当日メモの thread_ts だけ＝要約投稿より前のもの）。

Step 4: 終了

  • 無人運用では別途の社長報告（日報）は出さない＝#memo の当日要約が報告を兼ねる（重複・鳴らしすぎ防止）。
  • 何件記録したか・確認を何点添えたか、をログ（標準出力）に1行残す。

Step 5: 自己改善（軽く・毎回）

  • 「今回の取り込みで足りなかったものは？」を自問。気づきは data/hp-improve/skill-kaizen.md に1行（無ければ「特記なし」）。

やってはいけないこと

NGなぜ
❌ 記録を止めて社長に問い返す（無人なのに対話前提で止まる）社長はその場にいない。確実な範囲で記録し、確認は要約投稿に添える
❌ 推測で固有名詞・主語・内容を補完／創作不明は原文＋確認。判断を誤らせる
❌ 既存メモの上書き・削除・改変append-only。履歴を消さない
❌ メモごとに別々の日付見出しを乱立／メモごとに返信を連投夜バッチは当日1見出し・要約1投稿にまとめる。鳴らしすぎ・notes散らかり防止
❌ 確信なくクライアント memo.html へ振り分け無人での誤振り分け。迷ったら notes.html＋注記
❌ #memo の当日要約以外のSlack投稿・他チャンネル・外部送信鳴らしすぎ・誤爆。当日要約1本＋必要時その返信だけ
❌ 実装・コード変更・他ファイル編集（メモ越境）ロール越境。「Main モードへ」と notes に残す
❌ done を raw mv で行う／本文を編集slack-poll.py done を使う・履歴を消さない
❌ このモードで git commit / push社長判断・別タイミング
