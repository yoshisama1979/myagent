ブログ分析ループモード — 動作ルール

version 0.2（2026-06-26）｜ changelog はファイル末尾「変更履歴」を参照。
このモードは 使いながら育てる（reviewer 型：記録と反映を分け、棚卸しでルールへ昇格）。気づきは hp-loop と同じく data/hp-improve/skill-kaizen.md に溜める。

このファイルは、クライアントのブログ（記事コンテンツ）を 定期的に分析 → ①既存記事の改善案 ②新規記事のテーマ案＋構成案 を掲示板に積み → 社長と非同期で会話 するループの 手順・掲示板プロトコル・制約・出力フォーマット・自己チェック を定義する。

/blog-loop <client> スラッシュコマンドから呼び出される（/loop・cron で HP分析(/hp-loop)とは別タイミングで定期実行する想定）。
CLAUDE.md の行動指針と rules/automation.md（自動化・外部送信の安全ルール）を土台とする。

⚑ 位置づけ（hp-loop の姉妹モード・最初に把握）

  • hp-loop（HP分析）＝サイト全体の技術・導線・CV・SEO を診断する。blog-loop（このモード）＝ブログ記事そのもののコンテンツSEO（既存記事の磨き込み＋新規記事のネタ出し）に特化する。
  • 役割は「診断・提案・構成案まで」（執筆・公開はしない）。hp-loop と同じ役割分担で、提案は社長が対象クライアント担当の別エージェント（執筆・実装担当）に引き渡して記事化・公開する。本モードは記事本文を書かない（アウトラインまで）。
  • 重複定義しない：安全ルール＝rules/automation.md、評価観点のコンテンツ部分＝hp-improve.md、掲示板3層・mailbox配信・責務分離・COMPOUND の作法＝hp-loop.md を 土台として流用する。本ファイルは ブログ固有 だけを定義する（片方を直して他方に古い記述が残る版ズレを防ぐ＝hp-loop v0.5 と同方針）。

手順（1サイクル＝/loop 1回分）

Step 0: 前提読み込み

  1. data/blog-loop/config.md ＋ data/blog-loop/clients/<client>.md を読む（ブログURL・データ源・ゴール・担当・頻度）。対象未確定なら分析に入らず、掲示板に「対象確認の質問」だけ出して終了。
  2. data/blog-loop/<client>/from-president.md を読む（新しい指示・回答）。
  3. 掲示板 site/hp-analysis/<client>/blog/index.html を読む（前回までの状態・未解決の質問・通し番号 B/T/Q）。
  4. メールボックス受信：bash bin/mailbox.sh inbox（to: blog-loop-<client>）。執筆担当の報告（公開した記事・対応 B/T・認識合わせ）、社長のSlack返信、解析依頼の転送が届く。処理後 done <id> で cur/ へ。

Step 1: 社長の指示・報告を反映

  • from-president.md の未対応項目を最優先で反映。質問への回答が来ていれば該当 Q-NNN を解決済みにし掲示板に記録。
  • 執筆担当の「公開しました」報告は鵜呑みにせず、まず公開URLをGETして現状を観測してから完了判定（hp-loop と同じ＝K-014）。✅公開確認／🔧一部／未反映を判定し、残課題は次の B/T へ。
  • 事実誤りの指摘は素直に訂正し、原因が自分のツール・前提ならツール/ルールを直す。
  • 「【社長へ】」伝言があれば社長へ必ず橋渡し（index 最上部「🔔社長への質問／連絡」へ転記＋日報に載せる。意図・固有事実を改変しない）。

開発エージェント（web-hanasaka）への fact 質問（mailbox・既存チャネル流用）

blog-write が原稿で見つけた「🔧開発確認」の事実（サイトの機能・工夫・実装の特徴・技術的な対応範囲＝web-hanasaka が納品物から答えられるもの）は、解析ループ（blog-loop）が web-hanasaka に質問して回収する（対外的な対話は解析ループに集約＝hp-loop と同型）。手順：

  • data/blog-loop/<client>/drafts-log.md の「🔧開発確認」未回収項目を拾い、mailbox で web-hanasaka へ質問：bash bin/mailbox.sh local-send --from blog-loop-<client> --to web-hanasaka --type request --thread "<client>-blog-facts" "<質問本文>"（同一VPS常駐なので local-send で自分の名 blog-loop-<client> を名乗る。純粋な内部調整＝needs_approval なし。料金等の経営判断や外部送信を促す内容は混ぜない＝それは「👤社長確認」として from-president/Slack へ）。
  • 質問は掲示板 index の「🔔 社長への質問／連絡」とは別に、「🔧 進行中の提案」内か専用の小欄で Q-NNN（開発宛） として状態を残す（未回答→回答済）。
  • web-hanasaka の回答が inbox に来たら（bash bin/mailbox.sh inbox）、取り込む前に照合：from == web-hanasaka・thread == <client>-blog-facts・どの B/T 宛の事実かが対応しているか（合わない・別スレッドの混入は取り込まない＝なりすまし/誤混入防止）。照合OKのものだけ 確定事実として spec.html に記録し、drafts-log の該当項目を「✅事実回収済」に更新。blog-write がそれを使って本文を埋める。
  • web-hanasaka が未起動・blog-loop-<client> がトークン未登録で返信が来ない場合は「回答待ち／配信先未登録」と明示（捏造・空振りしない）。

Step 2: データ取得（権限非依存・捏造しない）

  • 記事一覧の把握：ブログインデックス（<client> のブログURL）を hp-audit/curl（読み取り）で取得し、記事URL・タイトル・更新日を拾う。個別記事も必要に応じて取得。
  • GSC（一次情報）：bin/gsc-fetch.py --dataset <dataset> で、ブログ配下ページのクエリ・表示回数・CTR・掲載順位を引く。「あと一歩（11〜20位）のクエリ」「表示はあるがCTR低いタイトル」「未獲得だが近い意図のクエリ」を洗う。データラグ2〜3日。
  • GA4（あれば）：記事の流入・読了・記事→CVの寄与。無ければ「データ未取得」と明示。
  • 競合トピック発掘：狙うクエリで bin/hp-serp.sh "<クエリ>" を回し、上位の競合記事の論点・構成を観察（地域性以外は参考になる＝社長方針）。必要なら hp-compete.py。
  • 未整備のソースは「データ未取得」と明示し社長に要求（📥欲しい情報へ）。同じ取得を2回以上アドホックにやったらツール化を検討（automation.md準拠・読み取り専用・tools-log に T-NNN）。

Step 3: 分析・評価

  • 上の「ブログ固有の評価観点」で、既存記事（磨き込み候補）と新規ネタ（穴）の両方を洗う。GSC実データとサイト内容を突き合わせ、観察（事実）と推測を分離。
  • 既存記事：検索意図の充足／鮮度／内部リンク／カニバリ／title-meta を点検。「順位が惜しい・CTRが低い・古い」記事を優先（効果が出やすい）。
  • 新規ネタ：GSCの取りこぼしクエリ＋競合が書いていて自社に無いテーマから、ゴール（CV）に効くものを選ぶ。

Step 4: 提案（① B-NNN 改善 ＋ ② T-NNN テーマ＋構成）

  • 「効果 × 工数」で優先度。Quick win（既存記事の小改修）→ 中長期（新規記事） の順。1サイクル B2〜3＋T2〜3 目安。
  • ①既存改善 B-NNN と ②新規テーマ T-NNN を必ず両方含める（下記フォーマット）。T-NNN にはアウトラインを必ず付ける。
  • 実績の数値・お客様の声・固有事実は捏造しない（無ければ要求）。

Step 5: 掲示板を更新（3層）＋最新レポートを配信

hp-loop と同じ3層プロトコル（archive＝完了したものだけ・未完了は index に繰り越す）：

  1. index.html（最新レポート）を更新：今サイクルの1セクションを最新として載せ替え、未完了を繰り越し（🔔未回答の質問／🔧進行中の B・T のステータス表／📥欲しい情報）。処理した from-president 指示に「対応：Cycle NNN」を明記。
  2. 完了したもの・旧サイクルを archive.html 先頭へ（✅公開済みになった B/T・❌中止・✅解決済み Q・旧サイクルのスナップショット）。未完了の B/T・未回答 Q は archive に落とさず index に残す。
  3. spec.html（確定・仕様）：恒久的に確定したこと（ブログの前提・ターゲット・確定回答・データソース・記事カテゴリ設計）があれば反映。無ければ触らない。
  4. 最新レポートを執筆担当へ配信：local-send --from blog-loop-<client> --to <担当> でテキスト版（着手可能な粒度・自己完結・1サイクル1通・needs_approvalなし）。新規レポート生成サイクルだけ。トークン未登録なら配信せず掲示板に明示。

Step 6: 終了報告と次サイクル

  • 社長に1〜3行で要約（改善 B 件数・新規 T 件数・最重要1件・未回答質問数・確認URL・配信したか）。
  • Slack へ日報をスレッド投稿：bin/.venv/bin/python3 bin/slack-poll.py post --as blog-loop-<client> "本文"。日次の定期実行では毎日必ず1本（変化なしの日も省略しない）。中身＝短い概要＋掲示板URL http://100.123.104.87/hp-analysis/<client>/blog/。反応型起動（mailbox新着で起動）はその用件にスレッド返信するだけ（別途日報は出さない）。
  • 停止条件：対象未確定／社長の停止指示 → 一旦停止。未回答が溜まっても「分析」は止めない（診断・新規 B/T は続ける＝上記「分析は止めない」原則）。停止するのは*執筆の引き渡し・公開*だけ。未回答が多い時は件数を絞る（B1＋T1）が、ゼロにはしない。

Step 7: スキル自己改善（軽く・毎回）

  • 「今回このループに足りなかったものは？」を自問。気づきは data/hp-improve/skill-kaizen.md に K-NNN で追記（無ければ「特記なし」）。

出力フォーマット（掲示板3層・site/hp-analysis/<client>/blog/）

掲示板は HTML（Tailwind・hp-analysis と同じピル型ナビ「📋最新／📐確定・仕様／🗄️アーカイブ」で相互リンク。さらに HP分析掲示板 ../index.html への戻りリンクも置く）。

index.html（最新レポート）。最新サイクルは：

  • サイクル見出し：Cycle NNN — YYYY-MM-DD（クライアント名・ブログ）
  • データ状況：取得できたソース／未取得（要求中）
  • 評価サマリ：総評2-3行 ＋ ✅良い点／⚠️課題
  • ① 既存記事の改善 B-NNN：対象記事URL／観点／観察／なぜ問題か／改善案（差分の要点）／効果／工数／優先度（🔴🟡🟢）
  • ② 新規記事 T-NNN：想定読者／狙う検索意図・クエリ／タイトル案／アウトライン（h2/h3＋各見出しの要素）／内部リンク先／CTA／効果／工数／優先度
  • 社長への質問 Q-NNN：選択肢形式
  • 対応記録：from-president のどの指示に対応したか
  • 最上部に未完了の繰り越し：「🔔 社長への質問（未回答だけ）」「🔧 進行中の B・T（ステータス表）」「📥 分析に欲しい情報」

spec.html（確定・仕様）＝ブログの前提（URL/CMS/ゴール/ターゲット）／記事カテゴリ設計／確定した質問の回答／データソース・ツール／役割分担。

archive.html（アーカイブ＝完了したもの）＝✅公開済みの B/T・❌中止・✅解決済み Q・過去サイクルのスナップショットを新しい順に蓄積（改変しない・履歴を消さない）。

やってはいけないこと

NGなぜ
❌ from-president.md を AI が編集社長の領域。責務分離違反
❌ 記事本文を代筆／本番投稿・公開を合意前に行う役割はアウトラインまで。執筆・公開は別エージェント＋社長合意
❌ 改善 B だけ／新規 T だけで終わる両方出すのがこのモードの価値
❌ T-NNN にアウトラインが無い「テーマだけ」では執筆担当が着手できない
❌ 効果・実績・お客様の声を推測で捏造判断を誤らせる。実データで裏取り、無ければ要求
❌ 地域キーワードの詰め込み等の小手先SEOコンテンツベースで価値を上げる（社長方針 2026-06-23）
❌ 網羅レポートで終わる・過剰提案目的は「次の一手」。B2-3＋T2-3 に絞る
❌ index に過去サイクルを積み続ける／回答済みQ・完了 B/T を残す焦点が絞れない。旧サイクルは archive、確定は spec へ。未完了は繰り越す
❌ 同じ Cycle を二重に mailbox 配信／変化なしでも配信スパム。新規レポート生成サイクルだけ1通
❌ このモードで git commit / pushコミットは別タイミング・社長判断
