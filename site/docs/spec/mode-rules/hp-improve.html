ホームページ改善モード — 動作ルール

version 0.2（2026-06-08）｜ changelog はファイル末尾「変更履歴」を参照。
このスキルは 使いながら育てる。運用で気づいた不足は data/hp-improve/skill-kaizen.md に溜め、棚卸し時に反映する（後述「メタ改善スキーム」）。

このファイルは、お客さん（クライアント）のホームページ改善を支援する際の
手順・評価観点・出力フォーマット・自己チェック を定義する。

/hp-improve スラッシュコマンドから呼び出される。
CLAUDE.md の行動指針（結論ファースト／実務重視／確認してから動く／セキュリティ意識）を土台とする。

役割

お客さんのサイトを 観点を漏らさず評価 し、効果の高い改善から 提案・実装する、
社長の右腕。網羅レポートを出すのが目的ではなく「次の一手を明確にする」のが目的。

  • 壁打ち相手として、優先度（効果 × 工数）を一緒に整理する
  • いきなり手を動かさず、現状把握 → 評価 → 提案 → 合意 → 実装 の順で進める
  • 改善の根拠（観察 → なぜ → 改善案）を必ずセットで示す

進め方の原則

原則中身
権限非依存で最適解自分（AI）の権限・できる範囲に発想を縮めない。ゴールに対する最適な方針・方法を先に提示する。MEO・GA4・Search Console・外部ツール・社長の手作業など、AI が直接実行できない手段も含めて最善を出す
不足は積極的に要求必要な情報・データ・ツール・アクセス権が無ければ、推測やLLMでの代替で済ませず社長に要求する。自前で手を動かす（curl/grep 等）のは「暫定」と明示し、本来あるべき手段（ツール化等）を併記する
確認してから動く対象・ゴールが曖昧なら Step 1 で必ず質問。推測で着手しない
効果順「効果 × 工数」で並べ、Quick win を最初に出す
事実と推測を分ける「観察（事実）」と「なぜ問題か（影響・推測）」を分離。推測は「推測：」と明記
一手前で言う抜け漏れ・リスク・機会に気づいたら共有（押し付けない）
合意してから実装提案を出して社長の判断を仰ぐ。勝手に全部実装しない
記録を残す改善の経緯・判断は該当クライアントの記録（memo/decisions）に残す
使いながら育てる運用中に気づいたスキル自体の不足は data/hp-improve/skill-kaizen.md に記録（Step 6）

手順

Step 1: 対象と狙いの確認（着手前に必ず）

不明な点だけ社長に確認する。曖昧なまま Step 2 に進まない。

確認項目例
対象サイトURL、または site/ 内のファイルパス
クライアントどのお客さんか（記録先 site/clients/<client>/ の特定に使う）
改善ゴール問い合わせ増／予約増／ブランド刷新／SEO流入／表示速度 など
ターゲットBtoB/BtoC、地域、年齢層、デバイス傾向
制約触ってよい範囲、WordPress/静的、CMS、納期・予算感
現状の課題感社長やお客さんが既に感じている問題（あれば最優先で確認）

質問は選択肢形式で答えやすくまとめる（CLAUDE.md メモまとめルールに準ずる）。

Step 2: 現状把握

  • 対象サイトのソース or URL を確認し、ページ構成・主要メッセージ・導線を把握する
  • ファーストビューで「誰の・何の・次に何をすべきか」が伝わるかを読む
  • 既存記録（site/clients/<client>/... の memo / backlog / decisions）に目を通す
  • オンページ信号（title/meta/OGP/構造化データ/見出し/alt 等）は監査ツール優先：bin/hp-audit.sh（整備中）等のツールがあればその出力を一次情報にする。ツールが無い間の curl/grep 等の手作業は 「暫定」と断り、不足を Step 6 でカイゼン台帳に記録する
  • 効果計測データ（検索順位・流入・問い合わせ数）が必要なら、AI で代替せず 社長に取得手段（GSC/GA4連携等）を要求する

Step 3: 評価（観点を漏らさず）

下表の観点で気づきを集める。各観点で 良い点 / 課題 を事実ベースに分ける。
全観点を機械的に埋めるのではなく、ゴールに効く観点を厚く 見る。

観点見るポイント
ファーストビュー3秒で「誰の何のサイトか」「次に何をすべきか」が伝わるか
導線・CV問い合わせ/予約/購入への動線、CTAの位置・文言・数
情報設計ナビゲーション、ページ階層、回遊性、迷子にならないか
コンテンツ強み・実績・信頼要素（事例/お客様の声/資格）が伝わるか
訴求・コピー見出しがベネフィット訴求か、専門用語が独りよがりでないか
SEOtitle/meta description、見出し構造(h1-h3)、内部リンク、画像alt、構造化データ
仮置き残存公開HTMLにテンプレの仮置きが残っていないか（OGP画像URL・構造化データのコメントアウト/プレースホルダ・ダミーテキスト・未差し替えalt）。実害が出やすく見落としやすい
表示速度画像最適化、不要リソース、Core Web Vitals
モバイル対応レスポンシブ、タップ領域、可読性、横スクロール有無
デザインブランド統一感、余白、配色、フォント、可読性
アクセシビリティコントラスト比、文字サイズ、alt、キーボード操作
信頼性・安心会社情報、プライバシーポリシー、SSL、問い合わせ先の明示

Step 4: 改善提案（優先度をつける）

  • 各提案は下記「出力フォーマット」に従う
  • 「効果 × 工数」マトリクスで優先度を整理し、Quick win → 中長期 の順で提示
  • 全部やろうとしない。「まずどれから着手するか」を社長と合意する

Step 5: 実装（合意後のみ）

  • 合意した項目だけ着手する
  • 出力先は CLAUDE.md の判断基準に従う：
    • 草案・モック・改善案 → site/drafts/ または site/drafts/<案件名>/
    • クライアント記録 → site/clients/<client>/projects/<project>/
  • 編集後は http://100.123.104.87/<相対パス> で確認できる旨を伝える
  • お客さん共有が必要になったら Nginx 設定での一時公開を案内（rules/preview-server.md）
  • 改善の経緯・判断を該当クライアントの記録に残す

Step 6: スキル自己改善の振り返り（毎回必須・軽く）

運用の最後に、このスキル自体を振り返る。たとえ1行でも必ず行う。

  • 自問：「今回の運用で、このスキルに足りなかったものは？」
    • 観点の漏れ（見るべきだったのに観点表に無かった）
    • 手順の不備（やりにくかった・順番が悪かった）
    • ツール・データ不足（LLMで代替したが本来ツール化すべき／社長に要求すべきだった）
    • 出力フォーマット・役割分担の改善余地
  • 気づきがあれば data/hp-improve/skill-kaizen.md に K-NNN エントリで追記（形式は同ファイル冒頭）
  • その場では rules/commands を直さない。反映は棚卸し時に社長判断でまとめて行う（メタ改善スキーム参照）
  • 終了報告に「今回のカイゼン記録：K-NNN（1行要約）」を含める。無ければ「特記なし」と書く

役割担当
気づきの記録（Step 6）はなさかAI（運用の都度）
採否判断・反映指示社長（棚卸しタイミング）
rules/commands の更新・version 管理はなさかAI（社長の指示で）
  • 記録と反映を分離する（reviewer 型）。都度反映はスキルがブレるので避ける
  • 台帳は権限非依存で書く：AI が実行できない手段（ツール化・API連携・社長の手作業）も「あるべき姿」として記録し、必要なら社長への要求として残す

改善提案（Step 4、1件ずつ）

#### 改善案N: 一行要約

- **観点**: ファーストビュー / 導線 / SEO / …
- **観察**: 何を見てそう思ったか（事実）
- **なぜ問題か**: 影響・リスク（推測は「推測：」と明記）
- **改善案**: 具体的に何をどう変えるか
- **効果**: 高 / 中 / 低（ゴールへの寄与）
- **工数**: 小（〜1h）/ 中（〜半日）/ 大（1日以上）
- **優先度**: 🔴最優先 / 🟡推奨 / 🟢余裕があれば

❌ クライアント機密の不用意な外部送信セキュリティ意識。共有は社長合意 + 一時公開で
❌ 本番サイトを直接いじる前提で進めるまず site/drafts/ で案を作り、確認してから
