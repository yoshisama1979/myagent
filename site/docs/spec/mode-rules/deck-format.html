スライド資料（デッキ）作成ルール — HTML → PDF

version 0.1（2026-07-13・初版）｜ changelog はファイル末尾「変更履歴」を参照。
別プロジェクト（hana-chat 営業提案・A4横21ページ）で実際に構築・運用した手順と、
そこで実際に踏んだ失敗を移植したもの。きれいな最終形ではなく失敗が本体。

クライアント向けのスライド資料（提案書・営業資料・報告デッキ）を **単一の HTML で作り、
ヘッドレス Chrome で A4横の PDF に書き出す** ときの共通ルール。PowerPoint も Figma も使わない。

HTML/CSS の骨格（1ページ＝1 <section class="slide">）

/* ★ @page はスタイルシートの「トップレベル」に置く（@media print の中に入れない） */
@page { size: 297mm 210mm; margin: 0; }   /* ★ キーワード(A4 landscape)でなく実寸で書く */

html, body { margin: 0; padding: 0; }
.slide {
  width: 297mm; height: 210mm;
  box-sizing: border-box;
  break-after: page;                      /* 1 section = 1 ページ（現行標準） */
  page-break-after: always;               /* 旧プロパティ＝互換用に併記 */
  overflow: hidden;                       /* あふれ＝改ページ崩れを画面で検知できる */
  -webkit-print-color-adjust: exact;      /* ★ 無いと背景色・装飾が印刷で消える */
  print-color-adjust: exact;
}
table { table-layout: fixed; width: 100%; }  /* ★ <colgroup> で列幅を明示（下記） */
  • 表は table-layout: fixed ＋ <colgroup><col style="width:30%">…</colgroup> で列幅を明示する。

指定しないと文字数で列幅がずれ、ページごとに表の見た目が揃わない。

  • 共通クラス化するのは「意味が同じで、まとめて変えたくなる部品」だけ（見出し帯・表の可否セル ✓/✗・

注釈ボックス等）。スライドごとの grid・余白・配置まで CSS クラスに畳まない＝個別スライドの
微調整が効かなくなり、押し出しで A4 の改ページが崩れる（実際にやって戻した）。

落とし穴カタログ（実際に踏んだもの・症状→原因→対処）

1. PDF が縦になる（3つの別原因がある。順に疑う）

#症状原因対処
1aHTML は横なのに PDF が縦@page { size: ... } を @media print { } の中に書いた（無視する環境がある）@page をスタイルシートのトップレベルに置く
1b同上size: A4 landscape のキーワード指定が無視される環境がある実寸で書く：size: 297mm 210mm
1cHTML も CSS も正しいのに縦印刷ダイアログの送信先が「Microsoft Print to PDF」等のプリンタドライバ＝Chrome は @page を捨ててドライバの用紙設定（既定 A4 縦）を使う@page が効くのは送信先「PDF に保存」のときだけ。手動印刷ではそれを選ぶ。ヘッドレス --print-to-pdf はこの罠を踏まない

2. 背景・装飾が印刷で消える

  • 症状：画面では出ている背景色・帯・アイコンが PDF で白くなる。
  • 原因：印刷時のインク節約既定で背景がスキップされる。
  • 対処：print-color-adjust: exact（-webkit-print-color-adjust: exact も併記）を .slide に指定。

3. 画像が入らない

#症状原因対処
3aPDF で画像だけ抜けるHTML を一時ディレクトリにコピーしてからレンダリングした＝相対パスの画像が解決されない必ずリポジトリ上の実パスからレンダリングする（コピーしない）
3bfile:// で画像がブロックされるヘッドレス Chrome のローカルファイルアクセス制限--allow-file-access-from-files を付ける

4. 表の列幅が揃わない

  • 症状：同じ構造の表なのにページによって列幅が違う。
  • 原因：既定の auto レイアウトは中身の文字数で幅を決める。
  • 対処：table-layout: fixed ＋ <colgroup> で幅を明示（上記骨格）。

5. 出力ファイルがまだ書き終わっていない

  • 症状：生成直後に PDF を読むと壊れている・サイズ 0。
  • 原因：レンダリングプロセスの完了を待たずに次へ進んだ。
  • 対処：完了を待ってから読む。Linux（この VPS）ではコマンドが同期実行なので通常は待てば済む。

Windows（PowerShell）では Start-Process -Wait -NoNewWindow で待つ。読む前に ls -la でサイズ確認。

実行コマンド（この環境＝VPS でそのまま動く実例。各環境で読み替え）

# HTML → A4横 PDF（リポジトリ上の実パスから・ヘッダ/フッタ無し）
/usr/bin/google-chrome --headless=new --disable-gpu \
  --allow-file-access-from-files --no-pdf-header-footer \
  --print-to-pdf=/home/vpsuser/projects/myagent/site/decks/<案件名>/deck.pdf \
  "file:///home/vpsuser/projects/myagent/site/decks/<案件名>/deck.html"
  • 待ち時間フラグの使い分け：JS のタイマー・アニメーション描画を早送りするのは --virtual-time-budget=5000

（仮想時間＝実時間の待機ではない）。外部フォント・遅延ロードなどネットワーク待ちには --timeout=5000
（最大待機の実時間）を使う。重要資料はそもそも外部 Web フォント依存を避けるのが確実。

  • Windows（事務所PC）では Chrome 実パスを読み替え、PowerShell なら

Start-Process -Wait -NoNewWindow で完了を待つ（落とし穴5）。

検証（「たぶん大丈夫」で終わらせない・2手順）

A. PDF の実寸とページ数を PDF 内部から読む

目視では「なんとなく横っぽい」しか分からない。/MediaBox と /Type /Page を数えて確認する：

python3 - <<'EOF'
import re
d = open('site/decks/<案件名>/deck.pdf','rb').read()
boxes = set(re.findall(rb'/MediaBox\s*\[([^\]]+)\]', d))
pages = len(re.findall(rb'/Type\s*/Page\b', d))
print('MediaBox:', boxes)   # A4横 ≒ 幅842 x 高さ595 pt（Chrome出力は 841.92 x 594.96 等の近似値）
print('Pages:', pages)      # 意図したページ数と一致するか
EOF

幅 > 高さ（≒842×595pt）とページ数の一致を確認して初めて「A4横で N ページ」と報告してよい。
※ これは Chrome --print-to-pdf 出力向けの簡易検証（オブジェクトストリーム圧縮の PDF では平文に
/Type /Page が現れず数え漏れる）。値が疑わしいときは pdfinfo・qpdf・PyMuPDF 等で確認する。

B. リファクタ前後のスクリーンショットを SHA256 で比較する

CSS を整理・共通化したとき「挙動不変」を主張するなら、ピクセル差ゼロで裏を取る：

  1. 特定スライドだけ表示する CSS を一時注入（例 .slide{display:none} .slide:nth-of-type(7){display:block}）
  2. 変更前・変更後で同条件のスクリーンショットを撮る（--screenshot --window-size=...。

本リポジトリなら bin/hp-shot.sh の流用も可）

  1. sha256sum before.png after.png → ハッシュ一致＝ピクセル差ゼロを確認してから「見た目不変」と書く

🚨 スクリーンショットの扱い（最重要・技術より先に読む）

実事故：管理画面のスクリーンショット（他クライアントの会話履歴・フリーダイヤル・顧客の質問内容が
写ったもの）を、別のクライアント向け資料に貼っていた。クロスレビューで発覚しコミット直前で止めた。
技術の失敗は直せば済むが、他社データの流用は直せない（コミットすれば履歴に残り、配布すれば取り返せない）。

ルール理由
管理画面のスクショには実データが写る前提で、撮ったら必ず中身を精読する「画面の見た目」を撮ったつもりでも中身はデータ
他社のデータは使わない。ぼかし・社名消しでは足りない掲載同意がない。質問内容から業種と当事者が推測できる
提供元を隠す資料（中間業者経由の提出等）では、本文だけでなくスクショ内の社名・連絡先も確認する表紙から連絡先を消しても画像に社名が残れば意味がない
架空のデモ用クライアントを1つ作り、そこで全画面を撮り揃えるのが最も安全で早い実データ混入が構造的に起きない
画像・スクショ・PDF は安全確認が済むまで .gitignore で追跡対象外にする（例：site/decks/<案件名>/deck-images/・site/decks/<案件名>/*.pdf を追加）一度コミットすると git 履歴から消せない
deck-images/README.md に画像ごとの台帳を残す：取得元／同意の有無 or デモ環境か／精査者／精査日レビュー時に「この画像は安全か」を都度目視し直さなくても判断できる

NGなぜ
❌ @page を @media print の中に書く／A4 landscape キーワードに頼る落とし穴1a/1b。環境で無視され縦になる
❌ HTML を一時ディレクトリへコピーしてレンダリング落とし穴3a。相対パス画像が抜ける
❌ スライドごとの grid・余白まで共通クラス化個別調整が効かず A4 改ページが崩れる。共通化は「まとめて変えたい部品」だけ
❌ 目視だけで「A4横・Nページ」と報告検証Aを通す（MediaBox・ページ数）
❌ 「見た目は変えていません」をハッシュ比較なしで主張検証Bを通す（SHA256一致）
❌ 実データ入りスクショの使用・同意なき他社データの掲載上記🚨。ぼかしでは足りない・履歴から消せない
❌ 未確認の画像・PDF をコミットgitignore で止めてから精査。コミット後は取り返せない
