# 依頼：エージェントの稼働状況を可視化する「3Dオフィス」ダッシュボードを作ってほしい

この環境で動いている複数のエージェント（定期実行ループ・受信箱で仕事を待つ担当など）の状態を、
**3Dのオフィスの部屋**として一目で見られるダッシュボードを作ってください。
同じものを別環境（VPS）で構築済みで、この指示はその実証済みの設計と教訓をまとめたものです。
**設計はこの指示に従い、データ源だけこの環境の実態に合わせて置き換えてください。**

## 完成イメージ（何が見えるか）

- 3Dの部屋に役割ごとの「島」（デスクのまとまり）が並び、各デスクにエージェントのアバターが座っている
- エージェントの状態が動きで分かる：
  - 🔨 **働き中**（いまセッション実行中）＝腕を高速に振って作業・モニタが緑に光る
  - 📬 **仕事待ち**（受信箱にメールあり）＝デスクに封筒の山（件数分・上限8＋「✉ N」バッジ）、そわそわ跳ねる
  - 💤 **待機中**＝居眠り（頭がこくり・💤が浮かぶ）。次の起床時刻をラベル表示
  - ⚠️ **異常**（直近実行がタイムアウト/エラー）＝体が揺れ、モニタが赤く点滅
  - 🏢 **別拠点**（この環境から中身が見えないリモートエージェント）＝ガラス壁の向こうに半透明シルエット。受信箱の山だけ見える
- 承認待ちキュー（あれば）は専用テーブルに書類の山として表示
- 画面上部HUD：ハートビート最終確認時刻・いま実行中のエージェント名・一覧ビューへのリンク・主要ページへのメニュー
- 5秒ごとに自動更新。`?demo=1` を付けると全状態を疑似表示できるデモモード（デザイン確認用）

## アーキテクチャ（3層・この形を守る）

```
[既存の運用ファイル群]                [読み取り専用API]              [ビュー]
ハートビート/実行ログ/受信箱 ──→ status API（JSON・件数と状態のみ）──→ list.html（一覧）
                                                          └────→ index.html（3D・Three.js）
```

1. **status API**（PHP でも何でも可・読み取り専用）：既存の運用ファイルを読んで JSON を返す。**書き込みは一切しない**
2. **一覧ビュー（Phase 1）**：カード形式のHTML。まずこれで API とデータの正しさを確認する
3. **3Dビュー（Phase 2）**：Three.js。一覧ビューと同じ API を叩く

**必ず Phase 1 → Phase 2 の順で作り、各フェーズで動作検証してから進むこと。**

## Step 0（最初にやる）：この環境のデータ源を棚卸しして仕様を提示

いきなり作らず、まずこの環境で以下に相当するものを調べて、**エージェント一覧表とデータ源の対応表を提示**してください：

| 必要な情報 | 例（VPS版での実体） | この環境では？ |
|---|---|---|
| エージェントの静的台帳（ID・表示名・島・起床時刻） | tick スクリプト内のラベル定義 | 要調査 |
| ハートビート（システム生存確認） | 2分ごとに更新されるテキストファイル | 要調査 |
| 「いま実行中」の判定材料 | tick ログの `[run]`/`[tick]` マーカー | 要調査 |
| 直近エラーの判定材料 | ログの `TIMEOUT`/`ERR<rc>` 文字列 | 要調査 |
| 受信箱の件数 | mailbox `new/*.json` の `to` フィールド集計 | 要調査 |
| 承認待ちの件数 | mailbox `hold/` のファイル数 | 要調査 |

該当が無い項目は「この環境には無い」でよい（その表示要素を省く）。捏造しない。

## status API の仕様（JSON の形）

```json
{
  "ok": true, "ts": "...",
  "tick": { "lastSeen": "...", "ageSec": 42, "mode": "...", "status": "ok",
            "runningNow": "hp-loop:ycom", "stale": false },
  "hold": 2,
  "agents": [
    { "id": "hp-loop-ycom", "label": "解析｜自社YCOM", "island": "analysis",
      "kind": "vps", "status": "working", "inbox": 0, "nextWake": "02:00" }
  ]
}
```

- `status` は `working / inbox / idle / error / remote` の5値
- `stale` ＝ハートビートが5分超停止 かつ 実行中でもない（システム自体の死活）
- ヘッダ：`Content-Type: application/json` / `Cache-Control: no-store` / `X-Content-Type-Options: nosniff`

## 🔒 セキュリティ要件（実際に踏んだ穴。必ず守る）

1. **ログ由来の文字列を信用しない（stored XSS 対策・最重要）**：
   実行ログにはエージェント自身の標準出力が混ざるため、ログから拾ったラベルをそのまま JSON に載せると、
   細工された出力（偽の `[run] label=<payload>` 行）が画面の innerHTML まで届く。
   対策＝①ログ行は**行頭のタイムスタンプ形式でアンカーした正規表現**でだけマッチ ②拾ったラベルは
   **静的なエージェント台帳（コード内定義）と照合し、台帳に無いものは捨てる** ③クライアント側でも
   動的値はすべて `esc()`（HTMLエスケープ）してから innerHTML に入れる（二重防御）
2. **受信箱の宛先も台帳照合**：任意の `to` 値を自動でデスク化・ラベル化しない。未知宛先は `unknownInbox` 件数に集約
3. **APIは件数と状態だけ返す**：メール本文・件名・トークン・秘密情報は絶対に返さない
4. **認証なしで置くのは、ネットワーク境界（Tailscale等）内限定 かつ 返す情報が既に同レベルで見える場合のみ**。
   条件を満たさないなら認証を付けるか公開しない
5. **ファイル走査の上限**：受信箱スキャンは件数上限（例500）・1ファイルサイズ上限（例64KB）・シンボリックリンク除外

## 3Dビューの技術仕様（VPS版で確定した作り方）

- **Three.js はベンダリング**（`vendor/three.module.min.js` に固定バージョンを同梱＋importmap）。CDN 依存にしない
- アバターはプリミティブ（カプセル胴体＋球の頭＋腕）で軽量に。リモートは opacity 0.4 程度の半透明
- **日本語ラベル・絵文字は CanvasTexture のスプライト**で描く（3Dテキストは使わない）
- 島＝色分けした床マット＋デスク群。デスクにはモニタ（状態で emissive 色が変わる：働き中=緑/異常=赤）
- カメラは簡易オービット（ドラッグで回転・ホイールでズーム）。初期値は部屋全体が入る引きの構図にする
- アニメーションは rAF ループ内で `agent.state` により分岐。アバターごとに位相オフセットを持たせ動きをずらす
- **⚠️ メモリリーク対策（実際に踏んだ）**：ポーリングごとにオブジェクトを作り直すと texture/geometry がリークする。
  ①再構築は**値が変わった時だけ**（前回値をキャッシュして比較）②破棄時は traverse して
  geometry/texture/material を dispose（**共有マテリアルは Set で除外**してから）
- ポーリングは「完了してから次を予約」型：`try { await refresh() } finally { setTimeout(loop, 5000) }`。
  fetch には `AbortSignal.timeout(8000)` と `r.ok` チェック
- `?demo=1` パラメータで全状態（働き中・仕事待ち・異常・hold件数）を疑似注入できるデモモードを付ける
- ダッシュボードをサイトトップにする場合は iframe ラッパーにし、**iframe 内から親ページへ戻るリンクには
  `target="_top"`** を付ける（入れ子表示バグを踏んだ）

## 検証（各フェーズで必ず・目視だけで終わらせない）

- API：構文チェック（`php -l` 等）→ CLI/HTTP で JSON を取得し形と値を確認
- ログ解析ロジック：正常行・偽装行（ログポイズニング）・エッジケースの3ケース以上でユニットテスト
- 3D：ヘッドレス Chrome でスクリーンショットを撮って実際に見る
  （WebGL は `--enable-unsafe-swiftshader` が必要な環境がある）。通常表示と `?demo=1` の両方
- 完成したら閲覧URLを報告
