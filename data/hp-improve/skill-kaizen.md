# hp-improve スキル改善ログ（カイゼン台帳）

`/hp-improve` を**使いながら、スキル自体を育てる**ための気づき蓄積ファイル。
運用中に出た「不足・摩擦・観点の漏れ・ツール/データ不足」をここに溜め、
**棚卸し時に社長判断でまとめて**スキル本体（`.claude/commands/hp-improve.md` / `.claude/rules/hp-improve.md`）へ反映する。

## 運用ルール

- **記録**：各運用の末尾（rules の Step 6）で、最低1行でも気づきを追記する（毎回必須・軽く）
- **反映**：都度は反映しない。溜めて棚卸しタイミングで社長が採否判断 → 採用分のみ rules/commands を更新
- **権限非依存**：自分（AI）の権限でできるかは無視し、「あるべき姿」を書く。実現手段（ツール・データ・権限）が要るなら、それも社長への要求として書く
- 反映した項目は `ステータス` を `✅反映済み(YYYY-MM-DD)` に更新。却下は `❌却下` で残す（再提案防止）

## エントリ形式

```
### K-NNN: 一行要約
- 日付 / 種別（観点漏れ｜手順｜ツール・データ不足｜出力フォーマット｜役割分担｜原則）
- 観察：運用中に何が起きたか（事実）
- あるべき姿：本来どうあるべきか（権限を問わず最適形）
- 必要なもの：実現に要るツール・データ・権限・社長への要求
- 反映先候補：rules / commands / 別ツール / 運用手順
- ステータス：⏳未反映 ｜ ✅反映済み(日付) ｜ ❌却下
```

### K-005: HP分析ループ（/hp-loop）でも監査ツール不在で手作業curl/grepに依存
- 2026-06-11 / ツール・データ不足
- 観察：/hp-loop Cycle 001（y-com.info）で、OGP・h1・canonical・viewport 等を `curl + grep + python` で都度抽出した。/hp-improve と同じく K-001 のツール（`bin/hp-audit.sh`）が無く「暫定」手作業。hp-loop は定期実行で回数が増えるため、手作業のままだと毎サイクルのトークン・取りこぼしリスクが /hp-improve より大きい
- あるべき姿：オンページ監査は `bin/hp-audit.sh <url>` のJSON出力を一次情報にし、/hp-loop は「前回JSONとの差分」だけを評価する（変化が無ければ提案を出さない＝定期実行の重複提案を抑止）
- 必要なもの：Tier1 監査ツール（K-001と同じ）＋ サイクル間でJSONを保存する場所（`data/hp-loop/cycles/<date>.json`）
- 反映先候補：別ツール（hp-audit.sh）／rules（hp-loop の Step 2 に「前回との差分評価」を追記）
- ステータス：🟡一部対応(2026-06-11)。`bin/hp-audit.sh`（T-001・`--json` 出力可）を作成し手作業を解消。差分比較（前回JSONとの比較・変化時のみ提案＝K-006）は未実装

### K-006: 定期実行ループは「変化検知 → 変化時のみ提案」の仕組みが要る
- 2026-06-11 / 手順
- 観察：頻度=1時間ごとの指示。HP本体もGSC/GA4も1時間では変わらず、素朴に回すと同じ提案を繰り返す。SSLチェック（cron・固定判定）と違い、HP分析は「前回から何が変わったか」を持たないと定期実行の価値が出ない
- あるべき姿：hp-loop は前サイクルの状態（監査JSON・出した提案・社長の回答）を保持し、(1)サイト/データに変化、(2)社長の新指示、のいずれかがある時だけ新提案を出す。無変化なら「変化なし」とだけ記録して静かに終わる
- 必要なもの：cycles/ への状態保存＋差分ロジック。頻度は本番は日次〜週次が妥当（1時間は動作テスト用）
- 反映先候補：rules（hp-loop の Step 2/4 に差分・無変化スキップを追記）
- ステータス：⏳未反映

### K-007: 改善提案は技術だけでなくコンテンツ・ページ・コピー・運用も含める
- 2026-06-11 / 観点漏れ（社長フィードバック）
- 観察：Cycle 001 が hp-audit.sh の技術信号中心になり、メタ/タグの指摘に寄った。社長より「技術だけでなく、作るべきページ・テキスト内容の変更など、コンテンツにも言及せよ」と指示
- あるべき姿：毎サイクル、技術提案とコンテンツ提案を両方出す。ページ本文を実際に読み、訴求・コピー・情報設計・コンテンツ鮮度（ブログ/お知らせ）・信頼要素（実績の中身/お客様の声）・不足ページを評価。実績の成果数値やお客様の声は事実が要る→捏造せず社長へ要求
- 必要なもの：なし（rules追記）。事実データ（成果/声）は社長提供
- 反映先候補：rules/hp-loop.md（Step 3/4・NG表）／hp-improve.md 側にも同趣旨あり
- ステータス：✅反映済み(2026-06-11)。hp-loop の Step 3/4 と NG表に追記。Cycle 002 でコンテンツ提案 R-005〜009 を実施

### K-008: hp-audit が HTMLコメント内の meta/og タグを「配信中」と誤検出していた

- あるべき姿：解析前に `<!-- -->` を除去し、ブラウザ/クローラが見るのと同じ状態で判定する。コメント内タグ数は別途報告
- 必要なもの：なし（実装済み）
- 反映先候補：別ツール（hp-audit.sh）
- ステータス：✅反映済み(2026-06-11)。コメント除去を実装し `commented_meta_tags` を追加（tools-log T-001）

### K-009: 静的取得の限界とCMS過剰推測（事実を断定しすぎない）
- 2026-06-11 / 観点漏れ・原則（社長フィードバック）
- 観察：(1)viewport が JS で `user-scalable=no` を再注入していたのを静的取得では見抜けず、提案が「静的metaの修正」に留まった。(2)`wp-content` 検出だけで「WordPress」と断定したが、実際は独自PHPテンプレート（WPは一部）。Yoast 前提の提案が構成に不適合だった
- あるべき姿：curl取得はDOM/JS挙動を見ないと明示し、重要な挙動（meta再注入・遅延読込）は「要・実コード/担当エージェント確認」と添える。CMSや技術構成は単一信号で断定せず「推測」と明示し、実装手段は構成に依存する旨を断る
- 必要なもの：なし（rules/出力姿勢）。hp-audit に限界注記は反映済み（tools-log）
- 反映先候補：rules（事実と推測の分離を技術構成の断定にも適用）／運用姿勢
- ステータス：⏳未反映（次の棚卸しで rules への一文追加を検討）

### K-010: 新規指示も新規データも無いサイクルは「展開状況の確認＋停止」に徹する

- 観察：Cycle 004 は from-president.md に新規指示なし・GSC/GA4 も未整備のままで、前回提案(R-010〜012)が引き渡し待ちの状態だった。ここで新規提案を量産すると「過剰提案＋同じ議論の蒸し返し」になりやすい。実際に効いたのは「ライブ再監査で展開状況を事実確認（全未反映）」「works h1=42 等の新規エビデンスだけ追加」「停止条件を明示して回答・引き渡し待ちにする」だった

- 必要なもの：なし（運用規律）。将来は K-006 の「変化検知（前回監査との diff）」ツールがあれば、無変化サイクルを機械的に判定でき空回りを防げる

### K-011: 実データ(GSC)で「総流入」と「事業成果に効く流入」を分けないと判断を誤る
- 2026-06-12 / 観点漏れ・原則（Cycle 005 で T-006 のGSC実データ投入時に判明）
- 観察：GSC実データを入れた途端、Cycle 002〜004 の前提が部分的に崩れた。y-com.info のオーガニッククリックの大半は事業外（地図素材・SSD交換・CSS等のハウツー）で、問い合わせにつながらない。一方「ホームページ制作 料金」等の買い手意図クエリは表示はあるが掲載11〜12位＝2ページ目でクリック0だった。総クリック数（348/4日）を成果指標にしていたら「集客好調」と誤認した可能性
- あるべき姿：SEO提案は最初から「総流入」と「事業関連クエリ（購入意図×地域×業種）の表示・順位・CV」を分けて見る。技術修正も全ページ一律でなく、買い手意図クエリの受け皿ページ（例 /price/）を優先度の根拠にする。データが入ったら過去サイクルの前提（推測ベースの提案）を再検証する規律を持つ
- 必要なもの：GSC実データ＝T-006で充足。次は「事業関連クエリ群」を定義した定点ビュー（gsc-fetch にクエリ分類フィルタ）と、効果検証用のGA4(CV)＝T-003
- 反映先候補：rules/hp-loop.md・hp-improve.md（評価観点に「流入の質＝事業関連/事業外の分離」を追加）／gsc-fetch の分類オプション
- ステータス：⏳未反映（次の棚卸しで評価観点へ追加を検討）

### K-012: 認証付き外部APIツールも「読み取り専用＋課金/秘密の安全装置」で hp-loop の制約内に収められた
- 2026-06-12 / 手順・ツール（T-006 gsc-fetch.py の作り方）
- 観察：BigQuery(GSC)連携は hp-audit の「HTTP GETのみ」より一段重い（認証JSON・課金・外部API）。だが automation.md に沿って「秘密はdata/secrets+.env のパス参照のみ・AIは中身を読まない／SELECTのみ・日付必須フィルタ＋maximum_bytes_billed でスキャン上限／--dry-run で事前見積り／専用venvでシステムを汚さない」とすれば、hp-loop の『作るツールは実行時に読み取り専用』制約を満たせた。実際 dry-run で 0.0009GB と確認してから本実行した
- あるべき姿：外部API連携ツールを作るときの定型＝(1)秘密はgitignore領域にパス参照 (2)読み取り操作に限定 (3)コスト/件数の上限を強制 (4)dry-run/手動検証→本実行 (5)tools-logに限界明記。これを hp-loop のツール作成ガイドに昇格
- 必要なもの：なし（運用知見）。rules/hp-loop.md のツール作成節に「外部API版の安全パターン」を一文追加すると再現性が上がる
- 反映先候補：rules/hp-loop.md（ツール作成の安全パターン）／automation.md との相互参照
- ステータス：⏳未反映（次の棚卸しで検討）
