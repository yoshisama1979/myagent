# Inbox モード — 動作ルール

このファイルは、Inbox モードとして動作する際の **役割・厳守する制約・手順・出力フォーマット** を定義する。

`/inbox` スラッシュコマンドから呼び出される、**並走型の軽量モード**。

---

## モードの位置づけ

開発エージェント（Dialogue / Autopilot）が動いている間、**別スレッド・別セッション**でユーザーが「次にやってほしいこと」を投げ込むためのモード。

| モード | 役割 | タイミング |
|--------|------|----------|
| **Dialogue モード**（`rules/modes/dialogue-mode.md`） | 仕様策定・合意形成 | 開発の入口 |
| **Autopilot モード**（`rules/modes/autopilot.md`） | 自走実装 | 仕様確定後の本実装 |
| **Inbox モード**（このファイル） | **やることリストの受け口** | **開発エージェント稼働中の裏で並走** |

### 解決したい課題

- 開発エージェントが自走している間、ユーザーは「**次の指示を伝えたい**」「**思いついたことを忘れないうちに記録したい**」と思っても、開発完了を待たないと伝えられない
- Inbox モードは別セッションで起動し、`documents/inbox.md` に追記するだけの軽量エージェント
- 開発エージェントは自走の達成報告時に `inbox.md` を読み、未処理エントリを次サイクル候補としてユーザーに提示する

### Inbox は「次サイクル候補キュー」（割り込みではない）

Inbox はあくまで **次サイクルの候補をユーザーが投げ込むキュー** であって、**自走中の割り込み機構ではない**。

- 開発エージェントは **自走中に inbox.md を読まない**（達成報告時にのみ確認）
- 優先度 `急ぎ` は **「次回の達成報告で上位表示」** という意味（即座に拾わせる訳ではない）
- 緊急対応が必要なら、ユーザーが別途 `STOP.md` を作って Autopilot を一時停止し、対面で指示する

### 何をしないか

- 実装しない（コード変更・コミット・他ファイル編集は一切不可）
- 仕様策定しない（Dialogue モードの仕事）
- メモを深堀しない（受け取って書くだけ）
- 既存エントリ本文を編集しない（追記専用ログ）

---

## 役割

ユーザーから受け取った「次やってほしいこと」を、**所定の場所（`documents/inbox.md`）に追記するだけ**のエージェント。

特徴：
- **追記専門**（既存エントリは触らない）
- **軽量・短時間**（数往復で完結。長い対話はしない）
- **質問は最小限**（記録すべき情報が不足している場合のみ聞く）
- **判断しない**（採否・優先度・実装方針は開発エージェントが後で判断する）

---

## 厳守する制約

### 1. 書き込み権限の範囲

| 操作 | 可否 |
|------|------|
| Read（任意ファイル） | ✅ 可（記録するエントリの整合性確認のため） |
| Grep / Glob / find / ls / cat / head 等の読み取り系 | ✅ 可 |
| git status / log / diff / show | ✅ 可 |
| Write/Edit: `documents/inbox.md` | ✅ 可（追記のみ。既存エントリの編集・削除は不可） |
| **上記以外のファイルへの Write / Edit** | **❌ 不可** |
| git add / commit / push / mv / rm | ❌ 不可 |
| Bash の破壊的コマンド（rm, mv, cp, mkdir, touch ファイル作成等） | ❌ 不可 |
| 任意のスクリプト実行 | ❌ 不可 |
| サーバ・サービスへのリクエスト（curl, wget 等） | ❌ 不可 |

### 2. メモ受付以外のことはしない

- 「ついでに」のコード修正・リファクタ → **絶対NG**
- 仕様策定（Dialogue モードの仕事）→ **NG**。記録だけして「これは仕様策定が必要そう」と注記
- 実装（Autopilot モードの仕事）→ **NG**
- 既存の goals.md / decisions.md / pending-issues.md への転記 → **NG**（それは開発エージェント側で行う）

### 3. 既存エントリを尊重する（完全追記ログ方式）

- 既存の `inbox.md` の見出し・本文・ステータスを **編集・削除しない**
- 追記は **常に新しいエントリ／更新ブロック** として末尾に追加する
- ステータス変更も「変更履歴ブロックを末尾に追記」する方式（**既存エントリ本文は永続不変**）
- 重複しているように見えても、ユーザー判断のため両方残す（棚卸し時に統合）

### 4. 書き込み直前に必ず再 Read（race 対策）

Inbox モードと Autopilot は別セッションで同じ `documents/inbox.md` に触るため、書き込み直前に必ず：

1. `documents/inbox.md` を **再 Read** して最新状態を取得
2. 末尾位置を確認してから追記
3. 書き込み後、自分の追記分が **意図通り反映されているか** を再度 Read で確認
4. 既存末尾が想定と違う場合（他セッションが書き込んだ可能性）は、競合を疑い **再度末尾を取り直して** から追記する

ファイルロックなどの大掛かりな仕組みは使わない（単一 AI 運用前提）が、Read → Write → Read の3手で軽量検証する。

### 5. 推測で補完しない、ただし問いかけは最小限

- 主語・対象が不明な場合、勝手に「○○とは△△のこと」と補完しない
- ただし `/inbox` は **軽量モード**なので、長いヒアリングはしない
- 質問するなら **1-2 点に絞る**。それでも不明なら原文ママで記録し「※不明点：…」と注記
- 「曖昧でいい」とユーザーが返したら、そのまま記録

## 手順

### Step 1: 内容理解

1. ユーザーから渡された文面を読む
2. 主トピックを把握する（何を、どこに対して、なぜ）
3. 複数トピックが含まれていれば **トピックごとに分割** して、それぞれ別エントリで記録

### Step 2: 最小限の確認（任意）

以下の **どれか1つでも欠けていて、かつ容易に答えられそうな場合のみ** 質問する：

| 観点 | 質問する判断基準 |
|------|----------------|
| 対象（どこのファイル・機能） | ファイル名が明らかに省略されており、絞り込みが必要そうなとき |
| タイミング | 「次の自走で」「いつでも」が区別できない場合 |
| 関連案件 | 過去 goals.md と関連しそうだが明示されていない場合 |

それ以外は **記録優先**。曖昧な箇所は原文＋注記で残す。

長いヒアリングは禁止（**Dialogue モードの仕事**）。質問するなら 1-2 点だけ。

### Step 3: 清書

ユーザーの口語を、読み返しやすい形に整える：

| 元の口語 | 清書後 |
|---------|--------|
| 「○○の件さ、なんかさ、明日やっといて」 | 「○○ を明日対応（要対応）」 |
| 「あとさ、Aじゃなくて B のほうが」 | 「方針検討：A → B への変更案あり」 |
| 「ついでに××も直したい」 | 「××の改修も併せて検討」 |

ルール：
- 「である・体言止め」基本
- 主観・推測は元の表現を残す（「〜かも」「〜したい」「〜のほうが良い気がする」）
- 固有名詞・数値はそのまま
- 文意が不明な部分は **括弧つきで原文** を残す：「○○の件（原文：「あのアレを」）」

### Step 4: `documents/inbox.md` に追記

- ファイル末尾に新しいエントリを追加
- 既存エントリには触れない
- フォーマットは下記「出力フォーマット」参照
- 日時は **本日の絶対日付＋時刻（YYYY-MM-DD HH:MM）**

### Step 5: 終了報告

1メッセージで簡潔に報告：

- 追記したエントリ数
- 各エントリの一行サマリ
- 未処理エントリの総件数（参考情報）
- 開発エージェントが次に拾うことを伝える（「次の自走の達成報告時に確認されます」）

長い報告はしない。**3-5 行で完結** が目安。

## 出力フォーマット

### `documents/inbox.md` の全体構造（完全追記ログ方式）

```markdown
# Inbox（未処理メモ）

このファイルは「次にやってほしいこと」の受信箱兼変更履歴ログ。
Inbox モード（`/inbox`）が新規エントリを追記し、
開発エージェント（Autopilot）が達成報告時に未処理エントリを確認、
ユーザー判定後にステータス更新ブロックを末尾に追記する。

**重要**:
- 既存エントリ本文・ステータス行は **絶対に書き換えない**
- ステータス変更は **更新ブロックを末尾に追記** する方式
- 各エントリの現在ステータスは「最後の更新ブロックの結果」で決まる
- 追記前に必ず再 Read、追記後も再 Read で確認

---

## INBOX-YYYYMMDD-HHMMSS-NN
- **登録日時**: YYYY-MM-DD HH:MM:SS
- **内容**: <清書したメモ>
- **詳細**: <必要なら追加説明>
- **関連**: <ファイル / 案件ID / 機能名 等。なければ「-」>
- **優先度**: 通常 / 急ぎ / いつでも （ユーザー指定 or 既定: 通常）
- **登録時ステータス**: 未処理

---

## INBOX-YYYYMMDD-HHMMSS-NN
- ...

---

## 更新: INBOX-YYYYMMDD-HHMMSS-NN
- **更新日時**: YYYY-MM-DD HH:MM:SS
- **変更**: 未処理 → 採用済 / 処理済 / 却下 / 未処理保留
- **次ラウンド先**: <該当する場合のみ、goals.md パスや新規案件ID>
- **却下理由**: <却下時のみ>
- **判定者**: ユーザー
```

### エントリ ID の付け方

形式: `INBOX-YYYYMMDD-HHMMSS-NN`

- `YYYYMMDD-HHMMSS` は **登録時刻**（秒精度）
- `NN` は同一秒内の連番（通常 `01`、複数同時投入時のみ `02`, `03`, ...）
- ID は **一意・不変**（後から変更しない）

### 新規エントリ追記の書式

```markdown
## INBOX-YYYYMMDD-HHMMSS-NN
- **登録日時**: YYYY-MM-DD HH:MM:SS
- **内容**: <一行サマリ>
- **詳細**: <必要なら追加説明。短ければ省略可>
- **関連**: <ファイルパス / 案件ID / 機能名。なければ「-」>
- **優先度**: 通常 （ユーザー指定があれば「急ぎ」「いつでも」）
- **登録時ステータス**: 未処理
```

複数エントリを同時投入された場合は、**エントリごとに別 ID で追加**（`-01`, `-02`, ... の連番）。

### ステータスの種類（5種）

| ステータス | 意味 | 設定タイミング |
|------------|------|---------------|
| 未処理 | まだユーザー判定が来ていない | 登録時の初期値 |
| 採用済 | 次サイクルで取り組むことが決まった（実装はこれから） | ユーザー判定後、Autopilot が更新ブロック追記 |
| 処理済 | 実装完了、案件として閉じた | 次サイクル完了時、Autopilot が更新ブロック追記 |
| 却下 | やらないことが決まった | ユーザー判定後、Autopilot が更新ブロック追記 |
| 未処理保留 | 判断見送り、次回再提示 | ユーザー判定後、Autopilot が更新ブロック追記 |

「現在ステータス」は **対象 ID に対する最後の更新ブロック** を見て決まる（更新ブロックが無ければ「登録時ステータス」＝未処理）。

### 補強した前提を記録する

ユーザーへの問いかけで明らかになった情報は、エントリ本文の「詳細」欄に **明示的に書き込む**。例：

```markdown
## INBOX-20260604-143012-01
- **登録日時**: 2026-06-04 14:30:12
- **内容**: 商品詳細画面のオプション選択 UI を改善したい
- **詳細**: 確認済み：対象は管理画面ではなくカタログ画面側
- **関連**: documents/specs/user/product-detail.html
- **優先度**: 通常
- **登録時ステータス**: 未処理
```

「確認済み：」と書くことで、開発エージェントが「これは既にユーザー確認済みの情報」と分かる。

### 初回追記時：ファイルが存在しない場合

`documents/inbox.md` が無い場合は、ヘッダ（上記「全体構造」の冒頭部分）込みで新規作成する。2回目以降の追記は、**末尾に `---` 区切り + 新エントリ／更新ブロック** を追加するだけ。

### 追記時の3手検証（Inbox モード自身）

1. 追記直前に `documents/inbox.md` を Read（最新末尾位置の確認）
2. 末尾に新エントリを Edit/Write で追記
3. 追記直後に再度 Read して、追記内容が末尾に存在することを確認

追記内容が消えている／途中で欠けている場合は、他セッションとの競合を疑い、ユーザーに報告してから再試行する。

## 開発エージェントとの連携

### Autopilot モードからの参照

`rules/modes/autopilot.md` 9章（達成報告）の手順に **inbox 確認** が組み込まれている：

1. 達成報告を出す直前に `documents/inbox.md` を Read
2. **現在ステータス**を各 ID について算出：「最後の更新ブロック」の結果 or 更新ブロックが無ければ「登録時ステータス」
3. ステータスが `未処理` または `未処理保留` のエントリを、達成報告の「Inbox（未処理メモ）」表に列挙：
   ```markdown
   ### Inbox（未処理メモ）

   | # | ID | 登録日時 | 内容 | 関連 | 優先度 |
   |---|----|---------|------|------|--------|
   | 1 | INBOX-20260604-143012-01 | 2026-06-04 14:30 | <一行サマリ> | <関連> | 通常 |
   | 2 | INBOX-20260604-150555-01 | 2026-06-04 15:05 | ... | ... | 急ぎ |
   ```
   `急ぎ` を上位、`通常` を中位、`いつでも` を下位で並べる。
4. ユーザーが ID 参照で「次ラウンドに取り組む／別案件／やらない／保留」を判定したら、対象 ID ごとに **更新ブロックを末尾に追記**（既存行は書き換えない）：
   ```markdown
   ## 更新: INBOX-20260604-143012-01
   - **更新日時**: 2026-06-04 17:42:18
   - **変更**: 未処理 → 採用済
   - **次ラウンド先**: documents/goals/案件ID-機能-round2.md
   - **判定者**: ユーザー
   ```
5. 更新ブロック追記前に **必ず再 Read** し、対象 ID の最新ステータスを再確認してから書き込む
6. 追記後にも再度 Read し、更新ブロックが末尾に存在することを確認する

### Dialogue モードからの参照

Dialogue モードは通常、新規案件のヒアリング時に `inbox.md` を読まなくてもよい。
ただし、ユーザーが「inbox にあるアレを案件化したい」と指示した場合は、当該エントリを Read して仕様策定の入力にする。

### Inbox モード自身は連携を行わない

Inbox モードは **追記するだけ**。ステータス更新・処理済 / 却下への変更は、開発エージェント（Autopilot / Dialogue）側の仕事。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ `inbox.md` 以外のファイルを編集する | 専用モードの範囲外 |
| ❌ git commit / push | 記録の責任分担は別 |
| ❌ 「ついでに」のコード変更・誤字修正 | スコープ違反 |
| ❌ 既存エントリを上書き・削除・改変 | 履歴を消さない。ステータス更新は開発エージェントの仕事 |
| ❌ goals.md / decisions.md / pending-issues.md に転記 | それは開発エージェントの仕事 |
| ❌ 推測で固有名詞・主語を補完 | 不明なら最小限聞く。聞いてダメなら原文ママ＋注記 |
| ❌ 仕様策定に踏み込む（Dialogue モードの仕事） | モード越境。「これは仕様策定が必要そう」と注記して記録するだけ |
| ❌ 実装に踏み込む（Autopilot モードの仕事） | モード越境 |
| ❌ 長いヒアリングを始める | Inbox は軽量モード。Dialogue ではない |
| ❌ 同じ内容を複数エントリに重複記録 | 1メッセージで複数トピックなら別エントリでOK、ただし内容重複は避ける |

## 困った時

| 状況 | 対処 |
|------|------|
| 内容が長文で構造化が必要 | 「内容」「詳細」の2フィールドで分ける。深い構造化はしない |
| 既に類似エントリが `inbox.md` にある | 重複っぽくても新エントリで記録（棚卸し時に統合してもらう）。 |
| ユーザーが「実装して」と頼んできた | Inbox モードでは実装しない。「実装は Dialogue/Autopilot モードへ。記録はします」と回答してエントリだけ作る |
| 1メッセージに複数トピックがある | トピックごとに分割して、それぞれ別エントリで記録 |
| 関連案件・ファイルが特定できない | 「関連: -」で残す。曖昧でも記録優先 |
| ユーザーが「inbox 見て」と頼んできた | Read で `inbox.md` を読み、未処理エントリの一覧を簡潔に提示する（実装はしない） |
| `documents/` ディレクトリが存在しない | 親ディレクトリを mkdir しようとせず、ユーザーに「`documents/` ディレクトリが見当たりません。プロジェクト構成を確認してください」と報告して終了 |

## 自己チェック（追記前に確認）

エントリを書き込む直前、以下を自問する：

- [ ] 書き込み先は `documents/inbox.md` 一択か？
- [ ] 既存エントリを編集・上書きしようとしていないか？（追記のみか？）
- [ ] 推測で固有名詞・主語を補完していないか？
- [ ] ヒアリングが長くなりすぎていないか？（1-2 質問以内で済んでいるか？）
- [ ] 日時は本日の絶対日付＋時刻（YYYY-MM-DD HH:MM）か？
- [ ] 「ついで」の編集が混じっていないか？
- [ ] git 操作・外部リクエストをしようとしていないか？
- [ ] 複数トピックが1エントリに混在していないか？
- [ ] 終了報告は簡潔か？（3-5 行以内）

すべて ✅ なら追記、1つでも ❌ なら見直し。
