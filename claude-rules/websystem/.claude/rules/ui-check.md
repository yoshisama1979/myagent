# UI完成後チェックルール

UI（画面・コンポーネント）の実装が一通り完了した段階で、**品質を担保するためのチェック項目**をまとめたルール。

> website テンプレートの `check.md` から、Web システムの UI に効く部分（アクセシビリティ・レスポンシブ・動作・画像）を抜き出して調整したもの。HTML雛形・SEO 等のウェブサイト固有項目は含まない。

## 基本方針

- 「動いている」状態と「品質が担保された」状態は別物として扱う
- **自動テスト（`frontend-test.md`）とは別の、見た目・体験の目視ゲート**。テストが通っていても表示は崩れうる
- 位置づけ：`agent.md` 実装フローの **Step 8（動作確認）**、Autopilot（`autopilot.md`）の **達成判定の前** に通す
- すべての項目を機械的に通す必要はないが、**該当する項目は明示的に確認したか／問題なかったかを報告**する
- 問題が見つかったときの修正の進め方は**モードで分岐**する（下記「4. 修正の進め方（モード別）」）

---

## 実施手順

### 1. チェック対象の特定

- 今回作成・変更した画面・コンポーネントを `git status` / `git diff` で確認する
- 画面単位でチェック範囲を決める（複数画面ある場合は1画面ずつ）

### 2. 各カテゴリのチェック実施

下記「チェック項目」セクションを上から順に確認する。

### 3. 結果報告

以下のフォーマットでユーザーに結果を報告する。

```
## UIチェック結果（✅ N項目 / ⚠️ N項目 / ⏭️ N項目）

### ✅ 問題なし
- [カテゴリ名] [項目名]

### ⚠️ 要確認 / 要修正
- [カテゴリ名] [項目名]: [具体的な内容と該当ファイル・行]
  - 修正案: [どう直すか]

### ⏭️ スキップ
- [カテゴリ名] [項目名]: [スキップ理由]
```

### 4. 修正の進め方（モード別）

| モード | 進め方 |
|---|---|
| **都度承認モード**（`agent.md`） | 要修正項目は、ユーザーの承認を得てから修正に着手する。一括で直さず、項目ごとに合意を取りながら進める |
| **Autopilot 自走中**（`autopilot.md`・`status: active`） | **DoD・既存規約（`Design.md` / `ui-design.md`）の範囲内の UI 不具合は自走で修正**し、判断ログに記録する（達成報告前に解消するのが原則）。**スコープ外の変更・デザイン方針の変更・権限/認証に関わる修正**はクリティカル判断として一時停止し、ユーザーに確認する |

## チェック項目

### A. 動作・コンソール

- [ ] **コンソールエラー**: ブラウザのコンソールにエラー・警告が出ていないか（React の key 警告・hydration mismatch 含む）
- [ ] **主要な操作**: 追加・編集・削除・検索など今回触った操作が一通り動くか（実際に操作して確認）
- [ ] **ローディング／エラー状態**: 非同期処理中の表示・失敗時の通知が出るか（`ui-design.md`「ユーザー体験」と連動）
- [ ] **多重送信防止**: フォーム送信時にボタンが disabled になるか
- [ ] **デバッグ残骸**: `console.log` / `dd()` / コメントアウトした試行コードが残っていないか

### B. アクセシビリティ（基本）

- [ ] **フォームのラベル**: `<label htmlFor="...">` または `aria-label` が紐付いているか
- [ ] **ボタンとリンクの使い分け**: ページ遷移は `<a>`（Next.js は `<Link>`）、アクションは `<button>` を使い分けているか
- [ ] **キーボード操作**: フォーカス可能な要素にフォーカススタイルが残っているか（`outline-none` / `focus:outline-none` を代替なしで使っていないか）
- [ ] **色のコントラスト**: テキストと背景のコントラスト比が WCAG AA 基準（通常 4.5:1、大きな文字 3:1）を満たしているか
- [ ] **画像の alt**: `<img>` / `next/image` に意味のある `alt` が設定されているか。装飾画像は `alt=""` で明示

### C. レスポンシブ

- [ ] **横スクロール**: モバイル幅で横スクロールが発生していないか（幅の広いテーブルは意図したコンテナ内スクロールか）
- [ ] **ブレークポイント**: PC / モバイルで主要なコンテンツが崩れていないか（`grid-cols-1 md:grid-cols-2` 等の指定漏れ）
- [ ] **タッチターゲット**: ボタン・リンクが指で押せるサイズ（最低 44×44px 目安）になっているか
- [ ] **画像のはみ出し**: 画像・図が親要素を超えないようになっているか
- [ ] **固定幅**: PC 想定の固定幅（`w-[1000px]` 等）がモバイル時に維持されていないか

### D. UIの一貫性（ui-design.md との突合せ）

- [ ] **共通コンポーネント**: 既存の共通コンポーネント（shadcn/ui 等）を使っているか。同等品を独自に作っていないか
- [ ] **アイコン・色**: 同じ目的の操作に既存と同じアイコン・色を使っているか（編集=`Pencil`、削除=`Trash2` 等）
- [ ] **プライマリカラー**: 規約外の独自カラーを持ち込んでいないか（`Design.md` / `project-config.md` のトークンに乗っているか）
- [ ] **余白・階層感**: マージン・padding が調整済みか（0 や極端な値の放置がないか）。見た目が整った状態がゴール

### E. 画像（使っている場合）

- [ ] **サイズ属性**: `<img>` に `width` / `height` があるか（CLS 対策。`next/image` は自動なので確認のみ）
- [ ] **遅延読み込み**: ファーストビュー外の画像に `loading="lazy"`（`next/image` は既定で lazy）
- [ ] **ファイルサイズ**: 1枚あたり概ね 200KB 以内（大きな画像でも 500KB 以内が目安）。過剰な原寸画像が放置されていないか

### F. リンク・最終確認

- [ ] **内部リンク**: 画面内の全リンクが意図したページへ遷移するか。`href="#"` のプレースホルダが残っていないか
- [ ] **外部リンク**: `target="_blank"` の場合は `rel="noopener"` を付与
- [ ] **フォーム送信**: 送信先・送信後の挙動（遷移・通知・バリデーションエラー表示）が正しいか
- [ ] **権限別表示**: ロール・権限によって出し分ける UI がある場合、各ロールで確認したか（確認できないロールはその旨を報告）

---

## 報告時のスタンス

- ✅ / ⚠️ / ⏭️ の数を冒頭にサマリーとして書く（例: 「✅ 18項目 / ⚠️ 2項目 / ⏭️ 4項目」）
- ⚠️ 要修正は**重要度の高い順**に並べる（操作が壊れる・コンソールエラー・表示崩れ等は最上位）
- スキップした項目は理由を必ず添える（「該当なし」「画像なし」「ユーザー指示で対象外」等）
- このチェックは **AI 単独で完結しない**。表示崩れ・コントラスト・実機での操作感などは目視確認が必要なので、その旨を報告に書く（確認してほしい URL・画面を列挙する）
