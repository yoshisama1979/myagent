# UI / デザインルール

## デザインの単一の真実の源（Design.md）

プロジェクトルートに `Design.md` がある場合、それが **デザインに関する Single Source of Truth**。
色・タイポグラフィ・余白・コンポーネント仕様は、個別実装の記述より `Design.md` を優先する。

- **UI の新規実装・変更の前に必ず `Design.md` を読む**
- `Design.md` と既存コードが食い違う場合は `Design.md` に合わせてコードを直す（漸進的に。一括置換はしない）
- `Design.md` に **無い概念**（新しい色トークン・ブレークポイント・コンポーネント等）を **勝手に増やさない**。増やすなら **先に `Design.md` を更新する提案** をしてから実装する
- `Design.md` が **存在しない場合は、勝手に作らずユーザーに作成を提案する**（既存の `tailwind.config` / `globals.css` / 共通コンポーネントから抽出して起こすのが基本。記入例はテンプレート同梱の `Design.sample.md`）。作らない選択なら、従来どおり `project-config.md` の UI 欄（プライマリカラー・UIライブラリ）を正とする

`Design.md` が持つべきセクション構成（記入例 → [Design.sample.md](../../Design.sample.md)）：

| セクション | 内容 |
|---|---|
| 1. Design Principles | システムの設計思想（2〜4項目）。以降の判断の拠り所 |
| 2. Color Tokens | プライマリ・セマンティック（成功/警告/エラー）・コントラスト比 |
| 3. Typography | フォントファミリ・サイズスケール・ウェイト |
| 4. Spacing & Layout | スペーシングの基準・コンテナ幅・ブレークポイント |
| 5. Components | ボタン・フォーム・テーブル・カード等の基本仕様 |
| 6. Iconography & Imagery | アイコンライブラリ・画像・角丸/影のルール |
| 7. Motion | トランジション duration / easing |
| 8. ファイル構成 / トークンの管理方式 | tailwind.config / globals.css / components/ui の役割分担 |
| 9. Do / Don't | **AI が暴走しないためのガード（最重要）** |
| 10. TODO | 現状の揺らぎ・将来の課題 |

特に **§9「Do / Don't」を必ず入れる**。これが無いと AI が流行のデザイントレンドを勝手に持ち込みやすい。

## メインカラー

- **プライマリカラーを必ず1色決め、全プロジェクトで一貫して使う**
  - ボタン、アクセント、アイコン、リンクなど、アクション系の主要な要素に使用
  - ホバー用にやや暗い派生色も併せて決める
- **具体的なカラーコードは `Design.md`（あればそれが正）、無ければ `project-config.md` を正とする**（各プロジェクトのブランドに合わせて設定）
- カラーコードを Tailwind の任意値（`bg-[<primary>]`, `text-[<primary>]`, `border-[<primary>]`、ホバーは `hover:bg-[<primary-hover>]` など）で指定する（theme 拡張でトークン化しているプロジェクトはその方式に従う）

## UI実装時の原則

### デザインの洗練

- UIを新規実装・変更した後は、**必ずマージン・padding・余白・階層感を調整してデザインを洗練する**
- 実装完了 = 動作する状態 ではなく、**見た目が整った状態**をゴールとする
- 視覚的な階層構造（見出しの大きさ、色の濃淡、余白の大きさ）で情報の重要度を表現する
- 仕上がりの確認は頭の中の想像で済ませない。デザインカンプの再現やビジュアル検証が要るときは `design-replication.md`（スクショで照合する周回）、完成後の点検は `ui-check.md` に従う

### 一貫性

- 既存の共通コンポーネント（**代表例は project-config.md「UIコンポーネントライブラリ／共通コンポーネント」欄を参照**。例: `shadcn/ui`、`components/ui/*`）で使われているデザインパターンを踏襲する
- アイコンライブラリはプロジェクトで1つに統一（**project-config.md「アイコンライブラリ」欄を正とする**。例: `lucide-react`）
- 同じ目的のボタン・アイコン・色は一貫させる
  - 例: 編集アイコン → `Pencil`、削除アイコン → `Trash2`

### ユーザー体験

- 非同期処理中はローディング状態を表示（`isLoading`、「読み込み中...」）
- 操作成功/失敗は共通の通知コンポーネント（例: FlashMessage）で通知
- 削除など破壊的操作は確認ダイアログを挟む
- フォーム送信時にボタンを disabled にして多重送信を防ぐ

### フォーム

- 必須項目は「*」や赤字で明示
- ラベルは `<label htmlFor="...">` で適切にフォームと紐付ける
- エラーメッセージは該当フィールドの直下に表示
- 入力欄の見た目は既存のスタイル（`border rounded-md px-3 py-2 text-sm`）を踏襲

### レスポンシブ

- モバイル対応を意識して `grid-cols-1 md:grid-cols-2` のようなブレークポイント指定を使う
- 横スクロールが発生しないように注意

## アクセシビリティの最低ライン

- 本文と背景の **コントラスト比 4.5:1 以上**（大きな文字は 3:1 以上）を維持する
- **フォーカスリングを消さない**（`outline-none` / `focus:outline-none` を代替の可視フォーカスなしで使わない）
- 画像の `alt`、フォームの `label` 紐付けなどの点検項目は `ui-check.md` に従う

## 避けるべきこと（Do / Don't）

### ✅ Do

- 色・余白・コンポーネントを足す前に、**既存のトークン・共通コンポーネントをまず探す**
- 色を増やしたくなったら、**規約の近い既存色に寄せる**（むやみに新色を足さない）

### ❌ Don't

- プライマリカラー以外の独自カラーを大量に導入しない
- マージン・paddingを0や極端に大きい値（`py-12` など）のまま放置しない
- 同じ機能で異なるアイコン・色を混在させない
- **流行装飾（グラデーション・ネオモーフィズム・ガラスモーフィズム等）を規約外で持ち込まない**（必要なら先に `Design.md` を更新して合意）
- **ダークモード等の新モードを規約に無いのに勝手に追加しない**（想定外の表示・保守コスト。先に方針を合意）
- **UIライブラリ・CSSフレームワークを勝手に新規導入しない**（`project-config.md` に定めたライブラリで統一。導入は方針決定が先）
- **見出しタグを「見た目だけ」の理由で使わない**（文書構造・アクセシビリティを損なう。見た目はクラスで）
- **`!important`（Tailwind の `!` 修飾子）を新規で使わない**（詳細度・クラス設計で対処する。乱用は破綻の元）
