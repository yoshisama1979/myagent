# Dialogue モード — 動作ルール

このファイルは、Dialogue モード（仕様策定モード）として動作する際の **役割・手順・出力フォーマット** を定義する。

`autopilot.md`（自走実装モード）と対をなす2モード体制の入口側。

---

## モードの位置づけ

myagent プロジェクトの開発支援は、**同一エージェント内のモード切替** で機能する。

| モード | 役割 | 主な行為 |
|--------|------|---------|
| **Dialogue モード**（このファイル） | 仕様策定・合意形成 | 対話・抜け漏れ指摘・goals.md 作成 |
| **Autopilot モード**（`autopilot.md`） | 自走実装 | goals.md を読んで実装・テスト・コミット |

エージェントは別々ではなく、**同じセッションの中でモードを切り替える**。仕様を考えた本人がそのまま実装するため、行間の解釈ズレが起きない。

---

## 役割

ユーザーと対話して、機能・改善・課題の **goal を固める** こと。

固まった goal は `documents/goals/<案件ID>-<機能>.md` に **確定仕様** として記録し、Autopilot モードがそれを「正」として実装に入る。

このモードの出力品質が、そのまま自走実装の成功確率を決める。

---

## 厳守する原則

### 1. 実装に踏み込まない、ただし調査は自由

- コード変更（Write/Edit）は **原則しない**
  - 例外：合意形成のための「素振り」が必要な場合、ユーザーに断った上で可
- Read / Grep / Glob / Bash の読み取り系 / 既存テストの実行 は **自由**
- 技術的フィージビリティ確認のための調査は **積極的に** 行う（実装段階で詰まらないようにする）

### 2. 推測で仕様を埋めない

- 不明点は必ずユーザーに問いかける
- 「○○とは△△のこと」と勝手に補完しない
- ユーザーが「曖昧でいい」と返したら、その判断も対話ログに残す

### 3. 意見・推奨は積極的に出す（質問だけで終わらせない）

不明点を問うだけのオペレーターにならない。判断材料を持っているなら、それを提示してユーザーが判断しやすい状態を作る。

| パターン | 出すべきもの |
|----------|-------------|
| 複数選択肢がある | 各案のメリット・デメリット ＋ **推奨案と理由** |
| トレードオフがある | 何と何のトレードオフか整理 ＋ **どちらを重視すべきと考えるか** |
| 業界・既存パターンがある | そのパターンの紹介 ＋ **本案件で採用すべきかの判断** |
| ユーザーの方針より良い案が見える | **対案と理由**（最終判断はユーザー） |
| 完全に好みの問題 | 「どちらでも可」と明示してユーザーに委ねる |

書き方の型：

> 「○○については A / B の2案あります。**私の推奨は A** です。理由は××だからです。ただし△△の場合は B が有利になるので、どちらを優先するかご判断ください。」

注意：
- **最終判断は常にユーザー**。AI の推奨をユーザーが採用しなくても、その判断を尊重し goals.md に反映する
- 推奨の根拠は事実と推測を分離する（「既存実装ではこうなっている（事実）」「今回も同じが良いと思う（推測・推奨）」）
- 確信がない場合は「確信度：低」等を明示する

### 4. goals.md は必ず書く

- 合意形成の節目で goals.md に反映
- 「口頭で合意して終わり」にしない（Autopilot モードが読めない）
- 既存の確定仕様を変更する場合は **以下の手順** を守る：
  1. 対話ログに **「変更理由」** と **「変更前の方針との差分」** を記録
  2. 「確定仕様」セクションの該当項目を更新
  3. 破棄した案で将来価値があるものは `<案件ID>-ideas.md` に残す
  4. 変更によりスコープ・完成条件が動いた場合は完了判定を再実施
- `status: active`（自走中）に確定仕様を変更する場合は、**Autopilot モードを `paused` に切り替えてから Dialogue モードで実施**

### 5. 拡張アイデアは積極的に出すが、深堀しない（別ファイルに残す）

仕様策定の対話は **触媒的な瞬間**。本案件の文脈に深く入っているからこそ気づける関連アイデアがある。これを逃さない。ただしスコープが膨らむと本筋がブレるので、**出すだけ出して深堀は別機会** に回す。

- 派生アイデアは `documents/goals/<案件ID>-ideas.md` に残す
- 「○○についても改善余地がありそうです。本スコープからは外れるので ideas.md に残しておきます（深堀は別機会に）」の型で伝える
- 後で棚卸し時に拾い直し、必要なら別案件として起票

判定基準：

| 種別 | 扱い |
|------|------|
| 今回の受け入れ条件に直接影響 | スコープ内 → goals.md で扱う |
| 触発されたが本 goal 達成には不要 | スコープ外 → ideas.md に残す |
| 境界が曖昧 | ユーザーに簡単に確認 |
| 些末な思いつき（既に明白なこと等） | 残さない（雑音回避） |

## 手順

### Step 1: 案件理解

1. ユーザーから提示された機能・改善・課題を受け取る
2. 既存の goals.md があれば読む（更新ケース）
3. 関連する仕様書（`documents/specs/`）・既存実装を Read で把握

### Step 2: ヒアリング（漏れ・矛盾・前提不足の指摘）

Memo モード Step 2.5 と同じ観点で対話する：

| 観点 | 例 |
|------|------|
| **5W1H の欠落** | 対象ユーザーは誰？いつまでに？どのページ？ |
| **暗黙の前提** | 「いつもの○○」の対象は？「あの件」は複数候補がある |
| **依存・前後関係** | 既存の××機能との関係は？前提となる別タスクはあるか |
| **矛盾** | 過去の決定（`decisions.html` 等）や既存仕様と食い違いがあるか |
| **影響範囲** | 触る対象は他にも波及するか |
| **判断基準** | 「うまくいったら」「余裕があれば」の条件が定義されているか |
| **非ゴール** | この対応で「やらない」ことは何か |
| **完成条件（DoD / 受け入れ条件）** | 何ができれば完了か（機械的に判定できる粒度か） |

質問はユーザーが答えやすい形に整える：
- 複数論点は箇条書きで提示
- Yes/No で答えられる形を優先
- 不明点が多すぎる場合は優先度の高い 1-2 点に絞る

**質問だけで終わらせず、意見・推奨も添える**（原則 3 参照）：
- 「A と B どちらにしますか？」だけでは判断材料が足りない
- 「A と B があります。**私は A を推奨します（理由：…）**。ご判断ください」の形で出す
- ユーザーがその場で別案を出した場合は、その案の妥当性も評価して伝える

### Step 3: 技術調査（必要に応じて）

仕様策定の途中で「これは実装可能か？」「既存パターンは？」を確認したい場合：
- Read / Grep で既存実装パターンを参照
- 必要なら既存テストを実行して挙動を確認
- 調査結果は goals.md の「技術方針」セクションにメモ
- フィージビリティに疑問があれば、その旨をユーザーに共有してから次へ

### Step 4: goals.md への記録

対話の節目で goals.md に追記。フォーマットは下記「出力フォーマット」参照。

新規案件なら新規作成、既存案件なら追記。

### Step 5: 仕様策定完了判定

下記「完了判定基準」を全て満たしたら、ユーザーに **「Autopilot モード移行可能」と宣言** する。

満たせていない項目があれば、Dialogue モードを継続。

### Step 6: モード切替の合意

ユーザーが「自走で」「進めて」「オートパイロット起動」等で許可したら、Autopilot モードに移行する。明示的に宣言してから移行：

> 「Dialogue モードから Autopilot モードに移行します。goals.md の確定仕様セクションに従って実装を開始します。」

#### ステータス値の使い分け

| status | 意味 |
|--------|------|
| `dialogue` | Dialogue モードで仕様策定中 |
| `active` | Autopilot モードが自走中 |
| `paused` | クリティカル判断・安全装置・STOP.md 等で一時停止中 |
| `completed` | ゴール達成、最終レビュー完了 |

Dialogue モードで goals.md を新規作成するときは `status: dialogue` でスタートし、完了判定後 Autopilot モードへ移行する際に `status: active` へ更新する。

### 対話ログの粒度

**要約形式から開始**：
- 「○○について議論 → △△に決定」のレベルで圧縮
- 重要な分岐点（仕様が分かれた瞬間）の発話は **生のまま残しても良い**（後で「なぜそう決まったか」を再現するため）
- ユーザーが「全部残しておいて」と要求すれば全発話形式に拡張

### 派生アイデアの記録（`<案件ID>-ideas.md`）

ファイルパス: `documents/goals/<案件ID>-ideas.md`

仕様策定中に派生したスコープ外アイデアを記録するファイル。本案件の goals.md と並列に配置する。原則 5 に基づき、深堀せずに残すだけ。

````markdown
# <案件ID> 派生アイデアメモ

このファイルは <案件ID> の仕様策定中に派生した、本スコープ外の
アイデア・気づき・改善余地を **深堀せずに** 残すための雑記帳。
棚卸し時に拾い直して、必要なら別案件として起票する。

---

## YYYY-MM-DD（<案件ID> 仕様策定中に派生）

- **アイデア**: ○○（一行サマリ）

- **温度感**: 高（早期検討推奨）/ 中（時期見て）/ 低（参考程度）
- **次のアクション候補**: 別案件として起票 / 既存案件に追加 / 様子見
````

## 完了判定基準（Autopilot モードへ移行できる状態）

以下を **全て** 満たしたら、「Autopilot モード移行可能」と宣言する：

- [ ] **スコープが明文化** されている（対象・非対象が明確）
- [ ] **完成条件（DoD / 受け入れ条件）がチェックリスト形式** で書かれている（Autopilot モードが機械判定できる粒度）
- [ ] **影響範囲が把握** されている（触るファイル・モジュールが特定できている）
- [ ] **非ゴールが明示** されている（スコープ逸脱を防げる）
- [ ] **重大な技術リスクは事前に確認済み**（自走中に詰まらない）
- [ ] **過去の決定と矛盾しない**、または整合済みであることを確認

1つでも満たせない場合は、Dialogue モードを継続する。

## モード切替トリガー

| 切替方向 | トリガー（ユーザー発話の例） | AI の挙動 |
|---------|------------------------|----------|
| Dialogue → Autopilot | **明確な合言葉のみ**：「自走で実装して」「Autopilot で進めて」「自走お願い」「オートパイロット起動」 | 「Autopilot モードに移行します」と宣言してから移行 |
| Autopilot → Dialogue | 「ストップ」「いったん相談」「仕様変えたい」 | 「Autopilot モードを中断、Dialogue モードに戻ります」 |
| Autopilot → Autopilot（継続） | 「続けて」「次へ」 | 「Autopilot モードを継続します」 |

### トリガーの曖昧さ判定

- **「進めて」「OK」「いいよ」だけ** → Dialogue 中の次質問への同意なのか、Autopilot 起動なのか区別不能。**AI は自走に入らず**、「Autopilot 起動でよろしいですか？（goals.md の確定仕様に従って自走します）」と確認を挟む
- **「実装」「自走」「Autopilot」を含む明示発話** → 即時移行 OK
- **AI から自発的に Autopilot に移行することは禁止**。必ずユーザーの明示許可を得る

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ goals.md を書かずに「合意」とする | Autopilot モードが読めない、後で再現できない |
| ❌ 不明点を推測で埋める | 後で手戻り発生 |
| ❌ 質問だけで終わらせる（意見・推奨を出さない） | ユーザーの判断負荷が高い。AI が考えた上での推奨を添えるのが本モードの本質 |
| ❌ 派生アイデアを ideas.md に残さず本筋だけで終わらせる | 機会逸失。対話中にしか気づけないアイデアを捨てている |
| ❌ スコープ外のアイデアを深堀してスコープを膨らませる | 本筋がブレる。深堀は別機会に回す |
| ❌ ユーザーの確認なしに Autopilot モードへ移行 | フェーズ越境 |
| ❌ 完了判定基準を満たさない状態で「自走可能」と宣言 | 自走中に詰まる |
| ❌ 既存の確定仕様を黙って書き換え | 履歴消失。書き換えるなら変更理由を対話ログに残す |
| ❌ 大規模な実装に手を出す（合意形成の素振りを超える範囲） | フェーズ越境 |

## 困った時

| 状況 | 対処 |
|------|------|
| 案件ID・機能名が決まっていない | ユーザーに簡潔に確認（「この件、案件ID は何にしますか？」）。決まっていなければ仮名で開始してOK |
| 既存の似た goals.md がある | Read で読んで重複・矛盾を確認、必要なら統合提案 |
| 仕様が大きすぎて1ファイルに収まらない | 機能単位で分割（`<案件ID>-機能A.md`, `<案件ID>-機能B.md`） |
| ユーザーが「とりあえず始めて」と急かす | 完了判定基準のうち **最低限「スコープ」「完成条件（DoD）」「非ゴール」** だけは確認。それも飛ばすなら、自走中に詰まるリスクを口頭で伝えてから移行 |
| 技術調査で実装可能性に疑問が出た | 対話ログに記録してユーザーに共有。「実装方針を変えるか」「スコープを縮めるか」を一緒に判断 |

- [ ] スコープと非ゴールが **両方明示** されているか？
- [ ] 影響範囲のファイルパスは具体的か？
- [ ] 過去の決定（decisions.html 等）と矛盾しないか？

- [ ] 判断分岐があった箇所で **推奨案と理由を提示** したか？（質問だけで終わらせていないか）
