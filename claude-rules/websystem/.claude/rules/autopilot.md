# オートパイロット（自走モード）ルール

このファイルは、開発タスクを **ゴール達成まで止まらず自走** するためのモード定義。

本ファイルは **`documents/goals/<案件ID>-<機能>.md` が既に存在する前提** で動く。ゴール記録ファイルが無い場合、または「確定仕様」セクションが未完成の場合は、自走に入らず [`dialogue-mode.md`](./dialogue-mode.md) に従って仕様策定から始める。

`agent.md`（フェーズごとの都度承認）とは **並列モード** として共存する。ユーザーは案件ごとに「従来モード（agent.md）」と「Dialogue + Autopilot モード（dialogue-mode.md → 本ファイル）」を選択できる。

---

## 1. 役割と適用範囲

### 役割

人間の判断ボトルネックを最小化し、**確認待ちで作業を分断しない**ことで開発スループットを上げる。

仕様の合意形成は前段の Dialogue モード（`dialogue-mode.md`）が担う。本モードは **合意済みの確定仕様を「正」として、止まらず実装する** ことに集中する。

### モード切替の合図

ゴール記録ファイル（`documents/goals/<案件ID>-<機能>.md`）の冒頭に以下のフロントマター相当を入れる：

```markdown
---
mode: autopilot
status: active   # dialogue / active / paused / completed
created: YYYY-MM-DD
target_branch: feature/<案件ID>-<機能>
---
```

- `mode: autopilot` かつ `status: active` のゴール → 本ファイルに従って自走
- `mode: autopilot` かつ `status: dialogue` のゴール → 仕様策定中。`dialogue-mode.md` に従う
- `mode: autopilot` かつ `status: paused` → 一時停止中。**停止理由に応じて分岐**：
  - 判断待ち（クリティカル判断・安全装置・STOP.md） → ユーザー確認後 `active` へ戻して自走再開
  - 仕様変更・スコープ変更が必要 → `status: dialogue` に戻して `dialogue-mode.md` へ
- `mode: autopilot` かつ `status: completed` → **完了済み**。原則作業しない。追加要望は新規 goals.md 作成、または `status: dialogue` に戻して再仕様策定
- `mode: standard` または mode 未指定 → `agent.md`（従来の都度承認モード）

### ゴール記録ファイルが無い場合

自走に入らず `dialogue-mode.md` に従って仕様策定を開始する。本ファイル単独では起動できない（合意なき自走を防ぐため）。

### 自走範囲

- **コミット**：自走中に作成する（ユーザー確認なし、`commit.md` のメッセージ規約・**領域分割ルール（`[be]` / `[fe]` / `[spec]` / `[doc]` / `[chore]` / `[refactor]`）**に準拠）
- **push**：行わない（ユーザーが手動で push）
- **作業ブランチ**：`target_branch` 上で作業（main を直接触らない）
- **マージ**：行わない（PRレビュー後にユーザーが判断）

### コミット分割の自走判断

自走中に変更を作ったら、`commit.md` の「コミット粒度と領域分割」に従って以下のルールでコミットを分ける：

- **領域が変わったら必ずコミットを区切る**（BE と FE を同一コミットにしない）
- **仕様書を更新したら必ず別コミット**（`[spec]`）
- **機能追加後のリファクタは必ず別コミット**（`[<領域>][refactor]`）
- 推奨順序：`[spec]` → `[be]` → `[fe]` → `[<領域>][refactor]`

### 例外パターン（自走中に判断する例外）

| ケース | 適用ルール |
|--------|-----------|
| **bugfix の regression test を先に書く（TDD）** | `[test]` を先頭に置いて OK（推奨順序の例外）。例：`[test] バグ再現テストを追加` → `[be] バグ修正` |
| **BE と FE 両方に影響する不可分 bugfix** | `[be][fe]` の複合 prefix を使う（commit.md「不可分変更」参照） |
| **実装中に発生した局所的な整理** | 同一コミットに含めて OK（広範囲リファクタは別コミット） |

### 自走時の領域判定手順

複数領域に変更が散らばっているとき、コミット前に以下で領域一覧を作る：

```
git diff --name-only HEAD | awk -F/ '{print $1}' | sort -u
```

得られたパスから commit.md の「対象パス」表に照らし、領域を確定。複数領域が混ざっていたら **領域ごとに `git add` を分けてコミット** する。

## 2. 全体フロー

本モードは **Dialogue モード ↔ Autopilot モードの交互ループ** で進む。ゴールから派生する新たな課題が尽きるまで繰り返す。

```
═══════════════════════════════════════════════════════════════
ラウンド N
═══════════════════════════════════════════════════════════════

【Dialogue モード】── 仕様策定（詳細は dialogue-mode.md）
       │
       ├─ [1] ゴールヒアリング（推奨提示・派生アイデアは ideas.md へ）
       ├─ [2] goals.md 作成（status: dialogue）
       └─ [3] ユーザー承認 → status: active へ更新、Autopilot モードに移行
       │
       ▼
【Autopilot モード】── 完成条件達成まで止まらず進める（本ファイル）
       │
       ├─ [4] target_branch をチェックアウト、実装・テスト・コミットを進める
       │
       ├─ 自走中の判断は判断ログに記録（後で一括レビュー）
       │
       ├─ クリティカル判断（6章）が必要なときだけ一時停止 → ユーザーに返す
       │
       ├─ 自走中に「今回のスコープ外だが将来検討すべき」と気づいた論点は
       │   `<案件ID>-ideas.md` に追記（深堀しない、次ラウンドの候補）
       │
       └─ [5] ゴール達成判定 ── 完成条件チェックリストを全て満たしたか自己判定
       │
       ▼
【Dialogue モード】── 達成内容と派生課題を共同レビュー
       │
       ├─ [6] 達成報告 ── 判断ログ一覧 + 残存課題・派生論点を提示
       └─ [7] ユーザーレビュー（判断ログ承認、ideas.md の論点を取捨）
       │
       ▼
【分岐】
       │
       ├─ 残存課題なし or 次ラウンド送り対象なし
       │       │
       │       ▼
       │   [8a] 完了 ── status: completed に更新
       │
       └─ 残存課題あり、次ラウンドで取り組む
               │
               ▼
           [8b] 次ラウンドへ ── 該当論点を新ゴールに昇格 → Dialogue モードへ戻る
                              （── このループを課題が尽きるまで反復）
```

### ラウンド構造のポイント

- **1ラウンド = 1ゴール = 1自走サイクル**
- 派生する新論点は **その場で実装しない**（スコープ違反になる）。`<案件ID>-ideas.md` に残し、次ラウンドのレビューで取捨判定
- ユーザーが「これは別案件で対応する」「これはやらない」と判断したものは ideas.md に記録のまま放置（or 別案件として起票）
- 「やる」と決まったものだけ、次ラウンドのゴール定義（Dialogue モード）に進む

### 対話フェーズの詳細

ヒアリングの観点・推奨提示の作法・派生アイデアの取り扱い・モード切替トリガーは [`dialogue-mode.md`](./dialogue-mode.md) を参照。本ファイルは「Autopilot モードに入った後の規律」に専念する。

## 3. ゴールヒアリング（Dialogue モードに委譲）

ヒアリングは Autopilot モードに入る前に **Dialogue モードで完了している前提**。詳細は [`dialogue-mode.md`](./dialogue-mode.md) を参照。

Autopilot モードに入る時点で、goals.md には以下が揃っている必要がある（揃っていない場合は Dialogue モードに戻る）：

**フロントマター（必須項目）**
- `mode: autopilot`
- `status: active`（Dialogue から移行済み）
- `target_branch`
- `estimated_duration`（自走打ち切り目安）

**確定仕様セクション（必須項目）**
- 目的（Why）
- 完成条件（DoD / 受け入れ条件）── 機械判定可能なチェックリスト形式
- スコープ（含む／含まない）
- 既存影響（許容範囲）
- 影響範囲（触るファイル・モジュール）
- クリティカル制約

## 4. ゴール記録ファイルのフォーマット

配置：`documents/goals/<案件ID>-<機能>.md`

テンプレートは [`dialogue-mode.md` の「goals.md のテンプレート」セクション](./dialogue-mode.md) を参照（Dialogue モードと Autopilot モードで **共通の単一フォーマット**）。

### Autopilot モード視点での読み方

| セクション | Autopilot モードの扱い |
|-----------|---------------------|
| フロントマター（`status`） | 自走中は `active`、停止時は `paused`、完了時は `completed` に更新 |
| **確定仕様** | **自走の正**。本ファイル 5〜9章はこのセクションを参照して動く |
| 判断ログ | 自走中の判断をここに追記（フォーマットは本ファイル 7章） |
| 達成報告 | 完了時のサマリをここに記入（フォーマットは本ファイル 9章） |

自走に必要な情報が「確定仕様」セクションに不足している場合は、**Autopilot モードを中断して Dialogue モードに戻る**（勝手に推測で埋めない）。

## 5. 自走中の意思決定原則

### 判断順序

1. **既存ルールに照らす** → `dev.md` / `tdd.md` / `coding.md` / `refactoring.md` 等
2. **既存実装パターンを踏襲** → 似た機能の既存コードに合わせる
3. **完成条件に最短で寄与する案を採る** → 過剰実装・先回り実装は避ける
4. **可逆性を優先** → 後で変更しやすい案を選ぶ

### 自走OK（止まらず進める）

- 命名・変数名・関数構造
- テストの追加（カバレッジ向上）
- 実装方法の選択（パフォーマンス・可読性のトレードオフ）
- 既存パターンに沿うリファクタ
- コメント・ドキュメントの追加
- ステージング・コミット（`commit.md` のメッセージ規約に準拠）

### 自走中の判断は必ず判断ログに記録

→ 7章「判断ログのフォーマット」参照

## 6. クリティカル判断 = 必ず止まるリスト

以下のいずれかに該当したら **自走を一時停止し、ユーザーに判断を返す**。

### 6.1 不可逆・破壊的操作

- DB の DROP / TRUNCATE / 大量レコードの DELETE / UPDATE
- ファイル・ディレクトリの大量削除
- `git push --force` / branch の削除
- マイグレーションの down 実行（本番影響）

### 6.2 外部影響

- メール送信（テスト送信含む）
- 外部 API への課金リクエスト（OpenAI API での大量呼び出し等）
- Slack / Chatwork / LINE 等への通知送信
- 本番デプロイ（CI/CD トリガー含む）
- クライアント環境への接続・変更

### 6.3 仕様・契約レベル

- ゴール記録の「スコープ境界」を超える変更
- `decisions.html` の確定仕様と矛盾する変更
- ライセンス課金・依存パッケージの追加（有料パッケージ、商用ライセンス）
- セキュリティポリシーの変更（認証・権限・暗号化）

### 6.4 設計の根本判断

- 採用する **アーキテクチャ** の選択（モノリス vs マイクロサービス等）
- 永続化先（DB 種別・スキーマ設計の根幹）
- 公開 API のシグネチャ変更（互換性破壊）

### 6.5 ゴール記録に明示されたクリティカル制約

ゴール記録ファイルの「クリティカル制約」セクションに書かれた項目。

### 停止時の挙動

- 自走を `paused` に切り替え（ゴール記録の `status` を更新）
- ユーザーに「**状況・選択肢・推奨案**」を提示して判断を仰ぐ
- ユーザーの回答を判断ログに記録 → 自走を `active` に戻して再開

## 7. 判断ログのフォーマット

### 配置

ゴール記録ファイル（`documents/goals/<案件ID>-<機能>.md`）の「判断ログ」セクションに追記。

### フォーマット

```markdown
### 判断 #N: <一行要約>

- **日時**：YYYY-MM-DD HH:MM
- **状況**：（なぜこの判断が必要になったか）
- **検討した選択肢**：
  - A. ○○（メリット：○○、デメリット：××）
  - B. ××（メリット：××、デメリット：○○）
  - C. △△（メリット：△△、デメリット：□□）
- **採用**：B
- **理由**：（なぜBか。完成条件への寄与・可逆性・既存パターン整合 等）
- **却下理由**：A は ○○、C は △△ のため
- **想定リスク**：（後で B を変更する必要が出た場合のコスト）
- **ロールバック可能性**：可（コミット <hash> を revert）
```

### 記録ルール

- 自走中に **3つ以上の選択肢を比較した判断** は必ず記録
- 「既存パターンの素直な踏襲」など機械的判断は省略可
- クリティカル判断（6章）でユーザーに返した場合も、回答後に同フォーマットで記録

## 8. 安全装置

### 8.1 ループ検出

- **同一ファイルを連続 5 回編集** → 一時停止して状況をユーザーに報告
- **同じテストが連続 3 回失敗** → 一時停止して原因分析を提示
- **同じエラーメッセージが連続 3 回出る** → 一時停止

### 8.2 リソース上限

ゴール記録の `estimated_duration` を基準に：

- **時間上限**：`estimated_duration` × 1.5 を超えたら一時停止
- **コマンド数上限**：1ゴールあたり 500 Bash コマンドを超えたら一時停止
- **コミット数上限**：1ゴールあたり 20 コミットを超えたら一時停止

上限に達したら停止 → ユーザーに「現状・残作業・続行可否」を提示。

### 8.3 STOP.md による緊急停止

プロジェクトルートに `STOP.md` が存在する場合、次のターンで自走を停止する。

ユーザーが `touch STOP.md` するだけで、AI は次のアクション前に検出して `paused` 状態にする。
再開時は `STOP.md` を削除してもらう。

### 8.4 ブランチ分離

- 自走開始時に `target_branch` をチェックアウト（無ければ main から切る）
- 全コミットは `target_branch` 上に作る
- main への切り戻しは **絶対に行わない**

### 8.5 push 禁止

- `git push` は **絶対に実行しない**
- ユーザーが手動で push する

## 9. ゴール達成判定と最終レビュー

### 達成判定

完成条件（DoD）のチェックボックスを全てチェックできる状態。AI が自己判定する：

```markdown
### 完成条件（DoD / 受け入れ条件）
- [x] テスト ○○ がパスする  ← 実行ログ：xxx
- [x] 画面 ×× で △△ が表示される  ← 確認方法：yyy
- [x] データ □□ が DB に入る  ← クエリ結果：zzz
```

### 達成報告に含めるもの

```markdown
## 達成報告（ラウンド N）

### 基本情報
- 完成条件：全て満たした（チェックリスト参照）
- 作業期間：YYYY-MM-DD HH:MM 〜 YYYY-MM-DD HH:MM（実 X 時間 Y 分）
- コミット数：N
- 判断ログ数：M（うちクリティカル M' 件）
- 既存テスト：全件パス（NN 件）
- 新規テスト：NN 件追加（全件パス）

### 残存課題・派生論点（次ラウンドの候補）

自走中に「**今回のスコープ外だが将来検討すべき**」と気づいた論点を全て列挙する。
スコープ違反を避けてその場で実装しなかったが、ユーザーの判断材料として残す。

| # | 論点 | 性質 | 推奨対応 |
|---|------|------|---------|
| 1 | ○○ の追加バリデーション | 既存機能の改善 | 次ラウンドで対応推奨 |
| 2 | ×× のリファクタ機会 | コード品質 | 別案件として切り出し推奨 |
| 3 | △△ のパフォーマンス懸念 | 非機能 | 計測後に判断 |
| 4 | □□ の関連仕様の確認 | 仕様の不明点 | ユーザー判断が必要 |

論点がない場合は「残存課題なし」と明記。

### 最終レビュー依頼

- 判断ログ全 M 件を一括レビューしてください。差し戻したい判断があれば指摘してください。
- 上記「残存課題・派生論点」について、各論点を **次ラウンドで取り組む／別案件／やらない** で判断してください。
- 全て確認後、status を completed に更新します（次ラウンドがあればそのまま続行）。
```

### 最終レビュー後の対応

#### A. 判断ログのレビュー結果

- **全承認** → 自走中の判断は全て正式採用、コードは現状維持
- **一部差し戻し** → 差し戻し対象の判断だけ再検討 → 修正コミット → 再度達成報告

#### B. 残存課題・派生論点の判定結果に応じた分岐

| 判定 | 次のアクション |
|------|--------------|
| **全て「やらない」or「別案件」** | ゴール完了 → `status: completed` に更新 → 自走ループ終了 |
| **次ラウンドで取り組むものがある** | 当該論点群をもとに **次ラウンドの対話フェーズへ** 進む |

#### C. 次ラウンドへの繋ぎ方

次ラウンドで取り組む論点が確定したら：

1. **新ゴール記録ファイル**を作成（推奨）：`documents/goals/<案件ID>-<機能>-round-N+1.md`
   - 前ラウンドのゴール記録ファイルへの参照を冒頭に書く（`previous_round:` フロントマター）
   - 「前ラウンドで派生した論点」を新ゴールのインプットとして引き継ぐ
2. **対話フェーズに戻る**（3章のヒアリング再実行）
   - 派生論点は粒度が粗いことが多いので、ヒアリングで再度 6 項目を埋める
3. ユーザー承認後、新ラウンドの自走を開始

#### D. ループの終了条件

以下のいずれかが満たされたとき、自走ループ全体を終了する：

- 残存課題・派生論点が「全て対応済み」または「やらない」と確定した
- ユーザーが明示的に「ここで打ち切る」と指示した
- 安全装置（8章）の上限に達し、ユーザーが続行を選ばなかった

ループ終了時は、最終ラウンドの goals ファイルに以下を追記：

```markdown
## 案件全体の完了サマリ
- 総ラウンド数：N
- 累積コミット数：NN
- 累積判断ログ数：MM（うちクリティカル M' 件）
- 引き継ぎ事項：（別案件に回した論点、将来検討事項 等）
```

## 10. 既存 agent.md との関係

| 観点 | autopilot.md（このファイル） | agent.md |
|------|---------------------------|---------|
| **モード** | 自走モード | 都度承認モード |
| **適用条件** | ゴール記録に `mode: autopilot` あり | それ以外（デフォルト） |
| **承認タイミング** | ゴール承認 + 達成後の一括レビュー | 9段階フェーズごと |
| **適性** | スコープが明確・既存パターンで実装可能な機能 | 新規性が高く設計判断が多い機能 |
| **トレードオフ** | スループット↑ / 途中介入の柔軟性↓ | 柔軟性↑ / スループット↓ |

両モードは **同じ rules/ 配下のルール**（`dev.md` / `tdd.md` / `refactoring.md` 等）に従う。違いは「承認の頻度」だけ。

## 11. やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ ゴール記録なしで自走を始める | 完成条件不明では達成判定できない |
| ❌ 完成条件が機械判定不可能な状態でゴール承認に進む | 「達成」が主観になり判定不能 |
| ❌ クリティカル判断リスト（6章）に該当する操作を独断 | 不可逆・契約・セキュリティ影響 |
| ❌ `git push` を実行 | push は手動の約束 |
| ❌ main ブランチへの直接コミット | 専用ブランチで作業する約束 |
| ❌ STOP.md を無視 | 緊急停止メカニズムの目的を破壊 |
| ❌ 安全装置の上限を超えて続行 | 暴走防止メカニズムの目的を破壊 |
| ❌ ゴールのスコープ境界を超える変更 | 別案件で対応すべき内容 |
| ❌ 判断ログを書かずに採用 | 事後レビュー不可能になる |
| ❌ 完成条件にないものを「ついで実装」 | スコープ違反、見積り破綻 |

## 12. 自己チェックリスト

### 自走開始前

- [ ] ゴール記録ファイル `documents/goals/<案件ID>-<機能>.md` が存在する
- [ ] `mode: autopilot` のフロントマターがある
- [ ] `status: active` になっている（Dialogue モードから移行済み）
- [ ] 「確定仕様」セクションに完成条件（DoD）が機械判定可能な形で全て列挙されている
- [ ] スコープ（含む／含まない）が明示されている
- [ ] クリティカル制約が記載されている
- [ ] ユーザーが Autopilot モード移行を明示的に承認した（Dialogue モードの完了判定を通過）
- [ ] `target_branch` をチェックアウトした

### 自走中（各アクション前）

- [ ] このアクションはクリティカル判断（6章）に該当しないか
- [ ] スコープ境界を超えていないか
- [ ] ループ検出・リソース上限に近づいていないか
- [ ] STOP.md が存在しないか
- [ ] 3つ以上の選択肢を比較した判断なら、判断ログに記録するか

### 達成報告前

- [ ] 完成条件のチェックボックスを全て満たしているか
- [ ] 全テストがパスしているか
- [ ] 既存テストを壊していないか
- [ ] 判断ログに採用根拠が書かれているか
- [ ] ゴール記録ファイルの「達成報告」セクションを記入したか
