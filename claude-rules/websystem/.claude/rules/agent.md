# エージェント向けルール

## 必ず最初に読むこと

- **このファイル（rules/agent.md）を最初に読み込んでから作業を開始すること。**
- プロジェクト実装時は、**rules ディレクトリ内の各ルールファイルを確認**したうえで作業すること。

## 仕様書の場所

- 本プロジェクトの仕様書は **documents/document.md** に記載されている。
- 機能要件・画面仕様・データ仕様などを確認する際は、必ず documents/document.md を参照すること。

## 大きめの機能を実装する前に（設計フェーズ）

複数テーブル変更・新画面・新エンドポイントが絡む機能、もしくは
「どう作るか」が一意に決まらない機能では、**Step 1（仕様の記載）に入る前に
設計フェーズを必ず通す**こと。

### 設計フェーズで決めること

1. **目的とポリシー**: その機能で何を解決したいか、誰がどう使うかをユーザーと合意
2. **技術的選択肢**: 主要な設計判断（ストレージ・ライブラリ・データ構造・
   権限モデル等）を選択肢として整理し、トレードオフを並列提示
3. **ロードマップ**: 実装を「1コミット = 1論点」で完結できるよう、
   Step 1〜N に分解する

### 進め方

- ユーザーの要望に対し、技術的観点で**最適と思う案を推奨**したうえで複数選択肢を提示する
- ユーザーの最初の指示をそのまま鵜呑みにせず、より良い方法があれば質問・提案する
- 主要決定が揃ったら、**規模に応じた段階分け**でロードマップを作りユーザー承認を得る。
  - ステップ数は固定ではない。1 ステップで済む小変更もあれば、複数ステップに分けるべき変更もある
  - 段階分けの目的は**「一気に大きな変更を進めないためのストッパー」**。
    その目的が満たされる粒度で、その都度規模を見て判断する
  - ここでいう「1 ステップ = 1 コミット = 1 論点」の粒度は、TDD の Red/Green/Refactor の
    内部サイクル（rules/tdd.md）とは別レイヤー。各ステップの中で TDD を回しても矛盾しない
- 設計フェーズが終わってから Step 1 の仕様書記載に着手する

## 一つの機能の実装フロー（進め方）

**重要**: 一つ一つのステップが完了するごとに、必ずユーザーにフェーズを報告し、次のステップへ進むか確認すること。

| 順  | ステップ                               | 実施内容                                                                                                                             | 完了時のアクション                                           |
| --- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------ |
| 1   | **仕様の記載**                         | ユーザーが「何を作りたいか」を伝える → その内容をもとに **`documents/specs/` 配下の該当HTMLページ**（`rules/spec-format.md` 準拠）に仕様を書き込む。新規ページは `documents/specs/_template.html` をコピーし、`documents/specs/index.html` の目次にもリンクを追加する。 | 仕様記載完了を報告し、**ユーザーにフェーズを移す**。         |
| 2   | **バックエンド実装（TDD）**            | 記載した仕様を基に、**バックエンドから TDD** で実装する。rules/tdd.md および rules/testcode.md に従う。                              | 実装完了を報告し、**ユーザーにフェーズを移す**。             |
| 3   | **コミット準備（バックエンド）**       | ユーザーがファイルをステージングする。**ステージングされたファイルの差分**を確認し、問題がなければ**コミットメッセージ**を提案する。 | 提案完了を報告し、**ユーザーにフェーズを移す**。             |
| 4   | **リファクタリング（バックエンド）**   | **rules/refactoring.md** を確認し、それに従って**ステージングされたファイル**（バックエンド）に対してリファクタリングを行う。        | リファクタリング完了を報告し、**ユーザーにフェーズを移す**。 |
| 5   | **フロントエンド実装（TDD）**          | フロントエンドの実装を行う。rules/tdd.md および rules/frontend-test.md に従い、**自動テスト作成 → 実装 → テスト仕様書(test.csv)更新** をセットで行う。 | 実装完了を報告し、**ユーザーにフェーズを移す**。             |
| 6   | **コミット準備（フロントエンド）**     | ユーザーがファイルをステージングする。**ステージングされたファイルの差分**を確認し、問題がなければ**コミットメッセージ**を提案する。 | 提案完了を報告し、**ユーザーにフェーズを移す**。             |
| 7   | **リファクタリング（フロントエンド）** | **rules/refactoring.md** に従って、**ステージングされたファイル**（フロントエンド）に対してリファクタリングを行う。                  | リファクタリング完了を報告し、**ユーザーにフェーズを移す**。 |
| 8   | **動作確認・仕様書更新**               | ユーザーが動作を確認する。その後、**`documents/specs/` 配下の該当HTMLページを再度更新**する（必要なら index.html の目次・関連ページ間のリンクも更新）。 | 更新完了を報告し、**ユーザーにフェーズを移す**。             |
| 9   | **テスト仕様書の作成**                 | 更新した仕様に対する**テスト仕様書**を **documents/test.csv** に作成する。rules/create_test.md に従う。                              | 作成完了を報告し、**ユーザーにフェーズを移す**。             |

以上で**一連の一つの機能の実装は完了**とする。次の機能に進む場合はステップ1から繰り返す。

## 機能実装時にAIがすべきこと（共通原則）

フローに加え、以下を守ること。

### 実装前

- **仕様の確認**: `documents/specs/` 配下の該当HTMLページ（旧仕様は `documents/document.md`）およびユーザーの指示を読み、実装範囲と受け入れ条件を把握する。
- **曖昧さの解消**: 仕様や要件が不明な点があれば、**実装せずに質問として列挙**し、ユーザーの回答を得てから進める。
- **既存コードの把握**: 既存のディレクトリ構成・命名規則・コーディング規約を確認し、それに合わせて実装する。
- **技術アドバイザーとして関わる**: ユーザーの意見をそのまま実装するのではなく、技術的に最適な選択肢があれば**推奨案として提示**し、トレードオフを並べて質問する。ユーザーの判断材料となる情報を揃えるのが AI 側の責務。

### 実装中

- **仕様の範囲内で実装**: 仕様書に書かれていない機能や仕様の拡張は、ユーザーが明示的に依頼するまで行わない。
- **推測禁止**: 判断が分かれる点・解釈が複数ある点は、推測で実装せず質問する（rules/tdd.md に準拠）。
- **テストの実行確認**: バックエンド・フロントエンドの実装後、該当するテストを実行し、**成功していることを確認**してからフェーズを移す。
- **エラー・境界の考慮**: 仕様に異常系・バリデーション・境界値が含まれる場合は、それらを実装とテストに反映する。

### セキュリティ・環境

- **秘密情報の扱い**: APIキー・パスワード・トークン等をソースコードにハードコードしない。環境変数や設定ファイルで管理し、必要に応じて .env.example に項目のみ記載する。
- **依存関係**: 新規パッケージ・ライブラリを追加する必要がある場合は、理由と用途を簡潔に説明する。

### コミット準備時（差分確認）

- **問題がある場合**: ステージングされた差分に、明らかなバグ・仕様違反・セキュリティリスク・不要な変更が含まれる場合は、**コミットメッセージを提案せず**、問題点を指摘してユーザーに報告する。
- **問題がない場合**: 差分が仕様に沿っており問題ないと判断したときのみ、コミットメッセージを提案する。
- **変更内容の簡潔なサマリ**: コミットメッセージを提案する際は、合わせて以下をユーザーが一目で把握できる形でまとめて報告する。
  - ファイル一覧（変更ファイルと変更行数）
  - 主要な変更点（バグ修正なら原因と修正、機能追加なら追加機能、リファクタなら抽出した責務 等）
  - テスト結果（BE/FE のテスト件数・tsc・既存 Codex 指摘の対応状況）。
    ドキュメントのみの変更などテストが該当しないコミットでは「該当なし」と明記する
