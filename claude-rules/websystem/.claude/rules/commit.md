# コミットメッセージ作成ガイドライン

このファイルは **言語・フレームワーク・プロジェクトに依存しない汎用ルール** として書かれている。
他のプロジェクトにそのままコピーして使えるようにする。

---

## 基本原則

### 1. コミットメッセージは「**なぜ**」を伝える

差分（`git diff`）は「**何が**」変わったかを語る。コミットメッセージは「**なぜ**」変えたかを語る場所。

- ❌ 悪い例: `update controller` ← 差分を見ればわかる
- ✅ 良い例: `Hash::make と mutator の二重ハッシュ化でログイン不能だったバグを修正`

「何が」しか書かれていないメッセージは、半年後の自分や他の開発者にとって**最低限の情報源として失格**になりやすい。

### 2. アトミック（1 コミット 1 トピック）

1 つのコミットは 1 つの明確な目的だけを持つこと。

- リファクタリング + 機能追加 + バグ修正 を 1 コミットにまとめない
- レビュアーが「このコミットは何のため？」と一言で答えられる粒度にする
- `git revert` / `git bisect` が機能する単位にする

例外: セッション開発の現実として「ひとまとめにコミット」した方がコストが低い場面はある。
チームの慣習・PR レビュースタイル・履歴の重要度に応じて判断する。

### 3. 命令形・現在形で書く

「Add」「Fix」「Refactor」のように動詞の命令形で書く（または同等の日本語動詞「追加する／修正する／整理する」）。

- ❌ `Added new feature` / `Adds new feature` / `Adding new feature`
- ✅ `Add new feature` / `新機能を追加`

理由: Git 自身が生成する `Merge`, `Revert`, `Cherry-pick` などのメッセージも命令形。コミットを「このコミットを適用すると◯◯する」と読める文体に統一すると、履歴全体のトーンが揃う。

## フォーマット

### 50/72 ルール（汎用標準）

```
<タイトル：50 文字以内>

<空行 1 行>

<本文：72 文字で改行、なぜ・背景・影響を箇条書きで>

<空行 1 行>

<フッター：BREAKING CHANGE、issue 参照、Co-authored-by など>
```

- **タイトル**: 50 文字以内（`git log --oneline` / GitHub の PR 一覧で省略されない上限）
- **タイトル末尾**: ピリオド／句点を付けない
- **タイトルと本文の間**: 必ず空行を 1 行入れる
- **本文**: 72 文字（または各環境の表示幅）で改行。長すぎる行は読みにくい
- **箇条書き本文**: ハイフン `-` を推奨

### Conventional Commits（オプション）

タイトルの先頭に `<type>(<scope>): ` のプレフィックスを付ける形式。CI で自動分類・CHANGELOG 生成・semver 自動推定 に向く。

```
<type>(<scope>): <subject>

<body>
```

`scope` はオプション。範囲が広い変更や横断的な変更ではスコープなしの形式も有効：

```
<type>: <subject>

<body>
```

主な type:

| type | 用途 |
|---|---|
| `feat` | ユーザー視点で見える新機能 |
| `fix` | バグ修正 |
| `refactor` | 挙動を変えないコード整理 |
| `perf` | パフォーマンス改善（挙動は変えず性能を向上させる変更） |
| `docs` | ドキュメントのみの変更 |
| `test` | テスト追加・修正のみ |
| `chore` | ビルド・依存・設定など雑務 |
| `build` | ビルドシステム・外部依存の変更 |
| `ci` | CI 設定の変更 |
| `style` | コードフォーマット・スペース・セミコロン等（挙動に影響なし） |
| `revert` | 過去のコミットを取り消す |

`scope` はオプションで、変更範囲（モジュール名・コンポーネント名など）を入れる。

例:
- `fix(auth): セッショントークンの有効期限が更新されないバグを修正`
- `refactor(api): エラーレスポンスのフォーマットを統一`
- `feat(user): プロフィール画像のアップロードに対応`

導入する場合は、チームで合意したうえで `CHANGELOG.md` や PR テンプレートと整合させる。

## タイトルの書き方

### 良いタイトルの条件

1. **そのコミットを適用すると何が起きるか**が 1 行で伝わる
2. **動詞**で始まる（命令形）
3. **具体的**：`update X` ではなく `update X to handle Y` のように何を狙ったかが見える
4. **50 文字以内**

### よくある悪いタイトルとその直し方

| ❌ 悪い | 直した例 |
|---|---|
| `update` | `セッションタイムアウトを 30 分に延長` |
| `修正` | `日付フォーマット崩れを修正` |
| `Fix bug` | `空入力時にクラッシュする問題を修正` |
| `WIP` | （WIP は基本コミットしない。ローカル退避なら `wip:` プレフィックスで明示） |
| `Updated AuthController.php` | `JWT 検証時の例外ハンドリングを追加` |
| `色々修正` | コミットを分割する |

## 本文の書き方

### 含めるべき情報

1. **なぜ変更したか**（動機・背景・きっかけ）
2. **何を変えたか**（差分の要約。差分から自明なら省略可）
3. **影響範囲**（API 変更・データ削除・互換性破壊など）
4. **代替案を選ばなかった理由**（必要なら）
5. **関連 issue / PR / ドキュメントへのリンク**

### 含めない方が良いもの

- ファイル名の列挙（差分から自明）
- 一行ずつの逐次解説（差分から自明）
- 設計上の長文議論（PR 説明欄か設計ドキュメントに分離）

### 本文の例

```
- ログイン時にトークンの ability 検証を追加し、admin/agent の混在を防止
- 既存テストは MySQL に依存していたため SQLite :memory: で再実行可能に変更
- マイグレーションのロールバック時に NOT NULL 列が復活して失敗していた問題を修正
```

## フッター（メタデータ）

### 1. BREAKING CHANGE

互換性を破壊する変更があるとき、本文末尾に明示する。

```
BREAKING CHANGE: agents テーブルから account カラムが削除されました。
account カラムを参照していた外部ツールは email を使うよう移行してください。
```

Conventional Commits の場合、これがあると semver の major bump とみなされる。

### 2. issue / PR 番号

issue や PR を関連付けるキーワードは用途で使い分ける。

| キーワード | 挙動 |
|---|---|
| `Closes #123` / `Fixes #123` / `Resolves #123` | GitHub / GitLab で **マージ時に該当 issue を自動クローズ**する |
| `Refs #123` / `See #123` / `Related to #123` | **参照のみ**（自動クローズはしない）。関連 issue を辿れるようにする目的 |

```
Closes #245
Refs #240, #241
```

`git log --grep="#245"` で後から検索できる。

クローズしてはいけない issue（親 issue や関連 issue）には `Refs` を、このコミットで終わる issue には `Closes` / `Fixes` を使う。

### 3. Co-authored-by

ペアプロ・モブプロ・AI 支援などで複数人/エージェントの寄与があるとき。

```
Co-authored-by: Alice <alice@example.com>
Co-authored-by: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

GitHub の contributor 表示にも反映される。

## アンチパターン集

### 1. 「色々」コミット

複数のトピックを 1 コミットに詰める。あとで bisect できなくなる。

```
- バグ修正、リファクタ、ドキュメント更新、依存パッケージ追加
```

→ 4 つのコミットに分ける。

### 2. 「コードコメント」レベルのメッセージ

```
fix typo
```

タイプミスがどこで何だったか、なぜ修正が必要だったかが何も伝わらない。最低でも `fix typo in user-facing error message` のように対象を書く。

### 3. メッセージが嘘をついている

タイトルが `Add feature X` なのに、差分には feature X とは無関係な他の変更も入っている。

→ そういうコミットは分割する。または嘘のないタイトルにする（`Add feature X and fix unrelated formatting`）。

### 4. 「what」だけの本文

```
Update auth.php and user.php
```

差分を見ればわかる情報しかない。本文は **なぜ** か **影響** を書く。

## チェックリスト（コミット前の自問）

- [ ] タイトル 50 文字以内・命令形・末尾ピリオドなし
- [ ] このコミットを 1 文で説明できるか（できなければ分割を検討）
- [ ] 「なぜ」が本文に書かれているか（または PR 説明欄に書く前提か）
- [ ] 差分にタイトルと無関係な変更が混ざっていないか
- [ ] 機密情報（パスワード・APIキー・PII）が含まれていないか
- [ ] 互換性破壊があるなら `BREAKING CHANGE:` フッターを書いたか
- [ ] 関連 issue があれば参照を入れたか

## プロジェクトごとに上書きしてよい設定

このルールは汎用ベース。プロジェクト固有で以下を上書きしてよい。

- **言語**: 日本語 / 英語（チームで統一）
- **Conventional Commits 採用の有無**
- **type の追加・削除**（業界・チームによっては `hotfix`, `release` 等を足すことがある）
- **タイトル文字数の上限**（厳密な 50 ではなく 60〜72 で運用するチームも多い）
- **issue 参照のフォーマット**（`Refs #` か `Closes #` かなど）
- **コミット署名（GPG / SSH キー）の必須有無**

プロジェクト側で上書きする場合は、CLAUDE.md または `.claude/rules/` 配下のプロジェクト固有ファイルに記載すること。

## エージェントへの指示

このルールに従ってコミットメッセージを提案・作成すること。

- プロジェクトの過去ログ（`git log --oneline -10`）のスタイルが**明確にこのルールから外れている**場合は、まずユーザーに「過去ログのスタイルに合わせるか、このルールを適用するか」を確認する
- ユーザーが「過去ログに合わせる」を選んだ場合、過去のスタイルを優先する
- ユーザーが「このルールを適用する」を選んだ場合、または特に指示がない場合、このルールに従う
- 機密情報・PII・認証情報をコミットメッセージに含めないこと
