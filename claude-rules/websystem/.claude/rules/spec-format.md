# 仕様書フォーマットルール

仕様書を作成・更新するときに参照する。

## 配置場所と命名

- 仕様書は **`documents/specs/`** 配下に配置する
- 画面分類で2階層構成:
  - `documents/specs/admin/` — 運営管理者画面
  - `documents/specs/user/` — 一般ユーザー画面
  - 横断的な仕様（用語集・共通機能など）が必要になった場合は `documents/specs/common/` を作成
- ファイル名は **kebab-case の英語**（例：`camera-link.html`、`order-history.html`）
- 日本語・全角記号・スペースを含めない（リンク張り・URL に乗せやすくするため）

## ページのフォーマット

- **HTML5 形式**で記述する（マークダウンは使わない）
- スタイリングは **Tailwind CSS（CDN参照）**
  - `<head>` に `<script src="https://cdn.tailwindcss.com"></script>` を入れる
  - ビルド作業不要、ブラウザで直接開けることが必須要件
- テンプレート: [`documents/specs/_template.html`](../../documents/specs/_template.html) をコピーして開始する

## ページ構造（必須要素）

各仕様書ページは以下の要素を持つこと:

- **パンくずナビ**: `目次 → セクション → ページ名`（テンプレート参照）
- **ページタイトル**（`<h1>`）
- **概要**: 1〜2文で「このページが扱う機能」を説明
- **セクション見出し**（`<h2>`）→ サブ見出し（`<h3>`）の階層
- **表示項目**は `<ul>` または `<table>` で整理
- **操作フロー**は番号付きリスト（`<ol>`）
- **注意事項**は黄色枠（`bg-yellow-50 border-l-4 border-yellow-400`）で強調
- **関連ページ**節を最後に置き、相互リンクで辿れるようにする

## 目次（index.html）の運用

- 仕様書ページを **追加・移動・削除した際は必ず `documents/specs/index.html` を更新**
- 各ページへのリンクを `<a>` タグで張る
- セクション見出しで分類（運営管理者画面 / ユーザー画面 / 共通）

## ページ間リンク

HTMLの利点を活かし、関連ページは積極的にリンクで接続する:

- 「会社アカウント管理」⇔「カメラ設定」「通信セット設定」（会社紐付けの関係）
- 「現場管理」⇔「デバイスセット」⇔「商品注文」「注文履歴」（業務フローの関係）
- 仕様の同一概念（「通信セット」「グループ」など）はクロスリンクして用語の定義ページに飛ばせるようにする

## マークダウンからの段階的移行

既存の `documents/document.md` の内容は、機能ごとに HTML へ段階的に移行する。

- 新規機能の仕様 → 最初から HTML で `documents/specs/` に作成
- 既存機能の仕様修正 → そのタイミングで HTML 化を併せて行う
- 移行が完了したセクションは `documents/document.md` から削除し、`documents/specs/index.html` の該当行に正式リンクを張る

両方が併存する移行期間中は、**HTML 側を正**とする（`document.md` の該当節には「→ documents/specs/admin/xxx.html へ移行済み」と注記する）。

## レビュー観点

仕様書を書いた・直したときは以下を確認:

- ブラウザで開いて表示崩れがないか（Tailwind が読まれているか）
- リンク切れがないか（`<a href="...">` のパスが正しいか）
- 目次（`index.html`）から該当ページへ辿れるか
- 仕様（記述内容）が実装と一致しているか（三者整合性チェック）
