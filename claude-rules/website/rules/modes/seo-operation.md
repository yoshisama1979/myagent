# 運用・SEO モード — 動作ルール

公開後のホームページを **「使いながら育てる」** ための、運用改善・SEO のルール。
制作（`rules/modes/start.md` / `rules/modes/check.md`）が終わった後の **継続フェーズ** を担当する。

`CLAUDE.md` の協働原則（不明点は都度質問 / より良い案は提案 / Design.md 優先）を土台とする。

---

## 役割

公開済みサイトを **観点を漏らさず評価** し、**効果の高い改善から** 提案・実装する。
網羅レポートを出すのが目的ではなく「**次の一手を明確にする**」のが目的。

- 優先度（効果 × 工数）を整理し、**Quick win から** 出す
- いきなり手を動かさず、**現状把握 → 評価 → 提案 → 合意 → 実装 → 記録** の順で進める
- 改善の根拠（**観察 → なぜ → 改善案**）を必ずセットで示す

---

## 進め方の原則

| 原則 | 中身 |
|------|------|
| **権限非依存で最適解** | 自分（AI）ができる範囲に発想を縮めない。**ゴールに対する最適策を先に提示**する。MEO・GA4・Search Console・外部ツール・ユーザーの手作業など、AI が直接実行できない手段も含めて最善を出す |
| **不足は積極的に要求** | 必要なデータ・ツール・アクセス権が無ければ、**推測やLLMでの代替で済ませずユーザーに要求**する。自前の `curl`/`grep` 等は **「暫定」と明示** し、本来あるべき手段（ツール化・API連携等）を併記する |
| 確認してから動く | 対象・ゴールが曖昧なら Step 1 で必ず質問。推測で着手しない |
| 効果順 | 「効果 × 工数」で並べ、Quick win を最初に出す |
| 事実と推測を分ける | 「観察（事実）」と「なぜ問題か（影響・推測）」を分離。推測は「推測：」と明記 |
| 合意してから実装 | 提案を出して判断を仰ぐ。勝手に全項目を実装しない |
| **すぐ直せない課題は残す** | その場で対応しきれない課題は `documents/pending-issues.md` に記録する（後述 Step 6・`CLAUDE.md`「残存課題の記録ルール」準拠） |
| 記録を残す | 改善の経緯・判断はプロジェクトの記録に残す |

---

## 手順

### Step 1: 対象と狙いの確認（着手前に必ず）

不明な点だけ確認する。曖昧なまま Step 2 に進まない。質問は選択肢形式で答えやすくまとめる。

| 確認項目 | 例 |
|---------|-----|
| 対象 | URL、または対象ページ・対象範囲 |
| 改善ゴール | 問い合わせ増／予約増／SEO流入／表示速度／ブランド刷新 など |
| ターゲット | BtoB/BtoC、地域、年齢層、デバイス傾向 |
| 制約 | 触ってよい範囲、静的/PHP/CMS、納期・予算感 |
| 現状の課題感 | ユーザーやお客さんが既に感じている問題（あれば最優先で確認） |

### Step 2: 現状把握

- ページ構成・主要メッセージ・導線を把握する
- ファーストビューで「誰の・何の・次に何をすべきか」が伝わるかを読む
- **オンページ信号（title/meta/OGP/構造化データ/見出し/alt 等）は監査ツール優先**。監査ツールがあればその出力を一次情報にする。ツールが無い間の `curl`/`grep` 等の手作業は **「暫定」と断り**、不足は Step 6 で残課題に記録する
- 効果計測データ（検索順位・流入・問い合わせ数）が必要なら、LLM で代替せず **GA4 / Search Console 等の取得手段をユーザーに要求** する

### Step 3: 評価（観点を漏らさず）

下表の観点で気づきを集める。各観点で **良い点 / 課題** を事実ベースに分ける。
全観点を機械的に埋めるのではなく、**ゴールに効く観点を厚く** 見る。

| 観点 | 見るポイント |
|------|------------|
| 技術SEO | title / meta description、見出し構造(h1-h3)、内部リンク、画像alt、構造化データ、canonical、sitemap.xml / robots.txt |
| コンテンツSEO | 検索意図に沿っているか、情報の鮮度・網羅性、重複・薄いコンテンツ、見出しがベネフィット訴求か |
| **仮置き残存** | 公開HTMLにテンプレの仮置きが残っていないか（OGP画像URL・構造化データのプレースホルダ/コメントアウト・ダミーテキスト・未差し替えalt）。**実害が出やすく見落としやすい** |
| 表示速度・CWV | 画像最適化、不要リソース、LCP / CLS / INP |
| モバイル対応 | レスポンシブ、タップ領域、可読性、横スクロール有無 |
| 導線・CV | 流入後の問い合わせ/予約への動線、CTA の位置・文言・数 |
| 計測・分析 | GA4 / Search Console の設定有無、計測タグ、コンバージョン計測ができているか |
| ローカルSEO（該当時） | Google ビジネスプロフィール、NAP（社名・住所・電話）の一貫性、地域キーワード |
| 信頼性・安心 | 会社情報、プライバシーポリシー、SSL、問い合わせ先の明示 |

### Step 4: 改善提案（優先度をつける）

- 各提案は下記「出力フォーマット」に従う
- 「効果 × 工数」で優先度を整理し、**Quick win → 中長期** の順で提示
- 全部やろうとしない。「まずどれから着手するか」を合意する

### Step 5: 実装（合意後のみ）

- 合意した項目だけ着手する
- いきなり本番を直す前提で進めない。案・モックを作って確認してから反映する
- `.scss` を編集した場合：コンパイル運用は [.claude/rules/scss-autocompile.md](../../.claude/rules/scss-autocompile.md)「自動コンパイル環境が有効なときの運用」に従う（自動環境=依頼不要／無ければ開発者に依頼。AIは手動でビルドを叩かない）。コンパイル後の目視確認をユーザーに明示する
- 改善の経緯・判断をプロジェクトの記録に残す

### Step 6: 残課題の記録（毎回・軽く）

その場で対応しきれなかった改善・施策は、放置せず **残存課題として記録** する。

- 記録先：`documents/pending-issues.md`（`CLAUDE.md`「残存課題の記録ルール」準拠）
- 各課題に「**経緯 / 未対応（何が残っているか）/ 今後の判断ポイント**」を明記
- 「計測してから判断」「ツールが無いので暫定対応」等は、本来あるべき手段（GA4/GSC連携・監査ツール化等）を併記して残す
- 終了報告に「今回の残課題：PI-NNN（1行要約）」を含める。無ければ「特記なし」

## 出力フォーマット

### 評価サマリ（Step 3 の結果）

```markdown
## 現状評価サマリ：<サイト名>

**ゴール**: （Step 1 で確認したゴール）
**総評**: 2-3行。良い点と最大の課題を結論ファーストで

### 観点別
- ✅ 良い点：…
- ⚠️ 課題：…（観点名つき）
```

### 改善提案（Step 4、1件ずつ）

```markdown
#### 改善案N: 一行要約

- **観点**: 技術SEO / コンテンツSEO / 導線 / 表示速度 / …
- **観察**: 何を見てそう思ったか（事実）
- **なぜ問題か**: 影響・リスク（推測は「推測：」と明記）
- **改善案**: 具体的に何をどう変えるか
- **効果**: 高 / 中 / 低（ゴールへの寄与）
- **工数**: 小（〜1h）/ 中（〜半日）/ 大（1日以上）
- **優先度**: 🔴最優先 / 🟡推奨 / 🟢余裕があれば
```

### 着手提案（Step 4 末尾）

```markdown
### 次の一手（提案）
まず「改善案X・Y」（Quick win）から着手するのが効果的です。着手してよいですか？
```

---

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 対象・ゴール未確認で実装に入る | 「確認してから動く」違反。手戻りの元 |
| ❌ 合意前に全項目を実装する | 判断を飛ばす。Quick win 合意が先 |
| ❌ 網羅レポートで終わる | 目的は「次の一手」。優先度なき列挙はNG |
| ❌ 観察と推測を混ぜる | 判断を誤らせる。「推測：」で分離する |
| ❌ 本番サイトを直接いじる前提で進める | まず案を作り、確認してから反映する |
| ❌ すぐ直せない課題を記録せず放置する | 取りこぼしの元。`pending-issues.md` に残す |

## 自己チェック（提案・実装の前に確認）

- [ ] 対象・ゴールを確認済みか？（曖昧なら Step 1 に戻る）
- [ ] 評価でゴールに効く観点を厚く見たか？
- [ ] 各提案に「観察 / なぜ / 改善案 / 効果 / 工数 / 優先度」が揃っているか？
- [ ] 観察（事実）と推測を分離したか？
- [ ] Quick win を先頭に、効果順で並べたか？
- [ ] 実装は合意した項目だけか？
- [ ] **権限に縛られず最適策を出したか？ 不足を自前で握りつぶさずユーザーに要求したか？**
- [ ] **すぐ直せない課題を `documents/pending-issues.md` に記録したか？（無ければ「特記なし」）**

すべて ✅ なら次へ、1つでも ❌ なら見直し。
