# CLAUDE.md

このプロジェクトは **通常のホームページ**（コーポレートサイト・ランディングページ・小規模Webサイト等）を想定したサイト制作プロジェクト。

## プロジェクト概要

- 想定対象: HTML / CSS（Sass） / JavaScript で構築する一般的なホームページ
- 制作物の例: コーポレートサイト、サービス紹介サイト、ランディングページ、ブランドサイト等
- 前提とするツールチェーン: 静的HTML or PHPテンプレート + Sass。Webアプリケーションフレームワーク（React/Vue/Next.js 等）は対象外
- 対象フェーズ: 新規制作だけでなく、**公開後の運用・SEO改善**（使いながら育てる継続的な改善サイクル）も対象に含む
- 事業ファクト集: `documents/company-facts.md`（サイトに載せる**確定事実**の SSOT。`Design.md`＝デザインのSSOT と対になる**コンテンツの情報SSOT**。ヒアリングで確定した事実を公開可否・出典つきで集約し、ページはここから展開する。`page-build.md`/`page-update.md` §5・§6 参照）
- 残存課題: `documents/pending-issues.md`（その場で対応しきれない未解決課題の記録先。下記「残存課題の記録ルール」参照）
- ルール改善メモ: `documents/rule-improvements.md`（ルールファイルへの気づき・改善案の記録先。下記「コラボレーションの原則 2」参照）
- プロジェクト固有の定義（サイト名・ディレクトリ構成・デプロイ先・公開URL・ブランチ運用・環境・コンパイル方針など）: `project-config.md`（コピー後まず記入する）

> プロジェクト固有の情報（サイト名・ディレクトリ構成・デプロイ先・公開URL・ブランチ運用など）は `project-config.md` に記入する（CLAUDE.md 本文に直書きしない）。

## 残存課題の記録ルール

会話・制作・運用の中で **その場で対応しきれず未解決のまま残った課題** は `documents/pending-issues.md` に記録すること。すぐ対応できるものは対応し、そうでないものを取りこぼさないための仕組み。

- ユーザーから「残存課題として記録して」と依頼された場合は必ず `pending-issues.md` に追記する
- 制作フェーズ（バグ・未実装・保留した設計判断）でも、運用・SEOフェーズ（今回は見送る改善・要計測の施策）でも、**同じファイルに記録する**
- 各課題には「**経緯**」「**未対応**（何が残っているか）」「**今後の判断ポイント**」を明記する
- 解決した課題は `pending-issues.md` から削除し、必要なら確定仕様・記録側へ転記する

## ルールファイル (.claude/rules/)

**方針**: トークン消費を抑えるため、ルールファイルは**タスクに関連するもののみ必要に応じて読み込む**。作業のフェーズに入った時点で、下表から該当するファイルを選んで読み込むこと。

| ファイル | いつ参照するか |
|---------|--------------|
| `start.md` | **プロジェクト開始時 / 新規ページ・UI作成依頼を受けた時。** `Design.md` の有無確認など、制作に入る前のゲート |
| `design.md` | **色・フォント・余白・コンポーネント等のデザイン判断をする時。** `Design.md`=SSOT の扱い方、Design.md が持つべきセクション構成、プロジェクト非依存の普遍的な Do/Don't（固有値は各プロジェクトの `Design.md` 側） |
| `design-replication.md` | **デザインカンプ／モックアップ（特にAI生成画像）を渡されてUIを再現する時。** 推測を実測に変え、実ブラウザ描画で照合する周回（色の実測・同一幅スクショ・寸法は意図から設計しSSOTに記録） |
| `page-build.md` | 新規ページ・画面の追加依頼を受けた時。既存ページを雛形に実装する手順とコンテンツの正確性担保 |
| `page-update.md` | 既存ページを更新して内容を充実させる依頼を受けた時。既存を壊さず加筆・改善する手順とコンテンツの正確性担保 |
| `scss-autocompile.md` | **AIが `.scss` を編集すると即コンパイルされる環境を作る／使う時。** フックで自動生成する再現レシピと、各ルールの「コンパイル前提」を条件分岐（自動環境あり＝依頼不要／なし＝開発者に依頼）させる土台 |
| `refactor.md` | リファクタリングの提案・実施時 |
| `check.md` | ウェブサイトを作り終えた後、納品・公開前にチェックするとき |
| `seo-operation.md` | **公開後の運用・SEO改善**を依頼された時。現状把握→評価→提案→合意→実装→記録の改善サイクル、SEO評価観点、残課題の記録 |
| `hp-loop-dialogue.md` | **外部の分析ループ（はなさかAI HP分析ループ）から改善提案を受け取った時**。検証→対応→提案元への報告の標準フロー（R-NNN/Q-NNN の扱い、提案元の限界の理解、報告文の作り方） |
| `mailbox.md` | **hp-loop など別マシンのエージェントとメールボックス（共有受信箱）でやり取りする時**。受信箱を読む／報告・質問を送る操作と、社長承認ゲート（`--needs-approval`）。報告の中身は `hp-loop-dialogue.md` に従う＝こちらは送受信の土管 |

※ コーディング規約等のルールは必要に応じて追加していくこと。

## スラッシュコマンド (.claude/commands/)

ユーザーが `/コマンド名` で呼び出せる定型タスク。

| コマンド | 用途 |
|---------|--------------|
| `/staging` | 変更ファイルを `git add` でステージング(コンパイル済み `.css` を巻き込まないよう注意) |
| `/commit` | ステージング差分をチェックし、日本語コミットメッセージを提案 |
| `/commit-pipeline` | staging → 差分確認 → Design.md整合チェック → Codexクロスレビュー → コミット → リファクタ提案 のフルパイプライン |
| `/refactor` | `.claude/rules/refactor.md` に従ってリファクタリング候補を列挙 |
| `/cleanup-permissions` | `.claude/settings.local.json` の permissions を整理 |

## デザインルールの単一の真実の源 (Design.md)

プロジェクトルートに `Design.md` がある場合、それが **デザインに関する Single Source of Truth**。
色・フォント・余白・コンポーネント仕様は SCSS 変数より Design.md を優先する。

- 新規 UI 実装の前に必ず `Design.md` を読む
- Design.md と既存コードが食い違う場合は Design.md に合わせて既存コードを直す(漸進的に)
- Design.md に無い概念(新しい色トークン、ブレークポイント等)を勝手に増やさない。先に Design.md を更新する提案をする
- `Design.md` が存在しない場合は `.claude/rules/start.md` の「1. Design.md の準備」セクションに従って作成手順を踏む

> **汎用ルールと固有値の分離**：「Design.md をどう扱うか／どのプロジェクトでも守る普遍的なデザイン規律」は `.claude/rules/design.md`（汎用）にある。**各プロジェクト固有の値**（色 hex・フォント・幅・換算ルール等）は、そのプロジェクトの `Design.md` 側に書く。本リポジトリには YCOM の値で書いた記入例 `Design.sample.md` を同梱している（実プロジェクトではこれを `Design.md` にコピーして値を差し替える）。

## コラボレーションの原則

### 1. 不明点・提案は都度質問する

ユーザーからの指示や記述をそのまま鵜呑みにして実装に進まないこと。

- **不明点があれば**: 仕様の解釈に複数パターンがある、デザインの意図が読み取れない、既存ページとの整合性が不明、などの場合は実装前に必ず質問する
- **より良い案があれば**: 提示された方針より適切な選択肢が見える場合は、推奨案と理由を添えて提案する（最終判断はユーザー）
- **判断が分かれる設計事項**: ファイル構成・命名・画像形式・スコープなどは、実装前に選択肢を整理して合意を取る

推測で進めて手戻りするより、一手前で確認する方がコストが低い。

### 2. ルールファイル自体の改善も提案する

`.claude/rules/` 配下のルールファイルや `CLAUDE.md` を参照していて、以下に気付いた場合は実装作業の合間でも提案する。

- 実態と乖離している記述
- 抜け落ちている運用ルール
- 矛盾している記述（複数のルールファイル間で食い違う指示）
- 改善できる構成・粒度

提案時は「どのファイルの、どの記述を、どう変えるか、なぜ」を明示する。ユーザーの判断後に修正に進む。

**ユーザーが明確な回答をしなかった場合も、気づいた点はその都度 `documents/rule-improvements.md` に記録する**。提案が会話に埋もれて流れ、忘れられるのを防ぐため、「記録」と「反映」を分離する（記録は気づいた都度・反映はユーザー判断時にまとめて）。

- 提案したが判断が出ていない／会話が先に進んでしまった場合も、観察を1行でも残す
- 各エントリに「対象ファイル・どの記述・どう変えるか・なぜ」を明記する（フォーマットは `documents/rule-improvements.md` 冒頭参照）
- 反映が済んだエントリは削除する（または「反映済み」と印を付ける）
- **このプロジェクト固有でなく、配布元の共通テンプレート全体に還元すべき**と判断した気づきは `[テンプレ還元]` と印を付け、テンプレ更新側（はなさかAI）へ届ける（`rule-improvements.md`「`[テンプレ還元]` 印のものはテンプレ更新側へ送る」／`.claude/rules/mailbox.md` 2-3.）。固有値はテンプレに持ち込まず `project-config.md`／`Design.md` 側に置く

### 3. 作業のまとまりごとに「頭の切り替えを助ける報告」をする

ユーザーはエージェントの作業中に別の作業をしていることが多い。ひとまとまりの作業（ページ作成・改善・運用対応など）が一区切りしたら、ユーザーがスムーズに頭を切り替えられるよう、次の4点を**要点だけ**まとめて報告する。

1. **大局（目的）** … いま何を目指して作業しているか
2. **課題** … その目的に対する現在の課題
3. **今回やったこと** … この作業で何を進めたか
4. **次にすべきこと** … 次の一手・提案（あれば）

長い経緯の再説明はせず、切り替えに必要な最小限に絞る（**冗長にしない**）。軽微なやり取りには不要＝実質的な作業の区切りで出す。
