# CLAUDE.md

このプロジェクトは **通常のホームページ**（コーポレートサイト・ランディングページ・小規模Webサイト等）を想定したサイト制作プロジェクト。

## プロジェクト概要

- 想定対象: HTML / CSS（Sass） / JavaScript で構築する一般的なホームページ
- 制作物の例: コーポレートサイト、サービス紹介サイト、ランディングページ、ブランドサイト等
- 前提とするツールチェーン: 静的HTML or PHPテンプレート + Sass。Webアプリケーションフレームワーク（React/Vue/Next.js 等）は対象外
- 対象フェーズ: 新規制作だけでなく、**公開後の運用・SEO改善**（使いながら育てる継続的な改善サイクル）も対象に含む
- 残存課題: `documents/pending-issues.md`（その場で対応しきれない未解決課題の記録先。下記「残存課題の記録ルール」参照）

> プロジェクト固有の情報（サイト名・ディレクトリ構成・デプロイ先・公開URL・ブランチ運用など）はこのセクションに追記してください。

## 残存課題の記録ルール

会話・制作・運用の中で **その場で対応しきれず未解決のまま残った課題** は `documents/pending-issues.md` に記録すること。すぐ対応できるものは対応し、そうでないものを取りこぼさないための仕組み。

- ユーザーから「残存課題として記録して」と依頼された場合は必ず `pending-issues.md` に追記する
- 制作フェーズ（バグ・未実装・保留した設計判断）でも、運用・SEOフェーズ（今回は見送る改善・要計測の施策）でも、**同じファイルに記録する**
- 各課題には「**経緯**」「**未対応**（何が残っているか）」「**今後の判断ポイント**」を明記する
- 解決した課題は `pending-issues.md` から削除し、必要なら確定仕様・記録側へ転記する

## ルールファイル (.claude/rules/)

**方針**: トークン消費を抑えるため、ルールファイルは**タスクに関連するもののみ必要に応じて読み込む**。作業のフェーズに入った時点で、下表から該当するファイルを選んで読み込むこと。

| ファイル | いつ参照するか |
|---------|--------------|
| `start.md` | **プロジェクト開始時 / 新規ページ・UI作成依頼を受けた時。** `Design.md` の有無確認など、制作に入る前のゲート |
| `refactor.md` | リファクタリングの提案・実施時 |
| `check.md` | ウェブサイトを作り終えた後、納品・公開前にチェックするとき |
| `seo-operation.md` | **公開後の運用・SEO改善**を依頼された時。現状把握→評価→提案→合意→実装→記録の改善サイクル、SEO評価観点、残課題の記録 |

※ コーディング規約等のルールは必要に応じて追加していくこと。

## スラッシュコマンド (.claude/commands/)

ユーザーが `/コマンド名` で呼び出せる定型タスク。

| コマンド | 用途 |
|---------|--------------|
| `/staging` | 変更ファイルを `git add` でステージング(コンパイル済み `.css` を巻き込まないよう注意) |
| `/commit` | ステージング差分をチェックし、日本語コミットメッセージを提案 |
| `/commit-pipeline` | staging → 差分確認 → Design.md整合チェック → Codexクロスレビュー → コミット → リファクタ提案 のフルパイプライン |
| `/refactor` | `rules/refactor.md` に従ってリファクタリング候補を列挙 |
| `/cleanup-permissions` | `.claude/settings.local.json` の permissions を整理 |

## デザインルールの単一の真実の源 (Design.md)

プロジェクトルートに `Design.md` がある場合、それが **デザインに関する Single Source of Truth**。
色・フォント・余白・コンポーネント仕様は SCSS 変数より Design.md を優先する。

- 新規 UI 実装の前に必ず `Design.md` を読む
- Design.md と既存コードが食い違う場合は Design.md に合わせて既存コードを直す(漸進的に)
- Design.md に無い概念(新しい色トークン、ブレークポイント等)を勝手に増やさない。先に Design.md を更新する提案をする
- `Design.md` が存在しない場合は `.claude/rules/start.md` の「1. Design.md の準備」セクションに従って作成手順を踏む

## コラボレーションの原則

### 1. 不明点・提案は都度質問する

ユーザーからの指示や記述をそのまま鵜呑みにして実装に進まないこと。

- **不明点があれば**: 仕様の解釈に複数パターンがある、デザインの意図が読み取れない、既存ページとの整合性が不明、などの場合は実装前に必ず質問する
- **より良い案があれば**: 提示された方針より適切な選択肢が見える場合は、推奨案と理由を添えて提案する（最終判断はユーザー）
- **判断が分かれる設計事項**: ファイル構成・命名・画像形式・スコープなどは、実装前に選択肢を整理して合意を取る

推測で進めて手戻りするより、一手前で確認する方がコストが低い。

### 2. ルールファイル自体の改善も提案する

`.claude/rules/` 配下のルールファイルや `CLAUDE.md` を参照していて、以下に気付いた場合は実装作業の合間でも提案する。

- 実態と乖離している記述
- 抜け落ちている運用ルール
- 矛盾している記述（複数のルールファイル間で食い違う指示）
- 改善できる構成・粒度

提案時は「どのファイルの、どの記述を、どう変えるか、なぜ」を明示する。ユーザーの判断後に修正に進む。
