# 既存ページ更新ルール

ウェブサイトの **既存ページ（画面）を更新して内容を充実させるとき** の手順をまとめたルール。

## このルールの位置付け

- 既存ページの更新・内容充実の依頼を受けたときに参照する
- **新規ページの追加は別ルール**（あれば `page-build.md`）。**公開後の運用・SEO改善ルール**（あれば `seo-operation.md` 等）があれば併用する
- 大まかな流れ：**現状把握 → 合意 → 既存を壊さず段階的に加筆 → コンテンツ確定 → 退行防止確認 → 残課題記録**

> このルールは「既存ページを更新・充実させるときの手順」に絞る。新規ページの起こし方は `page-build.md`、リファクタリングは `refactor.md` に委ねる。重複する観点（コンテンツの正確性等）は両ルールで矛盾させない。

---

## 1. 着手前に「現状把握」と「合意」をする

推測で書き換え始めない。

1. **現状把握**：まず対象ページの **目的・構成・既存コンテンツ** を読んで理解する（何を伝えるページか、どんな導線か、既存の文体・トーン）
2. **合意**：以下を **選択肢形式** でユーザーに確認し、合意してから着手する

| 確認項目 | 例 |
|---|---|
| 更新のゴール | 「(例: 情報追加／訴求強化／古い記述の更新)」のどれか |
| 範囲 | どのセクションを対象にするか／全体か一部か |
| 触ってよい箇所 | 変更可の範囲・触ってはいけない箇所（既存の重要導線等） |
| 素材の有無 | 追加する原稿・画像・数値が手元にあるか／後から差し替えか |

---

## 2. 既存を壊さない（最優先）

- 既存の **レイアウト・構造・動作・既存コンテンツを尊重**する。内容は変えても、**表示崩れや既存導線の破壊を起こさない**
- **意図が不明な既存記述を、理由なく削除・上書きしない**
- **自分が作ったものでない記述**や、**説明と実態が食い違う箇所**を見つけたら、**消す前にユーザーに報告して判断を仰ぐ**（勝手に消さない）

---

## 3. 既存の作りに合わせる

- そのページの **構成・トーン・文体・命名・既存コンポーネント**に揃える
- **新しいデザインパターンや独自概念を持ち込まない**（必要なら先に提案・合意してから）
- 共通部品（ヘッダ／フッタ／パンくず等のテンプレート機構）を使っている箇所は、その作法を維持する

---

## 4. 加筆・充実は段階的に

- **小さく刻んで**追記・改善する。**一度に大きく書き換えない**
- 追加した箇所と既存部分の **整合（重複・矛盾・トーンのズレ）を確認**する
- 同じ情報が既存にあるなら二重化しない。古い記述と新しい記述が食い違う場合は、どちらが正かをユーザーに確認

---

## 5. コンテンツを正確に入れる（最重要）

> この節は `page-build.md`（新規ページ追加）と **共通の考え方**。両ルールで矛盾させない。

- **もっともらしい仮の文章を「事実」として確定させない**
- **数値・仕様・固有情報は一次情報源で裏取りする**（手元になければ推測で埋めず、ユーザーに要求）
- 実際に扱う対象の **“正しい出典”** で確認する
  - (例: 販売しているのが再販品／OEM なら、**再販元の公式情報**に合わせる。製造元や類似品のデータで代用しない)
- **根拠のない数値・主張は載せない**
- **ダミーと確定情報をコメント等で明示的に区別する**（例: `<!-- TODO: 仮値・要確認 -->`）
- 外部リンクは **公式情報を参照**し、新規タブで開く場合は `rel="noopener"` を付ける

---

## 6. 要確認情報は確定しない

依頼元（クライアント）の確認が必要な情報は、**確定で載せない**。

- 対象例：価格・料金、実績・導入数などの成果数値、法的表記（特商法・免責・資格・許認可番号 等）、お客様の声・固有名詞
- 仮置きする場合は **ダミーと明示**し、**残課題として記録**（後述 9）してユーザーに確認を依頼する

---

## 7. 公開中ページ特有の注意（SEO・導線の退行防止）

既存ページは **既に検索評価・流入・導線を持っている**。更新でそれを毀損しない。

- `title`・meta description・**見出し階層（`h1`〜）**・**URL**・**内部リンク（アンカー文言・リンク先）** など、検索評価や導線に影響する要素を **不用意に変えない**
- 変更が必要なら、**影響を明示してユーザーと合意してから**行う（例: URL変更ならリダイレクトの要否、見出し変更なら検索評価への影響）
- 既存の構造化データ・canonical・OGP を壊さない
- 詳しい運用・SEO観点は `seo-operation.md` を参照

---

## 8. 仕上げ前の確認

- 使用言語の **構文／Lint を確認**する（タグ閉じ忘れ・属性ミス等）
- **変更が表示・既存導線を壊していないか**、目視確認をユーザーに依頼する（既存ページなので退行確認が特に重要）
- **ビルド／コンパイルはプロジェクトの運用に従う**：**自動コンパイル環境（[scss-autocompile.md](scss-autocompile.md)）が有効なら** `.scss` 編集でフックが `.css` を自動生成（追加操作不要）。**無ければ**開発者にコンパイルを依頼し、AIは手動でビルドコマンドを叩かない
- **完成後チェックのルールがあれば、そちらへ引き継ぐ**（`check.md`）

---

## 9. 残課題を記録

今回見送った改善・要確認事項は、放置せず **残課題として記録**する（`CLAUDE.md`「残存課題の記録ルール」・`documents/pending-issues.md` 準拠）。

- 段階的に継続する加筆・改善（今回スコープ外にした分）
- 要確認情報（価格・実績・法的表記）の **クライアント確認**
- 意図不明だったため保留した既存記述（消すか残すかの判断待ち）
- SEO・導線に関わる変更で、影響確認が必要なため見送った項目

各項目に「未対応の範囲・対象ファイル・確認すべき相手」を添える。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 現状把握・合意なしに書き換え始める | 既存の意図を壊す。手順 1 が先 |
| ❌ 既存コンテンツを理由なく書き換える | 既存ページの価値・導線を毀損する |
| ❌ 意図不明な既存記述を勝手に削除する | 背景がある可能性。消す前にユーザーに報告 |
| ❌ 一括置換で複数箇所を同時に変更する | 影響範囲が読めず事故の元。小さく刻む |
| ❌ 仮の文章・数値を「事実」として確定する | 誤情報の公開。一次情報で裏取りする |
| ❌ 別ブランド・類似品・製造元のデータを流用する | 対象（例: 再販品／OEM）の正しい出典と食い違う |
| ❌ 根拠のない数値・主張を載せる | 信頼性を損なう。出典を確認してから載せる |
| ❌ 要確認情報（価格・実績・法的表記）を確定で載せる | クライアント確認前。残課題にして確認を仰ぐ |
| ❌ SEO・導線に関わる要素（title・見出し・URL・内部リンク）を不用意に変更する | 検索評価・流入・導線の退行。影響を明示し合意してから |
| ❌ 規約外の独自デザイン・概念を勝手に持ち込む | 既存の作り・Design.md に揃える |
| ❌ AIが手動でビルド／コンパイルコマンドを叩く | コンパイルは自動環境ならフック・無ければ開発者。生成済み `.css` は手編集しない |
| ❌ 残課題を口頭報告だけで済ませ記録しない | 取りこぼし。`documents/pending-issues.md` に残す |

- [ ] 小さく刻み、追加箇所と既存部分の整合（重複・矛盾・トーン）を確認したか？
- [ ] 数値・仕様・固有情報を一次情報源（対象の正しい出典）で裏取りしたか？ダミーと確定を区別したか？
