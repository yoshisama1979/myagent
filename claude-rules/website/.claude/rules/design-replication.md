# デザイン再現ループ（design-to-code）ルール

デザイン（カンプ画像・モックアップ、特に **AI生成のモックアップ**）を渡されて UI を実装・再現するときの作業ループをまとめたルール。

## いつ参照するか（発動条件）

- デザインカンプ／モックアップ（画像）を渡されて、それに沿って UI を実装・再現するとき
- 既存実装を「このデザインに寄せて」と調整を依頼されたとき
- 色・サイズ・余白・位置が「なんとなく合っているが微妙にズレる」状態を詰めたいとき

## このルールが解く問題

AIエージェントがデザイン画像を渡されて UI を実装するとき、つい
**「デザイン画像を目で見て」「自分の出力スクショを目で見て」頭の中で照合し、値を推測** してしまう。
結果、色・サイズ・余白・位置が微妙にズレ、トライ&エラーが非効率になる。

このルールは **推測を実測に変え、実ブラウザ描画でフィードバックを閉じる** ことで再現精度を上げる。
判定のゴールは **ピクセル完全一致ではなく、「現実的な幅で見て同じ階層・印象になっているか」**。

> このルールは「デザインを忠実にコード再現するための周回の組み方」に絞る。色・余白・タイポの確定値そのものの管理は `Design.md`（デザイン規約の単一の真実の源）に委ねる。新規ページの起こし方は `page-build.md`、既存改修は `page-update.md` を併用する。

## 中核原則

### 1. デザインは「意図」、ブランド資産が「正」

- AI生成デザインは **見た目の方向性であって仕様書ではない**。画像の寸法・font-size・余白・色の絶対値は **生成時の偶発値** であり、そこから絶対値を逆算しても「精密に間違える」。
- 画像から拾うのは **相対関係・意図**（レイアウト構成、階層／強弱、色の役割、どの語を強調するか 等）。
- **クライアント提供の正式値（ブランド hex・指定フォント・実素材）が常に優先**。画像と正式値が矛盾したら **正式値を採用** し、その判断を記録する（`Design.md` / `documents/pending-issues.md`）。

### 2. 実ブラウザ描画でループを閉じる

- 実装した成果物を **ヘッドレスブラウザでスクリーンショット** し、それを **画像として読み込んで確認** する。頭の中の想像で済ませない。
- 比較は **参照デザインと同じビューポート幅** で撮る（幅が違うと比率判断が狂う）。

### 3. 「測れるもの」は目視でなく実測する

- 参照画像は **ディスク上のファイルとして用意してもらう**（チャットに貼り付けただけの画像は計測できない）。専用ディレクトリ（例 `documents/design-refs/`）に置く。
- **色の実測**：
  - ベタ塗り → **小ブロックの平均** を取る
  - アンチエイリアスする文字 → **「最も彩度の高い代表ピクセル」** で芯の色を取る（縁のぼけたピクセルに引っ張られない）
- 必要に応じて **領域をクロップ＋拡大** して細部（下線の有無／太さ／位置など）を読む。任意で差分（オーバーレイ）画像も作る。
- **重要**：AI生成デザインから測った **寸法値は、そのまま CSS の確定値として採用しない**（生成時の偶発値を写すだけになる）。寸法の計測そのもの（矩形・余白を測って **相対比・密度・要素の実在** を確認する）は有効だが、**最終値は原則4のとおり「Webとして妥当な設計値」を自分で決める**。実測値をそのまま確定値に使ってよいのは **色のパレット抽出** と **要素が実在するかの検証**。

### 4. サイズ・余白・レスポンシブ値は自分で決め、SSOTに記録する

- 画像の比率／意図を満たす **Webとして妥当な値**（PC/SP のブレークポイント込み）を **エージェントが設計** する。
- 決めた値は **デザイン規約の単一の真実の源（例: `Design.md` / トークン定義）に記録** し、以後は推測し直さない。
- 判定は「ピクセル一致」ではなく **「現実的な幅で見て同じ階層・印象になっているか」**。

### 5. AIデザインのアーティファクトを鵜呑みにしない

- 実在しない装飾を “見えた気がして” 足さない／本来必要な要素を消さない。各要素は **参照画像で実在を検証** してから実装する。
- リフローしないテキスト、破綻した余白、ありえない固定幅などは **そのまま写さず**、Web 向けに是正する。

### 6. ビルド境界（コンパイル等）を前提に周回を組む

- ソース→成果物の変換（例: Sass コンパイル）が **人手や別ツールに委ねられている** 場合、その境界を織り込んだ周回（**修正 → 変換 → スクショ確認**）にする。
- 可能なら **保存時自動変換（watch）** で周回を短縮する提案も含める（プロジェクトの運用に従う。AI が勝手にコンパイルを実行しない方針なら、変換は依頼する）。

## 推奨手順

1. **参照デザインをファイルとして受領** する（専用ディレクトリ、例 `documents/design-refs/`）。無ければ依頼する（チャット貼付のままでは計測できない旨を伝える）。
2. **正式なブランド資産（色 hex・フォント・実画像）の有無を確認** する。あれば **最優先で採用**。`Design.md` があればそれを先に読む。
3. **実装の現状を、参照と同一幅でスクショ** し、画像として確認する。
4. **参照から色を実測**（ベタ塗り＝ブロック平均／文字＝最大彩度）し、各要素の **実在を検証**。パレットを確定する。
5. **サイズ・余白・レスポンシブ値を設計** し、SSOT（`Design.md` / トークン）に記録する。判断（特に **画像と正式値の食い違い**）を明記する。
6. **修正 →（必要なら）変換 → スクショ → 比較** を、**効果の高い差分から** 周回する。ピクセル完全一致は目標にしない。

## ツール（例。本文はスタック非依存）

実装手段はプロジェクトの環境に合わせる。以下は一例。

| 用途 | 例 |
|------|----|
| スクリーンショット | ヘッドレスブラウザ（例: Chrome / Edge `--headless=new --screenshot --window-size=W,H`）。**参照と同じ幅 W** を指定する。出力はスクラッチ用ディレクトリ（例 `.preview/`）へ |
| 画像処理（クロップ／拡大／ピクセル取得） | 利用可能な画像ライブラリ（例: Windows なら .NET `System.Drawing`、他環境なら Pillow / ImageMagick） |

注意例：
- ヘッドレス起動が **プロファイル競合で失敗** する場合は、使い捨ての `user-data-dir` を指定する。
- **PowerShell / `System.Drawing`** で実装する場合、**PowerShell は変数の大小文字を区別しない** ため `$r` と `$R` を同時に使うと壊れる。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 画像を目視で照合して値を推測する | 「精密に間違える」元。測れるものは実測に置き換える |
| ❌ AI生成デザインの寸法・font-size を測って **そのまま CSS の確定値にする** | 生成時の偶発値。計測は相対比・実在確認に使い、最終値は設計して決める |
| ❌ チャット貼付画像のまま計測しようとする | 計測できない。ファイル保存を依頼する |
| ❌ ブランド正式値より画像の見た目を優先する | 正式値が「正」。矛盾したら正式値を採用し記録する |
| ❌ 実在しない装飾を足す／参照で未検証のまま実装する | アーティファクトの混入。要素は実在を検証してから |
| ❌ ピクセル完全一致を最終ゴールにする | 目的は現実幅での **印象・階層の一致** |
| ❌ 比較スクショを参照と違う幅で撮る | 比率判断が狂う。同一ビューポート幅で撮る |
| ❌ 決めた値を記録せず毎回推測し直す | ブレの元。SSOT（`Design.md` / トークン）に記録する |

## 自己チェック（再現の前後で確認）

- [ ] 参照デザインを **ファイルとして** 受領したか？（チャット貼付のままにしていないか）
- [ ] ブランド正式値（色 hex・フォント・実素材）の有無を確認し、あれば最優先で採用したか？
- [ ] 比較スクショを **参照と同一幅** で撮り、画像として読み込んで確認したか？
- [ ] 色は実測したか？（ベタ塗り＝ブロック平均／文字＝最大彩度）
- [ ] AI生成デザインの寸法を **そのまま CSS の確定値に写経していない** か？（測ってもよいが、最終値は意図から自分で設計したか）
- [ ] 実在しない装飾を足していないか？ 各要素を参照で実在検証したか？
- [ ] 決めたサイズ・余白・レスポンシブ値を SSOT（`Design.md` / トークン）に記録したか？
- [ ] 画像と正式値の食い違いの判断を記録したか？（要確認は `documents/pending-issues.md` へ）
- [ ] ピクセル一致でなく「現実幅での印象・階層の一致」で判定したか？
