# ウェブサイトリファクタリングルール

ウェブサイトのコード(HTML / PHP / SCSS / CSS / JavaScript / 画像)をリファクタリングする際のガイドライン。

## 基本方針

- リファクタリングは **「動作を変えず、可読性・保守性・パフォーマンスを上げる」** ための作業
- **見た目とDOMに副作用が出ないこと** が最優先。崩れたら本末転倒
- 一度に大きく変えず、**小さく刻んで** 段階的に進める
- 触る機会があれば寄せていく漸進的方針(一括置換しない)
- リファクタリング後は **必ず該当ページをブラウザで目視確認** する(または、ユーザーに確認を依頼)

---

## いつリファクタリングを検討するか

| タイミング | 判断基準 |
|---|---|
| コミット直後 | 直前の差分にコピペ・重複・マジックナンバーが残っていないか |
| 同じ修正を2回以上した時 | 共通化できるサインかもしれない |
| 新規ページ追加前 | 既存コンポーネントの揺らぎを揃えると新規が綺麗に書ける |
| Design.md と乖離を見つけた時 | 触る機会に寄せていく |

逆に**やらない方がよい**ケース:

- 動いている古いページ(`index_20170802.html` 等のレガシー)に手を入れる時は最小修正に留める
- 公開直前・キャンペーン期間中

## カテゴリ別チェックリスト

### A. SCSS / CSS

#### 共通化・整理
- [ ] **重複したカラー値**: `#6CAC3D` 等が直書きされていたら `_color.scss` 由来の変数に置換できないか
- [ ] **重複したセレクタブロック**: 同じプロパティ群が複数箇所にあれば `@mixin` または共通クラスに
- [ ] **マジックナンバー**: `margin: 37px` のような中途半端な値は、グリッド単位(10/20/30/40...)に揃える
- [ ] **未使用クラス・未使用変数**: HTML/PHP から参照されていない SCSS ルールは削除
- [ ] **ネスト深さ**: 4階層以上はフラット化を検討
- [ ] **ベンダープレフィックス**: 古いブラウザ向けの `-moz-` `-webkit-` で不要なものは削除

#### Design.md との整合
- [ ] 新規追加された色・余白・フォントサイズが Design.md に登録済みか
- [ ] Design.md に無い値を使っているなら、Design.md を先に更新するか、近い既存値に寄せる
- [ ] `1rem = 1px` 換算ルールに沿っているか(本サイト特有)

#### 触ってはいけないもの
- `.css` ファイルを直接編集しない(`.scss` を直す)
- レガシーCSS(`top_org.css` `price_org.css` `top_20170802.css` 等)は基本そのまま

### B. HTML / PHP

#### セマンティック
- [ ] `<div>` の入れ子を `<header>` `<nav>` `<main>` `<section>` `<article>` `<footer>` に置き換えられないか
- [ ] 見出し階層(`<h1>` → `<h2>` → `<h3>`)が論理的に並んでいるか
- [ ] 装飾目的の見出しタグになっていないか(見た目だけならクラスで)

#### 共通化
- [ ] ヘッダー・フッター・パンくず等の **PHP テンプレート (`$m_tpl->...`)** を使っているか。直書きされていれば置換
- [ ] 同じカードレイアウト・同じボタン群を複数ページで直書きしているなら、`include` で部品化検討

#### クリーンアップ
- [ ] コメントアウトされた古いコード(`<!-- ... -->`)を削除
- [ ] デバッグ用の `var_dump` `print_r` `console.log` が残っていないか
- [ ] `alt` 属性が空または冗長(`alt="画像"` 等)になっていないか

### C. JavaScript

- [ ] 同じセレクタを複数回 `$(...)` していたら変数化
- [ ] グローバル変数を新規に増やしていないか
- [ ] **jQuery で書かれた古い箇所を Vanilla JS に書き換えない**(リスクに見合わない)。新規追加分のみ Vanilla を推奨
- [ ] 外部CDNの読み込みで、未使用のライブラリがあれば削除

### D. 画像

- [ ] ファイルサイズが 500KB を超えていれば圧縮を検討(写真は WebP / JPG、ロゴはSVG)
- [ ] 表示サイズ200pxの画像に2000px幅の原寸を使っていないか
- [ ] 同じ画像を別ファイル名で複数持っていないか
- [ ] 未参照の画像ファイル(`images/` 配下で HTML/CSS から呼ばれていない)を検出して削除候補に

### E. ファイル構成

- [ ] バックアップファイル(`*.BAK` `*_org.html` `index_2017*.html` 等)が肥大化していないか
  → **削除はユーザーに確認してから**。古いABテスト等の意図がある可能性
- [ ] `custom.css` が肥大化していたら SCSS 側に逆移植

## リファクタリングの進め方

### Step 1. 対象を限定する

- `git diff --staged` または `git diff HEAD~1` で**今回の差分**に絞る
- 全ファイルを一気にリファクタしない

### Step 2. 提案を列挙する

リファクタ候補を見つけたら、**実行前にユーザーに提案** する:

```
## リファクタリング提案

### 1. [カテゴリ] [概要]
- 場所: [ファイル:行]
- 現状: [問題のあるコード]
- 提案: [どう変えるか]
- 影響範囲: [見た目への影響 / 他ページへの影響]
- リスク: [低 / 中 / 高]

### 2. ...
```

### Step 3. 承認後に着手

- ユーザーが OK と言った項目のみ実施
- 1つずつ small commit で進められるように分ける(まとめて1コミットにしない)

### Step 4. 動作確認

- SCSS を触ったら **「開発者側で `sass.bat` を実行してコンパイルしてください」** と伝える
- コンパイル後、該当ページの目視確認をユーザーに依頼
- HTML/PHP を触った場合は、影響を受けるページ全部を列挙して確認依頼

---

## やってはいけないこと

- ❌ 動いているコードを「綺麗にしたいから」だけの理由で書き換えない(理由を必ず述べる)
- ❌ 一括置換(`sed -i` / `Replace All`)で複数ファイルを同時に変更しない
- ❌ レガシーページ(`*_org.html` `index_2017*.html`)を勝手にリファクタしない
- ❌ コンパイル済み `.css` を直接編集しない
- ❌ AI 側で `sass.bat` を実行しない(コンパイルは開発者)
- ❌ Design.md に無い概念を「ついでに」追加しない
- ❌ ライブラリのアップデート(jQuery 1.x → 3.x 等)を勝手にやらない

---

## 完了報告

リファクタリングが完了したら以下を報告する。

```
## リファクタリング完了

### 変更内容
- [カテゴリ] [何を変えたか]

### 影響範囲
- 表示が変わる可能性のあるページ: [ページ名]
- 動作が変わる可能性のある機能: [機能名 / なし]

### 確認のお願い
- [ ] 開発者側で `sass.bat` を実行(SCSS変更時)
- [ ] 該当ページの目視確認: [URL or パス]
- [ ] [その他確認事項]
```
