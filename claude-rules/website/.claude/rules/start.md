# プロジェクト開始時ルール

ウェブサイト制作プロジェクトの **開始時** に確認・準備すべき事項をまとめたルール。
新しいプロジェクトに入った時、または新規ページ・UI 作成の依頼を受けた時に参照する。

## このルールの位置付け

- 「制作に入る前」のゲート。ここを飛ばすと後でデザイン・構造が場当たり的に増えて揃わなくなる
- 各セクションは独立しているので、状況に応じて必要なものだけ実施してよい
- 一度確認した項目は **同セッション内では再確認しない**(煩わしさ回避)

## 1. Design.md の準備

デザインルールの単一の真実の源(Single Source of Truth)である `Design.md` をプロジェクトルートに準備する。
これが無い状態で UI を作り始めると、色・余白・タイポが場当たり的に増えていき後で揃わなくなる。

### 1.1 基本方針

- `Design.md` は **AIコーディングエージェントと人間が共有するデザイン規約**
- 実装に入る前に必ず Design.md の有無を確認し、無ければユーザーに作成を提案する
- **勝手に作らない**。必ずユーザーの YES を得てから着手する

### 1.2 実施手順

#### Step 1. 存在確認

プロジェクトルートに `Design.md`(または `design.md` / `DESIGN.md`)が存在するか確認する。

- **存在する** → 内容を読み込み、以降の作業はそのルールに従う。本セクションはここで終了
- **存在しない** → Step 2 へ

#### Step 2. ユーザーへの確認

以下のように質問する(文言は状況に合わせて調整可)。

> このプロジェクトに `Design.md`(デザインルールの単一の真実の源)が見つかりませんでした。
> AIと人間で色・フォント・余白などの判断を揃えるために、最初に作成しておくことをおすすめします。
>
> 作成しますか?
> - **はい、既存ファイルから抽出して作成** (既存の HTML/CSS/SCSS/デザインカンプ等がある場合)
> - **はい、ゼロからヒアリングして作成** (新規プロジェクトで既存資産が無い場合)
> - **いいえ、今は作らない** (後で必要になったら相談)

回答に応じて Step 3-A / 3-B / 3-C へ。

#### Step 3-A. 既存ファイルから抽出して作成

プロジェクト内に以下のような既存資産があれば、それを根拠に Design.md のドラフトを作る。

| 参照すべき資産 | 抽出する情報 |
|---|---|
| `*.scss` / `_variables.scss` / `_color.scss` 等 | カラートークン・フォント変数・mixin |
| `*.css`(コンパイル済み) | 実際に使われている色・余白の頻度 |
| `*.html` / `*.php` | 見出し階層・コンポーネント構造・既存のクラス命名 |
| `tailwind.config.js` / `theme.json` 等 | 既存トークン定義 |
| Figma / XD / PSD 等のファイル名 | デザインカンプの有無(中身は別途ヒアリング) |
| `package.json` | フレームワーク・ライブラリ依存 |
| 既存の `images/` | アイコンセット・画像形式の傾向 |

**手順:**
1. `git status` で対象プロジェクトの規模感を把握
2. 上記資産を `Read` / `Glob` / `Grep` で機械的に抽出
3. 揺らぎ(同じ役割で異なる値が散在する場合 — 例: 緑系の色が6種類ある等)は **「現状を1〜4個に集約した推奨値」+「TODO に統合タスクとして記載」** の形でまとめる
4. ドラフトをユーザーに提示し、合意を得てから書き込む

#### Step 3-B. ゼロからヒアリングして作成

既存資産が無い、または極小の場合は、以下の最小限の項目を順番にヒアリングする。長くなりすぎないよう **1ターンで2〜3項目ずつ** 質問する。

##### 必須ヒアリング項目

1. **サイトの目的・対象ユーザー**
   - 何を伝えるサイトか?(コーポレート / LP / サービス紹介 / ブランド)
   - 誰に向けたサイトか?
2. **トンマナ(雰囲気)**
   - 例: 「清潔感がある」「親しみやすい」「高級感」「先進的」「ポップ」
3. **ブランドカラー**
   - メインカラー(1色)
   - サブ/アクセントカラー(0〜2色)
   - 既にロゴがある場合はその色を起点に
4. **フォント方針**
   - 日本語: ゴシック系 / 明朝系 / 両方
   - 英字: 指定あれば
   - Google Fonts 等の使用可否
5. **レイアウト幅**
   - PC時のコンテンツ最大幅(例: 1000 / 1200 / 1280px)
   - フルワイドのセクションを使うか
6. **ブレークポイント**
   - モバイル / タブレット / PC の3段階か、モバイル / PC の2段階か
7. **既存ライブラリ・フレームワーク**
   - Bootstrap / Tailwind / 素のCSS / Sass
   - jQuery / Vanilla JS / Vue / React

##### 任意ヒアリング項目(必要に応じて)

- ダークモード対応の予定
- アクセシビリティの目標水準(WCAG AA / AAA)
- アニメーション・モーションの方針
- アイコンライブラリ(FontAwesome / Material Icons / 独自SVG)
- 多言語対応の予定

ヒアリング結果をまとめて Design.md ドラフトを作成し、ユーザーに確認してもらってから書き込む。

#### Step 3-C. 今は作らない場合

ユーザーが「今は作らない」を選んだ場合:
- そのまま作業を続ける
- ただし、**色や余白を新規に決める場面が来たら都度ユーザーに確認**する(勝手に決めない)
- セッションの最後に「やはり Design.md を作っておきますか?」と提案してもよい

### 1.3 Design.md に含めるべき項目(テンプレート)

含めるべきセクション構成は [design.md](design.md) の「2. Design.md が持つべきセクション」を参照する(10セクション。汎用ルール側に一本化)。
実際に値を埋めた構成は記入例 [Design.sample.md](../../Design.sample.md) を参考にする。

特に **Section 9 「Do / Don't」を必ず入れる**。これが無いと AI が流行のデザイントレンドを勝手に持ち込みやすい。
そのプロジェクト固有のガード(換算ルール・固定幅・テンプレート機構の前提 等)は、その Design.md の §9 に書く。

### 1.4 完了報告

Design.md を作成したら以下を報告する。

```
## Design.md 作成完了

- ファイル: ./Design.md
- 抽出/ヒアリング元: [既存SCSS / ヒアリング / Figma 等]
- 主な決定事項: [ブランドカラー、レイアウト幅、ブレークポイント等のサマリ]
- TODO に残した課題: [N件]

今後の UI 実装はこの Design.md を Single Source of Truth として進めます。
ルールを変更したい場合は Design.md を更新してから実装してください。
```

### 1.5 注意事項

- Design.md と既存のコードが食い違う場合、**Design.md に合わせて既存コードを直すのが原則**。ただし一括置換はせず、触る機会があれば寄せていく漸進的方針で進める
- Design.md を作った後は、新規UI実装の前に必ず Design.md を読む(`CLAUDE.md` のルールファイル参照表に従う)
- このルール自身に過不足があれば、ユーザーに改善提案する(CLAUDE.md「2. ルールファイル自体の改善も提案する」に従う)
