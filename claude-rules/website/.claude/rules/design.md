# デザイン規約の運用ルール（各プロジェクトの Design.md を SSOT とする）

> このルールファイルは **汎用（`design.md`）**。各プロジェクト固有の値を書く **`Design.md`（大文字始まり）** とは別物。本ファイルは「Design.md をどう扱うか」を定義し、固有値は持たない。

色・フォント・余白・コンポーネント等の **デザイン判断を、プロジェクト共通の規約に揃える** ためのルール。
具体値そのものは各プロジェクトの `Design.md`（デザイン規約の単一の真実の源）に置き、本ファイルは
**「Design.md をどう扱うか」「どのプロジェクトでも守る普遍的なデザイン規律」** を定義する（プロジェクト非依存）。

## いつ参照するか

- 色・フォント・余白・コンポーネントなど **デザインに関わる判断・実装** をするとき
- `Design.md` の **構成を決める／見直す** とき
- 「規約に無い色・装飾・概念を足したくなった」とき（足す前にここを読む）

## 他ルールとの関係（役割分担）

| 担当 | ファイル |
|------|---------|
| Design.md の **有無確認と作成手順**（開始時ゲート） | `start.md` §1 |
| **具体値そのもの**（色 hex・フォント・幅・mixin 等） | 各プロジェクトの `Design.md`（本リポジトリの記入例 → [Design.sample.md](../../Design.sample.md)） |
| カンプ／モックアップを **コードで再現する周回** | `design-replication.md` |
| SCSS の **コンパイル運用・ビルド境界** | `scss-autocompile.md` / `refactor.md` |

本ファイルは「**デザイン規約として何を守るか**」に絞る。重複定義はせず、上記へ委ねる。

---

## 1. Design.md は単一の真実の源（SSOT）

- 色・フォント・余白・コンポーネント仕様は、**SCSS変数やコード実装よりも `Design.md` を優先**する。
- `Design.md` とコードが食い違う場合 → **`Design.md` に合わせてコードを直す**（漸進的に。一括置換はしない＝触る機会に寄せていく）。
- `Design.md` に **無い概念**（新しい色トークン・ブレークポイント・コンポーネント等）を **勝手に増やさない**。増やすなら **先に `Design.md` を更新する提案** をしてから実装する。
- **UI 実装の前に必ず `Design.md` を読む**。無ければ `start.md` §1 の手順で作成する（勝手に作らず、ユーザーの合意を得てから）。

---

## 2. Design.md が持つべきセクション（汎用構成）

新規に `Design.md` を起こすとき・見直すときは、最低限このセクションを揃える（記入例は [Design.sample.md](../../Design.sample.md)）。

| セクション | 内容 |
|---|---|
| **1. Design Principles** | サイトの設計思想（2〜4項目）。以降の判断の拠り所 |
| **2. Color Tokens** | ブランドカラー・アクセント・セマンティック（警告/成功）・コントラスト比 |
| **3. Typography** | フォントファミリ・サイズスケール・行間・ウェイト |
| **4. Spacing & Layout** | グリッド単位・コンテナ幅・ブレークポイント |
| **5. Components** | ボタン・リンク・リスト・フォーム・カード等の基本仕様 |
| **6. Iconography & Imagery** | アイコンセット・画像形式・角丸/影のルール |
| **7. Motion** | トランジション duration / easing |
| **8. ファイル構成 / ビルド** | SCSS のディレクトリ構成・コンパイル方法（`scss-autocompile.md` 参照） |
| **9. Do / Don't** | AI が暴走しないためのガード（**最重要**） |
| **10. TODO** | 現状の揺らぎ・将来の課題 |

特に **§9「Do / Don't」を必ず入れる**。これが無いと AI が流行のデザイントレンドを勝手に持ち込みやすい。
各プロジェクト固有のガード（換算ルール・固定幅・テンプレート機構の前提 等）は、その `Design.md` の §9 に書く。

---

## 3. 普遍的な Do / Don't（プロジェクト非依存）

どのプロジェクトでも効く原則。**各プロジェクト固有の値・方式は `Design.md` 側に書く**（ここには書かない）。

### ✅ Do

- 色・余白・コンポーネントを足す前に、**既存の変数・ユーティリティクラスをまず探す**。
- 色を増やしたくなったら、**規約の近い既存色に寄せる**（むやみに新色を足さない）。
- CSS の修正は **ソース（`.scss` 等）側のみ** を編集する（生成物は触らない → §4）。

### ❌ Don't

| NG | なぜ |
|----|------|
| ❌ 流行装飾（グラデーション・ネオモーフィズム・ガラスモーフィズム等）を規約外で持ち込む | 規約のトーンを壊す。必要なら先に `Design.md` を更新して合意 |
| ❌ ダークモード等の新モードを規約に無いのに勝手に追加する | 想定外の表示・保守コスト。先に方針を合意 |
| ❌ ユーティリティフレームワーク（Tailwind/Bootstrap 等）を勝手に新規導入する | 既存の書き方と二重化。導入は方針決定が先 |
| ❌ CSS変数（`--var`）・新トークンを規約の方式を無視して増やす | 命名・管理が分裂する。`Design.md` の方式に従う |
| ❌ `!important` を新規で書く | 詳細度（specificity）で対処する。乱用は破綻の元 |
| ❌ グリッド・レイアウト規約（基本単位・コンテナ幅・ブレークポイント）を崩す | 全ページの整合が崩れる。変えるなら `Design.md` を更新して合意 |
| ❌ 見出しタグを「見た目だけ」の理由で使う／装飾用に乱用する | 文書構造・SEO・アクセシビリティを損なう。見た目はクラスで |

---

## 4. ビルド境界（生成物を手編集しない）

- **コンパイル済みの生成物（`.css` 等）を直接編集しない**。必ずソース（`.scss` 等）を直す。
- コンパイルの実行は **プロジェクトの運用に従う**：自動コンパイル環境があれば AI の編集にフックが反応して生成、無ければ開発者が行う。**AI が手動でビルド／デプロイ系コマンドを叩く話ではない**（詳細 → `scss-autocompile.md` / `refactor.md`）。

---

## 5. アクセシビリティの最低ライン（汎用）

- 本文と背景の **コントラスト比 4.5:1 以上**（大きな文字は 3:1 以上）を維持する。
- **フォーカスリングを消さない**（`outline: none` 単独禁止。消すなら代替の可視フォーカスを置く）。
- 画像の `alt`、フォームの `label` 紐付けなど基本は `check.md` のチェック項目に従う。

---

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ `Design.md` を読まずに色・余白・フォントを決める | SSOT 無視。判断がブレる |
| ❌ 規約に無い概念を、先に `Design.md` を更新せず実装する | 規約と実装が乖離する。更新提案が先 |
| ❌ 普遍ルール（§3）にプロジェクト固有の値を書き込む | 汎用ルールが特定プロジェクトに汚染される。固有値は `Design.md` へ |
| ❌ 生成済み CSS を手編集する／AI が手動でビルドを叩く | §4 違反。ソースを直し、コンパイルは自動環境／開発者に委ねる |
