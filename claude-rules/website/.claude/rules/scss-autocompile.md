# SCSS リアルタイム自動コンパイル環境のセットアップ

AIエージェントが `.scss` を編集した **瞬間に `.css` を自動生成** する環境の作り方をまとめたルール。
これがあると「SCSS修正 →（自動でCSS生成）→ そのまま表示確認」まで一気に回せる。

## いつ使うか

- AIが **頻繁に SCSS を編集する制作**で、毎回の手動コンパイル（や「開発者にコンパイル依頼」）を省きたいとき
- デザイン再現ループ（`design-replication.md`）のように **修正 → 生成 → スクショ確認** を高速に周回したいとき

> この環境が無いプロジェクトでは、従来どおり **「`.scss` を編集 → 開発者にコンパイル（保存／ビルド）を依頼」** で運用する（各ルールの「フォールバック」記述参照）。

## 前提：なぜ「保存イベント」ではなく「AIの編集」に紐づくフックが要るか

- VS Code 拡張などの保存時コンパイラ（例: **Live Sass Compiler**）は **エディタの“保存イベント”でしか動かない**。
- AIエージェントは **プログラム的にファイルを書き込む**（Write/Edit）ため、保存イベントが発火せず、拡張ではコンパイルされない。
- よって **「保存」ではなく「AIのツール実行」に紐づくフック** でコンパイルする必要がある。
- なお、これは **AI が任意のビルドコマンドを叩く**話ではない。**ハーネスのフックが編集に反応して自動ビルドする**だけで、AI が勝手にデプロイ系コマンドを実行するものではない（区別する）。

## セットアップ手順（再現レシピ）

> 参照実装は **Windows + VS Code + standalone dart-sass + Claude Code の PostToolUse フック**。本文の主旨はスタック非依存。具体ツール名は「例」。

### 1. コンパイラはエディタ拡張に依存しない単体 CLI を使う

- 例：**standalone dart-sass**（GitHub の zip 1個。npm 不要）。
- **なぜ npm を避けるか**：`npm install` はネットワーク事情で極端に遅い／失敗しうる環境がある（HTTPS 取得が1件ごとに数十秒スタックする等）。**zip 1個で済む standalone を使えば回避できる**。
- **Windows 注意**：Node の `execFileSync` は `.bat` を直接起動できない（EINVAL）。同梱の実行ファイル（例 `dart.exe` + `sass.snapshot`）を **直接叩く**。

### 2. AIエージェントのハーネスのフックで起動する

- 例：Claude Code の **PostToolUse フック**（`Write|Edit|MultiEdit` にマッチ）→ コンパイル用スクリプトを実行。
- スクリプトの要件：
  - **編集ファイルが対象プロジェクト内の `.scss` のときだけ** コンパイルする（無関係な編集で走らせない）。
  - **各プロジェクトの既存コンパイル設定を読み取って出力先を決める**（例: `.vscode/settings.json` の Live Sass 設定 = 入力 glob／出力先／format）。これによりプロジェクトごとの個別設定なしに汎用で効く。
  - **失敗時はフックを非ゼロ終了させ、stderr にエラーを出す**。さらに **ハーネス側でその出力が AI に見える設定** にする（フック種別・設定で挙動が変わるため）。これで SCSS の構文エラーに AI が気付ける。

### 3. フック／スクリプトはユーザーレベルに1回置けば全プロジェクトで有効

- フックとスクリプトは **ホーム配下（例 `~/.claude/`）に1回** 置く。
- コンパイラも **マシンに1回** 入れるだけ。以後どのプロジェクトでも自動で効く。

### 4. ベンダープレフィックスの方針を一本化する

- standalone dart-sass には **autoprefixer が無い**。
- フックとエディタ保存の **二重コンパイラでプレフィックスが付いたり消えたりすると差分が暴れる**。これを防ぐため、まず **どちらか一方の生成系に統一** する。
- **プレフィックス無しで揃える場合**：エディタ側の autoprefix も無効化する。**対象ブラウザ次第**で、モダンブラウザ限定ならプレフィックス無しで揃えられることが多い。
- **プレフィックスが必要な場合**（古いブラウザ対象・既存方針がprefix前提）：**PostCSS/autoprefixer 込みの生成系を1本に統一**する（standalone dart-sass には autoprefixer が無い点に注意）。
- いずれにせよ **コミット用 CSS は片方のコンパイラに統一**すること（二系統で揃えないと差分が暴れる）。

### 5. AIが触れない設定ファイルがある点に注意

- 権限ルールを含むグローバル設定（例 `~/.claude/settings.json`）は **AI が自動編集できないことがある**。
- その場合は **「ユーザーが1回だけ手で貼り付ける」手順** として明示する（AIは内容を提示し、適用はユーザーが行う）。

## 自動コンパイル環境が有効なときの運用（各ルール共通）

- AIは **`.scss` を編集するだけでよい**。`.css` はフックが自動生成するので **「コンパイル依頼」は不要**。
- AIは必要なら **生成結果（スクショ／差分）で確認** する。
- **`.css` を直接編集しないのは不変**（`.scss` を直す）。自動生成された `.css` を手で書き換えると、次の生成で消える／差分が暴れる。
- Design.md 等の **Single Source of Truth 優先・変数/mixin の活用** も不変。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 自動環境が無いのにコンパイル依頼を省略する | `.css` が古いまま。フォールバック時は開発者にコンパイル（保存／ビルド）を依頼する |
| ❌ 2系統コンパイラでプレフィックス方針を揃えない | 差分が暴れる。CSS生成を片方に統一し、対象ブラウザ次第で「prefix無し」か「autoprefixer込み1本」に揃える |
| ❌ 生成済み `.css` を手編集する | 次の自動生成で消える／差分が暴れる。`.scss` を直す |
| ❌ 「自動ビルド」を口実に AI がデプロイ系コマンドを叩く | 自動化されるのは編集に反応したコンパイルのみ。デプロイ・公開は別判断 |
| ❌ AIが編集できないグローバル設定を黙って前提化する | 適用されず動かない。ユーザーが1回手で貼る手順として明示する |

## 自己チェック（環境構築・運用の前に）

- [ ] コンパイラは **エディタ拡張非依存の単体 CLI**（例 standalone dart-sass）か？
- [ ] フックは **AIの編集（Write/Edit）に反応** するか？（保存イベント頼みになっていないか）
- [ ] スクリプトは **対象プロジェクトの `.scss` だけ** をコンパイルし、**既存設定から出力先** を決めているか？
- [ ] 失敗時に **非ゼロ終了＋stderr** でエラーを出し、**ハーネス側でAIに見える設定** にしたか？
- [ ] **プレフィックス方針を一本化** したか？（二系統で揃えたか／片方に統一したか）
- [ ] AIが触れない設定は **ユーザーが手で貼る手順** として明示したか？
- [ ] 環境が無いプロジェクトでは **フォールバック（開発者コンパイル依頼）** に切り替えられるか？
