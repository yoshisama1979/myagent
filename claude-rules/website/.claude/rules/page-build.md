# 新規ページ作成ルール

ウェブサイトに **新しいページ（画面）を追加するとき** の実装手順をまとめたルール。

## このルールの位置付け

- 新規ページ／画面の追加依頼を受けたときに参照する
- 設計準備ルール（あれば `start.md` 等）と完成後チェックルール（あれば `check.md` 等）の **“間”＝実装フェーズ** を担う
- 大まかな流れ：**着手前合意 → 配置決定 → 雛形から作成 → 骨組み → コンテンツ確定 → 仕上げ確認 → 残課題記録**

> このルールは「新しくページを追加するときの手順」に絞る。ページ追加に限らない一般論（汎用的なコーディング規約・リファクタリング方針等）は `refactor.md` / `check.md` 側に委ねる。

---

## 1. 着手前に合意する

推測で進めない。以下を **選択肢形式** でユーザーに確認し、合意してから着手する。

| 確認項目 | 例 |
|---|---|
| 雛形にする既存ページ | 「(例: サービス紹介ページ) を下敷きにしますか／ゼロから作りますか」 |
| 配置場所 | どのディレクトリ・階層に置くか |
| 素材の有無 | 原稿・画像・ロゴ・実績数値などが手元にあるか／後から差し替えるか |
| URL・ルーティング | 公開パス（URL）、ナビゲーションに載せるか、内部リンク元 |

不明点が複数あれば箇条書きで、答えやすい形にまとめて確認する。

---

## 2. 配置を決める

- 既存の **命名規則・ディレクトリ規則に揃える**（新しい流儀を持ち込まない）
- 理想的な構成と既存の構成が食い違う場合：
  - **新規ページは最適な配置にする**
  - **既存ページの移設は別課題として切り出す**（今回のページ追加に巻き込まない）。残課題として記録（後述 9）

---

## 3. 既存ページを雛形に作る

- 似た既存ページを下敷きにする。**複数ページの良い部分を取り入れる**（ヘッダ／フッタ／導線の優れた箇所を借りる）
- ヘッダー・フッター・パンくず等は **共通コンポーネント／テンプレート機構を使う**（例: インクルードやテンプレート部品）。同じ内容の **直書き・重複を避ける**
- 既存テンプレートに無い独自の流儀を新規ページだけに持ち込まない

---

## 4. 骨組みから作る

- まず **骨組み（構造）** を作る：ダミー文言・プレースホルダ素材・`TODO` コメントで、セクションの並びと階層を確定させる
- スタイルは **近い既存ページのものを暫定流用してよい**。ただし構成が固まったら **専用ファイルへ分離** する（暫定流用のまま放置しない → 残課題に記録）
- **ビルド／コンパイルはプロジェクトの運用（担当者）に従い、勝手に実行しない**。コンパイルが要る変更をしたら、その旨をユーザーに伝える

---

## 5. コンテンツを正確に入れる（最重要）

骨組みのダミーを実コンテンツに置き換える段階。**ここで“もっともらしい嘘”を作らない。**

- **もっともらしい仮の文章を「事実」として確定させない**
- **数値・仕様・固有情報は一次情報源で裏取りする**（手元になければ推測で埋めず、ユーザーに要求）
- 実際に扱う対象の **“正しい出典”** で確認する
  - (例: 販売しているのが再販品／OEM なら、**再販元の公式情報**に合わせる。製造元や類似品のデータで代用しない)
- **根拠のない数値・主張は載せない**。載せるなら出典を確認する
- **ダミーと確定情報をコメント等で明示的に区別する**（例: `<!-- TODO: 仮値・要確認 -->`）
- 外部リンクは **公式情報を参照**し、新規タブで開く場合は `rel="noopener"` を付ける

---

## 6. 要確認情報は確定しない

依頼元（クライアント）の確認が必要な情報は、**確定で載せない**。

- 対象例：価格・料金、実績・導入数などの成果数値、法的表記（特商法・免責・資格・許認可番号 等）、お客様の声・固有名詞
- 仮置きする場合は **ダミーと明示** し、**残課題として記録**（後述 9）してユーザーに確認を依頼する

---

## 7. 仕上げ前の確認

- 使用している言語の **構文／Lint を確認**する（HTML/CSS/JS/テンプレート言語の文法エラー・タグ閉じ忘れ等）
- リンク先・画像パスが解決するか、プレースホルダ（`#` 等）が残っていないかを確認
- **完成後チェックのルールがあれば、そちらへ引き継ぐ**（`check.md` の画像／アクセシビリティ／レスポンシブ／SEO 等のゲートに通す）

---

## 8. 残課題を記録

その場でやり切らなかった項目は、放置せず **残課題として記録**する（`CLAUDE.md`「残存課題の記録ルール」・`documents/pending-issues.md` 準拠）。

典型的な残課題：

- 暫定流用したスタイルの **専用ファイルへの分離**
- **ナビゲーション／サイトマップ／内部リンクへの登録**
- プレースホルダ素材・ダミー文言の **実素材への差し替え**
- 要確認情報（価格・実績・法的表記）の **クライアント確認**
- 別課題に切り出した **既存ページの移設**（手順 2）

各項目に「未対応の範囲・対象ファイル・確認すべき相手」を添える。

---

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 雛形・配置・URL を確認せず着手する | 手戻りの元。手順 1 の合意が先 |
| ❌ 仮の文章・数値を「事実」として確定する | 誤情報の公開。一次情報で裏取りする |
| ❌ 別ブランド・類似品・製造元のデータを流用する | 対象（例: 再販品／OEM）の正しい出典と食い違う |
| ❌ 根拠のない数値・主張を載せる | 信頼性を損なう。出典を確認してから載せる |
| ❌ 要確認情報（価格・実績・法的表記）を確定で載せる | クライアント確認前。残課題にして確認を仰ぐ |
| ❌ ダミーと確定情報を区別せず混在させる | 後で「未確認なのに確定」になり判断を誤らせる |
| ❌ 共通部品を使わず同じ内容を直書きする | 重複・更新漏れの元。テンプレート機構を使う |
| ❌ 規約外の独自概念（色・命名・構造）を勝手に増やす | 既存の規則・Design.md に揃える |
| ❌ ビルド／コンパイルを勝手に実行する | プロジェクトの運用（担当者）に従う |
| ❌ 残課題を口頭報告だけで済ませ記録しない | 取りこぼし。`documents/pending-issues.md` に残す |

## 自己チェック（仕上げ前に確認）

- [ ] 雛形・配置・URL・素材の有無をユーザーと合意したか？
- [ ] 既存の命名・ディレクトリ規則に揃えたか？（理想と既存の食い違いは別課題に切り出したか）
- [ ] 共通コンポーネント／テンプレート機構を使い、直書き重複を避けたか？
- [ ] 数値・仕様・固有情報を一次情報源で裏取りしたか？（対象の正しい出典で確認したか）
- [ ] ダミーと確定情報をコメント等で区別したか？
- [ ] 要確認情報（価格・実績・法的表記）を確定で載せず、残課題にしたか？
- [ ] 構文／Lint を確認し、完成後チェック（`check.md` 等）へ引き継いだか？
- [ ] 残課題（スタイル分離・ナビ登録・実素材差し替え 等）を `documents/pending-issues.md` に記録したか？
- [ ] ビルド／コンパイルを勝手に実行していないか？

すべて ✅ なら次へ、1つでも ❌ なら見直し。
