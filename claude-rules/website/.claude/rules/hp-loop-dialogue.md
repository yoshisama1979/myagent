# 外部分析ループ（はなさかAI HP分析ループ）との対話ルール

外部の診断エージェント「**はなさかAI HP分析ループ**」から **改善提案を受け取り、検証 → 対応 → 報告（提案元へ返す）** するときの標準フロー。
このループは **診断・提案だけを行い、実装はしない**。実装はこのプロジェクト（managing側＝あなた）が担う、という分担が前提。

`CLAUDE.md` の協働原則（鵜呑みにしない／不明点は質問／より良い案は提案）と、`seo-operation.md`・`refactor.md`・`check.md` を土台とする。

> 提案の受け取り・報告の **送受信の経路（土管）** は [mailbox.md](mailbox.md)（メールボックス）に従う。本ファイルは **報告の中身（何を検証し・何を書くか）** を定義する。

---

## 1. このループとは何か（相手の素性を知る）

| 項目 | 内容 |
|------|------|
| 相手 | はなさかAI が回す HP分析ループ（`/hp-loop`）。定期/都度にサイトを分析し、改善提案と質問を「掲示板」に積む |
| 掲示板 | 提案・質問・履歴が並ぶページ（はなさか側 `site/hp-analysis/`。ユーザー＝社長が閲覧し、内容を転送してくる） |
| 提案の粒度 | 「観察（事実）／なぜ問題か／改善案／効果／工数／優先度」付き。番号は **提案＝`R-NNN`／質問＝`Q-NNN`／サイクル＝`Cycle NNN`** |
| 相手の役割 | **診断・提案まで**（本番改変・実装はしない）。あなたが実装する |
| やり取りの経路 | **メールボックス経由が基本**（→ [mailbox.md](mailbox.md)）：提案・質問が `hp-loop-ycom`（このサイト担当の分析ループ。旧 `hp-loop` から移行）から直接届き、報告も `bash bin/mailbox.sh send --to hp-loop-ycom` で直接返せる。メールボックス未整備の拠点では従来どおり **ユーザー（社長）が橋渡し**（提案を転送し、報告は `from-president.md` / `/hp-reply` で戻す） |
| 相手の次の動き | 後続サイクルで **公開HTMLを再取得（GET）して実装結果を再監査・検証** してくる。だから報告には「何を・どう直したか」を **公開HTMLの再取得で確認できる観測可能な変更点** として具体的に書く（commit は辿れない） |

### 相手の能力と限界（ここを理解しておくと検証が速い）

このループの監査は基本 **HTTP GET による静的HTML取得 + 解析**。次の前提・弱点があるので、提案を疑う材料にし、**かつ報告をこの観測面に合わせて書く**：

- **ソース・commit・ステージング・ローカルは見られない**：ループがアクセスできるのは **公開URLの実HTMLを HTTP GET で再取得した結果だけ**。「commit を見れば分かる」「ステージングで確認して」「ローカルの○○を見て」は通じない。検証の唯一の手がかりが公開HTMLなので、報告は **公開URLに反映済みの、GETで観測できる変化** で書く（→ Step 5-2）
- **JS による後処理が見えない（JSを実行しない）**：JSで meta を再注入（例 viewport の `user-scalable=no` 上書き）・遅延描画する箇所は静的取得では捉えられない。**逆に、JSでしか現れない変更を直しても、ループの再監査では「未反映」に見える** → 報告で「これは静的HTMLには出ない／JS実行後の挙動」と明示しないと、ループが誤って未対応と判定する
- **CMS/構成の判定はヒューリスティック**（`wp-content` 等の資産参照を見るだけ）。「WordPressだから」系の前提は外れることがある（本サイトのメインは独自PHPテンプレート、WPは一部のみ）
- **件数カウントは共有テンプレ分とページ固有分を区別しない**（「h1が12個」等は内訳を自分で数え直す）
- 実測データ（検索クエリ・順位・流入・CV）は相手も持っていないことが多い（GSC/GA4 未連携なら「データ未取得」と明示してくる）

## 2. 対話の全体像

```
（ループ）掲示板に R-NNN 提案・Q-NNN 質問を積む
   │  ユーザー（社長）が転送
   ▼
（あなた）Step1 検証 → Step2 影響範囲整理・合意 → Step3 実装 → Step4 残課題記録 → Step5 報告
   │  「提案元へ返す報告文」をユーザーに渡す
   ▼
（ユーザー）報告を from-president.md / `/hp-reply` でループへ戻す
   │
   ▼
（ループ）次サイクルで実装を再監査・検証し、残課題と次の提案を出す … 以降ループ
```

---

## 3. 基本方針

- **提案を鵜呑みにしない。** ループは実コードを直接見られず、静的取得＋テンプレ知識で書いている。必ず **実コードと突き合わせて検証** してから着手する
- 提案の **事実誤認は遠慮なく指摘** する（深刻度の過大評価、構成の誤認＝WordPress前提など）。ただし **「方向性は妥当」「ここは正しい」も明示** し、フェアに評価する（ループはこの指摘を歓迎し、次サイクルで前提を修正する）
- **着手前にユーザーの合意を取る。** 特に **影響範囲（1ページか／共有テンプレート＝全ページか）** と着手順を確認する
- 一度に全部やらない。**Quick win から段階的に**。やり切れない分は `documents/pending-issues.md` に残す
- 最後に **ユーザー向け完了報告** と **提案元ループへ返す報告文** の両方を用意する
- **内部アクセスという非対称の強みを活かす（受け身で実装するだけにしない）。** 分析ループは **公開HTMLを外から見る（GET）ことしかできず**、コード・構成・パフォーマンス・保守性・テンプレートの重複・セキュリティなど **内部からしか気づけない改善点は原理的に提案できない**。あなたは実コードを直接読める立場なので、**ループ提案の実装に留まらず、内部を見て気づいた改善点は自分から積極的に提案・改善する**（外形監査では出てこない領域こそ、あなたの価値）。気づいた内部課題は報告（§5-2）に添えて **ループにもフィードバック** し、次サイクルの提案精度を上げる。
- **あなたは「ループが調べられない内部」を確認できる情報源。依頼に応える／こちらからも聞く。** ループは内部を見られないため、判断に必要な内部事実（この実装は実際どうなっているか／JSで何をしているか／この値はどこ由来か／本当に重複しているか 等）を **`type: request` であなたに調査・確認依頼してくる** ことがある。コードを直接確認して `report`/`ack` で返す。逆に、外形的にどう見えているか・どのURLで何を観測したかは **あなたからループに聞ける**。**互いに見えている面が違う（あなた＝内部／ループ＝外形）ことを前提に、不足を問い合わせ合う**。

## 4. 実施手順

### Step 1: 提案の検証（実コードとの突き合わせ）

各 `R-NNN` について、**該当ファイル・行を実際に確認**し、主張が事実か判定する。

| 判定 | 意味 |
|---|---|
| ✅ 正確 | 主張どおり。対応する |
| ⚠️ 部分的に正しい | 方向性は妥当だが、深刻度・対処法・前提に誤りがある（修正して対応） |
| ❌ 不正確 | 事実と異なる。対応不要、または前提から見直し |

検証時の典型的な落とし穴（「相手の限界」と対応）：

- 「本番に出ている」とあっても **HTMLコメントで無効化** されていることがある（→ 配信されているか実DOM/実コードで確認）
- **構成の誤認**（WordPress前提のプラグイン導入提案など）。本サイトのメインは独自PHPテンプレート → 多くは不適合。テンプレート側で対処する
- 静的HTMLだけ見て、**JS側で上書き・再注入** している箇所を見落としている（viewport 等）
- 件数（「h1が12個」）は **共有テンプレート分とページ固有分を分けて** 数える

### Step 2: 影響範囲と着手順の整理 → ユーザー確認

各項目に付けて提示する：

- **影響範囲**：単一ページ / 共有テンプレート（＝全ページ）/ 特定機能
- **工数・リスク**：小/中/大、見た目への影響有無（クラス指定なら不変／タグ指定なら要注意）
- **着手順**：Quick win（小・低リスク・即効）を先頭に

ユーザーに確認すること：どこから着手するか（全部／一部／トップ先行）／ユーザー側で用意が要る素材（画像・原稿・アカウント連携・**実績の成果数値やお客様の声**等）／段階実施なら今回スコープと残す分の線引き。**合意を得てから着手する。**

### Step 3: 実装

- `refactor.md` の「やってはいけないこと」を遵守（`.css` 直接編集禁止／一括置換禁止／レガシーファイル放置／レガシーページの無断改変 等）
- **SCSS を触ったときのコンパイル**：**自動コンパイル環境（[scss-autocompile.md](scss-autocompile.md)）が有効なら**、`.scss` 編集でフックが `.css` を自動生成するため依頼不要。**無ければ開発者に依頼**（AIは手動で `sass.bat` を叩かない）。タグ変更・meta追加など **SCSS非変更の作業はコンパイル不要** である旨も明示する
- **共有テンプレートを変更する場合は「全ページに影響する」ことを必ず明記**する（このループの提案は「トップで効いた修正を全ページへ」という展開系が多い）
- 公開前チェックが要る規模なら `check.md` を通す

### Step 4: 残課題の記録

その場でやり切らない項目は `documents/pending-issues.md` に記録する（`CLAUDE.md`「残存課題の記録ルール」・`seo-operation.md` Step 6 準拠）。

- 完了した範囲 / 未対応の範囲を分けて書く（例：R-011 h1整理＝トップ完了、共有テンプレ＋サブページが未対応）
- 未対応の影響範囲（どのファイル・どのページ）／着手時の安全性メモ（クラス指定で見た目不変 等）
- 対応する `R-NNN` を併記し、ループ側の提案と突合できるようにする

### Step 5: 報告（2系統）

#### 5-1. ユーザー向け完了報告
実施項目（`R-NNN` と対応内容の対応表）／検証で判明した提案の誤り／残課題（pending-issues 記録済みの旨）／ユーザーに依頼する目視確認・素材配置。

#### 5-2. 提案元ループへ返す報告文（ユーザーがそのまま転送できる形）
ループは後続サイクルでこれを読んで **再監査・検証** する。次の要素を入れる：

- **実施済み**：`R-NNN` ごとに、対応結果を **ループが公開URLを HTTP GET で再取得して確認できる「観測可能な変化」** として書く。ループはリポジトリ・commit・ステージング・ローカルを見られず、検証の手がかりはこれだけ。次を満たす：
  - **どのURLの・何が・どう変わったか**を具体値で：対象の **公開URL** ＋ 観測できる信号を **「旧→新」の値**で書く（例：`h1数: 2→1`、`title: 旧文言→新文言`、`canonical: なし→https://…`、OGPタグの値）。可能なら対象要素の識別子も添える（`meta[name="description"]`／`link[rel="canonical"]`／`script[type="application/ld+json"]`）。ループはこの値を再GETして直接照合する
  - **公開URLでGET確認済みかを明記**：「反映済み」は *Managing側のデプロイ操作* ではなく **公開URLを実際に GET して現値を確認できた** ことを指す。デプロイしたがGET未確認なら「デプロイ済み・GET未確認」と区別して書く。本番に出ていない（ステージング止まり・コンパイル/デプロイ待ち）ならその旨と反映予定を。CDN/キャッシュがあるなら **GET確認した時刻 or キャッシュ反映待ち** を添える（書かないとループの再監査で「未反映」と誤判定される）
  - **静的HTML(GET)では確認できない変更は、その旨を明示**：JSでしか現れない変更（例：viewport 再注入の解除）・ログイン後・画像バイナリの内容などは、ループが再監査しても観測できない。「これは静的取得には出ない」と添える。これらは **ループの `R-NNN` 完了判定には使えない＝別確認扱い**（観測できないものを観測前提で報告しない）
  - **commit ハッシュは載せない**：ループの検証には使えない。commit は当方の社内記録用にとどめ、報告文では検証手段として頼らない
  - **対応した `R-NNN` の併記は必須**（ループ側の提案番号と突合するため）
- **認識合わせ**：提案の事実誤りと正しい実態（**フェアに、正しかった点も明記**）。例「R-002：本サイトは独自PHPテンプレート。Yoastは不適合、テンプレートにcanonical出力で対応」
- **残課題**：段階的に継続する項目（`R-NNN`・影響範囲つき）
- **データ補完依頼**：内部改善だけでは「どのページ/クエリを伸ばすか」決められない。GSC/GA4 の検索クエリ・順位・流入元・CV数の共有を、ループ側の `Q-NNN`（データ要求）に答える形で依頼/相談する

> 報告文は「`/hp-reply <本文>` でそのまま渡せます」と添えると、ユーザーがループへ戻しやすい。
> コミットが必要なら `/commit-pipeline` に流す。

## 5. やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 提案を検証せずそのまま実装する | ループは静的取得＋推測。実コードと食い違うことがある |
| ❌ 影響範囲（特に共有テンプレ＝全ページ）を確認せず着手する | 全ページ事故の元。展開系の提案が多い |
| ❌ 構成前提（WordPress等）を確かめずプラグイン導入等を実行する | 独自PHPテンプレートには不適合。テンプレートで対処する |
| ❌ 残課題を口頭報告だけで済ませ記録しない | 取りこぼし。`documents/pending-issues.md` に残す |
| ❌ 提案元への報告で誤りの指摘だけして正しかった点を無視する | フェアでない。ループの前提修正を妨げる |
| ❌ 報告を「公開HTMLで確認できない形」で書く（commitハッシュ頼り等） | ループはHTTP GETで公開HTMLを再取得して再監査するため、観測可能な変更点で書かないと検証できない。対応 `R-NNN` の併記は引き続き必須 |
| ❌ ループが観測できない状態を前提に報告する（未公開／ステージング止まり／JS依存・ログイン後 等） | ループが見るのは公開URLの実HTMLの静的GETのみ。本番反映前やJSでしか現れない変更は再監査で観測できず「未反映」と誤判定される。公開URL・本番反映済みか・静的取得で見えるかを明示する |
| ❌ AI が手動で `sass.bat` を叩く／生成済み `.css` を直接編集する | `refactor.md` 違反。コンパイルは自動環境ならフック・無ければ開発者。`.css` は手編集しない |
| ❌ ループ提案の実装だけで終わり、内部からしか気づけない改善を黙っている | 内部（コード・構成・保守性・パフォーマンス・セキュリティ）を見られるのは開発側だけ。外形監査では出ない改善が埋もれる。気づいたら提案・改善し、ループにも共有する（§3） |
| ❌ ループに「内部を見て確認して」と求める | ループは公開HTMLの外形しか見られない。内部の確認・調査は **依頼されたら開発側が行う**。ループへ尋ねてよいのは外形的観測（どのURLで何が見えるか）だけ |

## 6. このルール自体の改善

ループとの対話で「掲示板の番号体系が変わった」「報告フォーマットの過不足」等に気付いたら、`CLAUDE.md`「2. ルールファイル自体の改善も提案する」に従い、ユーザーに改善提案する。
