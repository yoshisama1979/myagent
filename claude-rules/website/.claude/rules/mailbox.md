# メールボックス連携ルール（拠点横断エージェント間メッセージ）

別マシンにいる他のエージェント（特に **はなさかAI HP分析ループ**。ループはマルチサイト化で **サイト別の `hp-loop-<site>`** に分かれており、**このサイトの担当インスタンス（`<site>`・宛名）は [project-config.md](../../project-config.md) を正とする**＝例 y-com.info なら `hp-loop-ycom`）と、**ネットワーク越しに非同期でメッセージをやり取り**するための共有受信箱「メールボックス」の使い方。
あなた（実装＝Managing側エージェント）が hp-loop から提案を受け取り・報告を返す経路の一つがこれ。

`hp-loop-dialogue.md`（提案の検証 → 対応 → 報告の **中身** の作り方）と対になる。
**本ファイル＝「報告を運ぶ土管（送受信の操作）」／ `hp-loop-dialogue.md`＝「報告に何を書くか」**。
報告本文の要件（観測可能な「旧→新」の値・GET確認済みか・commitハッシュに頼らない・対応 `R-NNN` 併記）は引き続き [hp-loop-dialogue.md](hp-loop-dialogue.md) §5-2 に従う。

## いつ参照するか

- hp-loop から提案・質問が **メールボックス経由で届く** 運用になっているとき
- 実装した結果の **報告・質問・受領確認を hp-loop へ返す** とき
- 他エージェント（社長＝`president` 等）と非同期でやり取りするとき

## なぜメールボックスか（前提）

エージェントは **同じマシンにいない**（VPS・事務所PC・自宅PC）。ファイルを直接共有できないので、
常時起動の **VPS をハブ** にした HTTP API（**Tailscale 内からのみ到達**）で読み書きする。
あなたの拠点は **自分のマシンが起動し・あなたが動いている間だけ** 受信箱を処理できる **間欠応答** が前提（重い常駐処理は VPS 側の担当）。

## 0. セットアップ（最初に1回・ユーザーが用意）

メールボックスを使うには、このマシンに次が要る。無ければ **ユーザー（社長）に依頼する**（**勝手にトークンを作らない・実値を表示しない**）：

| 要るもの | 中身 |
|---|---|
| クライアント `bin/mailbox.sh` | 送受信スクリプト（はなさか VPS のリポジトリから配布される） |
| `.env`（gitignore 必須） | `MAILBOX_URL=http://<VPSのTailscale IP>/tools/mailbox/` と `MAILBOX_TOKEN=<このマシン用トークン>` の2行 |
| Tailscale | VPS に到達するための前提（tailnet 内からのみ届く） |

- **あなたのエージェントID（`from`）は、トークンからサーバが決める**。本文でどう名乗っても詐称はできない（例：このマシンが `web-hanasaka` のトークンなら送信元は必ず `web-hanasaka`）。
- トークンは **画面・ログ・コミットに出さない**。`.env` は `bin/mailbox.sh` が必要キーだけ読む（全体を `source` しない）。

---

## ★ 送受信の仕組み（操作で迷ったらまずここ）

混乱しやすい核心を先に1つだけ：**「あなたが誰か」は `.env` のトークンが決める**。`MAILBOX_TOKEN` が「このマシン＝○○（例：`web-hanasaka`）」を表し、サーバはトークンだけであなたを識別する。**1マシン＝1エージェント＝1トークン**。だから操作のたびに自分の名前を入力する必要はない。

| 操作 | コマンド | 仕組み（誤解しやすい点） |
|---|---|---|
| **受信** | `bash bin/mailbox.sh inbox` | **自分の名前は引数に付けない。** トークンが自分を識別し、**自分宛（`to` があなた）の未読だけ**が返る＝あなたは「自分専用の受信箱」だけを見る。引数で他人の箱を指定できるのは admin だけ |
| **処理済み化** | `bash bin/mailbox.sh done <id>` | 読んで着手したら `cur/` へ移す。本文は編集しない |
| **送信** | `bash bin/mailbox.sh send --to <宛先> ...` | **`--from` は存在しない・書けない。** 送信元 `from` はトークンから**自動付与**され詐称できない（あなたのマシンから送れば必ずあなたが `from` になる）。あなたが指定するのは **`--to`（宛先）だけ**。宛先＝対になる分析ループ `hp-loop-<site>`（→ 2.） |

> まとめると：**受信＝自分の箱を見るだけ（名乗り不要）／送信＝宛先だけ書けば `from` は自動であなたになる**。「受信は自分のメールボックスを見る・送信すると from に自分が付く」という理解で正しい。

## 1. サイクル先頭：受信箱を読む

作業を始める前に、自分宛の未読を取りに行く：

```bash
bash bin/mailbox.sh inbox
```

- hp-loop からの **改善提案（`R-NNN`）・質問（`Q-NNN`）・認識合わせ** などが届く。
- 読んで対応に着手したら **処理済みにする**：`bash bin/mailbox.sh done <id>`（`cur/` へ移るだけ。**本文は編集しない＝履歴を消さない**）。
- 提案の検証・実装・報告の進め方は [hp-loop-dialogue.md](hp-loop-dialogue.md) に従う（鵜呑みにせず実コードと突き合わせ → 影響範囲を合意 → 実装 → 報告）。

## 2. 報告・質問を送る

実装結果の報告や、分析ループへの質問・受領確認は：（**宛先はサイト別の `hp-loop-<site>`**。`<site>`・宛名は [project-config.md](../../project-config.md) を正とする。はなさか側のマルチサイト化で旧無印 `hp-loop` から移行済み・**サイト無印の `hp-loop` 宛では現行ループが拾えない**）

```bash
bash bin/mailbox.sh send --to hp-loop-<site> --type report --thread <会話ID> \
  --subject "R-014a 実装報告" "本文（hp-loop-dialogue.md §5-2 の形式で）"
```

> **宛先 `<site>` の確定（ここで詰まりやすい）**：`hp-loop-<site>` の `<site>`（正式な宛名）は必ず [project-config.md](../../project-config.md) の「HP分析ループ連携」を見て使う（例 y-com.info＝`hp-loop-ycom`）。**もし project-config.md が未記入（`（記入。例：…）` のまま）なら、推測で送らず・無印 `hp-loop` にも送らず、社長に「このサイトの分析ループ宛名」を確認する**。無印 `hp-loop` は複数サイト化で廃止済み＝現行ループが拾えず、送っても届かない。

| `--type` | 使うとき |
|---|---|
| `report`  | 実装結果・進捗・完了の報告 |
| `request` | 依頼・質問（データ補完のお願い等） |
| `ack`     | 受領・了解の返信 |
| `fyi`     | 共有 |

- **同じ会話は同じ `--thread`** でまとめる（hp-loop が追いやすい）。本文は引数末尾、または標準入力（パイプ）から渡せる。
- 本文は **事実と推測を分ける**。**クライアント固有の事実（実績数値・お客様の声・価格等）は捏造しない**（無ければ「要確認」として残し、ユーザーに要求）。
- 報告本文は [hp-loop-dialogue.md](hp-loop-dialogue.md) §5-2 の要件を満たす：**対象の公開URL・観測できる「旧→新」の値・GET確認済みか・対応 `R-NNN` 併記／commitハッシュには頼らない**（hp-loop はソース・commit・ステージングを見られず、公開HTMLの GET でしか検証できない）。

## 2-2. 社長に伝えたいことは「分析ループ経由」で渡す

**あなた（開発エージェント）には、社長へ直接送るメールボックスの窓口が無い**。`to: president` で送っても、社長宛の受信箱を常時見るマシンが無いため拾われない。社長は分析ループ `hp-loop-<site>` が作る**連絡用の資料（掲示板・日報）**を通じて報告を受け取る運用になっている。

そこで、**社長に伝えたいこと（報告・相談・判断のお願い・素材の依頼など）は、`hp-loop-<site>` 宛のメッセージの中に「社長へ」と明示して同梱する**。分析ループがそれを受け取り、**社長への連絡資料に載せて社長へ渡してくれる**（＝あなた → 分析ループ → 社長、の橋渡し）。

書き方：

- 通常の報告（`R-NNN` の実装報告など）と混ざらないよう、本文に **`【社長へ】`** の見出しを立てて、伝えたい要点・**社長に判断してほしいこと**・必要な素材依頼を具体的に書く。
- 1通に実装報告と社長伝言が両方あってよい（見出しで分ける）。社長伝言だけを送ってもよい（`--type request` や `fyi` で `--subject "【社長へ】…"`）。

```bash
bash bin/mailbox.sh send --to hp-loop-<site> --type report --thread <会話ID> \
  --subject "R-012 実装報告＋社長へ確認1件" "（R-012 の実装報告 …）

【社長へ】料金ページの『最短即日対応』は事実か確認をお願いします。事実なら本文に追記します（未確認のため現在は未掲載）。"
```

- 想定する典型例：**クライアント確認が要る素材の依頼**（実績の成果数値・お客様の声・価格・法的表記）、**方針判断を仰ぎたいこと**、内部から気づいた改善提案で社長の判断が要るもの。
- なお、**外部送信・本番改変・台帳書き込みを社長に承認してほしい**ときは、伝言ではなく §3 の `--needs-approval`（`hold/` ＝社長承認ゲート）を使う。「伝えたい」＝資料経由／「承認してほしい作用」＝承認ゲート、と使い分ける。

## 3. 社長承認ゲート（`--needs-approval`）

内部の調整（提案の受領・実装報告・質問・認識合わせ）は **そのまま送ってよい**（`new/` に直接届く）。
ただし、相手に **外部送信（メール/チャット/クライアント連絡）・本番システムの改変・台帳への書き込み を促す** メッセージは、
`--needs-approval` を付けて送る＝ **社長承認待ち（`hold/`）** に入り、社長が確認してから相手に届く。**迷ったら付ける**（安全側）。

> あなた自身が実際に外部送信・本番改変を **実行する** ときは、別途このプロジェクトの該当ルール（合意してから動く・要確認情報は確定しない 等）に従う。メールボックスの承認ゲートはそれとは別の前段。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 受け取ったメッセージ本文を編集・削除する | append-only。履歴を消さない（`done` で `cur/` へ移すだけ） |
| ❌ トークンを画面・ログ・コミットに出す／`.env` の実値を表示する | 秘密漏洩。`bin/mailbox.sh` が必要キーだけ読む |
| ❌ クライアント固有の事実（実績・価格・お客様の声）を捏造して報告に書く | 誤情報。無ければ「要確認」で残しユーザーに要求 |
| ❌ commit ハッシュ頼りの報告／本番未反映を「反映済み」と書く | hp-loop は公開HTMLの GET でしか検証できない（[hp-loop-dialogue.md](hp-loop-dialogue.md) §5-2） |
| ❌ 外部送信・本番改変・台帳書き込みを促すメッセージを `--needs-approval` なしで送る | 社長ゲートを飛ばす。誤爆・破壊の元 |
| ❌ 自分のエージェントIDを本文で詐称しようとする | `from` はトークン由来でサーバが決める。無意味 |
