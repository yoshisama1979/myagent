以下の手順を**一つずつ順番に**実行してください。
各ステップで何をしたかを報告すること。
**問題がある場合のみユーザーに確認を取って停止すること。**

---

## ステップ1: ステージングと差分確認

変更したファイルを `git add <path>` でステージングする。
- `git add .` / `git add -A` は使わない(コンパイル済み `.css` やバックアップを巻き込むため)
- `.scss` 編集時は対応する `.css` を一緒にステージしない(コンパイルは開発者側)

`git diff --staged` を実行し、ステージング差分を確認する。

以下に該当する問題があれば指摘して**停止**:

- バグ・仕様違反
- セキュリティリスク(XSS、入力サニタイズ漏れ、ハードコードされた認証情報等)
- 不要な変更(デバッグ用 `console.log` `var_dump`、コメントアウトされた古いコード)
- **コンパイル済み `.css` の直接編集**(`.scss` 側を直すべき)
- **画像・バイナリの誤コミット**(過大サイズ、`*.BAK` 等)

問題がなければステップ1.5へ。

---

## ステップ1.5: Design.md 整合性チェック

差分に CSS / SCSS / HTML / PHP の変更が含まれる場合のみ実施。それ以外はスキップしてステップ2へ。

`Design.md` を読み込み、差分が以下のルールから逸脱していないか確認する:

- **色**: 新規追加された色値が Design.md の Color Tokens にあるか、または近い既存値に寄せられないか
- **タイポ**: フォントサイズ・行間が Section 3 のスケールに沿っているか(`1rem = 1px` 換算)
- **レイアウト**: 1000px グリッド・768px ブレークポイントを崩していないか
- **コンポーネント**: ボタン・リスト等の既存ユーティリティを使っているか
- **Don't違反**: グラデーション・`!important` 新規追加・CSS変数(`--var`)導入等が混入していないか

逸脱が見つかれば、対応を**併記してステップ2に進む**(自動で修正しない):

```
### Design.md との不一致
- [ファイル:行] [問題内容]
  - 推奨対応: [どう直すか / Design.md を更新するか]
```

逸脱がなければ「Design.md整合: OK」と報告してステップ2へ。

---

## ステップ2: 報告とコミットメッセージ提案

以下をまとめて報告し**停止**:

- ステージしたファイル一覧
- 差分確認結果
- Design.md整合性チェック結果(ステップ1.5 の出力)
- コミットメッセージ提案

コミットメッセージは日本語で **見出し1行 + 必要に応じて箇条書きの本文** で提案すること。
過去のコミット履歴(`git log --oneline -10`)のフォーマットに合わせること。

ユーザーの承認を得てから次へ進む。

---

## ステップ3: コミット実行

承認されたコミットメッセージでコミットを実行し、`git status` で結果を報告する。

- コミットメッセージは HEREDOC で渡す(改行を保持するため)
- `--no-verify` 等のフックスキップは使わない

---

## ステップ4: リファクタリング検討

`.claude/rules/refactor.md` を読み込んでガイドラインを確認する。

`git diff HEAD~1` で直前のコミット差分を確認し、ガイドラインに従って **リファクタリング候補を列挙** する:

- 重複・マジックナンバー・未使用クラス等の検出
- その場で書き換えず、候補を提案する

### 候補なしの場合
「リファクタリング不要」と報告して完了。

### 候補ありの場合
ガイドラインの「Step 2. 提案を列挙する」フォーマットで提案して**停止**。
ユーザーが承認した項目のみ、ステップ1に戻って小さなコミットとして実行する。

---

## SCSS変更時の最終確認

ステップ1〜4 を通して `.scss` を編集した場合、最後に以下を明示する:
