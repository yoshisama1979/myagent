以下の手順を**一つずつ順番に**実行してください。
各ステップで何をしたかを報告すること。
**問題がある場合のみユーザーに確認を取って停止すること。**

---

## ステップ1: ステージングと差分確認

変更したファイルを `git add <path>` でステージングする。
- `git add .` / `git add -A` は使わない(コンパイル済み `.css` やバックアップを巻き込むため)
- `.scss` 編集時の `.css` の扱いはプロジェクト方針に従う(既定はステージしない＝古い/二系統の差分混入を防ぐ。含める運用なら生成元を一本化。`scss-autocompile.md`/`staging.md`参照)

`git diff --staged` を実行し、ステージング差分を確認する。

以下に該当する問題があれば指摘して**停止**:

- バグ・仕様違反
- セキュリティリスク(XSS、入力サニタイズ漏れ、ハードコードされた認証情報等)
- 不要な変更(デバッグ用 `console.log` `var_dump`、コメントアウトされた古いコード)
- **コンパイル済み `.css` の直接編集**(`.scss` 側を直すべき)
- **画像・バイナリの誤コミット**(過大サイズ、`*.BAK` 等)

問題がなければステップ1.5へ。

---

## ステップ1.5: Design.md 整合性チェック

差分に CSS / SCSS / HTML / PHP の変更が含まれる場合のみ実施。それ以外はスキップしてステップ2へ。

`Design.md` を読み込み、差分が以下のルールから逸脱していないか確認する:

- **色**: 新規追加された色値が Design.md の Color Tokens にあるか、または近い既存値に寄せられないか
- **タイポ**: フォントサイズ・行間が Design.md のタイポスケール（換算ルール含む）に沿っているか
- **レイアウト**: Design.md のグリッド単位・コンテナ幅・ブレークポイントを崩していないか
- **コンポーネント**: ボタン・リスト等の既存ユーティリティを使っているか
- **Don't違反**: Design.md の §9 Do/Don't（グラデーション・`!important` 新規追加・規約外の CSS変数導入等）に反していないか

逸脱が見つかれば、対応を**併記してステップ2に進む**(自動で修正しない):

```
### Design.md との不一致
- [ファイル:行] [問題内容]
  - 推奨対応: [どう直すか / Design.md を更新するか]
```

逸脱がなければ「Design.md整合: OK」と報告してステップ1.6へ。

---

## ステップ1.6: Codex クロスレビュー

ユーザー全体スコープのスラッシュコマンド `/codex-review` の手順に従い、
ステージング済みの差分を Codex(OpenAI GPT-5系)に投げてクロスレビューを実施する。

Web制作の差分なので、特に以下の観点を重点的に見てもらう:

- XSS・入力サニタイズ漏れ・ハードコードされた認証情報
- 外部リンクの `rel="noopener"`・フォームの送信先/バリデーション
- アクセシビリティ(alt 欠落・ラベル紐付け・コントラスト)、SEO(title/meta/見出し階層)の退行
- レスポンシブ崩れ・コンパイル済み `.css` の直接編集

レビュー結果は次のステップでコミットメッセージ提案と一緒にユーザーに報告する。

**Codex の指摘の扱い:**

- 🔴 Critical(バグ・セキュリティ)が出た場合は、コミットメッセージ提案より先に
  問題点を強調してユーザーに報告し、対応するか・無視するかを確認する
- 🟡 Warning や 🟢 Suggestion は参考情報として併記し、ユーザーが判断する
- ✅ Approved の場合はその旨を簡潔に報告する

Codex MCP が利用不可の場合(接続エラー等)はその旨を伝えて、Claude のレビューのみで進める。

---

## ステップ2: 報告とコミットメッセージ提案

以下をまとめて報告し**停止**:

- ステージしたファイル一覧
- 差分確認結果
- Design.md整合性チェック結果(ステップ1.5 の出力)
- Codex クロスレビュー結果(ステップ1.6 の出力)
- コミットメッセージ提案

コミットメッセージは日本語で **見出し1行 + 必要に応じて箇条書きの本文** で提案すること。
過去のコミット履歴(`git log --oneline -10`)のフォーマットに合わせること。

ユーザーの承認を得てから次へ進む。

---

## ステップ3: コミット実行

承認されたコミットメッセージでコミットを実行し、`git status` で結果を報告する。

- コミットメッセージは HEREDOC で渡す(改行を保持するため)
- `--no-verify` 等のフックスキップは使わない

---

## ステップ4: リファクタリング検討

`.claude/rules/refactor.md` を読み込んでガイドラインを確認する。

`git diff HEAD~1` で直前のコミット差分を確認し、ガイドラインに従って **リファクタリング候補を列挙** する:

- 重複・マジックナンバー・未使用クラス等の検出
- その場で書き換えず、候補を提案する

### ステップ4.1: Codex クロスレビュー(条件付き)

ステップ1.6 の Codex レビュー結果を確認する:

- 🔴 Critical または 🟡 Warning が出ていた場合 → 続行
- 🟢 Suggestion のみ・または ✅ Approved の場合 → スキップして候補列挙へ

**Suggestion 単独では起動しない**(軽微な改善提案だけで Codex を再度呼ぶのはコストに見合わないため)。

続行する場合、`mcp__codex__codex` を以下のパラメータで呼び出す:

- `prompt`: `git diff HEAD~1` の内容と `.claude/rules/refactor.md` の原則(重複・マジックナンバー・未使用クラス・命名・可読性)を含める。「リファクタリング不要なら NO_ACTION、提案があれば具体的な箇所と内容を返す」と指示する
- `sandbox`: `read-only`
- `approval-policy`: `never`

Codex の提案を Claude 自身の評価と統合し、下記のリファクタリング候補リストに反映する。

### 候補なしの場合
「リファクタリング不要」と報告して完了。

### 候補ありの場合
ガイドラインの「Step 2. 提案を列挙する」フォーマットで提案して**停止**。
ユーザーが承認した項目のみ、ステップ1に戻って小さなコミットとして実行する。

## SCSS変更時の最終確認

ステップ1〜4 を通して `.scss` を編集した場合、最後に以下を明示する:

- **自動コンパイル環境（`.claude/rules/scss-autocompile.md`）が有効なら**：`.scss` 編集時にフックが `.css` を自動生成済み。コンパイル依頼は不要で、該当ページの目視確認のみ依頼する: [URL or ファイルパス]
- **環境が無ければ（フォールバック）**：

  > **開発者側で `sass.bat` 等を実行してコンパイルしてください。**
  > コンパイル後、該当ページの目視確認をお願いします: [URL or ファイルパス]

なお `.css` をコミットに含めるかはプロジェクト方針に従う（含めるなら、二系統コンパイラでプレフィックスが暴れないよう生成元を一本化すること。`scss-autocompile.md` の「ベンダープレフィックス方針」参照）。
