# 残存課題（pending-issues）

このファイルは、制作・運用・SEO の中で **その場で対応しきれず未解決のまま残った課題** の記録先。
すぐ対応できるものは対応し、そうでないものを取りこぼさないための仕組み（`CLAUDE.md`「残存課題の記録ルール」準拠）。

## 記録フォーマット

各課題は以下の3点を明記する：

- **経緯**: なぜこの課題が出たか／現状どうなっているか
- **未対応**: 何が残っているか（具体的に）
- **今後の判断ポイント**: 着手の判断材料・前提・あるべき手段

解決した課題はこのファイルから削除し、必要なら確定仕様・記録側へ転記する。
通し番号 `PI-NNN` は連番で続ける。

## オープン課題

### PI-001: オンページ監査の自動化（監査ツール化）

- **経緯**: 運用・SEO評価で title/meta/OGP/構造化データ/見出し/alt 等のオンページ信号を見る必要がある。現状は `curl`/`grep` 等の手作業（暫定）で確認している。
- **未対応**: 監査を自動化するツール（例: 対象URLを渡すと SEO 信号と仮置き残存を一覧出力するスクリプト）が未整備。
- **今後の判断ポイント**: 暫定の手作業を恒久ツールに置き換えるか。対象が静的HTMLか PHP/CMS かで取得方法が変わる。`seo-operation.md` Step 2 の「監査ツール優先」を実体化する位置づけ。

### PI-002: GA4 / Search Console 連携の具体手順

- **経緯**: 改善の効果計測（検索順位・流入・コンバージョン）には GA4 / Search Console のデータが要る。`seo-operation.md` では「LLM で代替せず取得手段を要求する」と原則化したが、取得・参照の具体手順はルール化できていない。
- **未対応**: 計測データをどう取得し、評価サイクルにどう取り込むかの手順・フォーマットが未定義。
- **今後の判断ポイント**: API 連携で自動取得するか、ユーザーがエクスポートを渡す運用にするか。計測の最小セット（流入・CV・主要KW順位）を先に定義する。

### PI-003: 業種別の訴求・SEOテンプレ

- **経緯**: 士業／飲食／製造／クリニック等、業種ごとに刺さる訴求軸・SEOキーワード・信頼要素が異なる。汎用ルールだけでは毎回ゼロから組み立てになる。
- **未対応**: 業種別の訴求・SEO定型テンプレが未整備。
- **今後の判断ポイント**: 受託の多い業種から優先して定型化するか。汎用ルール（`seo-operation.md`）の評価観点を業種別に肉付けする形が候補。

### PI-004: Before/After の効果記録フォーマット

- **経緯**: 「使いながら育てる」改善サイクルでは、施策の前後で効果を振り返れることが重要。現状その記録様式が無い。
- **未対応**: 改善前後の指標（流入・CV・速度スコア等）を残す Before/After フォーマットが未定義。
- **今後の判断ポイント**: PI-002（計測連携）と一体で設計する。記録粒度（施策単位／月次）を先に決める。
