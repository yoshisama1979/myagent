# ルール改善メモ（rule-improvements）

ルールファイル（`CLAUDE.md` ・`.claude/rules/` 配下のルール定義 等）について **気づいた改善点・気になった点** の記録先。
`CLAUDE.md`「コラボレーションの原則 2. ルールファイル自体の改善も提案する」準拠。

**ユーザーが明確な回答をしなかった場合も、気づきはその都度ここに記録する**（記録と反映を分離）。
提案が会話に埋もれて流れ、忘れられるのを防ぐための仕組み。反映はユーザー判断のタイミングでまとめて行う。

## 記録フォーマット

各エントリは以下を明記する（通し番号 `RI-NNN` で連番）：

- **対象**: どのファイルの、どの記述か
- **気づき**: 何が気になったか（実態との乖離／抜け／矛盾／構成・粒度）
- **どう変えるか**: 改善案
- **なぜ**: 根拠
- **状態**: 未反映 / 反映済み（反映済みは削除してよい）
- **還元**: `[テンプレ還元]` を付ける（このプロジェクト固有でなく、**はなさか共通テンプレ `claude-rules/` 全体に還元すべき**と判断したとき）／付けない（このプロジェクト内で閉じる話。多くは `project-config.md`／`Design.md` 側の調整）
- **還元状態**（`[テンプレ還元]` のときだけ）: `未送信` → `送信済み(M-ID)` → （反映されたら削除）。受け口未整備で送れなかったものは `受け口未整備・要社長連絡` と書く。**この状態を見て二重送信・取りこぼしを防ぐ**（`送信済み` を再送しない・`未送信`/`受け口未整備` を放置しない）

反映が済んだエントリは削除する（または「反映済み」と印を付ける）。

## 現場で問題を感じたら「指示書」形式で書く（テンプレ還元の標準形式・2026-07-21 決定）

ルールを実際に使っていて問題を感じた（実態との乖離・危険な記述・抜け・現場で手戻りが起きた）ときの `[テンプレ還元]` エントリは、1行メモでなく **「指示書」**＝ルール開発エージェント（このプロジェクトの文脈を知らない汎用エージェント）が**それだけ読んで修正作業に入れる自己完結の依頼文**として書く。ユーザー（社長）がそのままルール開発エージェントへ渡して変更してもらう運用（全体像は `CLAUDE.md`「ルールファイルの出自と改善ループ」参照）。mailbox で `rule-kaizen` 宛に送る場合も本文はこの指示書形式で書く。

運用は二段階でよい：**発見時は従来どおり `RI-NNN` の1行仮記録**（低摩擦・作業を止めない）→ **`[テンプレ還元]` として送信・持ち帰りに出す前に、指示書形式へ展開する**（還元状態を「未送信→送信済み」へ進める前が展開のゲート）。

指示書に含めること：

- **対象**: どのテンプレート（website／websystem）の、どのファイルの、どの記述か（**該当箇所の原文を引用**する。相手はこのプロジェクトのファイルを開けない前提）
- **現場で何が起きたか**: 事実ベースで（いつ・どの作業で・何が問題になったか・実害や手戻りの内容。推測は「推測：」と分離）
- **変更案**: どう直すべきか（可能なら**修正後の文面まで**書く）
- **なぜ**: 根拠。このプロジェクト固有でなく**汎用的に効く**理由
- **書かないもの**: クライアント名・秘密情報・実パス・公開URL・ブランド色は書かない（匿名化。例示が要るなら「例:」と明示した架空値にする）
- **書くもの（再現情報は必須）**: 製品名・バージョン（例: PHP 8.3・Sass 1.x）・実行コマンド・エラーメッセージなど、**再現条件・適用範囲の判定に必要な情報は書く**（無いと汎用エージェントが自己完結で判断できない）。テンプレ本文へ反映するときに固有バージョンを一般化するのはルール開発側の仕事

> 指示書は鵜呑みにされない（ルール開発エージェント側で検証・社長承認を経てテンプレートに反映される）。事実と提案を分けて書くほど検証が速く、採用されやすい。

### `[テンプレ還元]` 印のものはテンプレ更新側へ送る

`[テンプレ還元]` と判断した気づきは、**テンプレートを更新する側（はなさかAI）へ届ける**とテンプレ自体が良くなる。送り方は [`.claude/rules/mailbox.md`](../.claude/rules/mailbox.md)「2-3. テンプレ改善提案を送る（`rule-kaizen` 宛）」に従う（`bash bin/mailbox.sh send --to rule-kaizen …`）。mailbox 未設定・受け口未整備なら送らず、ここに残して社長に伝える（記録は消さない）。
