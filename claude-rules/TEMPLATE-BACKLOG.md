# テンプレート整備バックログ

`claude-rules/` テンプレート自体の未完課題（宿題）の置き場。
**配布物（各テンプレートの `documents/pending-issues.md`）には入れない** — コピー先プロジェクトが他人の課題を初期状態で抱えないため（2026-07-02 に website/documents/pending-issues.md から移設）。

通し番号 `TB-NNN`。解決したら削除する。

> **現場からの提案の受け皿でもある**：開発エージェントが `rule-kaizen` 宛に送ってきた（または `rule-improvements.md` の `[テンプレ還元]` 印で持ち帰った）テンプレ改善提案のうち、**その場で反映しない・要検討のもの**をここに `TB-NNN` で起票する。選別〜反映の流れは [README.md](README.md)「テンプレート自体の育て方」③参照（反映は社長承認後）。

## オープン

### TB-001: オンページ監査の自動化（監査ツール化）〔website〕

- **経緯**: 運用・SEO評価で title/meta/OGP/構造化データ/見出し/alt 等のオンページ信号を見る必要がある。現状は `curl`/`grep` 等の手作業（暫定）で確認している。
- **未対応**: 監査を自動化するツール（例: 対象URLを渡すと SEO 信号と仮置き残存を一覧出力するスクリプト）が未整備。
- **今後の判断ポイント**: 暫定の手作業を恒久ツールに置き換えるか。対象が静的HTMLか PHP/CMS かで取得方法が変わる。`seo-operation.md` Step 2 の「監査ツール優先」を実体化する位置づけ。※はなさか側には `bin/hp-audit.sh`（T-001）が既にあり、テンプレ同梱化・配布方法が論点。

### TB-002: GA4 / Search Console 連携の具体手順〔website〕

- **経緯**: 改善の効果計測（検索順位・流入・コンバージョン）には GA4 / Search Console のデータが要る。`seo-operation.md` では「LLM で代替せず取得手段を要求する」と原則化したが、取得・参照の具体手順はルール化できていない。
- **未対応**: 計測データをどう取得し、評価サイクルにどう取り込むかの手順・フォーマットが未定義。
- **今後の判断ポイント**: API 連携で自動取得するか、ユーザーがエクスポートを渡す運用にするか。計測の最小セット（流入・CV・主要KW順位）を先に定義する。

### TB-003: 業種別の訴求・SEOテンプレ〔website〕

- **経緯**: 士業／飲食／製造／クリニック等、業種ごとに刺さる訴求軸・SEOキーワード・信頼要素が異なる。汎用ルールだけでは毎回ゼロから組み立てになる。
- **未対応**: 業種別の訴求・SEO定型テンプレが未整備。
- **今後の判断ポイント**: 受託の多い業種から優先して定型化するか。汎用ルール（`seo-operation.md`）の評価観点を業種別に肉付けする形が候補。

### TB-004: Before/After の効果記録フォーマット〔website〕

- **経緯**: 「使いながら育てる」改善サイクルでは、施策の前後で効果を振り返れることが重要。現状その記録様式が無い。
- **未対応**: 改善前後の指標（流入・CV・速度スコア等）を残す Before/After フォーマットが未定義。
- **今後の判断ポイント**: TB-002（計測連携）と一体で設計する。記録粒度（施策単位／月次）を先に決める。

### TB-005: SafeTestCase 等の同梱コード資産の配布方法〔websystem〕

- **経緯**: `testcode.md` は `Tests\SafeTestCase` 継承を必須とするが、テンプレートにはルールだけがあり実装（PHP クラス）は各プロジェクトで用意する前提。雛形コードはルール内に追記済み（2026-07-02）。
- **未対応**: 雛形をコピペで済ませるか、テンプレートに `src/tests/SafeTestCase.php` 相当のファイルとして同梱するかが未決。
- **今後の判断ポイント**: Laravel 以外のスタックを採るプロジェクトが出たときの扱いと合わせて決める。
