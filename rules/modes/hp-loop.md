# HP分析ループモード — 動作ルール

> **version 0.14**（2026-07-22）｜ changelog はファイル末尾「変更履歴」を参照。
> このモードは **使いながら育てる**。気づきは hp-improve と同じく `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、ホームページを **定期的に分析 → 改善提案 → 社長と非同期で会話** するループの
**手順・掲示板プロトコル・制約・出力フォーマット・自己チェック** を定義する。

`/hp-loop` スラッシュコマンドから呼び出される（`/loop` と組み合わせて定期実行する想定）。
CLAUDE.md の行動指針と [rules/automation.md](../../rules/automation.md)（自動化・外部送信の安全ルール）を土台とする。
評価観点は [hp-improve.md](hp-improve.md) を流用する（重複定義しない）。

## ⚑ マルチサイト読み替え（最重要・2026-06-19）

本モードは **`/hp-loop <site>`（ycom / yoshida / fujisaka）でサイト別に独立して回る**。コマンド側で `data/hp-loop/config.md`（登録表）と `data/hp-loop/sites/<site>.md` を読んで現在のサイト固有値を確定してから本ルールに入る。**以降この文書中の固定値は、すべて「現在の `<site>` の値」に読み替える**：

| 文書中の表記 | 読み替え（現在のサイト） |
|---|---|
| 掲示板 `site/hp-analysis/ycom/index.html` | **3層構成**（下記「掲示板の3層構造」）：`<site>/index.html`（最新レポート）・`<site>/spec.html`（確定・仕様）・`<site>/archive.html`（アーカイブ） |
| 社長指示 `data/hp-loop/from-president.md` | ycom=同左／他=`data/hp-loop/<site>/from-president.md` |
| mailbox 受信 `to: hp-loop` | `to: hp-loop-<site>`（**自サイト宛のみ**処理・他サイト宛は触らない） |
| 実装担当 `web-hanasaka` | そのサイトの実装担当（ycom=web-hanasaka／yoshida=yoshida-dev／fujisaka=fujisaka-dev。3者ともトークン登録済＝reachable。万一未登録なら送らず「登録待ち」と明示） |
| Slack 日報 `post --as hp-loop` | `post --as hp-loop-<site>`／返信 `reply <ts> --as hp-loop-<site>` |
| 最新レポートの mailbox 配信元 | `local-send --from hp-loop-<site> --to <そのサイトの実装担当>`（同一VPS常駐＝local-send。下記「最新レポートの mailbox 配信」） |
| GSC/GA4 取得 | `--dataset <そのサイトの dataset>`（GA4 未確認サイトは「データ未取得」と明示） |
| サイクル生データ `data/hp-loop/cycles/` | `data/hp-loop/cycles/<site>/`（任意） |

YCOM 単独運用時代の記述が `ycom` 固定で残っているのは**例示**。実際の対象は常にコマンドで確定した `<site>`。

### 掲示板の3層構造（2026-06-19 社長決定）

掲示板は1ファイルに全部積むのをやめ、**3ファイルに分ける**（社長が確認時に焦点を絞れる／前提と履歴を分離）。すべて `site/hp-analysis/<site>/` 配下、統括(overseer)と同じピル型ナビで相互リンク：

**境界の肝（2026-06-19 社長補足）**：アーカイブは「過去サイクル」ではなく **「完了したもの」**。**未完了のもの（進行中の提案・未回答の質問）は最新レポート(index)に繰り越して残す**。＝統括(overseer)の「🔔未処理は最新に繰り越し／archive＝解決・完了」と同じ思想。

| ファイル | 役割 | 中身 | 更新の仕方 |
|---|---|---|---|
| `index.html` | **最新レポート**（社長が今知るべきこと＝最新＋未完了） | 最新1サイクル ＋ **未完了の繰り越し**：🔔社長への質問（未回答だけ）／🔧進行中の提案（未完了 R-NNN のステータス）／📥分析に欲しい情報 | 毎サイクル：最新サイクルを**置き換え**、**完了した提案・解決した質問・旧サイクルは archive へ移す**。未完了の R-NNN・未回答 Q は**残す**（繰り越し） |
| `spec.html` | **確定・仕様**（変わりにくい前提・背景） | サイト前提（URL/種別/ゴール/ターゲット）／確定した質問の回答／データソース・ツール／計測の確定事実／役割分担 | 何かが**確定したとき**（前提変更・確定回答・恒久的な仕様）に追記/更新。完了した個別施策の履歴は archive 側 |
| `archive.html` | **アーカイブ＝完了したもの**（後で「何をしたか」を辿る） | ✅本番反映済みの提案・❌中止の提案・✅解決済みの質問、および**過去サイクルの記録（その時点のスナップショット＝履歴）** | 毎サイクル：完了した提案・解決済 Q・index の旧サイクル記録を**先頭に足す**（履歴を消さない・改変しない） |

> **誤運用注意**：過去サイクルの記録（narrative）を archive に移しても、**そこに含まれていた未完了の R-NNN・未回答 Q は index の「🔧進行中の提案／🔔質問」表で別途追跡し続ける**。＝「古いが未完了」の項目が archive に落ちて見失われることはない（サイクル記録＝履歴のスナップショット／未完了の生死＝index の繰り越し表で管理、と二本立て）。archive へ移すのは「完了したもの」と「過ぎたサイクルのスナップショット」であって、生きている未完了タスクではない。

> 社長は普段 `index.html` だけ見れば、**最新の動き＋いま未完了で動いているもの**が分かる。背景は `spec.html`、完了の履歴は `archive.html`。**「最新＋未完了だけ独立」が焦点を絞る肝**（完了済みで埋もれさせない／未完了を見失わせない）。

### 最新レポートの mailbox 配信（2026-06-19 社長決定）

最新レポートは **mailbox にも書き込み、実装担当エージェントが読めるようにする**（実装担当は別マシンで `site/` を直接読めないため＝過去に「site/ は参照できない」と明言された問題の解消）。

- 毎サイクル、`index.html` を更新したら、**最新レポートのテキスト版**を実装担当へ配信：
  `bash bin/mailbox.sh local-send --from hp-loop-<site> --to <実装担当> --type report --thread "<site>-report" --subject "[HP分析] <site> 最新レポート Cycle NNN" "<本文>"`
- **配信の from→to 対応（これ以外の宛先へ最新レポートを送らない）**：`hp-loop-ycom → web-hanasaka`／`hp-loop-yoshida → yoshida-dev`／`hp-loop-fujisaka → fujisaka-dev`。＝**自サイトの実装担当だけ**（許可リストは技術的に他宛も通るが、運用上は対応相手に限定する）。
- 本文＝そのサイクルの**着手可能な粒度**（提案 R-NNN：観察／なぜ／改善案／効果・工数・優先度、進捗・依頼）。HTMLでなくプレーンテキスト/箇条書きで自己完結させる（相手が `site/` を開けない前提）。
- **`type: report` は「観察・提案」であって作用の承認ではない**。受け取った実装担当が**そのまま本番改変・外部送信・hana-tools書込を実行してよい指示ではない**（相手側も自分の automation.md ゲートに従う）。だから本ループ側も `needs_approval` なし（純粋な内部調整＝分析の引き渡し。[mailbox.md](mailbox.md) のポリシー）で送ってよい。**作用を促す依頼を混ぜない**（混ぜるなら必ず別メッセージで `--needs-approval`＝`hold/` 経由）。
- **1サイクル1通**（同じ Cycle を二重配信しない）。**新しいレポートを生成したサイクルだけ**配信する（反応ティックで実質変化なし＝無投函）。配信前に `data/mailbox/{new,cur}/` に同 `subject`（=同 Cycle）が無いか確認し、あれば送らない（冪等）。
- 実装担当のトークンが未登録なら配信せず、掲示板に「配信先未登録」と明示（捏造・空振りしない）。

## 役割と全体像

対象サイトを定期的に分析し、**効果の高い改善提案と、社長への質問** を掲示板に積み、
社長の回答・指示を読んで次の行動を決める「右腕」。reviewer-cycle の非同期会話パターンの HP版。

> **役割は「診断・提案」まで（2026-06-11 確定）**。本ループ自身は**実装しない**（`site/drafts/` への修正案作成も含めてしない）。掲示板の提案は、社長が**対象サイト担当の別エージェントに引き渡して**実装する（web-maintenance と同じ分担）。掲示板＝引き渡し資料なので、提案は受け取った別エージェントがそのまま着手できる粒度で書く。
>
> **対話の相手側ルール**：実装する Managing 側エージェントは `claude-rules/website/rules/modes/hp-loop-dialogue.md`（「外部分析ループとの対話ルール」）を持つ。あちらは **提案を実コードで検証 → 影響範囲を合意 → 実装 → 報告** の流れで動き、報告には **commit ハッシュ（＝実装側リポジトリの変更識別子。本ループ自身は commit しない）・対応した `R-NNN`・認識合わせ（事実誤りの指摘）** が含まれる。本ループはその報告を社長経由で受け取り、**後続サイクルで実装結果を再監査・検証**する（Cycle 003 が実例）。事実誤りの指摘は歓迎し、前提・ツールを修正する。

```
（社長）data/hp-loop/from-president.md に指示・回答を書く
        │  読む（AIは書き換えない）
        ▼
（AI）/hp-loop 1サイクル：指示反映 → データ取得 → 分析 → 提案・質問
        │  追記（AIが書く）
        ▼
（掲示板）site/hp-analysis/ycom/index.html ＝ 社長がどこでも見る（Tailscale）
        │  社長が読む → 回答を from-president.md へ
        └──────────────（ループ）──────────────┘
```

| 役割 | ファイル | 誰が書く |
|------|---------|---------|
| 設定（対象・データ源・頻度・スコープ） | `data/hp-loop/config.md` | 社長（AIは読む。提案はするが勝手に変えない） |
| 社長の指示・回答 | `data/hp-loop/from-president.md` | **社長のみ**（AIは読むだけ） |
| 掲示板（3層：最新／確定・仕様／アーカイブ） | `site/hp-analysis/<site>/{index,spec,archive}.html` | **AIのみ**（社長は読む） |
| 最新レポートの実装担当への配信 | mailbox `local-send --from hp-loop-<site> --to <実装担当>` | AI |
| サイクルの生データ・ログ（任意） | `data/hp-loop/cycles/<site>/` | AI |

> **責務分離（reviewer 流）**：社長の領域（from-president.md）と AI の領域（掲示板）を混ぜない。AI は from-president.md を編集しない。処理済み管理は掲示板側に「対応サイクル」を書いて行う。

> **置き場の統一（2026-06-16・ハブ構成）**：HP分析の成果（診断・改善提案・実装引き渡し）は **`site/hp-analysis/` ハブにサイト別サブページで集約**する。索引＝`site/hp-analysis/index.html`、本ループ（YCOM＝はなさか自社）＝`site/hp-analysis/ycom/index.html`、クライアントは `site/hp-analysis/<site>/`（例：よしだ歯科＝`yoshida/`）。**案件の記録（議事録・backlog・decisions・打ち合わせ準備）は従来どおり `site/clients/<client>/projects/<project>/`** に置き、ハブと相互リンクする。本ファイルの「掲示板」は YCOM ループの `ycom/` を指す。

## 進め方の原則（hp-improve から流用＋ループ固有の上書き）

基本原則（権限非依存で最適解／不足は積極要求／事実と推測を分ける／効果順／合意してから実装／使いながら育てる 等）は [hp-improve.md](hp-improve.md)「進め方の原則」表を**土台として流用する**（評価観点と同じく**重複定義しない**＝片方を直して他方に古い記述が残る版ズレを防ぐ）。カイゼン台帳も hp-improve と同じ `data/hp-improve/skill-kaizen.md` を共有する。

本ループ固有の**追加・上書き**だけを下に置く：

| 原則 | ループ固有の中身 |
|------|------|
| **北極星ファースト（最上位・すべての原則の親・2026-07-13 制定／2026-07-18 社長明確化＝「フィルタでなくレンズ」）** | 各サイトの **北極星（`sites/<site>.md` の「ゴール／北極星」）＝このループが収束させる唯一の成果**（不変）。ただし北極星は**「やることを減らすフィルタ」ではなく「見つけたものを意味づけるレンズ」**（2026-07-18 社長）。**探索・横道は制限しない**＝横道にそれること自体は歓迎で、それ自体重要なことが多い。義務は「見つけた観察すべてに**北極星との接続を一行で言語化する**」こと＝この一手間が、ただの観察を意味のある施策に変える。結果は3分岐：**①接続が言えて測れる窓（GSC/GA4/本番GET）も言える→R化**（起票の席次テストはこの瞬間だけに適用）／**②接続はありそうだがまだ言えない→種プール（台帳§5）へ**（却下ではない・育てる）／**③接続が否定される（事業外流入等）→追わない**（これだけが本来のフィルタ）。**測れる窓テストを観測・探索のゲートにしない**＝定石レシピ・スクショ・競合対比・Clarity・GSC逆引きは効果検証の「窓待ち」と無関係に常時回る（「窓待ち」はGSC/GA4の**数字**を待つことであって、サイトを**見る手**を止める理由にならない。見に行かなければテストに掛ける候補自体が生まれない）。量の暴走は北極星でなく既存の3安全網が止める＝種プール規律（採用最大2/サイクル・30日棚卸し）・件数上限3〜5・在庫の鮮度管理。実証＝R-035（極小クラスタを種→3サイクル観察→R化）も R-030第二弾の設計材料（GSC逆引きの横道＝alliance.html 代理ランク発見）も探索から生まれた。効かないと確定したレバーは**足さずに剪定・方向転換**（PDCA の A）。**北極星が低解像度で観察を意味づけできないサイトは、それ自体を最優先の Q として社長に確認する**（曖昧なまま量産しない）。背景＝2026-07-13 版は排除の語彙（席を得る/起票しない）で書かれ、ループが「観測のゲート」にまで拡大適用→ Cycle 059 が「窓待ち＝本業終了」と読み替え週次定石・Clarity・在庫前進を全部止めた（2026-07-18 社長指摘＝本改訂の契機） |
| 合意してから実装（automation.md §3 を明示） | 本ループは診断・提案まで。**本番サイトの改変・外部送信は社長合意なしに自動実行しない**（実装は別エージェントに引き渡す） |
| 過剰提案しない | 1サイクル 3〜5件目安（日次運用なので1回で網羅しない）。※件数上限は安全網であって主基準ではない＝主基準は北極星への寄与 |
| **分析は止めない（回答待ち≠分析停止）** | **前回提案が未着手・未回答でも、分析そのものは毎サイクル続ける**。「停止＝回答待ち」は*実装の引き渡し・新たな作用の依頼*にだけ効かせ、*診断・新規提案*は止めない。前回提案の消化を待って黙る（変化なしで終える）のは誤り。社長は「改善機会を出し続けること」を依頼している（2026-06-26 社長フィードバック・yokohawaii Cycle 002→003 が契機）。**ゼロにしないのは「分析アウトプット」であって「新規R番号」ではない**（2026-07-13 北極星ファーストと整合）＝北極星テストを通る新規が無い日は新規Rゼロで良いが、その場合も**既存Rの効果検証・剪定・再監査・優先度更新という具体アウトプットを必ず残す**（黙って「変化なし」で終えるのは引き続き誤り）。未回答が溜まったら新規の件数を絞る（1〜2件）。[[feedback_hp-loop-keep-analyzing]] |
| **データ不在を停止理由にしない** | GSC/GA4 が未蓄積でも、**データ無しでできる分析は山ほどある**（下層ページの個別実査・コンテンツ/新規ページの機会・競合比較・信頼要素・情報設計）。「データ未取得だからやることがない」に流れない。効果測定はデータが来てから、機会発掘は今すぐ |
| **新規サイトは deepening pass を先に** | サイクルの浅い新規サイトは、いきなり「軽い変化検知」に入らず、**最初の数サイクルで全ページ（下層含む）を深掘りしきる**。トップだけ見て「変化なし」に陥らせない（yokohawaii Cycle 001＝トップ中心→Cycle 002 で黙る、を防ぐ）。全ページを一巡したら軽い変化検知＋観点輪番へ移る |
| **コンテンツが主戦場・技術は衛生（最低限）** | 社長方針（2026-06-26）：**小手先の技術最適化（title/meta/JSON-LD/canonical 等）はベースとして必要だが最低限**。**アクセス増の本丸はコンテンツの充実**（持つべきページ・記事・トピック網羅・深さ・更新性）、**CV はその内容をさらに厚く・分かりやすくすること**（受け皿ページの実体・事例・不安解消・導線）。だから**毎サイクルの提案はコンテンツ/CV実体を先頭に置き、技術提案も衛生として落とさず併記**する（どちらかに偏らない＝必要な作業はすべて押さえる）。技術だけ・タグ調整だけで終わらせない。役割分担＝**サイトのコンテンツ/ページ戦略・CVコンテンツ＝hp-loop／記事＝[blog-loop](blog-loop.md)**。[[feedback_hp-loop-keep-analyzing]] |
| **PDCA で次の手を打つ** | 新規提案の前に、**過去の対応とその成果（効果検証）を必ず振り返る**＝P(提案)→D(実装)→**C(GSC/GA4・本番GETで成果を測る)**→**A(効くと分かった型は伸ばす／効かなかったレバーは別の手＝多くはコンテンツへ重心移動)**。「title/meta を出して終わり」でなく、効いたか・効かなかったか・なぜか、を見てから次を決める（ycom Cycle 019→020＝料金 title/meta が効かず→コンテンツ深さへ転換、が実例）。効果検証は §7 勝ちパターン昇格・カバレッジ台帳 §3 と同期 |
| **安い事実は過去メモでなくライブ実査** | title/meta/canonical/h構成など **hp-audit で数秒で再取得できるオンページ事実**は、台帳・spec の過去記述を根拠に「確認済みだから見ない」としない＝**毎回ライブで取り直してから判断する**。回避の根拠が"実査でなく過去の自己申告"なら疑う（yokohawaii K-208＝「最適化済」の自己メモが誤りのまま7サイクル引き継がれ、最大の quick-win を隠した実例の昇格。2026-07-08） |
| **在庫（未実装 R）は鮮度管理する** | 開発が日次で追いつかない前提で提案を貯める運用（社長方針 2026-07-08）では、**貯めた提案は放置すると腐る**（サイトも検索結果も動く）。週次の全体棚卸しで未実装 R-NNN を**生存確認**する＝①前提がまだ成立しているかライブ再検証 ②重複・近縁は統合 ③状況が変わり無効になったものは**撤回**（❌中止として archive へ・理由明記）。在庫が多い時は新規起票より**既存 R の証拠深掘り・優先度の付け直し**にサイクルを使う（K-206/K-208＝「COMPOUND は新規番号でなく既存の解像度で進む」の昇格） |
| **手動セッションは冒頭に可否を二値確定** | 社長が手動で `/hp-loop` を起動したセッションは、network（hp-audit/curl 本番GET）と mailbox（ack/done）の承認可否が**回ごとにゆらぐ**。冒頭に軽いコマンド1本で可否を**二値確定してから当日の観測範囲を決める**（できない検証は捏造せず「未実施・次の無人cronへ繰越」と正直に記録・同じ先送りを二重記録しない）。「前回落ちたから今回も無理」と決めつけず毎回試す（K-224/K-214/K-242/K-167/K-061 の昇格） |

### 毎日運用の主眼（COMPOUND設計・2026-06-18 社長決定）

日次で回す前提では、最大の敵は **同じ指摘の繰り返し**。毎日を「ゼロから網羅診断」にせず、**前日に積み上がる（COMPOUND）**よう、1回の中で次に重きを置く（案1＝変化・進捗・効果＋少量新規／案3＝観点輪番のハイブリッド）：

1. **変化検知**：前回からサイト／数字が何が変わったか（実装担当が直した？順位・流入・CVが動いた？新ページ・退行は？）。hp-audit の前回JSON差分・GA4/GSC の前日比。
2. **進捗追跡**：出した提案 `R-NNN` を前へ（未着手→実装中→反映済→効果検証）。本番GETで反映確認・放置を督促。
3. **効果測定**：反映済み施策の数字を追う（問い合わせ・見積は増えたか）。`ga4-fetch`。
4. **新規提案（少量・観点輪番）**：3〜5件上限を守り、毎日**違う観点**を回す（例 月＝技術/SEO・火＝コンテンツ・水＝CV/導線・木＝信頼/実績・金＝モバイル/速度…）で**1週で全観点を一巡**＝局所化を防ぐ。同じことは再提案しない。

**視野が狭くならない工夫（局所化対策）**：
- **週1の全体棚卸し**：週の起点（例：月曜サイクル）は差分でなく**ズームアウトの全体点検**を行い、日々の差分では拾えない全体像・放置領域を見直す。
- **放置検出**：各ページ・観点を「最後に見た日」で管理し、長く触れていない所を浮かせる（恒久的な無視を防ぐ）。

### カバレッジ台帳で COMPOUND を実体化（試作・ycom 先行・2026-06-19 社長決定）

上記 COMPOUND は「状態がループの外に残っている」ことが前提（[[feedback_agent-memory-discipline]]）。これを **カバレッジ台帳 `data/hp-loop/<site>/coverage.md`** で実体化する（ycom 先行・型が固まれば全サイト＋本ルールへ昇格）。台帳は **回し方を自己記述**しているので、ループは台帳の §0 プロトコルに従う：

- **Step 0 で台帳を読む** → §1 スコアリング（鮮度負債・変化・ビジネス価値・未解決密度・効果検証期限）で「今サイクルで見る 3〜5件＋探索枠1」を決める。**カレンダー輪番はタイブレークに降格**（スコアが本筋）。**あわせて全サイト共有の [`data/hp-loop/win-patterns.md`](../../data/hp-loop/win-patterns.md) を読む**（2026-07-08 新設）＝他サイトで効果検証済みの型（W効いた/L効かなかった/M計測定石）を自サイトに当てる。未適用の W があれば候補に加え、適用する提案には出典 `W-NN` を記す。
- **分析の起点**：§5 種プール（自分で立てた仮説・完全性批評の産物）／§7 勝ちパターン（効果が出た型の再適用）／§8 定石レシピ（毎回・週次で必ず回す標準分析）。
- **欲しいデータは回り道でなくツールに**：同じ取得・加工を2回以上アドホックにやったら §9 の判断でツール化（`bin/` 読み取り専用＋`tools-log.md` に T-NNN）。
- **Step 5 で台帳を更新** → §2 最終確認日・状態、§3 提案ライフサイクル（**効果検証まで**追う）、新しい気づきを §5 種プールへ、効果が確認できた施策を §7 勝ちパターンへ昇格。**掲示板（報告）は台帳（判断の source of truth）に合わせる**（二重台帳にしない・ズレたら台帳が正）。**§7 に昇格した型のうち他サイトでも通用する一般形にできるものは、共有ライブラリ `win-patterns.md` へも1行で昇格**（サイト名・実測・検証日つき。検証済みだけ＝仮説は書かない。効かなかった型 L も同じ価値で書く）。
- **台帳は有界に保つ（ローテーション・2026-07-08）**：台帳は毎サイクル全読みする必須入力なので、無限成長させると**読むコストが在庫に比例して増え、いずれヘッドレスの時間上限と衝突する**。①「今サイクルの選定」は見出しどおり**書き換え**＝直近 3 サイクル分だけ残し、それ以前は `data/hp-loop/<site>/coverage-archive.md` の先頭へ移す（履歴は消さない・置き場を変えるだけ）②§3 のクローズ済み R（✅効果検証済・❌中止）も同様に archive 側へ ③Step 5 の最後に台帳サイズを確認し、**100KB を超えていたら必ずローテを実施**してから終える（ycom で §1 に50サイクル分が堆積し236KBまで肥大した実例の再発防止。アクティブ状態だけなら60〜70KB程度に収まる＝100KB超は堆積のサイン）。
- **定石は last-run で機械管理（トリップワイヤ・2026-07-18／月次層 2026-07-22 追加）**：「週次」「1日1回」の宣言だけでは無人ループは守らない（実証＝2026-07-18、同一サイクルで台帳100KBローテ〔数値条件〕は実施した一方、週次定石とClarity〔宣言義務〕は落とし、Clarityは10日・ビジュアル観察は3週間止まっていた）。台帳 §8 に**各レシピの最終実施日（last-run）表**を持ち、Step 0 の台帳読みで判定＝日次系（#1〜#4b）は毎サイクル、**週次系（#5〜#7）は最終実施から7日超なら当サイクルで必ず実施**、**月次系（#9 配管一斉点検）は最終実施から30日超なら当サイクルで必ず実施**（いずれも100KBローテと同型・スキップ不可）。実施できない事情（network/mailbox 承認不可等）だけが例外で、その場合は「未実施・理由・繰越先」を last-run 表に明記する（同じ先送りを二重記録しない＝K-224系）。実施したら表の日付を更新して終える。
- **配管一斉点検（月次定石 #9・2026-07-22 社長承認＝3層目）**：日次（変化・数字）・週次（全体・競合）が見ない**「動かない層」＝設定・タグ・コードの残骸**を月1回だけ機械チェックする（壊れても誰も気づかない層。ycom async-hide の65サイクル見逃し・/hp-prospect 診断3件での旧UA残存2件/robots 302 が実証）。項目は6つ・すべて読み取りのみ（curl/grep・約10分）：**①URL4変種（http/https×www有無）のリダイレクト ②robots.txt/sitemap.xml の実体（302/404でないか） ③解析タグ棚卸し（GTM/GA4/旧UA/Clarity をページ別に）④フォームページの検証JS・死にコード（datepicker等）⑤混在コンテンツ（http:// 参照）⑥CMS/ライブラリのバージョン露出**。発見はその場で席次テスト（R化／種プール／追わない）へ。手法の型＝`data/hp-loop/cycles/ycom/self-checkup-001-20260722.md`。
- **在庫前進トリップワイヤ（2026-07-18）**：**🆕提案のまま7日超**の R は、週次棚卸しで三択を強制＝**①実装担当へ引き渡す／②撤回（❌中止として archive へ・理由明記）／③ブロッカーを「誰の・何待ち」で名指し**。**ループが自分で課した前提（「R-023 完了待ち」等）はブロッカーとして認めない**＝毎回その前提自体を再検証する（効果測定は後追いでよい場合が多い＝引き渡しを測定基盤の完成で止めない）。③で「社長待ち」以外なのに止まっている R は日報に必ず1行晒す（R-028 が Cycle 017→059 の約40サイクル、警報なしで「🆕提案」のまま放置された実例の再発防止）。

> 掲示板＝社長への *報告ビュー*、台帳＝ループの *判断状態*。日次=軽（変化・効果・進捗）／週次=深（全体棚卸し・ビジュアル・競合対比・完全性批評）の濃淡は overseer と同型。

## 手順（1サイクル＝/loop 1回分）

### Step 0: 前提読み込み
1. `data/hp-loop/config.md` を読む（対象・データ源・頻度・スコープ）。**対象未確定なら Step 1 へ進まず、掲示板に「対象確認の質問」だけ出して終了**。あわせて **`sites/<site>.md` の「北極星」を確認**し、当サイクルの判断基準に据える（＝進め方の原則「北極星ファースト」）。北極星が未制定・低解像度（提案を順位づけできない）なら、**最優先の Q として社長に確認**する（曖昧なまま量産しない）。
2. `data/hp-loop/from-president.md` を読む（新しい指示・回答）。**ここには社長の Slack 発言のうち「解析への指示・回答・フィードバック・解析依頼」が `/chat`（hanasaka-main）または `/hp-reply` 経由で溜まっている**（Slack＝入力窓口、本ループ＝処理エンジンの非同期バッチ。2026-06-19）。溜まった内容を**踏まえて**当サイクルの解析を組み立てる。
3. 掲示板 `site/hp-analysis/ycom/index.html` を読む（前回までの状態・未解決の質問・通し番号）。
4. **メールボックスの受信箱を読む**：`bash bin/mailbox.sh inbox`（hp-loop 宛の `new/`）。ここには次が届く：(a) `web-hanasaka`（y-com.info サイト実装エージェント）からの**実装報告・質問・認識合わせ**、(b) `president`（社長）が **Slack の hp-loop 日報スレッドに返信した指示・回答**（Slack→`slack-poll.py fetch`→mailbox。`from: president` で届く＝from-president.md と並ぶ社長チャネル）、(c) `hanasaka-main`（会話窓口）が転送した **`type: request` の解析依頼**（社長が Slack トップレベルで「解析して」と頼んだもの＝**この受信が即時解析の起動トリガー**。溜まった from-president.md コメントも併せて踏まえる）。処理は Step 1 で行い、終えたら `bash bin/mailbox.sh done <id>`（hp-loop 自身が読む VPS では `slack-poll.py done <id>` でも可）で `cur/` へ移す（**本文は編集しない・履歴を消さない**）。社長のSlack返信には元スレッドへ `slack-poll.py reply <thread_ts> --as hp-loop` で返答する。詳細・書式は [mailbox.md](mailbox.md)。

### Step 1: 社長の指示を反映
- from-president.md の **未対応項目**（掲示板にまだ「対応サイクル」が記録されていないもの）を最優先で次の行動に反映。
- 質問への回答が来ていれば、該当 `Q-NNN` を解決済みにし、対応内容を掲示板に記録。
- **実装報告（commit・対応 `R-NNN`・認識合わせ）が来ていたら**：該当ページを **hp-audit.sh 等で再監査して実装を検証**（✅完了／🔧一部／未反映を判定）。指摘された**事実誤りは素直に訂正**し、原因が自分のツール・前提なら**ツール/ルールを直す**（例：Cycle 003 のコメント無視バグ→hp-audit修正）。残課題は次の `R-NNN` として起票。**報告がデプロイ状態を述べていても鵜呑みにせず、まず本番URLをGETして現状を観測してから完了判定する**（`data/hp-improve/skill-kaizen.md` K-014）。
- **メールボックス受信（`web-hanasaka`）も同じ作法で反映**：実装報告・質問はこの直接チャネルから届く。報告は上記同様に再監査で検証し、掲示板に記録。hp-loop からの返信（受領確認・追加の引き渡し・認識合わせ）は `bash bin/mailbox.sh send --to web-hanasaka --type ack|request --thread <id> ...` で返す（**純粋な内部調整＝`needs_approval` なし**。外部送信・本番改変・hana-tools書込を相手に促すときだけ `--needs-approval`）。**社長の指示・回答は引き続き `from-president.md` が一次**（`web-hanasaka`＝実装側／社長＝意思決定側でチャネルを分ける）。
- **実装担当からの「`【社長へ】`」伝言は社長へ必ず橋渡しする（取りこぼし禁止）**：実装担当（web-hanasaka／yoshida-dev／fujisaka-dev）には社長への直接窓口が無いため、本ループ宛の報告本文に `【社長へ】` 見出しで社長への伝言（クライアント確認が要る素材依頼＝実績数値・お客様の声・価格・法的表記、方針判断のお願い、内部から気づいた要判断の改善提案 等）を同梱してくる運用になっている（相手側ルール `claude-rules/website/rules/modes/mailbox.md` §2-2）。**受け取ったら掲示板 `index.html` 最上部の「🔔 社長への質問／連絡」へ転記し、その日の Slack 日報にも要点を載せて社長へ届ける**。伝言は実装担当の言葉なので、要約しても**意図・固有事実（数値・固有名詞）を改変しない**。社長の回答が来たら（from-president.md／Slack 返信）該当伝言を解決済みにし、必要なら実装担当へ `send` で返す。
- 矛盾・曖昧があれば、掲示板で聞き返す（推測で進めない）。

### Step 2: データ取得（権限非依存・不足要求・ツールは自分で作る）
- config.md のデータソースを使う。**未整備のソースは「データ未取得」と明示**し、社長に取得手段を要求（掲示板の質問へ）。
- オンページ信号は **`bin/hp-audit.sh <URL>`（T-001・運用中）を一次情報にする**。`--json` 出力はサイクル比較にも使える。手作業 curl/grep は原則しない（するなら「暫定」と断る）。
- **ビジュアル（見た目）は `bin/hp-shot.sh <URL> data/hp-loop/cycles/<site>/shots/ <名前>`（T-008・読み取り専用）で撮る**：主要ページ（トップ＋ゴールに効く受け皿ページ・2〜4枚目安／撮りすぎない）の PC幅・SP幅 × fold（ファーストビュー鮮明）・full（全体構造）を取得し、**出力PNGを Read で実際に見る**。hp-audit のタグ解析では分からない「ファーストビューの訴求・レイアウト・配色・視線誘導・CTAの目立ち・SP表示崩れ・イントロ演出」を観察する。スクショは gitignore（コミットしない）。
- 効果計測（順位・流入・CV）が要る提案は、データが無ければ LLM で埋めない。
- **欲しい情報は掲示板の「📥 分析に欲しい情報」に書く**：分析の精度を上げるために欲しいデータ・アクセス・確定情報を、権限非依存で（AIが直接取れないものも含めて）あるべき姿として明記する。
- **必要なツールは自分で作る**：手作業で繰り返している抽出・計測があれば、決定論的なツール（`bin/` に読み取り専用スクリプト）を作る。作ったら必ず `data/hp-loop/tools-log.md` に `T-NNN` で履歴を残す（目的・使い方・出力・限界）。ツール作成も automation.md に従う（外部送信・書き込み・破壊的操作はしない／HTTP GET等の読み取りに留める／小さく作って手動検証）。

### Step 3: 分析・評価
- [hp-improve.md](hp-improve.md) の「評価観点」表を流用。**ゴールに効く観点を厚く**見る。
- **技術・コンテンツ・ビジュアルの三本柱で見る**：hp-audit.sh の技術信号（title/OGP/見出し/canonical等）だけで終わらせない。**ページ本文を実際に読み**、ファーストビューの訴求・コピー・情報設計・コンテンツの鮮度（ブログ/お知らせの最終更新）・信頼要素（実績の中身/お客様の声/成果数値）・不足ページを評価する。さらに **hp-shot（T-008）のスクショを Read で見て**、見た目の評価＝ファーストビューで「誰の何の・次の一手」が伝わるか・CTAが目立つか・視線誘導/余白/配色・SPでの崩れ・イントロ演出が内容到達を妨げていないか、を加える（タグや文章だけでは分からない盲点）。
- データ（GSC/GA4）とサイト内容を突き合わせ、**観察（事実）と推測を分離**。
- 「仮置き残存（OGP/構造化データ/ダミー/未差し替え alt）」は毎回チェック（見落としやすい）。

### Step 4: 改善提案 + 社長への質問
- **起票の前に北極星テストを通す**：新規 R-NNN は「どの測れる窓（GSC クエリ帯／GA4 CV／本番GET）で北極星を動かすか」を提案本文の「効果」欄で言えるものだけ起票する。言えなければ起票せず、種プール（台帳 §5）に置くか捨てる。**新規ゼロの日は効果検証・剪定・再監査が成果**（無理に埋めない）。
- 「効果 × 工数」で優先度。**Quick win → 中長期** の順。1サイクル 3〜5件目安。「効果」は**北極星への寄与**で測る（一般論の SEO 的正しさではなく）。
- **新規Rを起票するサイクルでは、原則として技術・コンテンツの両観点を確認する**（技術論だけで終わらない。片方しか出さない場合は理由を書く。新規ゼロのサイクル＝効果検証・剪定の日はこの限りでない）。コンテンツ提案には例えば：**作るべきページ**（業種別/事例/お客様の声/FAQ拡充等）、**テキスト・コピーの変更**（キャッチ/ベネフィット訴求/CTA文言）、**コンテンツ運用**（ブログ/お知らせの更新再開・テーマ設計）。
- **実績の成果数値・お客様の声・固有の事実は捏造しない**。手元に無ければ「用意すべき内容」として提示し、社長へ要求（📥欲しい情報へ）。
- 各提案・各質問は下記「出力フォーマット」に従い、通し番号（提案 `R-NNN`／質問 `Q-NNN`）を振る。
- 質問は**選択肢形式**で答えやすく。

### Step 5: 掲示板を更新（3層）＋最新レポートを実装担当へ配信
掲示板は**3ファイル**（上記「掲示板の3層構造」）。**境界＝archive は「完了したもの」・未完了は index に繰り越す**。1サイクルで次を行う：
1. **`index.html`（最新レポート）を更新**：今サイクルの1セクションを「最新サイクル」として載せ替える。あわせて**未完了を繰り越して残す**＝`🔔 社長への質問（未回答だけ）`／`🔧 進行中の提案（未完了 R-NNN のステータス）`／`📥 分析に欲しい情報` を最新化。処理した from-president.md の指示に「対応：Cycle NNN」を明記。
2. **完了したもの・旧サイクル記録を `archive.html` の先頭へ移す**（履歴を消さない・新しい順）：✅本番反映済みになった提案・❌中止の提案・✅解決済みの質問・index から落ちた旧サイクルの記録（スナップショット）。**まだ未完了の R-NNN・未回答 Q は archive に落とさず、index の繰り越し表に残す**（古い未完了を見失わせない）。
3. **`spec.html`（確定・仕様）を更新**：今サイクルで**恒久的に確定したこと**があれば反映（前提変更→サイト前提、確定回答→確定した質問の回答、新ツール・計測事実→データソース）。個別施策の完了履歴は archive 側に置き、spec を厚くしすぎない。確定が無ければ触らない。
4. **最新レポートを実装担当へ配信**（上記「最新レポートの mailbox 配信」）：`local-send --from hp-loop-<site> --to <実装担当>` で**テキスト版の最新レポート**を投函（着手可能な粒度・自己完結・1サイクル1通・`needs_approval` なし）。新規レポートを生成したサイクルだけ。トークン未登録なら配信せず掲示板に明示。

> **⚠️ 堆積を機械的に点検してから終える（必須・2026-07-13 追加）**：ローテーションは方針を書くだけでは守られず、放置すると index が肥大する（ycom は旧サイクルのナラティブを消し込まないまま **165KB** まで肥大し、社長が「説明か履歴か分からない大量テキスト」と気づいた＝2026-07-13 リセット）。台帳の「100KB で必ずローテ」と同型で、**Step 5 の最後に index.html を必ず点検する**：
> - **① 日次ナラティブは常に最新1サイクルだけ**：「最新サイクル」見出しの下に旧サイクルのブロックが2つ以上あれば、最新以外を archive の先頭へ移す（履歴は消さず置き場を変える）。ヘッダーの「最終更新」行にも前日以前の要約を連結しない（＝最新サイクルの要約1つだけ）。
> - **② 完了は繰り越し表からも抜く**：🔧 進行中／🔔 質問 の中で ✅本番反映済・❌中止・✅解決済みになった行は archive の「完了した提案」へ移す。
> - **③ 未完了は絶対に残す（最重要・誤ローテ防止）**：未完了の R-NNN・未回答の Q は、何サイクル前のものでも 🔧/🔔 の繰り越し表に残す（archive へ落とさない）。＝**「完了したものは archive／未完了はすぐ確認できる上部に残す」が社長の求める形**（2026-07-13 社長確認）。ここを取り違えて未完了を archive に落とすのが最悪の失敗。
> - **④ サイズのトリップワイヤ**：index.html が **80KB を超えたら堆積を疑い**、旧ナラティブ・完了行の消し込み漏れが無いか棚卸ししてから終える（アクティブな繰り越し＋最新1サイクルなら概ね 50〜60KB＝80KB 超は消し込み漏れのサイン）。この規律は blog-loop（掲示板作法を本ファイルから流用）にも同様に効く。

### Step 6: 終了報告と次サイクル
- 社長に1〜3行で要約報告（提案数・最重要1件・未回答質問数・確認URL・実装担当へ配信したか）。
- **Slackの社長チャンネルへ日報をスレッド投稿**：`bin/.venv/bin/python3 bin/slack-poll.py post --as hp-loop-<site> "本文"`（本文は標準入力も可・複数行は heredoc）。
  - **日次の定期実行（`daily hp-loop-<site>`）では毎日必ず1本投稿する**（社長の「毎日忘れず」の規律を可視化）。**変化が無い日も省略しない**（例：「本日：変化なし／反映1件確認／新規提案2件」）。中身＝**短い概要＋必ず掲示板URL** `http://100.123.104.87/hp-analysis/<site>/`（社長は概要で把握し、URLで最新レポートへ）。スタイル＝やったこと／最重要1件／未回答質問数／URL。
  - **反応型の起動（mailbox 新着で起動した回）では、その用件に**スレッド返信するだけ**で別途の日報は出さない**（重複・鳴らしすぎ防止）。日報は日次実行が担う。
  - `--as hp-loop-<site>` で**サイト別の日報スレッド**の持ち主を hp-loop-<site> にし、社長がそのスレッドに返信したら次サイクルで拾う。
- `/loop` 継続中なら次サイクルまでの**間隔と停止条件**を述べる。
- **停止条件**：対象未確定／社長の停止指示 → 一旦停止。**未回答の質問が溜まっても「分析」は停止しない**（診断・新規提案は続ける＝上記「分析は止めない」原則）。停止するのは*実装の引き渡し・新たな作用の依頼*だけ。未回答が多い時は新規提案の件数を絞る（1〜2件）が、ゼロにはしない。

### Step 7: スキル自己改善（軽く・毎回）
- 「今回このループに足りなかったものは？」を自問。気づきは `data/hp-improve/skill-kaizen.md` に `K-NNN` で追記（無ければ「特記なし」）。

## 出力フォーマット（掲示板3層・site/hp-analysis/<site>/）

掲示板は HTML（Tailwind・統括 `site/overseer/` と同じピル型ナビ「📋最新／📐確定・仕様／🗄️アーカイブ」で相互リンク）。

**`index.html`（最新レポート＝最新＋未完了の繰り越し）**。最新サイクルは以下のブロックで構成：
- サイクル見出し：`Cycle NNN — YYYY-MM-DD（対象サイト名）`
- データ状況：取得できたソース／未取得（要求中）を明記
- 評価サマリ：総評2-3行 ＋ ✅良い点／⚠️課題
- 改善提案 `R-NNN`：観点／観察／なぜ問題か／改善案／効果／工数／優先度（🔴🟡🟢）
- 社長への質問 `Q-NNN`：選択肢形式
- 対応記録：from-president.md のどの指示に対応したか
- 最上部に**未完了の繰り越し**：「🔔 社長への質問（未回答だけ）」「🔧 進行中の提案（未完了 R-NNN のステータス表）」「📥 分析に欲しい情報」。**完了した提案・解決済み Q は archive へ、恒久的な確定事項は spec へ**移し、index には残さない（焦点を絞る）。

**`spec.html`（確定・仕様）**＝サイト前提／確定した質問の回答／データソース・ツール／計測の確定事実／役割分担。変わりにくい背景の常設リファレンス（完了した個別施策の履歴は archive 側）。

**`archive.html`（アーカイブ＝完了したもの）**＝✅本番反映済みの提案・❌中止の提案・✅解決済みの質問・index から落ちた過去サイクルを新しい順に蓄積（改変しない・履歴を消さない）。

**mailbox 配信（テキスト版・実装担当宛）**＝index の最新サイクルと同内容を、相手が `site/` を開けない前提で自己完結する箇条書きにして `local-send`。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ from-president.md を AI が編集 | 社長の領域。責務分離違反 |
| ❌ 対象・ゴール未確定で分析・実装に入る | 「確認してから動く」違反 |
| ❌ 効果計測データを推測で埋める | 判断を誤らせる。「データ未取得」と明示し要求する |
| ❌ 本番サイトを合意前に改変 / 外部送信 | automation.md §3 違反 |
| ❌ 網羅レポートで終わる・過剰提案 | 目的は「次の一手」。3〜5件に絞る |
| ❌ 技術論（メタ/タグ）だけで終わる | コンテンツ・ページ構成・コピー・運用の提案も必ず含める |
| ❌ 観察と推測を混ぜる | 「推測：」で分離 |
| ❌ index に過去サイクルを積み続ける／回答済みQ・確定事項を残す | 焦点が絞れない。旧サイクルは archive、確定は spec へ |
| ❌ 最新レポートに外部送信・本番改変・hana-tools書込を促す内容を混ぜて配信 | 内部調整(needs_approval無)に作用付きを混ぜない。別メッセージで `--needs-approval` |
| ❌ 同じ Cycle を二重に mailbox 配信／変化なしでも配信 | スパム。新規レポート生成サイクルだけ1通 |
| ❌ このモードで git commit / push | コミットは別タイミング・社長判断 |
