# 見込み客HP単発解析（営業用レポート）モード — 動作ルール

> **version 0.1**（2026-07-22・初版）｜ changelog はファイル末尾。
> このモードは **使いながら育てる**。気づきは hp-loop と同じく `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、**見込み客（まだ契約していない相手）のホームページを1回で解析し、社長が営業に使える「クライアント提示用レポート」を発行する** 単発モードの **手順・制約・出力・自己チェック** を定義する。

`/hp-prospect <URL>` スラッシュコマンドから呼ばれる（**手動・単発**。cron/ループには載せない）。
土台は CLAUDE.md と [rules/automation.md](../../rules/automation.md)（**衝突したら automation.md 最優先**）。評価観点は [hp-improve.md](hp-improve.md) を流用する（重複定義しない）。

## ⚑ 位置づけ（hp-loop との違い・最初に把握）

| | hp-loop（継続解析） | 本モード（営業用単発） |
|---|---|---|
| 対象 | 登録済みサイト（自社・契約クライアント） | **見込み客**（未契約・データアクセスなし） |
| データ | GSC/GA4 実データ＋台帳の蓄積 | **公開情報のみ**（本番GET・スクショ・検索結果） |
| 読者 | 社長・実装担当（社内向け） | **クライアント本人**（です・ます調・専門用語に補足・ビジネスインパクト） |
| 目的 | 改善の継続運用 | **「ここまで見える」を見せて継続解析サービスの受注につなげる** |
| 頻度 | 毎日 | 1回きり（受注したら hp-loop / blog-loop へ正式登録＝卒業） |

- **レポート自体が営業ツール**。網羅チェックリストではなく「この会社に頼むと違う」と思わせる**具体的で刺さる指摘**を厚くする。Codex クロスレビュー（別モデルの別視点）を標準組み込みにするのはそのため。
- **GSC/GA4 が無いことを弱点にせず営業導線にする**：レポート末尾に「実データ（検索・アクセス解析）を接続すると、さらに◯◯が分かります」の節を必ず置く＝継続契約の提案へ橋渡し。

## 厳守する制約

| 操作 | 可否 |
|------|------|
| Read / Grep / Glob / git status・log・diff | ✅ 可 |
| 対象サイトの**読み取り取得**（`bin/hp-audit.sh`・`curl` GET・`bin/hp-shot.sh` スクショ・`bin/hp-serp.sh` 検索結果・`bin/hp-compete.py`） | ✅ 可（**GETのみ・少ページ＝トップ＋主要5〜10ページ目安・連続アクセスは間隔を空ける**） |
| Codex MCP（`mcp__codex__codex`）でのクロスレビュー（読み取り専用サンドボックス・素材はスクラッチパッドに隔離） | ✅ 可（利用不可なら「Codexレビュー未実施」と明記して続行＝ブロックしない） |
| Write: `site/hp-analysis/prospects/<domain>/`（レポート・スクショ。**gitignore 済**＝クライアント分析はgit外） | ✅ 可（AIの担当） |
| Write: `site/hp-analysis/index.html` へのハブ行追加（prospects 節） | ✅ 可（VPS上のみ・既存行は消さない） |
| **フォーム送信・ログイン試行・脆弱性スキャン・大量クロール** | ❌ 絶対不可（未契約の他社サイト。公開ページの通常GETだけ） |
| 流入数・順位・CV数など**実データが無い数値の推定・捏造** | ❌ 不可（「実データ未接続のため未確認」と明示。一般論は「一般に」と明示） |
| 競合・他社への誹謗的表現／他社サイトのスクショをレポートに流用 | ❌ 不可（競合比較は事実ベースの構造比較まで。スクショは対象サイト自身のみ） |
| **レポートの外部送信・公開**（メール・URL共有・Nginx一時公開） | ❌ 不可（提示は社長。共有が要る時は [rules/preview-server.md](../../rules/preview-server.md) の一時公開を**社長が**判断） |
| 対象サイトの改変・営業メールの送信 | ❌ 不可 |
| git commit / push | ❌ このモードではしない（そもそも出力は gitignore 済） |

## 手順（1回＝1レポート）

### Step 0: 前提確認
- 引数の URL を確認。**URL が無ければ質問して終了**。任意で聞けるなら：業種・商圏（地域）・サイトのゴール（問い合わせ/来店/EC等）・気にしている競合。**不明でも止まらない**＝サイトから読み取れる範囲で進め、推測は「推測：」と明示。
- `<domain>` を決め（例 `example-co-jp`）、出力先 `site/hp-analysis/prospects/<domain>/` を作る。

### Step 1: データ取得（公開情報のみ・読み取り）
1. `bin/hp-audit.sh <URL> --json` でオンページ信号（title/meta/OGP/h構成/canonical/構造化データ）。
2. トップ＋主要ページ（サービス・料金・実績・会社概要・問い合わせ等、サイト構成から5〜10ページ）を curl GET で取得。
3. `bin/hp-shot.sh` で PC幅・SP幅 × fold/full を撮影（トップ＋主要2〜3ページ）→ **PNG を Read で実際に見る**。
4. `bin/hp-serp.sh "<主要と思われるクエリ>"` を2〜3本（例：業種×地域）＝検索結果での見え方・競合の顔ぶれ。
5. 表示速度の外形観察（HTMLサイズ・画像重量・レンダリングブロック要素）。

### Step 2: Codex クロスレビュー（標準組み込み）
- 取得素材（HTML・スクショ・SERP結果・hp-audit JSON）をスクラッチパッドの隔離フォルダにコピーし、Codex（読み取り専用サンドボックス）に「フレッシュアイズ監査」を依頼（2026-07-21 ycom 手動テストと同型のプロンプト：見逃し発掘・計測/実装視点・数字引用必須・捏造禁止）。
- **Codex の指摘は鵜呑みにしない**：レポートに載せる前に必ず現物で裏取りする（裏取りできないものは載せないか「要確認」と明示）。
- Codex が使えない環境なら省略し、レポートの内部メモに「Codexレビュー未実施」と記録（クライアント向け本文には書かない）。

### Step 3: 分析（hp-improve の評価観点を流用）
- [hp-improve.md](hp-improve.md) の観点表＋ビジュアル（スクショ）＋検索の見え方で、**観察（事実）と推測を分離**して気づきを集める。
- **出し惜しみしない（2026-07-22 社長指示）**：1回の解析で分かったことは**全件レポートに載せる**。絞るのは「量」ではなく「強調」＝先頭に「見せられて刺さる TOP5〜7件」を厚く置き（①具体的（スクショ・現物引用つき）②ビジネスインパクトが言える ③直せば変わると想像できる、の3条件）、残りの指摘も**観点別の詳細節で全件**（観察→なぜ→改善の方向性の粒度で）書く。フラットな羅列にせず強弱をつける、が唯一の制約。

### Step 4: レポート発行（クライアント提示用 HTML）
`site/hp-analysis/prospects/<domain>/index.html` に単一HTMLで出力（Tailwind・印刷でも読める体裁）。**です・ます調・専門用語に補足・ビジネスインパクト**（CLAUDE.md クライアント向けルール）。構成：

1. **表紙ヘッダ**：「ホームページ健康診断レポート」＋対象サイト・実施日。**発行元の社名は書かない**（2026-07-22 社長指示＝代理店の営業ツールとしても使うため。中立表記「AIウェブサイト診断サービス」等。本文中の「当社確認」等も「診断時実測」に言い換え、社名・自社特定表現を残さない）。**表紙に注意書きを必ず置く**（2026-07-22 社長指示）＝「公開情報のみに基づく初期診断であること」＋「サーチコンソール・GA4・Microsoft Clarity 等と連携すればより詳しいレポートを発行できること」を数行で。**配色は対象サイトのメインカラーに合わせる**（対象サイトのCSSから主要色を抽出し、ヘッダ・見出しバー・表ヘッダ等の基調色に使う。重要度バッジの赤/橙は意味色として維持）
2. **総評**（3〜4行・結論ファースト。良い点も必ず1つ以上＝欠点羅列にしない）
3. **重要な指摘 TOP5〜7**：各件＝現状（スクショ/現物引用）→ なぜ機会損失か（ビジネス言葉）→ 改善の方向性 → 期待できる効果（定性でよい・数値は捏造しない）
4. **観点別サマリ表**（ファーストビュー/導線/コンテンツ/SEO/モバイル/速度/信頼性…を ✅⚠️❌ で一覧）
5. **すべての指摘（詳細）**：TOP に入らなかった指摘も**全件**、観点別に「観察→なぜ→改善の方向性」の粒度で載せる（出し惜しみしない。ボリュームは制限しない＝単一HTMLに縦に長く載せてよい）
6. **検索での見え方**（主要クエリでの表示のされ方・競合の顔ぶれ＝事実ベース）
7. **実データを接続すると、さらに分かること**（**GSC・GA4・Microsoft Clarity** の3点セットで可能になる分析の予告＝**継続解析サービスへの営業導線・必須**。各ツールが何を明らかにするかを一言ずつ添える：GSC=検索語／GA4=来訪・問い合わせ数／Clarity=ヒートマップ・行動録画）
8. 免責注記（公開情報のみに基づく初期診断である旨）

- スクショ・生データは同フォルダ（`shots/` 等）に置く。ハブ `site/hp-analysis/index.html` に prospects 行を追加（無ければ節を作る）。

### Step 5: 報告
- 社長に要約：総評・TOP指摘3件・レポートURL（`http://100.123.104.87/hp-analysis/prospects/<domain>/`）・営業トークの示唆（どの指摘を入口にすると刺さりそうか）。
- **提示・送付は社長**。PDF が要る場合は [deck-format.md](deck-format.md) の手順で **A4縦の文書PDF** として書き出す（依頼があれば）。**PDF は紙印刷しない前提＝枚数を気にせず、見出し（縦バーつきセクション見出し）ごとに改ページする**（`main > section { break-before: page }`）。**カードは改ページ制御せずページをまたいで流し込む**（切れてよい＝余白を作らない。画像だけは分割不可なので `max-height` を1ページ内に抑える。2026-07-22 社長指示）。

### Step 6: 自己改善（軽く・毎回）
- 「営業ツールとして足りなかったものは？」を `data/hp-improve/skill-kaizen.md` に1行（無ければ「特記なし」）。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ フォーム送信・ログイン・脆弱性スキャン・大量クロール | 未契約の他社サイト。公開ページの通常GETのみ |
| ❌ 流入・順位・CVの数値を推定で書く | 捏造。信頼を壊す。「実データ未接続」を営業導線に変える |
| ❌ 強弱のないフラットな羅列で終わる／逆に量を絞って出し惜しみする | 営業ツール。**全件載せた上で**刺さる TOP5〜7＋良い点を先頭に置き「頼みたい」と思わせる |
| ❌ 社内向けの口調・略語のまま出す | 読者はクライアント。です・ます調・補足・ビジネスインパクト |
| ❌ 競合の誹謗・他社スクショの流用 | 法的/信頼リスク（deck-format 🚨と同根） |
| ❌ レポートを勝手に外部送信・公開 | 提示は社長の仕事。automation.md §3 |
| ❌ 出力を git 管理下に置く | クライアント分析はgit外（2026-07-16 社長決定）。prospects/ は gitignore 済 |

## 自己チェック（レポート発行前に）
- [ ] 取得は GET のみ・少ページ・スクショは対象サイト自身のみか？
- [ ] Codex の指摘を現物で裏取りしてから載せたか？（未検証を載せていないか）
- [ ] 数値の捏造なし・一般論は「一般に」・不明は「未確認」か？
- [ ] 分かったことを**全件**載せたか？（出し惜しみなし）その上で TOP5〜7 を先頭で強調し、良い点を1つ以上入れ、TOP各件にスクショ/現物根拠があるか？
- [ ] です・ます調・専門用語補足・ビジネスインパクト表現か？
- [ ] 「実データを接続すると分かること」の営業導線の節はあるか？
- [ ] 出力先は `site/hp-analysis/prospects/<domain>/`（gitignore済）でハブに行を追加したか？
- [ ] 外部送信・公開をしていないか？（提示は社長）
- [ ] skill-kaizen に1行残したか？編集ファイルを末尾にリンク一覧したか？
