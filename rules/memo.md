# プロジェクトメモ運用ルール

## 概要

社長から「メモを残しておいて」「プロジェクトメモを追加」等の指示を受けた際の、保存先・書式・記録方法に関するルール。

## 保存先（必須）

クライアント・プロジェクト単位のディレクトリに `memo.md` として保存する。
**3階層構造（`projects/` を中間に挟む）** で `clients/README.md` の構成に揃える。

```
clients/<クライアント名>/projects/<プロジェクト名>/memo.md
```

### 命名規則

- クライアント名・プロジェクト名は **英小文字 + ハイフン区切り**（例: `nara-forester-academy`, `coverall-owners-club`）
- 既存の `clients/` 配下のディレクトリ命名と整合させる

### 保存例

```
clients/nara-forester-academy/projects/corporate/memo.md
clients/osaka-toyopet/projects/omoide-photo/memo.md
```

### 関連ファイル

- `client-info.md`: クライアント単位の基本情報（`clients/<client>/client-info.md`）
- `project-info.md`: プロジェクト単位の基本情報（`clients/<client>/projects/<project>/project-info.md`）
- `memo.md`: 打合せ・メモ・進捗等の記録（本ルールが対象）

## プロジェクト横断のメモ

特定のクライアント・プロジェクトに紐づかない社長の日常的な思いつきは、プロジェクトルートの **`notes.md`** に追記する。

- 新しいエントリは日付の新しい順（上が最新）で追加
- 不具合・アイデア・気づき等を区別せず、見出し+ラベルで整理
- ある程度たまってプロジェクト化したら、該当 `clients/<client>/projects/<project>/memo.md` へ転記

## 記録ルール

### 1. 客観的に補足質問してから書く

- 社長が渡してくれる情報は **当事者視点** のため、第三者が読むと前提が抜けがち
- 書き始める前に、以下のうち欠けているものを **1〜3個に絞って** 確認する
  - 関係者（誰が依頼者・実行者・窓口か）
  - 目的・KPI
  - 期限・優先順位（確定 or 目安）
  - 予算・コスト負担者
  - 既存環境・前提条件
  - フェーズ（提案中・受注済・進行中 等）
- ※ 詳細は [feedback_memory_objective_review.md](C:/Users/yoshi/.claude/projects/d--xampp-htdocs-myagent/memory/feedback_memory_objective_review.md) と整合

### 2. 書式の基本

- 見出し構造でスキャンしやすくする
- **プロジェクト概要** / **作業内容** / **スケジュール** / **次のアクション** / **補足・未確認事項** を基本ブロックにする
- 確定情報と未確認情報を **明確に区別**（未確認は「未記載」「目安」等で示す）

### 3. 記録すべきこと

- 受注ステータス・スコープ・納期
- 関係者・連絡窓口
- 進行上の依存関係（〇〇待ち、等）
- 打合せで決まった具体事項（曖昧な意向ではなく、決定・宿題）

### 4. 記録すべきでないこと

- ソースコード由来で確認できる事実（CLAUDE.md・既存ファイルで参照可能なもの）
- 一時的な作業メモ（タスクで管理すべきもの）
- 機密性の高い個人情報・パスワード等
