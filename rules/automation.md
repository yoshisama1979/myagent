# 自動化・システム開発の安全ルール

スクリプト（bash/python）・定期実行（`/loop`・cron）・外部送信（Slack/Chatwork/メール）・
API 連携（hana-tools 等）を **作る・動かす** ときに従うルール。

これまでの myagent は「HTML 記録・ドキュメント生成」が中心だったが、`bin/` のツールや
定期通知のように **実際に動いて外部へ作用するシステム** を作る場面が出てきた。
記録物と違い、自動化は **誤爆・スパム・秘密漏洩・暴走** という固有のリスクを持つ。
このファイルはそのリスクを抑えるための作法を定める。

> 土台は CLAUDE.md の行動指針（結論ファースト／確認してから動く／セキュリティ意識／一手前で言う）。
> websystem の開発ルール（クリティカル判断・秘密情報・安全装置）を myagent の実態に翻訳したもの。
> コミット・デプロイの規律は [development.md](development.md) を併せて参照。

---

## 0. 適用範囲

| 対象 | 例 |
|------|-----|
| スクリプト | `bin/*.sh`、`bin/*.py`、ワンライナーでも繰り返し使うもの |
| 定期実行 | `/loop`、cron、スケジュール起動するジョブ |
| 外部送信 | Slack / Chatwork / メール / LINE 等への通知 |
| 外部 API | hana-tools API（読み取り・**書き込み**）、その他 SaaS API |

単発の読み取り（`curl` で1回確認する等）は軽量に扱ってよい。
**繰り返す・書き込む・外に送る** ものほど、本ルールを厳格に適用する。

## 1. 着手前に確定する（確認してから動く）

自動化は「動き続ける」ので、曖昧なまま作ると曖昧なまま暴走する。**着手前に次を確定**する。
不明なら推測で埋めず、選択肢形式で社長に確認する。

| 確定する項目 | 例 |
|------------|-----|
| 目的（何のために） | ToDo の取りこぼし防止 / サイト死活監視 |
| トリガー・頻度 | 毎朝1回 / 5分おき / 手動実行のみ |
| 送信先・宛先 | どの Slack チャンネルか、誰に届くか |
| 停止条件 | いつ止めるか、どうなったら止めるか |
| 失敗時の挙動 | 黙って次へ / エラーを通知 / 止める |
| 影響範囲 | 書き込み先・上書き対象・課金が発生するか |

- **権限非依存で最適解を出す**：自分が直接できる手段に縮めず、本来あるべき方法（ツール化・API連携・社長の手作業）も含めて最善を提示する。
- **技術アドバイザーとして**：社長の指示をそのまま実装せず、より安全・確実な案があれば理由付きで推奨する（最終判断は社長）。

## 2. 秘密情報の扱い

| ルール | 中身 |
|--------|------|
| `.env` に集約 | API トークン・Webhook URL・パスワード等は `.env`（gitignore 済み）に置く |
| ハードコード禁止 | スクリプト本体・HTML・コミットに秘密値を直書きしない |
| `.env.example` に枠だけ | 新しい秘密情報を足したら、プレースホルダだけ `.env.example` に追記する |
| 出力に出さない | ログ・通知本文・コミットメッセージ・エラー出力に秘密値を混ぜない（特に `set -x` やフルURL の出力に注意） |
| 安全に読む | スクリプトで `.env` を使うときは **全体を `source` せず、必要なキーだけ抽出**する（`source` は任意コード実行・変数汚染の温床）。`bin/slack.sh` の `get_env_value` を参考にする |

`.env` の実値は社長が設定する。AI は **実値を読まない・表示しない・コミットしない**。

### 通知本文に載せてよい情報の分類

「秘密情報を漏らさない」だけでは足りない。チェック結果には顧客名・取引情報・個人情報・APIレスポンス断片が混ざりうる。**Slack 等の外部チャンネルに送る本文は、以下を原則とする**：

- **送ってよい**：件数・状態（OK/NG）・対象IDの一部・社内システムへの参照リンク・基準時刻
- **送らない**：顧客名や個人情報そのもの・認証情報の断片・APIレスポンス本文・機密データの中身
- 詳細が必要なときは **本文を送らず、社内システム（hana-tools 等）側で参照させる** 導線にする
- 新しい通知を作るときは「この情報をそのチャンネルの全員が見てよいか」を必ず確認する

## 3. 外部送信・破壊的操作は「必ず確認 / 合意してから」

以下に該当する操作は、**自動で勝手に実行しない**。実行前に社長へ状況・選択肢・推奨を提示して合意を取る
（websystem autopilot の「クリティカル判断＝必ず止まる」を myagent 用に翻訳）。

### 必ず止まるリスト

- **外部送信**：Slack / Chatwork / メール / LINE 等への通知送信（テスト送信を除く初回稼働）
- **外部書き込み**：hana-tools への登録・更新（`create-todo` 等）、その他 API への書き込み
- **破壊的操作**：ファイル/ディレクトリの大量削除・上書き、`data/` 配下の改変
- **本番反映・デプロイ**：`site/` の公開・Nginx 設定変更・サーバ反映
- **課金リクエスト**：有料 API の大量呼び出し
- **クライアント環境への接続・変更**

### 自動ループから送信する場合（最重要）

定期実行で外部送信する仕組みは、誤爆すると **繰り返しスパムする**。次の順で進める：

1. **まず手動で1通テスト送信** → 内容・宛先を社長と目視確認
2. **頻度・宛先・本文フォーマットを明示合意**
3. **重複送信防止と停止条件を実装**（§4）してから
4. はじめて `/loop`・cron へスケジュール化する

「とりあえず回す」は禁止。**小さく送って確認 → 合意 → 自動化** の順を必ず守る。

#### 通知先の取り違え防止（Webhook URL＝送信先権限）

Incoming Webhook は **URL 自体が送信先の権限**。開発用/本番用・社内用/外部共有チャンネルを取り違えると情報漏洩に直結する。

- `.env` に送信先を識別する **`SLACK_DESTINATION_LABEL`** を併せて持たせ、`bin/slack.sh` は成功時・dry-run 時にラベルを表示する
- 環境が分かれる場合は **環境ごとに Webhook を分け**、どの URL がどのチャンネルかを記録する
- 本番の自動送信を開始する前に、**dry-run で送信先ラベルを確認**してから回す

## 4. 自動化スクリプトの作法（確実性）

### bash

- 先頭に `set -euo pipefail`（未定義変数・パイプ失敗を検知）
- 成否は **終了コード**で表す（成功 0 / 失敗 非0）。エラーは標準エラー(`>&2`)へ
- 必須の前提（`.env` の値・引数）が無ければ **明示エラーで即終了**（黙って続行しない）
- 外部呼び出しは **タイムアウト**を付ける（ハング防止）

### 信頼性の作法

| 作法 | なぜ |
|------|------|
| **冪等性** | 同じ入力で何度実行しても安全な結果にする（二重登録・多重通知を防ぐ） |
| **dry-run 対応** | 送信・書き込み系は「送らずに内容だけ表示」モードを用意し、検証に使う |
| **重複送信防止** | 「前回と同じ状態なら送らない」等の状態判定を持たせる（ループのスパム防止） |
| **レート制限** | 連続送信に上限・間隔を設ける |
| **二重起動防止** | `flock` 等でロックし、前回実行が終わっていなければ起動しない（cron と `/loop` の重複・前回の長時間化による多重送信/多重書き込みを防ぐ） |
| **タイムゾーン** | 基準を **`Asia/Tokyo`** に固定（サーバTZ・cronのTZに依存しない）。通知本文には基準時刻を明示する |
| **ログを残す** | いつ・何を・成否を記録。**保存場所・ローテーション・保存期間・マスク対象**を決める（ログが機密の集積場所にならないよう、通知本文とログ本文を分け、秘密値はマスク）。cron 実行なら stdout/stderr の配送先も決める |
| **キルスイッチ** | プロジェクトルートに `STOP` 系ファイルがあれば動かない等、止める手段を用意 |

### 既存資産の踏襲

- 新規スクリプトは `bin/` に置き、`bin/README.md` に1行追記する
- `.env` 読込・`curl` の組み立て・JSON エスケープは `bin/hana-api.sh` / `bin/slack.sh` のパターンに合わせる

## 5. 小さく作って検証してから回す

- **手動単発実行で検証** → 出力を社長と確認 → それから `/loop`・cron へ載せる
- ループ化する前に「暴走しない設計」を確認：**最大回数・実行間隔・停止条件**が決まっているか
- `/loop` のセルフペース（間隔省略）は便利だが、**外部送信を伴うものは間隔・停止条件を明示**してから使う

## 6. 監視・定期通知ジョブの死活（サイレント故障対策）

定期チェックは「異常を通知する」だけだと、**ジョブ自体が止まったことに気づけない**。
スクリプト停止・cron 未実行・Webhook 失効・ネットワーク断などの **サイレント故障** に備える。

- **ハートビート / 最終成功時刻の記録**：実行成功のたびに「最終成功時刻」を残す（ファイル等）
- **更新が途絶えたら別経路でアラート**：一定時間「最終成功」が更新されない場合に気づける仕組みを用意（社長への別チャンネル通知など）
- **送信失敗の可視化**：Slack 送信自体が失敗したときに、その事実をどこに残すか（ログ・別経路）を決めておく
- 「異常時だけ通知」より「**正常も含め定期的に生存を示す**」方が、止まったことに気づける

> いきなり完全な死活監視を作る必要はない。まずは最低限「最終成功時刻を残す」から始め、必要に応じて強化する。

## 7. コミット・デプロイの規律

- コミット済みのものだけサーバへ反映、ステージング後に `git diff --staged` で確認 → 詳細は [development.md](development.md)
- 秘密情報・一時ファイル・自動生成物をコミットに含めない
