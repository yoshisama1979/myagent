# 開発ルール

社内の開発作業における共通ルール。クライアント案件・自社案件問わず、すべての開発に適用する。

> スクリプト・定期実行（`/loop`・cron）・外部送信（Slack等）・API連携を **作る/動かす** ときは、
> 本ファイル（デプロイ・コミット規律）に加えて [automation.md](automation.md)（自動化の安全ルール）を必ず参照する。

## ルール・ドキュメントファイルの設計方針

ルールやドキュメントを新設・拡張するときは、二層に分ける：

1. **誤解防止の核（3〜4行）** … 読まなくても間違えないための最小限。`CLAUDE.md` に直接埋め込む。
2. **肉付け（詳細）** … 必要なときだけ読みに行く参照ファイルに外出しし、`CLAUDE.md` からリンクで誘導する。

理由：`CLAUDE.md` は毎ターン丸ごとコンテキストに載るため、太らせるとトークンを浪費し重要な行が埋もれる。
ただし「リンクだけ」にもしない（タスクに必要と判断されないと読まれないため、核は必ず埋め込む）。
全体像は [OVERVIEW.md](../OVERVIEW.md) に集約。

## 仕様書の更新（機能追加とセット・必須）

**機能を追加・変更したら、対応する仕様書・設計ドキュメントを必ず同じ作業内で更新する**。コードだけ直して仕様書を置き去りにしない（版ズレは「読んだ人が古い仕様で判断する」事故の元）。

| 変更した内容 | 更新する仕様書 |
|---|---|
| ディレクトリ構成・置き場の追加変更 | [OVERVIEW.md](../OVERVIEW.md) |
| モードの新設・役割/連携の変更（ループ・cron・mailbox 等） | [SYSTEM.md](../SYSTEM.md) ＋ 該当モードの `rules/modes/*.md`（version と変更履歴も上げる） |
| AIの基本動作・出力先ルールの変更 | [CLAUDE.md](../CLAUDE.md)（核のみ・肉付けは参照ファイルへ） |
| `site/` ページ・`data/` の追加 | `AI-INDEX.md`（VPSローカル生成物・git外。無ければ `python3 bin/build-ai-index.py` で生成） |
| クライアント案件の機能追加 | その案件の仕様書・記録（`site/clients/<client>/projects/<project>/` や spec.html 等） |

- 更新は**機能追加と同じコミットに含める**（後でまとめて、はやらない＝忘れる）
- 仕様書（OVERVIEW/SYSTEM/CLAUDE/rules/・rules/modes/）の .md を更新したら、**`python3 bin/build-spec-html.py` を実行して HTMLビュー（`site/docs/spec/`・社長がブラウザで読む版）も再生成する**。HTML側を直接編集しない（原本は常に .md・ビューは自動生成＝二重台帳にしない）
- どの仕様書に当たるか迷ったら、コミット前に確認する
- 無人モード（cron 実行）は例外：ゲート対象ファイル（SYSTEM.md・CLAUDE.md・rules 等）を直接編集できないため、**更新案を提案として残し**、有人セッションで反映する（[overseer.md](modes/overseer.md) の運用と同じ）

## デプロイ・アップロード関連

### 1. コミットしたもののみアップ

- **コミット済みのファイルだけ** をサーバーにアップロード（デプロイ）する
- **特に、ステージング前のファイルはアップしない**
- 理由：
  - 未コミットの変更がサーバーに反映されると、ローカルとサーバーの差分が追跡不能になる
  - 障害発生時に「いつの版が動いているか」を特定できなくなる
  - 別の担当者がデプロイしたとき、未コミット分が戻る・上書きされる事故が起きる

### 運用フロー

```
1. ローカルで作業
2. git add（ステージング）
3. git commit（コミット）
4. その後にサーバーへアップロード
```

未確定の修正をサーバーで確認したい場合は、**ステージング環境** を使う（本番に直接上げない）。

## コミット時のルール

### 1. ステージング後に差分を確認してからコミットする

`git add`（ステージング）後、必ず `git diff --staged` で差分を確認してから `git commit` する。**問題ないことを確認してからコミットする**。

### 確認項目

- **不足している情報がないか** — 追加すべきファイル・記述が漏れていないか
- **差分による情報欠落がないか** — 削除予定でない内容が消えていないか、上書きで失われた情報はないか
- **機密情報の混入がないか** — APIキー・パスワード・個人情報等
- **コミットすべきでないファイルが含まれていないか** — 一時ファイル・キャッシュ・自動生成物

### コミットフロー

```
1. git status            … 変更内容を確認
2. git add <file>        … 必要なファイルだけステージング
3. git diff --staged     … 差分を確認
4. 確認項目をチェック    … 不足・欠落・機密混入を再確認
5. 問題なし → git commit
   問題あり → 修正 or git restore --staged <file> → 2 に戻る
```

## 次のアクション

- 各案件のサーバー情報に「本番／ステージング」の区分を `project-info.md` に明記する
- ルール違反が起きやすいタイミング（急ぎの修正依頼等）でも、コミット → アップの順序を守る
