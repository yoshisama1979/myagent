# 開発ルール

社内の開発作業における共通ルール。クライアント案件・自社案件問わず、すべての開発に適用する。

## デプロイ・アップロード関連

### 1. コミットしたもののみアップ

- **コミット済みのファイルだけ** をサーバーにアップロード（デプロイ）する
- **特に、ステージング前のファイルはアップしない**
- 理由：
  - 未コミットの変更がサーバーに反映されると、ローカルとサーバーの差分が追跡不能になる
  - 障害発生時に「いつの版が動いているか」を特定できなくなる
  - 別の担当者がデプロイしたとき、未コミット分が戻る・上書きされる事故が起きる

### 運用フロー

```
1. ローカルで作業
2. git add（ステージング）
3. git commit（コミット）
4. その後にサーバーへアップロード
```

未確定の修正をサーバーで確認したい場合は、**ステージング環境** を使う（本番に直接上げない）。

## コミット時のルール

### 1. ステージング後に差分を確認してからコミットする

`git add`（ステージング）後、必ず `git diff --staged` で差分を確認してから `git commit` する。**問題ないことを確認してからコミットする**。

### 確認項目

- **不足している情報がないか** — 追加すべきファイル・記述が漏れていないか
- **差分による情報欠落がないか** — 削除予定でない内容が消えていないか、上書きで失われた情報はないか
- **機密情報の混入がないか** — APIキー・パスワード・個人情報等
- **コミットすべきでないファイルが含まれていないか** — 一時ファイル・キャッシュ・自動生成物

### コミットフロー

```
1. git status            … 変更内容を確認
2. git add <file>        … 必要なファイルだけステージング
3. git diff --staged     … 差分を確認
4. 確認項目をチェック    … 不足・欠落・機密混入を再確認
5. 問題なし → git commit
   問題あり → 修正 or git restore --staged <file> → 2 に戻る
```

## 次のアクション

- 各案件のサーバー情報に「本番／ステージング」の区分を `project-info.md` に明記する
- ルール違反が起きやすいタイミング（急ぎの修正依頼等）でも、コミット → アップの順序を守る
