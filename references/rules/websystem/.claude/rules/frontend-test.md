# フロントエンド自動テスト作成ルール

## 三者整合性チェック（必須）

フロントエンドの自動テストを作成・修正する際は、以下の **3つの整合性** を必ず確認すること。

### 確認対象

| # | 対象 | 参照先 |
|---|---|---|
| 1 | **仕様書** | `documents/document.md` |
| 2 | **テスト仕様書** | `documents/test.csv` |
| 3 | **実装コード** | `next/src/` 配下の該当コンポーネント・関数 |

### 確認手順

1. **テスト作成前**: 仕様書の該当セクションを読み、期待される挙動を把握する
2. **テスト作成時**: テストの期待値が仕様書の記載と一致しているか確認する
3. **テスト作成後**: 実装コードの挙動がテスト（=仕様書）と一致しているか確認する

### 不整合が見つかった場合

以下のいずれかの不整合が見つかった場合は、**テスト作成を一旦止め、ユーザーに報告**すること。

| 不整合パターン | 報告内容 |
|---|---|
| 仕様書 ≠ 実装コード | 仕様書の記載内容と実装の挙動の差異を具体的に示す |
| 仕様書 ≠ test.csv | test.csv の期待値が仕様書と矛盾している箇所を指摘する |
| test.csv ≠ 実装コード | test.csv の期待値と実装の実際の挙動の差異を示す |

**推測で「正しい方」を判断してテストを書かないこと。** ユーザーの判断を仰ぐ。

## テスト作成の原則

### テストの根拠は仕様書

- テストの期待値は **仕様書（documents/document.md）を正** とする
- コードの現在の挙動をそのままテストにしない（コードにバグがある可能性がある）
- 仕様書に記載がない挙動をテストする場合は、その旨をコメントに残す

### test.csv との連携

- 自動テストでカバーしたテスト項目は、test.csv のチェック欄に「自動テスト」と記載する
- test.csv に該当するテスト項目がない場合は、その旨をユーザーに報告する

---

## テスト環境

- フレームワーク: Vitest + React Testing Library + jsdom
- 設定: `next/vitest.config.*`（拡張子はプロジェクトの ESM/CJS 構成に合わせる）
- テストファイル配置: `next/src/__tests__/`
- 使い方の詳細: `next/src/__tests__/README.md`

---

## コマンド

Docker コンテナ内で実行する。
`<next-container>` は実プロジェクトの Next.js コンテナ名に置き換える
（`docker compose ps` または `docker-compose.yml` の `container_name` で確認）。

```bash
docker exec <next-container> npm test                       # 全テスト実行
docker exec <next-container> npm run test:watch             # watchモード
docker exec <next-container> npx vitest run <ファイル>       # 特定ファイルのみ
```

## テストを書くタイミング

以下のタイミングで関連するテストを作成・追加する。

### 1. 不具合を修正したとき（リグレッションテスト）

- 不具合の再現ケースをテストとして記述してから修正する
- 修正後、テストがGreenになることを確認する
- 将来同じ不具合が再発した際に即座に検知できる状態を保つ

### 2. 新機能を実装したとき

- 機能の主要なユースケース（正常系）と境界条件をテストする
- 仕様書に基づいて期待値を設定する

### 3. コードをリファクタリングしたとき

- 既存テストを実行して挙動が変わっていないことを確認する
- カバーされていない挙動を発見したら、リファクタ前にテストを追加する

---

## テスト実行のタイミング

- **コミット前**: 変更に関連するテストを実行し、Greenであることを確認する
- **リファクタリング後**: 関連テストが全てGreenであることを確認する（必須）
- **不具合修正後**: 追加したテストがGreenであり、かつ既存テストも壊していないことを確認する

---

## テスト環境の方針

テスト環境の**目的・原則・判断基準**を定める。
具体的なバージョン・コマンド・設定値はここに書かず、
`package.json` / `package-lock.json` / config ファイルを一次情報とする。

### 目的

自動テストは、誰がどのマシンで実行しても、
**追加の手動セットアップなしに同じ結果が出る**こと。

### 守るべき原則

1. **単一コマンドで完結**
   `docker exec ... npm test` 等の 1 コマンドで全テストが走る状態を維持する。
   実行に必要な環境変数・プール指定・互換フラグなどは package.json の
   scripts に埋め、利用者が手動で設定する必要をなくす。

2. **lockfile による再現性**
   `package-lock.json` を git に含め、`npm ci` で再現可能にする。
   互換性の都合で特定バージョンに固定する場合は、**固定理由**と
   **解除できる条件**を併記する。

3. **ネイティブ依存の確実な導入**
   `optionalDependencies` 経由のネイティブバインディングは、配布側や
   パッケージマネージャのバグで入らないことがある。入らないと test/build
   が失敗する依存は、`devDependencies` に明示するなど確実に取り込む。

4. **ESM/CJS の整合性**
   設定ファイル（vitest config、next config 等）は、import するパッケージの
   Module Type と整合させる。ESM-only パッケージを使うなら、設定ファイル側も
   ESM（`.mts` 拡張子 or `package.json` の `"type": "module"`）にする。

5. **ホスト環境非依存**
   テストはコンテナ内で完結する。開発者の OS・Node バージョン・
   グローバル npm 設定に依存しない。

### 壊れた / 改善したいときの判断手順

1. **どの原則が破られているか**を特定する
2. **複数の解決策を比較**する。
   その時点の最小修正 vs より根本的な対応（ランタイムや主要依存の更新等）の
   両方を選択肢に入れ、影響範囲と将来の負債化リスクを評価する
3. **採用した解決策の理由**と、**将来再評価する条件**を記録する
4. **テストが通ること**を確認してから完了とする

### 改善のスタンス

回避策は**その時点での最良の妥協案**であって最適解とは限らない。
ランタイム・ツール・依存パッケージの更新で状況が変われば、回避策を
取り除いてシンプルにできる場合がある。
**「いま入っている回避策はまだ必要か」を定期的に見直し、
不要になれば整理する**こと。
