# テスト仕様書作成ルール

## 基本方針

- CSV形式でテスト仕様書を作成する
- 大項目から小項目まで最大5段階の階層構造で整理する
- 各テストケースには期待される動作結果と備考を記載する

## CSVフォーマット

### 列構成（11列）

```
チェック, 通常系/異常系, 大項目, 中項目, 小項目, 手順1, 手順2, 手順3, 手順4, 動作結果, 備考
```

### 各列の説明

1. **チェック**: チェックマーク用（空欄可）
2. **通常系/異常系**: テストの種類（通常系 or 異常系）
3. **大項目**: テスト対象の大分類（機能グループ）
4. **中項目**: テスト対象の中分類（機能名）
5. **小項目**: テスト対象の小分類（具体的なテスト項目）
6. **手順1-4**: テスト実行手順（最大4つ、空欄可）
7. **動作結果**: 期待される結果・確認すべき事項
8. **備考**: 補足情報（空欄可）

### 空セルの扱い

- 手順や備考は必ずしも全て埋める必要はない
- 空セルでもカンマは必ず記載（スプレッドシート貼り付け時の体裁維持）
- 例: `通常系,,大項目,中項目,小項目,手順1,手順2,,,動作結果,`

## 記述ルール

### 1. 項目名の設定

- **大項目・中項目**: 仕様書の項目名から適切に設定し、テスト対象の機能が明確にわかるようにする
- 機能の階層構造を反映させる

### 2. 手順の記述

- **細かくしすぎない**: テスターが理解できる範囲でまとめる
- **同じ手順はまとめる**:
  - ❌ 悪い例: `Aを入力,Bを入力,Cを入力,Dを入力`
  - ✅ 良い例: `A,B,C,Dを入力`
- **内部処理は不要**: ユーザーが実際に確認できる手順のみ記載
- **表示関係のテスト**: 確認すべき項目をまとめて1行に記載
  - 例: `商品名、商品画像、商品価格、商品キャッチコピー、商品説明、オプション選択が正しく表示される`

### 3. 動作結果の記述

- テストの判断基準となる確認すべき事項を明確に記載
- 期待される結果を具体的に記述

### 4. 冗長性の排除

- **縦に連続する同名の手順内容は省略**（入れ子形式で整理）
- 親項目で定義された手順は子項目で繰り返さない

### 5. 並び順

以下の順序でソートする:

1. 大項目
2. 中項目
3. 系の種類（通常系 → 異常系）
4. 手順

### 6. 列数の制限

- 上記11列を超えないこと

## 記述例

### 例1: 基本的なテストケース

```
通常系,,ログイン,認証,正しい認証情報でログイン,メールアドレスを入力,パスワードを入力,ログインボタンをクリック,,ダッシュボード画面に遷移する,
```

### 例2: 表示確認テスト（項目をまとめる）

```
通常系,,商品詳細,商品情報表示,基本情報表示,商品詳細画面を表示,,,商品名、商品画像、商品価格、商品キャッチコピー、商品説明、オプション選択が正しく表示される,
```

### 例3: 複数手順をまとめる

```
通常系,,商品登録,商品情報入力,必須項目入力,商品名・価格・カテゴリを入力,保存ボタンをクリック,,,商品が登録され、一覧画面に遷移する,
```

### 例4: 空セルの扱い

```
異常系,,ログイン,認証,メールアドレス未入力,パスワードを入力,ログインボタンをクリック,,,エラーメッセージが表示される,
```
