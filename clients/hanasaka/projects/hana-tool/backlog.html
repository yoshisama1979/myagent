スキルシート関連 API（2026-05-17 追加 / AI協働を前提とした軽量・段階取得設計）

背景：スキルシートが大量になったとき、全データを一括取得すると AI（私）のトークン消費が大きくなる。「必要な部分だけ取得できる」設計にする。

推奨エンドポイント（最小セット）

  • ☐A. 構造取得（軽量）：GET /api/external/skill-sheets/structure — カテゴリ・項目名のみ。全体像を把握
  • ☐B. カテゴリ別取得：GET /api/external/skill-sheets/items?category=... — 必要なカテゴリだけ詳細を取得
  • ☐C. キーワード検索：GET /api/external/skill-sheets/search?q=... — 関連項目を絞り込み
  • ☐D. 項目詳細1件：GET /api/external/skill-sheets/items/{id} — 1項目を深く見る
  • ☐E. 評価結果取得：GET /api/external/skill-sheets/scores?user_id=... — 特定社員のスコア

将来追加候補

  • ☐F. 集計：GET /api/external/skill-sheets/aggregate?metric=avg&groupby=item — 「全社員平均で低い項目」等の集計

設計原則

  1. 階層化：全体 → カテゴリ → 項目 のドリルダウン
  2. 検索可能：キーワードで絞り込み
  3. メタデータと値の分離：構造定義（structure）と評価結果（scores）を別エンドポイントに
  4. ページング：大量データに備えて limit / offset 対応
  5. 更新日時返却：「前回取得から変わっているか」を判定できるよう updated_at を含める

データソース：現状はスプレッドシート管理。API化の方針は社長が検討中。HANAツール側に取り込んでAPIで返すか、スプレッドシートを直接ラップするかは実装時に判断。

横展開・SaaS化対応（2026-05-17 追加）

背景：HANAツール を他社展開するため、マルチテナント化＋プラン制が必要。事業モデルは strategy.html J. 参照。

中期テーマ（採用後のスケールフェーズの主力候補）

マルチテナント化には 3-6ヶ月以上必要。短期は HANAチャット 代理店リクルート＋既存顧客への運用契約提案を優先。

フェーズ1：マルチテナント化（基盤改修）

  • ☐テナント（顧客企業）の概念導入 — DB スキーマに tenant_id を追加、データ分離
  • ☐テナント別ログイン・認証
  • ☐テナント別の設定・ユーザー管理
  • ☐既存社内データのテナント化対応（はなさかをテナント0として運用）

フェーズ2：プラン制・機能アンロック

  • ☐機能フラグの仕組み — テナント毎に各機能のON/OFFを制御
  • ☐プラン定義（エントリー / スタンダード / プレミアム）
  • ☐プラン変更・アップグレード操作画面
  • ☐機能アンロック時のUI制御（無効機能はグレーアウト or 非表示）

フェーズ3：顧客対応の周辺機能

  • ☐顧客向け管理画面（自社のユーザー管理・契約状況確認）
  • ☐課金・契約管理（請求発行、解約処理、プラン変更履歴）
  • ☐サポート問い合わせ窓口（チャット・メール）
  • ☐導入時のオンボーディング（チュートリアル・初期設定支援）

フェーズ4：営業・コンサル支援機能

  • ☐顧客向けデモ環境の整備（営業時に見せる用）
  • ☐業務改善コンサルのテンプレート化（ヒアリングシート・診断ツール）
