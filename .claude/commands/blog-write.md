あなたは現在 **ブログ執筆（対応）モード** で動作します。

ブログ分析ループ（`/blog-loop`）が出した提案——**既存記事の改善 `B-NNN`／新規記事のテーマ＋構成 `T-NNN`**——に実際に対応し、**完成原稿ドラフト（本文まで）** を作る実行モードです（blog-loop の対になる「分析↔実装」の実装側）。将来は `/loop`・cron でループ化する想定ですが、**まずは手動で動かして検証**します。

## 対象クライアントの決定（最初に必ず）

引数 `<client>`（client-key）：**$ARGUMENTS**（例 `ycom`。無指定なら後方互換で `ycom`）
1. `data/blog-loop/config.md`（登録表）と `data/blog-loop/clients/<client>.md` を読み、固有値を確定（ブログURL・ゴール・執筆/実装担当・GSC/GA4 dataset・WP情報）。
2. **client-key が登録表に無いなら、対応に入らず**「未登録クライアント」と報告して終了。

## 必読ファイル（順番に読む）

1. `rules/modes/blog-write.md` — 本モードの詳細ルール（公開の境界＝段階導入・手順・品質基準・自己チェック）
2. `rules/modes/hp-improve.md` — コンテンツ品質の評価観点を流用
3. `rules/automation.md` — 本番書き込み・外部送信・秘密情報の安全ルール（**衝突したら最優先**）
4. `CLAUDE.md` — 行動指針・出力先の判断基準（既読なら省略可）

## 入力と出力

- **入力**：blog-loop 掲示板 `site/hp-analysis/<client>/blog/index.html` の未完了 `B-NNN`／`T-NNN` ＋ `data/blog-loop/<client>/from-president.md` の社長指示（着手順・優先・回答済みの固有事実）。
- **出力**：完成原稿ドラフトを `site/drafts/blog/<client>/<slug>.html` に（1ドラフト1ファイル）。対応ログを `data/blog-loop/<client>/drafts-log.md` に追記。

## 公開の境界（段階導入・automation.md）

最終形は「本番WPに**下書き(status=draft)投稿**まで・公開は人間」だが、いきなり自動投稿しない：
- **Phase 1（現在）**：ドラフトを `site/drafts/` に**ファイル生成のみ**。本番に触れない。
- **Phase 2（creds設定済＋手動テスト＋社長合意の後）**：WP REST API で `status=draft` 投稿。認証は `.env`（gitignore・社長手元設定）。**creds 未設定なら投稿せず「保留」と明示**（捏造・空振りしない）。
- **Phase 3**：ループ化。

## 実行

1. `rules/modes/blog-write.md` を読み、ルールを把握する
2. その「手順」セクションに従って Step 0 〜 Step 6 を実行する
3. **対象未確定／事実待ちで着手できないなら、対応に入らず質問・保留を報告**して終了

## 重要（厳守）

- **成果物＝完成原稿ドラフト（本文まで）**。新規 `T` は本文込み記事、既存 `B` は「差し替える title/meta・追記本文・内部リンク・CTA」を具体テキストで。
- **固有事実（料金・補助金・実績数値・お客様の声・受賞歴）を創作しない**＝手元に無ければ `【要・社長確認：…】` プレースホルダで要求。
- **地域キーワード詰め込み・競合の劣化コピー・量産はしない**＝コンテンツベースで独自価値（社長方針 2026-06-23）。
- **無断で本番WPに公開/投稿・既存記事を本番上書きしない**（下書きまで・公開は人間・合意ゲート）。
- 既存改善は**現行記事を取得してから**差分を作る（現状を見ずに直さない）。
- `from-president.md`・blog-loop 掲示板は**編集しない**（責務分離。対応の事実は drafts-log と報告で渡す）。
- 1回で大量に作らない（新規 `T` 1本＋既存 `B` 1〜2件目安）。
- `.env` 実値を表示しない。このモードでは **git commit / push をしない**。
- 編集・作成したファイルは回答末尾に markdown リンクで一覧表示する。
