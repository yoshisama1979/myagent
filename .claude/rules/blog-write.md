# ブログ執筆（対応）モード — 動作ルール

> **version 0.1**（2026-06-23・初版＝型）｜ changelog はファイル末尾「変更履歴」を参照。
> このモードは **使いながら育てる**（reviewer 型：記録と反映を分け、棚卸しでルールへ昇格）。気づきは `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、**ブログ分析ループ（`/blog-loop`）が出した提案（既存記事の改善 `B-NNN`／新規記事のテーマ＋構成 `T-NNN`）に実際に対応し、完成原稿ドラフト（本文まで）を作る** 実行モードの **手順・制約・出力フォーマット・自己チェック** を定義する。

`/blog-write <client>` スラッシュコマンドから呼び出される（将来は `/loop`・cron でループ化する想定。**最初は手動で動かして検証する**）。
CLAUDE.md の行動指針と [rules/automation.md](../../rules/automation.md)（自動化・外部送信・本番書き込みの安全ルール）を土台とする。**衝突したら automation.md が最優先。**

---

## ⚑ 位置づけ（blog-loop の対になる実行モード・最初に把握）

- **blog-loop（診断・提案）＝「何を直す／何を書くか」を構成案まで** 出す。**blog-write（このモード）＝その提案に対応して「実際の原稿・改善テキストを作る」実行側**。hp-loop ↔ web-hanasaka と同じ「分析↔実装」の分担を、ブログ執筆について1システム内で担う。
- **成果物＝完成原稿ドラフト（本文まで）**（社長決定 2026-06-23）。新規記事 `T` は本文込みの記事ドラフト、既存改善 `B` は「差し替える title/meta・追記する本文・内部リンク・CTA」を具体テキストで。**そのまま公開前レビューできる粒度**。
- **重複定義しない**：安全ルール＝[rules/automation.md](../../rules/automation.md)、コンテンツ品質の評価観点＝[hp-improve.md](hp-improve.md)、対象クライアントの固有値（ブログURL・ゴール・担当・dataset）＝blog-loop と同じ `data/blog-loop/clients/<client>.md` を**流用**する。本ファイルは **執筆・対応の実行** 固有だけを定義する。

## ⚑ マルチクライアント読み替え

`/blog-write <client>` でクライアント別に動く。`data/blog-loop/config.md`（登録表）と `data/blog-loop/clients/<client>.md` を読んで固有値を確定してから本ルールに入る。以降の固定値は「現在の `<client>` の値」に読み替える：

| 表記 | 読み替え |
|---|---|
| 入力（提案） | blog-loop 掲示板 `site/hp-analysis/<client>/blog/index.html` の `B-NNN`／`T-NNN`（＋ `data/blog-loop/<client>/from-president.md` の社長指示） |
| ドラフト出力先 | `site/drafts/blog/<client>/`（公開前の草案。プレビューURLで確認可） |
| 対応ログ | `data/blog-loop/<client>/drafts-log.md`（どの B/T に対応してどのドラフトを作ったか） |
| WP下書き投稿（Phase 2） | そのクライアントの WordPress REST API（認証は `.env`・gitignore。**creds 未設定なら投稿しない**） |

## 公開の境界（段階導入・automation.md：小さく作って手動検証→合意→自動化）

成果物の最終形は「**本番WPに下書き(status=draft)投稿まで**」（社長決定 2026-06-23）。だが本番WPへの書き込みも**外部・本番への書き込み**なので、いきなり自動投稿しない。次の段階で進める：

| Phase | 内容 | ゲート |
|-------|------|-------|
| **1（現在）** | 完成原稿ドラフトを `site/drafts/blog/<client>/` に**ファイルとして生成**。本番には一切触れない。社長がプレビューURLでレビュー | なし（ローカル草案） |
| **2** | WP REST API で `status=draft` 投稿（**下書き止まり・公開(publish)は人間**）。認証(Application Password等)は社長が `.env` に設定 | **creds 設定済 ＋ 手動で1本テスト ＋ 社長合意** が揃って初めて有効。creds 未設定なら投稿せず「WP認証未設定のため下書き投稿は保留」と明示（捏造・空振りしない） |
| **3** | `/loop`・cron でループ化（毎日ドラフト生成→下書き投稿） | Phase 2 の運用が安定＋社長合意後 |

> **publish（公開）は常に人間**。本モードは下書きまで。下書き投稿でも、その回ごとに automation.md の合意ゲートに従う（無断で本番WPに書かない）。
>
> **🔒 本番保護の要：無人ループの `.env` は「公開権限を持たない専用WPユーザー」で（強く推奨・Codex指摘 2026-06-23）**。`wp-draft.py` は client 側で「draft のときだけ更新」をガードするが、GET→更新の間に公開される TOCTOU は client 側だけでは塞げない。**publish_posts／edit_published_posts を持たない寄稿者(Contributor)相当の専用ユーザー**でアプリパスワードを発行すれば、万一でも WP 側が公開記事の上書き・公開を物理的に拒否する。`wp-draft.py check` はこの権限を持つユーザーだと警告を出す。

## 厳守する制約

| 操作 | 可否 |
|------|------|
| Read（任意）/ Grep / Glob / git status・log・diff | ✅ 可 |
| blog-loop 掲示板・from-president・clients/<client>.md を読む | ✅ 可（入力） |
| 既存記事・競合の取得（`curl`・`bin/hp-audit.sh`・`bin/hp-serp.sh`・読み取り） | ✅ 可（執筆の裏取り。手作業は「暫定」と明示） |
| Write: `site/drafts/blog/<client>/`（完成原稿ドラフト） | ✅ 可（AIの担当・Phase 1） |
| Write: `data/blog-loop/<client>/drafts-log.md`（対応ログ） | ✅ 可 |
| **WP REST API で `status=draft` 投稿** | **⚠️ Phase 2・creds設定済＋手動テスト＋社長合意の後のみ**。creds は `.env` から読む（実値はAIが表示しない）。**publish は不可（人間）** |
| **固有事実（料金・補助金制度・実績数値・お客様の声・受賞歴等）を創作** | **❌ 不可**。手元に無ければ**プレースホルダ**（例 `【要・社長確認：◯◯の数値】`）にして要求に回す |
| **地域キーワード詰め込み・競合記事の劣化コピー・量産スパム** | ❌ 不可（コンテンツベースで独自価値を出す。社長方針 2026-06-23） |
| **本番WPの公開(publish)・既存記事の本番上書き** | ❌ 不可（下書きまで・公開は人間。既存改善Bも「ドラフト/差分案」として出し、本番反映は担当/人間） |
| **`data/blog-loop/<client>/from-president.md` の編集** | ❌ 不可（社長の領域・読むだけ） |
| **blog-loop 掲示板（index/spec/archive）の編集** | ❌ 不可（blog-loop の責務）。対応の事実は drafts-log と報告で渡し、blog-loop が次サイクルで反映 |
| 外部送信（Slack/メール等） | ❌ 合意前は不可（automation.md §3） |
| `.env`・`data/secrets/` の実値を **AI が直接読む・表示する** | ❌ 不可（automation.md §2）。※承認済みスクリプト（`wp-draft.py`）が必要キーだけ実行時に読むのは可＝AIは値を閲覧・出力しない |
| git commit / push | ❌ このモードではしない（社長判断・別タイミング） |

## 執筆の品質基準（hp-improve のコンテンツ観点を土台に）

ドラフトは「検索上位を取りに行ける・読者の意図を満たす・CVに繋がる」ものにする：

| 基準 | 中身 |
|------|------|
| 検索意図の充足 | 提案 `T`/`B` の「狙うクエリ」の意図（知りたい/比べたい/買いたい）に過不足なく答える |
| E-E-A-T・独自性 | 一般論の寄せ集めにしない。YCOMの実務視点・具体例で独自性を出す（固有事実はプレースホルダで要求） |
| 構成 | 提案のアウトライン（h2/h3）に沿う。導入で結論、見出しで意図を構造化、読みやすい段落 |
| **読みやすさ・ビジュアル** | **箇条書きの羅列で終わらせない（無機質禁止）。比較・一覧・手順・料金などは表(`<table>`)で見せる。段落・表・番号/箇条書き・コールアウトを使い分けて単調にしない**（下記「読みやすさの作り込み」） |
| 内部リンク・CTA | 受け皿ページ（/price/・制作実績・問い合わせ）への文脈リンクと、記事内の自然なCTAを必ず入れる |
| SEO基礎 | SEO向けタイトル（クエリ含む・32文字前後）・meta description（120文字前後）・slug 案・適切な見出し階層・alt案 |
| 捏造しない | 数値・実績・補助金制度・お客様の声は創作せず `【要・社長確認：…】` で明示 |

### 読みやすさの作り込み（無機質にしない・必須）

社長フィードバック（2026-06-23）：「箇条書きだけだと無機質。表などを使って分かりやすく」。記事は**読み手が"流し読みでも要点を掴める"視覚構造**にする。

**表(`<table>`)を積極的に使う**——次のような内容は箇条書きより**表が圧倒的に分かりやすい**ので、原則テーブル化する：

| こういう内容 | 表の形（例） |
|---|---|
| 比較（A vs B・自作vs依頼・プラン比較） | 行＝比較軸／列＝選択肢。各セルに○△×や一言 |
| 料金・相場 | 目的/プラン × 費用感（＋含まれるもの）。`【要・社長確認】`の数値はプレースホルダ |
| チェックリスト・選び方の観点 | 観点 × 見るポイント（＋チェック欄□） |
| メリット・デメリット | メリット／デメリットの2列 |
| 手順・ステップ | ステップ × やること × ポイント |
| 用途別の使い分け（画像形式など） | 種類 × 向いている用途 × 注意 |
| Before/After・改善前後 | 項目 × 変更前 × 変更後 |

**構造に変化をつける**：導入（結論ファースト・2〜3文）→ 見出しごとに「短い段落＋表 or 番号リスト」を組み合わせる。**同じ形式（箇条書き）を延々続けない**。要点は**太字**、長文は短段落に割る。

**表で表しきれない関係・流れ**（フロー・相関図など）は、無理に文章で書かず **SEOメタの「差し込み素材依頼」に図版を明記**して人間/デザイン側に依頼する（捏造の作図はしない）。

**WordPress 前提**：シンプルな `<table>`（必要なら軽い枠線スタイル）で書く。**スマホでも崩れない列数（2〜4列目安）**に抑える。装飾過多にしない（中身で分からせる）。

## 手順（1サイクル＝1回の対応）

### Step 0: 前提読み込み
1. `data/blog-loop/config.md`＋`data/blog-loop/clients/<client>.md` を読む（ブログURL・ゴール・担当・dataset）。**未登録クライアントなら対応に入らず報告して終了**。
2. blog-loop 掲示板 `site/hp-analysis/<client>/blog/index.html` を読み、未完了の `B-NNN`／`T-NNN`（🔧進行中の表）を把握。
3. `data/blog-loop/<client>/from-president.md` を読む（着手順・優先・回答済みの固有事実）。
4. `data/blog-loop/<client>/drafts-log.md`（あれば）を読み、**既に対応済み・レビュー中のものを二重に作らない**。

### Step 1: 対応対象を選ぶ
- **社長の指示（from-president の優先・Q-001の回答）があればそれに従う**。無ければ blog-loop の優先度（🔴→🟡→🟢）順に、**1回で 新規記事 `T` 1本＋既存改善 `B` 1〜2件 目安**（やり過ぎない・毎日積み上げる前提）。
- **固有事実が必須でまだ未提供の提案（例：料金記事で価格未確定）は、本文の事実部分をプレースホルダにして書くか、事実待ちなら着手を保留**して掲示板の質問へ回す（捏造しない）。

### Step 2: 裏取り（読み取り）
- 既存改善 `B`：対象記事を `curl`/`hp-audit` で取得し、現行の title/meta/本文/内部リンクを確認してから差分を作る（現状を見ずに直さない＝K-014 と同じ思想）。
- 新規記事 `T`：必要なら `hp-serp` で上位記事の論点を確認し、**それを超える独自の切り口**を設計（劣化コピーにしない）。

### Step 3: ドラフト作成（完成原稿・本文まで）
- 出力フォーマット（下記）に従い、`site/drafts/blog/<client>/<slug>.html` に1ドラフト1ファイルで作る。
  - 新規 `T`：SEOメタブロック（title/meta/slug/カテゴリ/内部リンク先/差し込み素材依頼）＋**記事本文（HTML・見出し・段落・CTA）**。
  - 既存 `B`：対象URL＋「差し替え前→差し替え後」の title/meta・追記する本文ブロック・挿入する内部リンク/CTA を具体テキストで。
- 固有事実は `【要・社長確認：…】` プレースホルダ。地域キーワード詰め込み・量産はしない。

### Step 4: 対応ログ＋（Phase 2 なら）下書き投稿
- `data/blog-loop/<client>/drafts-log.md` に追記：対応した `B/T`・作ったドラフトのパス・状態（✍️ドラフト完成／⏳事実待ち）・**残課題＝必要な事実を「🔧開発確認／👤社長確認」に分類**して記録（下記）。
- **必要な事実の振り分け（捏造しないための調達先）**：
  - **🔧 開発確認**（web-hanasaka が納品物・実装から答えられる）：サイトの機能・工夫・実装の特徴・技術的な対応範囲 等。→ **blog-loop が mailbox で web-hanasaka に質問**して回収する（blog-write は drafts-log に「開発確認」と書くだけ。自分で web-hanasaka に送らない＝対外的な対話は解析ループに集約）。
  - **👤 社長確認**（経営判断・固有数値）：料金方針・数値の実績成果・お客様の声の可否 等。→ from-president / Slack（社長へ）。
  - どちらに振っても、回答が来るまでは本文を**プレースホルダのまま**にし、勝手に埋めない。
- **下書きは毎サイクル WP に反映して社長レビューに供する（社長決定 2026-06-23 改）**：新規 `T` の記事は、社長が WP 管理画面で体裁・内容を確認できるよう **`status=draft` で投稿/更新する**。**`【要・社長確認】` が残っていても投稿/更新してよい**（プレースホルダは下書き上に残し、社長が WP で見て指示・事実を返す＝レビュー駆動）。**公開(publish)は常に人間**。
  - **新規 or 更新の判定（重複を作らない）**：drafts-log にその記事の **WP下書きID** が無ければ `bin/wp-draft.py post …`（新規作成）し、**返ってきたIDを drafts-log に記録**。既にIDがあれば `bin/wp-draft.py update --id <id> …`（同じ下書きを更新＝デザイン改訂・事実追記を反映）。`update` は対象が **draft のときだけ**実行（公開済みは上書きしない＝ツール側でもガード）。
  - **既存改善 `B` は WP 自動投稿の対象外**：`wp-draft.py` は新規記事の下書き作成/更新用。`B` は「差分案」として drafts-log＋掲示板で web-hanasaka／人間に引き渡す（本番反映は人間）。
  - **1サイクル＝新規は最大1本**（毎日1本供給）。既存ドラフトの更新は随時。creds 未設定／許可未付与なら投稿せず「保留」と明示（捏造・空振りしない）。

> **fact質問の橋渡しは blog-loop が担う**：drafts-log に「🔧開発確認」と書いた事実は、blog-loop が次サイクルで web-hanasaka に mailbox 質問して回収し、回答を spec.html / 確定事実に記録する。blog-write はその確定事実を使って本文を埋め、下書きを更新する（[blog-loop.md](blog-loop.md)「開発エージェントへの fact 質問」）。

### Step 5: 報告
- 社長に1〜3行で要約：対応した `B/T`・作ったドラフト（プレビューURL `http://100.123.104.87/drafts/blog/<client>/<slug>.html`）・`【要・社長確認】`の事実一覧・WP下書き投稿の状態（保留/完了）。
- 停止条件：対象クライアント未確定／事実待ちで着手できない／社長の停止指示 → 一旦停止。

### Step 6: 自己改善（軽く・毎回）
- 「今回足りなかったものは？」を自問。気づきは `data/hp-improve/skill-kaizen.md` に `K-NNN`（無ければ「特記なし」）。

## 出力フォーマット（site/drafts/blog/<client>/<slug>.html）

各ドラフトは HTML（Tailwind・レビューしやすく）。先頭に**SEOメタブロック**（レビュー用の枠）、その下に**記事本文プレビュー**を置く：

- **メタ**：対応 `B/T`・SEOタイトル案・meta description 案・slug 案・カテゴリ/タグ案・狙うクエリ・内部リンク先・差し込み素材（画像/図）依頼・`【要・社長確認】`の事実一覧
- **本文**（新規 `T`）：見出し（h2/h3）・段落・表・記事内CTA・受け皿ページへの内部リンクを実テキストで（WordPress 貼り付けを想定した素直なHTML）
- **差分**（既存 `B`）：対象URL、`title`：現行→改善案、`meta`：現行→改善案、追記する本文ブロック、挿入する内部リンク/CTA

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 数値・実績・補助金・お客様の声を創作 | 信頼を壊す。プレースホルダで要求する |
| ❌ 無断で本番WPに公開/投稿・既存記事を本番上書き | 本番書き込み。下書きまで＋合意ゲート（automation.md §3）。公開は人間 |
| ❌ creds 未設定なのに「投稿した」と装う | 捏造・空振り。保留と明示する |
| ❌ 競合記事の劣化コピー・地域KW詰め込み・量産 | コンテンツベースで独自価値（社長方針） |
| ❌ from-president.md・blog-loop 掲示板を編集 | 責務分離違反。対応の事実は drafts-log と報告で渡す |
| ❌ 現状記事を見ずに既存改善を作る | 的外れになる。必ず現行を取得してから差分（K-014） |
| ❌ 1回で大量に作る | 毎日積み上げる前提。T1本＋B1〜2件目安 |
| ❌ `.env` 実値の表示・このモードで commit/push | 秘密漏洩・責務外 |
