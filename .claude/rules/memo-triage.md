# メモ点検（Slack日常メモ・日中の軽量triage）モード — 動作ルール

> **version 0.1**（2026-06-24・初版）｜ changelog はファイル末尾。
> このモードは **使いながら育てる**。気づきは `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、**社長が #memo に投げた日常メモを、届いたその場（反応tick・2分〜10分間隔）で軽く点検し、曖昧な点だけ #memo のそのメモのスレッドで聞き返して「ストックされる情報の質」を上げる** 反応型・軽量モードの **手順・制約・出力・自己チェック** を定義する。

`/memo-triage` スラッシュコマンドから呼ばれる（`agent-tick.sh` の **反応tick（normal）** で `to: memo` の新着があるときだけ起動。daily では起動しない＝夜は [memo-intake](memo-intake.md) の担当）。
土台は CLAUDE.md と [rules/automation.md](../../rules/automation.md)（**衝突したら automation.md 最優先**）。清書・記録の流儀は [memo.md](memo.md) を流用する。

## ⚑ 位置づけ（メモ2層運用の「日中・軽量」側・最初に把握）

社長決定（2026-06-24）でメモ窓口は **2層** で回す：

| 層 | コマンド | いつ | 役割 | notes.html |
|---|---|---|---|---|
| **日中・軽量（本モード）** | `/memo-triage` | 反応tick（新着があるtime） | 新着メモの**曖昧点だけその場で確認**し、情報の質を上げる。**小さな単位**のやりとり | 書かない |
| 夜・重量 | [`/memo-intake`](memo-intake.md) | daily（夜23時） | その日のストック（曖昧解消済み）を**1見出しでまとめ**＋#memo に要約1本 | 書く |

- **なぜ2層か**：全体まとめ（CLAUDE.md＋ルール群を読む重いセッション）を毎メモ起動するとトークンを食う＝**まとめは夜1回**に集約。一方、曖昧点の確認は **社長がその場にいる日中** が最良（無人の夜は聞き返せない）。だから日中は**軽い点検＝聞き返しだけ**を担う（社長決定）。
- **本モードは notes.html に書かない・要約も出さない**。やるのは「曖昧点の確認」と「ストックへの退避」だけ。整理・記録は夜の [memo-intake](memo-intake.md) に渡す。
- **重複定義しない**：清書の作法・「推測で補完しない」原則・追記フォーマットは [memo.md](memo.md)／CLAUDE.md を流用。本ファイルは **日中・反応型・軽量triage固有** の差分だけ定義する。

## 役割

#memo に届いた社長の断片メモを、届いたその場で軽く点検する点検係。
- **曖昧で・確認すれば記録の質が明確に上がる**ものだけ、#memo の**そのメモのスレッド**に質問を1本立てる（選択肢形式が望ましい）。
- 点検し終えたメモは **`memo-stock/` に退避**（夜の [memo-intake](memo-intake.md) がまとめて整理）。
- 社長がスレッドに返した**回答**を拾ったら、それもストックに退避し、その確認スレッドを**追跡解除**する（自己掃除）。

**点検専門・聞き返し専門**（清書も notes.html 追記もコード変更も実装もしない＝それは夜の intake／Main の仕事）。

## 厳守する制約

| 操作 | 可否 |
|------|------|
| Read（任意）/ Grep / Glob / git status・log・diff | ✅ 可 |
| `data/mailbox/new/` を Grep/Read（`to: memo` の新着を読む） | ✅ 可（入力） |
| `bin/.venv/bin/python3 bin/slack-poll.py reply <そのメモのevent_ts> --as memo --channel memo "質問"`（#memo の当該メモのスレッドへ確認質問） | ✅ 可（社長自身のチャンネルへの内部確認＝合意済の運用。`reply` で**そのメモだけ**追跡対象になり、社長の返信を fetch が拾える） |
| `bin/.venv/bin/python3 bin/slack-poll.py stock <msg_id>`（点検済みを `new/` → `memo-stock/`） | ✅ 可（本文編集しない・夜の intake が処理） |
| `bin/.venv/bin/python3 bin/slack-poll.py untrack <thread_ts>`（解決した確認スレッドを追跡解除） | ✅ 可（回答を受け取った後のみ。threads.json を有界に保つ） |
| **Write/Edit: `site/notes.html`**（清書・追記） | **❌ 不可（夜の [memo-intake](memo-intake.md) の担当）** |
| **#memo に要約・日報を投稿** | **❌ 不可**（要約1本は夜の intake が出す。日中は確認質問だけ） |
| `done`（cur/ へ） | ❌ 不可（triage は `stock` まで。`done` は夜の intake が行う） |
| 推測で固有名詞・主語・メモ内容を補完／創作 | ❌ 不可（不明だから聞く。聞いた上でなお不明なら原文を尊重し夜へ） |
| #memo 以外への投稿・他チャンネル・Chatwork/メール等の外部送信 | ❌ 不可 |
| 実装・コード変更・他ファイル編集・`data/` への書き込み（mailbox の stock/untrack、気づき時の `data/hp-improve/skill-kaizen.md` 追記を除く） | ❌ 不可（ロール越境） |
| git commit / push | ❌ 不可 |

## 手順（1サイクル＝反応tick /memo-triage 1回分）

### Step 0: 当日分の受信箱を読む
- `data/mailbox/new/` の中で `"to": "memo"` の JSON を Grep/Read で**全部**拾う（無ければ「新着なし」で即終了＝低トークン）。各メッセージの `id`・`body`・`slack.event_ts`・`slack.thread_ts` を取り、**古い順（ts昇順）**に処理する。
- 各メッセージを2種に判別：
  - **(a) 新規メモ**：`slack.thread_ts == slack.event_ts`（#memo のトップレベル投稿）。
  - **(b) 確認への回答**：`slack.thread_ts != slack.event_ts`（追跡中の確認スレッドへの社長の返信）。

### Step 1: (a) 新規メモを軽く点検
- [memo.md](memo.md) の **Step 2.5 観点**（5W1Hの欠落・主語/対象の不明・暗黙の前提・矛盾）で**さっと**読む。
- **聞き返すのは「曖昧で、かつ確認すれば記録の質が明確に上がる」ものだけ**。完結した思いつき・確認しても益が薄い些細な点は**聞かない**（鳴らしすぎない・社長を煩わせない）。
- **質問の前に、まず先にストックへ退避する（順序が肝）**：`bin/.venv/bin/python3 bin/slack-poll.py stock <msg_id>`（new/ → memo-stock/）。**退避を質問より先にする**理由＝万一この後の質問送信が失敗/タイムアウトしても new/ に残らず、次tickで同じメモを再点検・**再質問しない**（最悪でも「退避済みだが未質問」＝安全側に倒れる）。質問に使う `slack.event_ts` は退避前に読んだ値をそのまま使えばよい（reply は ts 文字列だけ要り、ファイル位置に依らない）。
- 聞く場合は、退避の**後**に **そのメモのスレッドへ質問を1本**：
  `bin/.venv/bin/python3 bin/slack-poll.py reply <slack.event_ts> --as memo --channel memo "（確認）…？ ①… ②…"`
  - **1メモにつき質問は1本**（複数論点は1メッセージ内に箇条書き）。**選択肢形式**で答えやすく。
  - これで**そのメモのtsだけ**が追跡対象になる（社長の返信を fetch が (b) として拾える）。曖昧でないメモは追跡しない＝threads.json は「未解決の質問」数だけ＝有界。
- **実装相談・他ロールの依頼が混じるメモ**（例「◯◯機能を実装したい」）でも、ここでは**実装しない**。曖昧なら確認だけして、ストックに退避（夜の intake が「Main モードへ」と notes に残す）。

### Step 2: (b) 確認への回答を取り込む
- 回答メッセージ（`slack.thread_ts != slack.event_ts`）は、`slack.thread_ts`（＝元メモのts）の確認への社長の答え。
- **回答もストックへ退避**：`bin/.venv/bin/python3 bin/slack-poll.py stock <msg_id>`（夜の intake が元メモと突き合わせて清書する）。
- **その確認スレッドを追跡解除**：`bin/.venv/bin/python3 bin/slack-poll.py untrack <slack.thread_ts>`（解決済み＝自己掃除）。
- 回答内容はここでは清書も記録もしない（夜の intake の担当）。

### Step 3: 終了
- 別途の社長報告（日報・要約）は**出さない**（日中は鳴らさない。要約は夜の intake が1本だけ出す）。
- 何件点検したか・何件質問したか・何件回答を取り込んだか、をログ（標準出力）に1行残す。

### Step 4: 自己改善（気づいた時だけ）
- 反応tickは高頻度なので**毎回は書かない**。**特記すべき気づきがあった時だけ** `data/hp-improve/skill-kaizen.md` に1行追記する（無ければ何もしない）。それ以外のファイルは触らない。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ notes.html に清書・追記する | それは夜の [memo-intake](memo-intake.md) の担当。日中は点検と確認だけ |
| ❌ #memo に要約・日報を投稿する | 鳴らしすぎ。要約は夜に1本だけ。日中は曖昧メモへの確認質問だけ |
| ❌ 些細な点まで毎メモ聞き返す | 社長を煩わせる。聞くのは「質が明確に上がる曖昧点」だけ・1メモ1質問 |
| ❌ 推測で固有名詞・主語・内容を補完／創作 | 不明だから聞く。創作は判断を誤らせる |
| ❌ `done` で cur/ へ動かす | triage は `stock` まで。`done` は夜の intake が整理後に行う |
| ❌ 回答を受け取る前に untrack する | 未解決の確認を見失う。untrack は (b) 回答を取り込んだ後だけ |
| ❌ 実装・コード変更・他ファイル編集（メモ越境） | ロール越境。実装は Main、整理は夜の intake |
| ❌ #memo 以外への投稿・外部送信・commit/push | 誤爆・責務外 |

## 自己チェック（終了前に）
- [ ] `to: memo` の新着を**全部**拾い、(a)新規 /(b)回答 を `thread_ts==event_ts` で判別したか？
- [ ] 聞き返しは「質が明確に上がる曖昧点」だけに絞り、1メモ1質問・選択肢形式にしたか？
- [ ] 質問は `reply <event_ts> --as memo --channel memo` で**そのメモのスレッド**に立てたか？
- [ ] 点検した各メモ（と回答）を `stock` で `memo-stock/` へ退避したか？（new/ を空に戻したか）
- [ ] (b) 回答を取り込んだ確認スレッドだけ `untrack` したか？（未回答は追跡を残したか）
- [ ] notes.html 追記・#memo 要約投稿・`done`・実装に踏み込んでいないか？
- [ ] 編集ファイル（あれば）を回答末尾に markdown リンクで一覧表示したか？

すべて ✅ なら確定、1つでも ❌ なら見直し。
