# ブログ分析ループモード — 動作ルール

> **version 0.1**（2026-06-23・初版＝型）｜ changelog はファイル末尾「変更履歴」を参照。
> このモードは **使いながら育てる**（reviewer 型：記録と反映を分け、棚卸しでルールへ昇格）。気づきは hp-loop と同じく `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、クライアントのブログ（記事コンテンツ）を **定期的に分析 → ①既存記事の改善案 ②新規記事のテーマ案＋構成案 を掲示板に積み → 社長と非同期で会話** するループの **手順・掲示板プロトコル・制約・出力フォーマット・自己チェック** を定義する。

`/blog-loop <client>` スラッシュコマンドから呼び出される（`/loop`・cron で **HP分析(`/hp-loop`)とは別タイミング**で定期実行する想定）。
CLAUDE.md の行動指針と [rules/automation.md](../../rules/automation.md)（自動化・外部送信の安全ルール）を土台とする。

---

## ⚑ 位置づけ（hp-loop の姉妹モード・最初に把握）

- **hp-loop（HP分析）＝サイト全体の技術・導線・CV・SEO** を診断する。**blog-loop（このモード）＝ブログ記事そのもののコンテンツSEO**（既存記事の磨き込み＋新規記事のネタ出し）に特化する。
- **役割は「診断・提案・構成案まで」（執筆・公開はしない）**。hp-loop と同じ役割分担で、提案は社長が**対象クライアント担当の別エージェント（執筆・実装担当）に引き渡して**記事化・公開する。本モードは記事本文を書かない（アウトラインまで）。
- **重複定義しない**：安全ルール＝[rules/automation.md](../../rules/automation.md)、評価観点のコンテンツ部分＝[hp-improve.md](hp-improve.md)、掲示板3層・mailbox配信・責務分離・COMPOUND の作法＝[hp-loop.md](hp-loop.md) を **土台として流用**する。本ファイルは **ブログ固有** だけを定義する（片方を直して他方に古い記述が残る版ズレを防ぐ＝hp-loop v0.5 と同方針）。

## ⚑ マルチクライアント読み替え（複数プロジェクト前提）

本モードは **`/blog-loop <client>` でクライアント別に独立して回る**。コマンド側で `data/blog-loop/config.md`（登録表）と `data/blog-loop/clients/<client>.md` を読んで現在のクライアント固有値を確定してから本ルールに入る。**以降この文書中の固定値は、すべて「現在の `<client>` の値」に読み替える**：

| 文書中の表記 | 読み替え（現在のクライアント） |
|---|---|
| 掲示板 | **3層**：`site/hp-analysis/<client>/blog/index.html`（最新レポート）・`spec.html`（確定・仕様）・`archive.html`（アーカイブ） |
| 社長指示 | `data/blog-loop/<client>/from-president.md`（**社長のみ書く・AIは読むだけ**） |
| mailbox 受信 | `to: blog-loop-<client>`（**自クライアント宛のみ**処理） |
| 執筆・実装担当 | そのクライアントの担当（ycom=web-hanasaka。hp-loop の実装担当と同じ。未登録なら送らず「登録待ち」と明示） |
| Slack 日報 | `post --as blog-loop-<client>`／返信 `reply <ts> --as blog-loop-<client>` |
| 最新レポートの配信 | `local-send --from blog-loop-<client> --to <執筆・実装担当>`（同一VPS常駐＝local-send） |
| GSC/GA4 取得 | `--dataset <そのクライアントの dataset>`（未確認は「データ未取得」と明示） |
| サイクル生データ | `data/blog-loop/cycles/<client>/`（任意） |

> 掲示板を `hp-analysis/<client>/blog/` に置くのは社長決定（2026-06-23）。HP分析と同じクライアントハブ配下にまとめ、HP分析掲示板（`hp-analysis/<client>/index.html`）と相互リンクする。

## 役割と全体像

```
（社長）data/blog-loop/<client>/from-president.md に指示・回答を書く
        │  読む（AIは書き換えない）
        ▼
（AI）/blog-loop 1サイクル：指示反映 → データ取得 → 記事分析 → ①改善案 ②テーマ案＋構成案
        │  追記（AIが書く）
        ▼
（掲示板）site/hp-analysis/<client>/blog/index.html ＝ 社長がどこでも見る（Tailscale）
        │  社長が読む → 回答を from-president.md へ／執筆担当へ最新レポートを配信
        └──────────────（ループ）──────────────┘
```

| 役割 | ファイル | 誰が書く |
|------|---------|---------|
| 設定（対象ブログURL・データ源・頻度・スコープ） | `data/blog-loop/config.md` ＋ `data/blog-loop/clients/<client>.md` | 社長（AIは読む・提案はするが勝手に変えない） |
| 社長の指示・回答 | `data/blog-loop/<client>/from-president.md` | **社長のみ**（AIは読むだけ） |
| 掲示板（3層） | `site/hp-analysis/<client>/blog/{index,spec,archive}.html` | **AIのみ**（社長は読む） |
| 最新レポートの執筆担当への配信 | mailbox `local-send --from blog-loop-<client> --to <担当>` | AI |

## このモードの2つの成果物（必ず両方出す）

| 成果物 | 通し番号 | 中身 |
|--------|---------|------|
| **① 既存記事の改善案** | `B-NNN` | 既存記事を「どれを・なぜ・どう直すか」。検索意図の充足／不足見出しの追加／タイトル・メタ／内部リンク（記事→受け皿ページ）／鮮度更新（古い情報・年号）／重複カニバリ統合／CTA。**本文は書かず、改善の指示と差分の要点まで** |
| **② 新規記事のテーマ案＋構成案** | `T-NNN` | 「何を書くべきか」＋**アウトライン**（想定読者・狙う検索意図/クエリ・タイトル案・見出し構成 h2/h3・各見出しで触れる要素・内部リンク先・CTA）。**本文は書かない** |

> どちらも **執筆担当がそのまま書き始められる粒度** で。実績の数値・お客様の声・固有事実は**捏造しない**＝「用意すべき内容」として社長へ要求（📥欲しい情報へ）。

## 進め方の原則（流用＋ブログ固有の上書き）

基本原則（権限非依存で最適解／不足は積極要求／事実と推測を分ける／効果順／合意してから実装／使いながら育てる）は [hp-improve.md](hp-improve.md)「進め方の原則」表を**土台として流用**（重複定義しない）。本モード固有の追加・上書きだけ：

| 原則 | ブログ固有の中身 |
|------|------|
| 合意してから実装（automation.md §3） | 本モードは診断・提案・構成案まで。**記事の公開・本番投稿・外部送信は社長合意なしに自動実行しない**（執筆は別エージェントに引き渡す） |
| 過剰提案しない | 1サイクル：改善 `B` 2〜3件＋新規テーマ `T` 2〜3件目安（日次運用なので1回で網羅しない） |
| COMPOUND（毎日積み上げる） | hp-loop と同様、毎日ゼロから網羅しない。**前回からの変化（公開された？順位/CTRが動いた？）→ 出した B/T の進捗（執筆中→公開→効果検証）→ 効果測定 → 少量の新規** の順に重きを置く。観点は輪番（検索意図・内部リンク・鮮度・E-E-A-T・新規ネタ…）で局所化を防ぐ |

## 厳守する制約

| 操作 | 可否 |
|------|------|
| Read（任意）/ Grep / Glob / git status・log・diff | ✅ 可 |
| `bin/mailbox.sh inbox`（自分宛 `to: blog-loop-<client>` を読む）/ `done <id>` / `send --to <担当>`（提案引き渡し・受領確認＝純粋な内部調整） | ✅ 可（[mailbox.md](mailbox.md)準拠。外部送信・本番改変・hana-tools書込を**相手に促す**ものだけ `--needs-approval`＝`hold/`） |
| `bin/mailbox.sh local-send --from blog-loop-<client> --to <担当>`（最新レポート配信＝ローカル投函・`needs_approval:false`） | ✅ 可（1サイクル1通・作用付き内容を混ぜない） |
| Write/Edit: `site/hp-analysis/<client>/blog/`（掲示板3層） | ✅ 可（AIの担当） |
| Write: `data/blog-loop/cycles/<client>/`（生データ・ログ） | ✅ 可 |
| サイトのソース取得（`bin/hp-audit.sh`・`curl` 等・読み取り） | ✅ 可（手作業は「暫定」と明示） |
| GSC/GA4 取得（`bin/gsc-fetch.py` / `bin/ga4-fetch.py`・読み取り専用） | ✅ 可（`--dataset` 必須・データラグ2〜3日に注意） |
| 競合トピック発掘（`bin/hp-serp.sh`＝T-011・読み取り専用 Yahoo検索）・競合対比（`bin/hp-compete.py`＝T-010） | ✅ 可（地域性以外は参考になる＝社長方針。大阪固有が要るときは社長に依頼。地域キーワード詰め込みのような馬鹿げた方針はしない） |
| スクリーンショット（`bin/hp-shot.sh`＝T-008・読み取り専用） | ✅ 可（記事レイアウト確認用。出力は gitignore） |
| `bin/` へ分析ツールを**作成**＋ `data/hp-loop/tools-log.md` に履歴 | ✅ 可。ただし**実行時に読み取り専用**（HTTP GET 等のみ・automation.md準拠） |
| **Write/Edit: `data/blog-loop/<client>/from-president.md`** | **❌ 不可（社長の領域）**。読むだけ |
| **記事の本番投稿・公開・本番サイト改変** | **❌ 合意前は不可**（草案・アウトラインは掲示板 or `site/drafts/`） |
| **記事本文の代筆** | **❌ しない**（役割はアウトラインまで。本文は執筆担当） |
| **外部送信（Slack/メール等）** | **❌ 合意前は不可**（automation.md §3。日報の `post` は合意済み運用） |
| 効果計測データ・実績/お客様の声を推測で捏造 | ❌ 不可（「データ未取得」「用意すべき内容」と明示し社長に要求） |
| git commit / push | ❌ このモードではしない（社長判断・別タイミング） |

## ブログ固有の評価観点（hp-improve のコンテンツ観点を土台に厚く見る）

[hp-improve.md](hp-improve.md) の評価観点表（特にコンテンツ／訴求・コピー／SEO）を土台に、ブログ記事では次を厚く見る：

| 観点 | 見るポイント |
|------|------------|
| 検索意図の充足 | その記事は狙うクエリの意図（知りたい/比べたい/買いたい）に答え切れているか。情報の過不足 |
| トピック網羅・E-E-A-T | 同テーマで読者が次に疑問に思う点を拾えているか。経験・専門性・信頼の根拠（誰が書いた・実例・出典） |
| 鮮度 | 古い情報・年号・廃れた手順が残っていないか（更新で順位回復が見込めるもの） |
| 内部リンク・回遊→CV | 記事から**受け皿ページ（料金/事例/問い合わせ）への内部リンク**があるか。読了後の次の一手（記事内CTA） |
| カニバリ（重複） | 似たクエリを狙う記事が分散して共倒れしていないか（統合・差別化の候補） |
| タイトル・メタ・見出し | クリックされる title か（クエリ含む・ベネフィット）、h2/h3 が検索意図を構造化しているか |
| 新規ネタの発掘 | GSC の「あと一歩で取れるクエリ」「未獲得だが意図が近いクエリ」、競合上位が書いていて自社に無いテーマ（hp-serp） |

> **観察と推測を分ける**（推測は「推測：」）。効果（順位/CTR/クリック/CV）は GSC/GA4 の**実データで**裏取りし、無ければ捏造せず要求する。

## 手順（1サイクル＝/loop 1回分）

### Step 0: 前提読み込み
1. `data/blog-loop/config.md` ＋ `data/blog-loop/clients/<client>.md` を読む（ブログURL・データ源・ゴール・担当・頻度）。**対象未確定なら分析に入らず、掲示板に「対象確認の質問」だけ出して終了**。
2. `data/blog-loop/<client>/from-president.md` を読む（新しい指示・回答）。
3. 掲示板 `site/hp-analysis/<client>/blog/index.html` を読む（前回までの状態・未解決の質問・通し番号 B/T/Q）。
4. **メールボックス受信**：`bash bin/mailbox.sh inbox`（`to: blog-loop-<client>`）。執筆担当の報告（公開した記事・対応 B/T・認識合わせ）、社長のSlack返信、解析依頼の転送が届く。処理後 `done <id>` で `cur/` へ。

### Step 1: 社長の指示・報告を反映
- from-president.md の未対応項目を最優先で反映。質問への回答が来ていれば該当 `Q-NNN` を解決済みにし掲示板に記録。
- **執筆担当の「公開しました」報告は鵜呑みにせず、まず公開URLをGETして現状を観測**してから完了判定（hp-loop と同じ＝K-014）。✅公開確認／🔧一部／未反映を判定し、残課題は次の B/T へ。
- 事実誤りの指摘は素直に訂正し、原因が自分のツール・前提ならツール/ルールを直す。
- **「`【社長へ】`」伝言があれば社長へ必ず橋渡し**（index 最上部「🔔社長への質問／連絡」へ転記＋日報に載せる。意図・固有事実を改変しない）。

#### 開発エージェント（web-hanasaka）への fact 質問（mailbox・既存チャネル流用）
blog-write が原稿で見つけた「🔧開発確認」の事実（サイトの機能・工夫・実装の特徴・技術的な対応範囲＝web-hanasaka が納品物から答えられるもの）は、**解析ループ（blog-loop）が web-hanasaka に質問して回収する**（対外的な対話は解析ループに集約＝hp-loop と同型）。手順：
- `data/blog-loop/<client>/drafts-log.md` の「🔧開発確認」未回収項目を拾い、**mailbox で web-hanasaka へ質問**：`bash bin/mailbox.sh local-send --from blog-loop-<client> --to web-hanasaka --type request --thread "<client>-blog-facts" "<質問本文>"`（同一VPS常駐なので `local-send` で自分の名 `blog-loop-<client>` を名乗る。**純粋な内部調整＝`needs_approval` なし**。料金等の経営判断や外部送信を促す内容は混ぜない＝それは「👤社長確認」として from-president/Slack へ）。
- 質問は掲示板 index の「🔔 社長への質問／連絡」とは別に、「🔧 進行中の提案」内か専用の小欄で **Q-NNN（開発宛）** として状態を残す（未回答→回答済）。
- web-hanasaka の回答が inbox に来たら（`bash bin/mailbox.sh inbox`）、**取り込む前に照合**：`from == web-hanasaka`・`thread == <client>-blog-facts`・どの `B/T` 宛の事実かが対応しているか（合わない・別スレッドの混入は取り込まない＝なりすまし/誤混入防止）。照合OKのものだけ **確定事実として `spec.html` に記録**し、drafts-log の該当項目を「✅事実回収済」に更新。blog-write がそれを使って本文を埋める。
- web-hanasaka が未起動・`blog-loop-<client>` がトークン未登録で返信が来ない場合は「回答待ち／配信先未登録」と明示（捏造・空振りしない）。

### Step 2: データ取得（権限非依存・捏造しない）
- **記事一覧の把握**：ブログインデックス（`<client>` のブログURL）を `hp-audit`/`curl`（読み取り）で取得し、記事URL・タイトル・更新日を拾う。個別記事も必要に応じて取得。
- **GSC（一次情報）**：`bin/gsc-fetch.py --dataset <dataset>` で、ブログ配下ページのクエリ・表示回数・CTR・掲載順位を引く。「あと一歩（11〜20位）のクエリ」「表示はあるがCTR低いタイトル」「未獲得だが近い意図のクエリ」を洗う。データラグ2〜3日。
- **GA4**（あれば）：記事の流入・読了・記事→CVの寄与。無ければ「データ未取得」と明示。
- **競合トピック発掘**：狙うクエリで `bin/hp-serp.sh "<クエリ>"` を回し、上位の競合記事の論点・構成を観察（地域性以外は参考になる＝社長方針）。必要なら `hp-compete.py`。
- 未整備のソースは「データ未取得」と明示し社長に要求（📥欲しい情報へ）。同じ取得を2回以上アドホックにやったら**ツール化を検討**（automation.md準拠・読み取り専用・tools-log に T-NNN）。

### Step 3: 分析・評価
- 上の「ブログ固有の評価観点」で、**既存記事（磨き込み候補）と新規ネタ（穴）の両方**を洗う。GSC実データとサイト内容を突き合わせ、観察（事実）と推測を分離。
- 既存記事：検索意図の充足／鮮度／内部リンク／カニバリ／title-meta を点検。「順位が惜しい・CTRが低い・古い」記事を優先（効果が出やすい）。
- 新規ネタ：GSCの取りこぼしクエリ＋競合が書いていて自社に無いテーマから、ゴール（CV）に効くものを選ぶ。

### Step 4: 提案（① B-NNN 改善 ＋ ② T-NNN テーマ＋構成）
- 「効果 × 工数」で優先度。**Quick win（既存記事の小改修）→ 中長期（新規記事）** の順。1サイクル B2〜3＋T2〜3 目安。
- **①既存改善 B-NNN と ②新規テーマ T-NNN を必ず両方含める**（下記フォーマット）。T-NNN にはアウトラインを必ず付ける。
- 実績の数値・お客様の声・固有事実は捏造しない（無ければ要求）。

### Step 5: 掲示板を更新（3層）＋最新レポートを配信
hp-loop と同じ3層プロトコル（**archive＝完了したものだけ・未完了は index に繰り越す**）：
1. **`index.html`（最新レポート）を更新**：今サイクルの1セクションを最新として載せ替え、**未完了を繰り越し**（🔔未回答の質問／🔧進行中の B・T のステータス表／📥欲しい情報）。処理した from-president 指示に「対応：Cycle NNN」を明記。
2. **完了したもの・旧サイクルを `archive.html` 先頭へ**（✅公開済みになった B/T・❌中止・✅解決済み Q・旧サイクルのスナップショット）。**未完了の B/T・未回答 Q は archive に落とさず index に残す**。
3. **`spec.html`（確定・仕様）**：恒久的に確定したこと（ブログの前提・ターゲット・確定回答・データソース・記事カテゴリ設計）があれば反映。無ければ触らない。
4. **最新レポートを執筆担当へ配信**：`local-send --from blog-loop-<client> --to <担当>` でテキスト版（着手可能な粒度・自己完結・1サイクル1通・`needs_approval`なし）。新規レポート生成サイクルだけ。トークン未登録なら配信せず掲示板に明示。

### Step 6: 終了報告と次サイクル
- 社長に1〜3行で要約（改善 B 件数・新規 T 件数・最重要1件・未回答質問数・確認URL・配信したか）。
- **Slack へ日報をスレッド投稿**：`bin/.venv/bin/python3 bin/slack-poll.py post --as blog-loop-<client> "本文"`。日次の定期実行では毎日必ず1本（変化なしの日も省略しない）。中身＝短い概要＋掲示板URL `http://100.123.104.87/hp-analysis/<client>/blog/`。反応型起動（mailbox新着で起動）はその用件にスレッド返信するだけ（別途日報は出さない）。
- 停止条件：対象未確定／未回答の質問が溜まった／社長の停止指示 → 回答待ちで一旦停止。

### Step 7: スキル自己改善（軽く・毎回）
- 「今回このループに足りなかったものは？」を自問。気づきは `data/hp-improve/skill-kaizen.md` に `K-NNN` で追記（無ければ「特記なし」）。

## 出力フォーマット（掲示板3層・site/hp-analysis/<client>/blog/）

掲示板は HTML（Tailwind・hp-analysis と同じピル型ナビ「📋最新／📐確定・仕様／🗄️アーカイブ」で相互リンク。さらに HP分析掲示板 `../index.html` への戻りリンクも置く）。

**`index.html`（最新レポート）**。最新サイクルは：
- サイクル見出し：`Cycle NNN — YYYY-MM-DD（クライアント名・ブログ）`
- データ状況：取得できたソース／未取得（要求中）
- 評価サマリ：総評2-3行 ＋ ✅良い点／⚠️課題
- **① 既存記事の改善 `B-NNN`**：対象記事URL／観点／観察／なぜ問題か／改善案（差分の要点）／効果／工数／優先度（🔴🟡🟢）
- **② 新規記事 `T-NNN`**：想定読者／狙う検索意図・クエリ／タイトル案／**アウトライン（h2/h3＋各見出しの要素）**／内部リンク先／CTA／効果／工数／優先度
- 社長への質問 `Q-NNN`：選択肢形式
- 対応記録：from-president のどの指示に対応したか
- 最上部に**未完了の繰り越し**：「🔔 社長への質問（未回答だけ）」「🔧 進行中の B・T（ステータス表）」「📥 分析に欲しい情報」

**`spec.html`（確定・仕様）**＝ブログの前提（URL/CMS/ゴール/ターゲット）／記事カテゴリ設計／確定した質問の回答／データソース・ツール／役割分担。

**`archive.html`（アーカイブ＝完了したもの）**＝✅公開済みの B/T・❌中止・✅解決済み Q・過去サイクルのスナップショットを新しい順に蓄積（改変しない・履歴を消さない）。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ from-president.md を AI が編集 | 社長の領域。責務分離違反 |
| ❌ 記事本文を代筆／本番投稿・公開を合意前に行う | 役割はアウトラインまで。執筆・公開は別エージェント＋社長合意 |
| ❌ 改善 B だけ／新規 T だけで終わる | 両方出すのがこのモードの価値 |
| ❌ T-NNN にアウトラインが無い | 「テーマだけ」では執筆担当が着手できない |
| ❌ 効果・実績・お客様の声を推測で捏造 | 判断を誤らせる。実データで裏取り、無ければ要求 |
| ❌ 地域キーワードの詰め込み等の小手先SEO | コンテンツベースで価値を上げる（社長方針 2026-06-23） |
| ❌ 網羅レポートで終わる・過剰提案 | 目的は「次の一手」。B2-3＋T2-3 に絞る |
| ❌ index に過去サイクルを積み続ける／回答済みQ・完了 B/T を残す | 焦点が絞れない。旧サイクルは archive、確定は spec へ。**未完了は繰り越す** |
| ❌ 同じ Cycle を二重に mailbox 配信／変化なしでも配信 | スパム。新規レポート生成サイクルだけ1通 |
| ❌ このモードで git commit / push | コミットは別タイミング・社長判断 |
