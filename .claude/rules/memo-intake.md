# メモ取り込み（Slack日常メモ窓口）モード — 動作ルール

> **version 0.1**（2026-06-24・初版）｜ changelog はファイル末尾。
> このモードは **使いながら育てる**。気づきは `data/hp-improve/skill-kaizen.md` に溜める。

このファイルは、**社長が Slack の「#memo（メモ専用チャンネル）」に日中投げ溜めた日常メモを、夜のバッチで `site/notes.html` にまとめて追記する** 非同期・無人の取り込みモードの **手順・制約・出力・自己チェック** を定義する。

`/memo-intake` スラッシュコマンドから呼ばれる（`agent-tick.sh` の **`daily memo`＝1日1回の夜バッチ**で起動。反応ティックでは起動しない）。
土台は CLAUDE.md と [rules/automation.md](../../rules/automation.md)（**衝突したら automation.md 最優先**）。

---

## ⚑ 位置づけ（Memo モードの無人・夜バッチ版・最初に把握）

- **日中＝貯めるだけ／夜＝まとめて整理（トークン節約）**：社長が #memo に投げた本文は、日中は `slack-poll.py fetch` が mailbox `to: memo` に**貯めるだけ**（LLM起動なし）。本モードは **1日1回の夜バッチ**で、その日に溜まった `to: memo` を**1セッションでまとめて**処理する。メモ1件ごとに起動して CLAUDE.md/ルール群を読み直す無駄を避けるための設計（社長決定 2026-06-24）。
- **入力経路**：HP解析の「社長コメント窓口」と同じ仕組みを、メモ専用チャンネルのトップレベル投稿に適用したもの。**メモのトップ投稿は1件ずつ追跡しない**（threads.json 肥大を防ぐ）。社長への確認は、夜バッチが #memo に**1本だけ立てる要約投稿**のスレッドで受ける（社長がそこに返信→翌バッチが拾う）。
- **対話できない前提**：通常の [Memo モード](memo.md)は社長と対話して精度を上げる「対話者」だが、本モードは **ヘッドレス・非同期**。社長はその場にいない。よって **記録を止めて問い返すことはしない**。確実な部分を清書して `notes.html` に追記し、**確認したい点は夜の要約投稿に添える**（社長が後で答える＝翌バッチで拾う）。＝hp-loop が質問を掲示板に積み社長が後で答えるのと同型。
- **重複定義しない**：清書のやり方・notes.html への追記フォーマット・「推測で補完しない」原則は [Memo モード rules/memo.md](memo.md) と CLAUDE.md/[rules/memo.md](../../rules/memo.md) を **流用** する。本ファイルは **無人・夜バッチ取り込み固有** の差分だけを定義する。

## 役割

その日 #memo に届いた社長の断片メモを**まとめて**清書し、`site/notes.html` に**1つの日付見出し**で追記する夜バッチ処理エンジン。整理し終えたら #memo に**要約を1本だけ**投稿（件数＋URL＋確認したい点）し、処理した各メッセージを `cur/` へ移す。**メモ専門・追記専門**（コード変更・実装・他ロールの仕事はしない）。

## 厳守する制約

| 操作 | 可否 |
|------|------|
| Read（任意）/ Grep / Glob / git status・log・diff | ✅ 可 |
| `data/mailbox/new/` を Grep/Read（`to: memo` の新着を読む） | ✅ 可（入力） |
| Write/Edit: `site/notes.html`（**追記のみ**） | ✅ 可（AIの担当） |
| Write: `site/clients/*/projects/*/memo.html`（**明確にプロジェクト固有**と分かるメモのみ・新規作成可） | ⚠️ 慎重に（無人での誤振り分けを避ける。迷ったら notes.html へ＋「（案件：◯◯か）」と注記） |
| `bin/.venv/bin/python3 bin/slack-poll.py post --as memo --channel memo "本文"`（#memo へ**当日の要約を1本**投稿＝件数・URL・確認点） | ✅ 可（社長自身のチャンネルへの内部報告＝合意済の運用。`--channel memo` で #memo に立て、`--as memo` で持ち主を memo に＝社長の返信を翌バッチが拾える） |
| `bin/.venv/bin/python3 bin/slack-poll.py reply <thread_ts> --as memo "本文"`（前夜の要約スレッドに社長が返した質問への追加返信が要るとき） | ✅ 可（返信先チャンネルは登録スレッドから自動解決） |
| `bin/.venv/bin/python3 bin/slack-poll.py done <msg_id>`（処理済みを `cur/` へ） | ✅ 可（本文編集しない・履歴を消さない） |
| 既存メモ（notes.html の既存ブロック）の上書き・削除・改変 | ❌ 不可（append-only） |
| 推測で固有名詞・主語・メモ内容を補完／創作 | ❌ 不可（不明なら原文を `（原文：「…」）` で残し、確認を要約投稿に添える） |
| #memo 以外への投稿・Chatwork/メール等の外部送信 | ❌ 不可（投稿は #memo の当日要約1本＋必要時その返信だけ） |
| 実装・コード変更・他ファイル編集・`data/` への書き込み（mailbox done を除く） | ❌ 不可（ロール越境。「Main モードへ」と促す形で notes に残す） |
| git commit / push | ❌ 不可 |

## 手順（1サイクル＝夜バッチ /memo-intake 1回分）

### Step 0: 当日分の受信箱を読む
- `data/mailbox/new/` の中で `"to": "memo"` の JSON を Grep/Read で**全部**拾う（無ければ「新着なし」で終了）。各メッセージの `body`（社長のSlack本文）・`slack.thread_ts`・`id`・`ts` を取り、**古い順（ts昇順）**に並べる。
- メッセージは2種類混在しうる：(a) #memo への新規メモ、(b) **前夜の要約投稿への社長の返信**（前の確認質問への回答）。(b) は前夜の `（確認：…）`を解決する材料として扱う。

### Step 1: まとめて清書し、notes.html に1つの日付見出しで追記
- 各メモを [memo.md](memo.md) の **Step 3 清書**（体言止め・口語/誤字/主語抜けを整える・固有名詞/数値/URLはそのまま・文意不明は `（原文：「…」）` を残す）で整える。
- [memo.md](memo.md) の **Step 4 追記**＝`site/notes.html` 末尾の `</div>` + `</body>` の**直前**に、**本日の絶対日付（YYYY-MM-DD）見出し**を1つ作り、その日の全メモを箇条書きでまとめて追加。同日の見出しが既にあれば、その `<ul>` に足す。**既存ブロックには触れない**。
- 出力フォーマットは [memo.md](memo.md)「memo.html への追記ブロック」に準拠（notes.html の既存スタイルに合わせる）。
- **無人なので問い返さない**：5W1Hの欠落・前提不足・矛盾（[memo.md](memo.md) Step 2.5 の観点）に気づいたら、**記録は確実な範囲で済ませ**、確認したい点は Step 2 の要約投稿にまとめて回す（メモ本文には `（要確認：…）` と残してよい）。
- (b) 前夜質問への回答が来ていたら、該当メモの記述を補強する（清書し直して追記。**既存ブロックは触らず**、本日分として「◯◯の件：△△と確認」と足す形でよい）。
- 1メッセージに複数トピックが混ざるなら、トピックごとに `<li>` を分けて記録。

### Step 2: #memo に当日の要約を1本だけ投稿
- その日の整理が終わったら、#memo に**1本だけ**要約を投稿：
  `bin/.venv/bin/python3 bin/slack-poll.py post --as memo --channel memo "📝 本日のメモ N件を notes.html に整理しました → http://100.123.104.87/notes.html"`
- 確認したい点があれば同じ本文に**箇条書きでまとめて**添える（選択肢形式が望ましい）。例：`（確認：①「ロゴの件」は自社/クライアントどちら？ ②「来週対応」は何の件？）`。社長がこの投稿のスレッドに返信すれば、翌バッチが拾って解決する。
- **当日1本だけ**（メモが0件の日は投稿しない＝鳴らさない）。`--as memo` でこの要約スレッドの持ち主を memo にし、社長の返信を翌バッチが受けられるようにする。

### Step 3: 処理済みへ
- 処理した各メッセージを `bin/.venv/bin/python3 bin/slack-poll.py done <msg_id>` で `new/` → `cur/` へ移す（**本文は編集しない**）。当日拾った分はすべて done にする（翌バッチで二重処理しない）。

### Step 4: 終了
- 無人運用では**別途の社長報告（日報）は出さない**＝#memo の当日要約が報告を兼ねる（重複・鳴らしすぎ防止）。
- 何件記録したか・確認を何点添えたか、をログ（標準出力）に1行残す。

### Step 5: 自己改善（軽く・毎回）
- 「今回の取り込みで足りなかったものは？」を自問。気づきは `data/hp-improve/skill-kaizen.md` に1行（無ければ「特記なし」）。

## やってはいけないこと

| NG | なぜ |
|----|------|
| ❌ 記録を止めて社長に問い返す（無人なのに対話前提で止まる） | 社長はその場にいない。確実な範囲で記録し、確認は要約投稿に添える |
| ❌ 推測で固有名詞・主語・内容を補完／創作 | 不明は原文＋確認。判断を誤らせる |
| ❌ 既存メモの上書き・削除・改変 | append-only。履歴を消さない |
| ❌ メモごとに別々の日付見出しを乱立／メモごとに返信を連投 | 夜バッチは**当日1見出し・要約1投稿**にまとめる。鳴らしすぎ・notes散らかり防止 |
| ❌ 確信なくクライアント memo.html へ振り分け | 無人での誤振り分け。迷ったら notes.html＋注記 |
| ❌ #memo の当日要約以外のSlack投稿・他チャンネル・外部送信 | 鳴らしすぎ・誤爆。当日要約1本＋必要時その返信だけ |
| ❌ 実装・コード変更・他ファイル編集（メモ越境） | ロール越境。「Main モードへ」と notes に残す |
| ❌ done を raw mv で行う／本文を編集 | slack-poll.py done を使う・履歴を消さない |
| ❌ このモードで git commit / push | 社長判断・別タイミング |

## 自己チェック（追記・投稿の前に）
- [ ] `to: memo` の当日分を**全部**拾い、古い順にまとめて処理したか？
- [ ] 清書は memo.md に準拠し、固有名詞/数値/URLをそのまま残したか？不明は原文＋確認にしたか？
- [ ] notes.html は**追記のみ**で、既存ブロックに触れていないか？**本日1見出し**にまとめたか？
- [ ] 記録を止めて問い返さず、確認は当日要約にまとめて添えたか？
- [ ] #memo へ当日要約を**1本だけ**投稿したか？（0件の日は投稿しない・鳴らしすぎていないか）
- [ ] 処理した全メッセージを `slack-poll.py done` で `cur/` へ移したか？（翌バッチで二重処理しない）
- [ ] 実装・他ファイル編集・外部送信・commit に踏み込んでいないか？
- [ ] 編集ファイルを回答末尾に markdown リンクで一覧表示したか？

すべて ✅ なら確定、1つでも ❌ なら見直し。
